## 10. 关键决策与依据

### 10.1 为什么 F11 从"违反 MVP 零后端"变为 v1.1.0 Phase A

**v1.0.0-report.md 判断**：F11 需要"新建 `/api/events` 路由 + 数据存储方案；LocalStorage 手动导出够用"——违反 MVP 零后端原则。

**v1.0.0 实际状态**：production 模式已具备完整 server 能力（SQLite + `/api/data/*` + `/api/migrate/*`），加 `/api/events` 仅需 1 个新路由 + 1 张表。

**结论**：F11 可提前到 v1.1.0 Phase A，与 F08 / F20 / F33 一起做可观测性主题，复用 analytics pipeline。

### 10.2 为什么 F04 实现方案从 IndexedDB 改为 SQLite staging

**v1.0.0-report.md 判断**：F04 需要"IndexedDB checkpoint + pipeline 改造"——大工作量。

**v1.0.0 实际状态**：production 模式已有 SQLite + staging 表（迁移工具用的 `migration_staging` 表），可复用模式新建 `compile_checkpoint` 表。

**结论**：F04 用 SQLite 比 IndexedDB 更可靠（不易被浏览器清缓存丢失）、更一致（与现有持久化层统一）、开发量更小。

### 10.3 为什么 Module 上限 12 题不再是一个问题

**v1.0.0 之前**：LocalStorage 5MB 限制导致 `STORAGE_MAX_HISTORY_MODULES = 12` 软警告。

**v1.0.0 之后**：production 模式下 quota.ts 按 `isShowcaseMode` 条件取值——showcase 保留 12，production 取消上限。该限制仅对 showcase 模式（Vercel 部署）生效，作者本地 dev 已无此约束。

### 10.4 为什么 v1.0.1 不引入新功能

v1.0.0 持久化升级是重大架构变更（448 单测中 107 个新增），且 E2E 29 个测试**未实跑**。此时引入新功能的风险远大于收益。v1.0.1 应聚焦：
1. 验证现有功能（E2E 实跑）
2. 修复已知问题（M1 / M4 / Q3）
3. 内测准备（响应式 / 浏览器兼容 / a11y / 编译成功率测试）
