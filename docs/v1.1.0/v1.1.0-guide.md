> **适用版本**：v1.1.0（含 v1.0.x 衍生项）
> **部署平台**：Netlify（GitHub 自动部署）
> **编写日期**：2026-07-15
> **关联文档**：[`v1.1.0-Review.md`](./v1.1.0-Review.md) §4 已知限制、[`docs/v1.0.0/Deploying.md`](../v1.0.0/Deploying.md)（Vercel 版）

## 0. 一句话结论

**Netlify 自动部署请使用 `showcase` 模式构建。** 该模式零后端、零数据库，全部数据留存在浏览器 LocalStorage，与 Netlify Functions 的无状态/短暂文件系统模型完美契合。`production` 模式（SQLite）**不适合任何 Serverless 平台**（Vercel 或 Netlify），仅适用于自建 Node 服务器或本地开发。

---

## 1. 平台选型：为什么 Netlify 只能跑 showcase 模式

本项目有两种构建模式，由构建期常量 `NEXT_PUBLIC_APP_MODE` 决定：

| 模式 | 数据存储 | 是否调用 LLM API | 是否依赖服务端持久化 |
|------|----------|------------------|----------------------|
| `showcase`（默认） | 浏览器 LocalStorage | 仅 env fallback 时（可选） | 否 |
| `production` | 服务端 SQLite (`better-sqlite3`) | 是 | **是** |

### 1.1 production 模式为什么不能上 Netlify

`production` 模式的核心是服务端 SQLite 持久化层（`src/lib/persistence/server/`）。SQLite 依赖**持久可写的文件系统**，而 Netlify Functions（基于 AWS Lambda）有两个致命限制：

1. **文件系统是短暂的**：每次函数调用可能在全新容器中执行，`/tmp` 写入的内容不保证在下一次调用时还在。SQLite 数据文件会在调用间丢失。
2. **`better-sqlite3` 是原生 C++ addon**：需要与 Lambda 运行时匹配的预编译二进制。安装和加载在 Netlify 构建环境可能失败。

> 这不是 Netlify 独有问题——Vercel Serverless 同样有短暂文件系统。`production` 模式实际只在 **本地 `bun run dev`** 或 **自建 Node 服务器（`bun run start`）** 下可靠工作。详见 `v1.1.0-roadmap.md` §11.1 风险登记。

### 1.2 showcase 模式为什么适合 Netlify

- **零服务端状态**：全部数据在浏览器 LocalStorage，刷新不丢，多设备各自独立
- **编译是 mock 回放**：showcase 编译不调用 LLM，前端播放预录制的 SSE 事件流（`src/lib/showcase/mock-compile-events.ts`），不依赖 `/api/compile`
- **API 路由可选**：`/api/feedback` 和 `/api/feynman-eval` 仅在配置了 LLM Key 的 env fallback 时触发；不配置则功能降级而非崩溃
- **SSR 安全**：所有 Zustand store 用 `useHydrated()` 守卫，SSR 阶段不访问 LocalStorage

### 1.3 函数超时对比（关键差异）

`vercel.json` 配置的路由超时在 Netlify **不生效**，需用 `netlify.toml` 重新配置。更重要的是 Netlify 各档函数超时远短于 Vercel：

| 平台 / 计划 | 函数最大超时 | 能否满足 `/api/compile`（60s） | 能否满足 `/api/feedback`（10s） |
|-------------|-------------|-------------------------------|-------------------------------|
| Vercel Hobby | 300s | 是 | 是 |
| Netlify Free | **10s** | 否（但 showcase 不调用此路由） | 临界（刚好 10s） |
| Netlify Pro | 26s | 否 | 是 |

> **结论**：showcase 模式部署到 Netlify 时，若不配置 LLM Key（纯展示），**完全不调用任何 API 路由**，超时不是问题。若配置了 env fallback Key 用于反馈评分，`/api/feedback` 在 Netlify Free 可能超时——建议升级 Netlify Pro 或不配置 Key（反馈功能降级）。

## 2. 部署前手动操作清单（必做）

以下操作源自 `v1.1.0-Review.md` §4 已知限制。**第 1 项是阻断性问题**（字体引用已断），其余为体验优化。

### 2.1 【阻断】执行字体分包（F10）

**现状（2026-07-15 确认）**：`src/app/globals.css:4` 已引用 `/fonts/source-han-serif-sc/result.css`，但 `public/fonts/source-han-serif-sc/` 目录下**只有 `README.md`**，分包产物（`result.css` + woff2 文件）**尚未生成**。当前线上会 404 请求这个 CSS，思源宋体无法加载，回退到系统衬线字体。

**操作步骤**：

```bash
# 1. 下载源字体（仅首次）
#    访问 https://github.com/adobe-fonts/source-han-serif/releases
#    下载 SourceHanSerifSC.zip，解压出 SourceHanSerifSC-Regular.otf
#    放到 public/fonts/source-han-serif-sc/source/SourceHanSerifSC-Regular.otf

# 2. 执行分包脚本
bun run scripts/split-fonts.ts
#    或：npx tsx scripts/split-fonts.ts

# 3. 验证产物
#    public/fonts/source-han-serif-sc/result.css 应存在
#    public/fonts/source-han-serif-sc/ 下应有 index-*.woff2 分包文件

# 4. 提交产物到 git（分包文件需入库，Netlify 构建时不会重新执行分包）
git add public/fonts/source-han-serif-sc/
git commit -m "build(fonts): add Source Han Serif SC subset output (F10)"
```

**若无法获取源字体**（许可证或体积考虑）：临时方案是注释掉 `globals.css:4` 的 `@import`，回退到系统宋体。但中文排版质量会下降。

> **为什么分包文件必须入库**：Netlify 构建从 GitHub 拉取代码后跑 `next build`，不会执行 `scripts/split-fonts.ts`。分包产物是静态文件，必须在 commit 里。总分包体积约 2-5MB（按字符频次切分），git 能接受。

### 2.2 【建议】OG 图片 SVG 转 PNG（D2）

**现状**：`public/og-image.svg`（1200×630，2KB）已存在，`layout.tsx` metadata 已引用。但部分社交平台（Facebook / LinkedIn / 微信）**不支持 SVG 格式的 OG 图片**，会显示空白或文字链接。

**操作步骤**：

```bash
# 方案 A：用 SVG 渲染工具转 PNG（推荐）
#    任意 SVG→PNG 工具（如 https://cloudconvert.com/svg-to-png）
#    输出 public/og-image.png，1200×630，<300KB

# 方案 B：用 sharp 在本地转（若已装）
bun add -d sharp
bun -e "import sharp from 'sharp'; await sharp('public/og-image.svg').resize(1200,630).png({quality:90}).toFile('public/og-image.png')"
```

转完后更新 `src/app/layout.tsx` 两处引用：

```typescript
// layout.tsx 第 29 行附近
url: '/og-image.png',   // 原为 '/og-image.svg'

// layout.tsx 第 40 行附近
images: ['/og-image.png'],   // 原为 ['/og-image.svg']
```

提交：

```bash
git add public/og-image.png src/app/layout.tsx
git commit -m "build(seo): convert OG image to PNG for platform compatibility (D2)"
```

### 2.3 【可选】合并前跑 E2E 回归（Review §4.5）

v1.1.0 的 pipeline 改造（PB.2）和 storage schema 升级可能影响 showcase 回归。合入 main 前建议本地跑：

```bash
bun run e2e
# 期望：v1.0.1 建立的 51 tests 基线全绿，无回归
```

若 E2E 环境不可用（Windows 上 Playwright 偶有兼容问题），至少跑：

```bash
bun run typecheck && bun run lint && bun run test
```

## 3. Netlify 项目配置

### 3.1 连接 GitHub 仓库

1. 登录 [Netlify Dashboard](https://app.netlify.com)
2. **Add new site → Import an existing project**
3. 选择 GitHub，授权后选 `Milo-isLearning` 仓库
4. 分支：`main`（或你的发布分支）

### 3.2 构建配置

| 配置项 | 值 | 说明 |
|--------|-----|------|
| Base directory | （留空=根目录） | 项目根 |
| Build command | `npm run build` | Netlify 默认用 npm；无需 bun |
| Publish directory | `.next` | Next.js 输出目录 |
| Functions directory | （自动） | `@netlify/plugin-nextjs` 自动处理 |

> **包管理器**：Netlify 会检测 `package-lock.json` / `bun.lockb` / `pnpm-lock.yaml`。项目用 Bun，若仓库含 `bun.lockb`，Netlify 会自动用 bun install；否则用 npm。两者对 `next build` 输出无影响。

### 3.3 环境变量（关键）

在 **Site settings → Environment variables** 配置。**最重要的一个**：

| Key | Value | 作用 | 必填 |
|-----|-------|------|------|
| `NEXT_PUBLIC_APP_MODE` | `showcase` | 强制 showcase 构建（默认值，但建议显式设置防 `.env.local` 污染） | **是** |

**LLM 配置（可选——决定 showcase 是否带反馈评分）**：

| Key | 示例值 | 作用 |
|-----|--------|------|
| `DEEPSEEK_API_KEY` | `sk-xxxx` | env fallback 时用于 `/api/feedback` + `/api/feynman-eval` 评分 |
| `DEFAULT_LLM_PROVIDER` | `deepseek` | fallback 供应商 |
| `DEFAULT_LLM_MODEL` | `deepseek-chat` | fallback 模型 |
| `GLM_API_KEY` | （智谱 key） | GLM 备用通道 |
| `OPENAI_COMPAT_API_KEY` | （可选） | OpenAI 兼容端点 |
| `OPENAI_COMPAT_BASE_URL` | `https://openrouter.ai/api/v1` | 必填（openai-compat 无 baseURL 会返回 null，D4 已修） |

**v1.1.0 新增（可选）**：

| Key | 默认值 | 作用 |
|-----|--------|------|
| `PROMPT_VERSION` | `v1` | Prompt 版本切换（F30，showcase 不用编译故无影响，留空即可） |
| `PROMPT_LOCALE` | `zh` | Prompt 语言（F31，同上） |

**配置策略建议**：

- **纯展示部署**（不配任何 LLM Key）：用户能完整体验编译流程（mock）、答题、费曼，但填空题的语义判分和费曼评分会降级。最省心。
- **带评分部署**（配 DeepSeek Key）：填空题语义判分 + 费曼评分可用。但需 Netlify Pro（26s 超时）保障 `/api/feedback` 不超时。

### 3.4 `netlify.toml`（可选，推荐）

在项目根创建 `netlify.toml` 固化配置（替代 Dashboard 手动设置，纳入版本管理）：

```toml
[build]
  command = "npm run build"
  publish = ".next"

[build.environment]
  NEXT_PUBLIC_APP_MODE = "showcase"
  NODE_VERSION = "20"
  # LLM Key 按需取消注释（敏感值建议放 Dashboard，不放这里）
  # DEFAULT_LLM_PROVIDER = "deepseek"
  # DEFAULT_LLM_MODEL = "deepseek-chat"

# Next.js 插件自动启用，无需显式声明 [[plugins]]
# 函数超时：Netlify Free=10s / Pro=26s，无法达到 Vercel 的 60s
# showcase 模式不调用 /api/compile，故无需为该路由配超时
```

> **注意**：API Key 等敏感值**不要**写进 `netlify.toml`（会进 git），放 Netlify Dashboard 的 Environment variables 里。

### 3.5 `better-sqlite3` 构建处理

showcase 模式**不使用** SQLite（无 `ALC_STORAGE_BACKEND=sqlite`），但 `better-sqlite3` 仍是 `dependencies`，`npm install` 会尝试编译这个原生模块。Netlify 的 Lambda 构建环境通常能编译成功（基于 Amazon Linux），若失败：

**方案 A（推荐）**：在 `netlify.toml` 加忽略标记，让 Netlify 跳过 native addon 构建：

```toml
[build.environment]
  NEXT_PUBLIC_APP_MODE = "showcase"
  # better-sqlite3 在 showcase 模式下不被 import，构建失败时可忽略
  npm_config_build_from_source = "false"
```

**方案 B**：若构建仍报错，临时把 `better-sqlite3` 和 `@types/better-sqlite3` 移到 `optionalDependencies`（showcase bundle 不引用它）：

```jsonc
// package.json
"optionalDependencies": {
  "better-sqlite3": "^12.11.1"
}
```

> 这两个方案仅在构建失败时才需要。先试默认配置，看 Build log。

## 4. 部署后验证清单

部署成功后，访问 Netlify 分配的域名，逐项验证：

### 4.1 基础渲染

- [ ] 首页正常加载，无 SSR 报错（查浏览器 Console）
- [ ] 暗色主题正确（warm amber 强调色 + 衬线字体）
- [ ] **中文字体正常**（思源宋体已加载——若 §2.1 字体分包完成；否则回退系统宋体）
- [ ] 导航到 `/settings`，确认页面可访问

### 4.2 showcase 核心流程

- [ ] 首页展示题库卡片可点击
- [ ] 点击展示题库 → 进入模拟编译页 → 看到 8 阶段进度事件逐步推送（mock 回放）
- [ ] 编译完成 → 进入 Module 学习 → 答题（choice / sorting / fill_blank）
- [ ] 答题反馈面板（FeedbackPanel）正常显示解析
- [ ] 费曼 6 步教学可走完
- [ ] 完成页显示掌握度 + 评分

### 4.3 持久化（showcase = LocalStorage）

- [ ] 刷新浏览器后，学习进度和题库数据不丢
- [ ] 答题后再刷新，作答历史仍在
- [ ] 多标签页打开同一 Module，数据一致

### 4.4 LLM 评分（仅当配置了 env Key）

- [ ] 填空题作答后，反馈文本含语义判分（非纯精确匹配）
- [ ] 费曼最终评分有数值返回
- [ ] 若超时（Netlify Free 10s 限制），反馈降级但不崩溃（显示"评估失败，请重试"）

### 4.5 SEO / 分享

- [ ] 用 [opengraph.xyz](https://www.opengraph.xyz) 检查 OG 卡片预览
- [ ] 分享链接到微信/Twitter，卡片图片正常（§2.2 PNG 转换后）

## 5. 与 Vercel 部署的差异速查

若团队同时维护 Vercel 部署，下表是两平台关键差异：

| 维度 | Vercel | Netlify |
|------|--------|---------|
| 路由超时配置 | `vercel.json` `functions.maxDuration` | `netlify.toml`（但上限更低） |
| Free 函数超时 | 300s | 10s |
| `better-sqlite3` | 需 `serverExternalPackages` 配置 | 同左 + 可能需 optionalDependencies |
| SQLite 持久化 | 短暂 FS，不可靠 | 同左，不可靠 |
| Next.js 插件 | 内置 | `@netlify/plugin-nextjs`（自动） |
| `outputFileTracingIncludes` | 生效 | 生效（插件兼容） |
| SSE 流式 | 原生支持 | 支持（但受函数超时限制） |
| Production 模式 | 仅本地 dev / 自建 server 可靠 | 同左 |

**结论**：两个平台都应跑 `showcase` 模式。`production` 模式留给本地开发与自建服务器。

## 6. 故障排查

### 6.1 构建失败

| 症状 | 原因 | 解决 |
|------|------|------|
| `Error: ENOENT ... prompts/*.md` | Prompt 文件未打包 | 确认 `next.config.ts` 的 `outputFileTracingIncludes` 未被删除 |
| `better-sqlite3` 编译失败 | native addon 环境 | 见 §3.5 方案 A/B |
| `next build` OOM | Netlify Free 构建内存 2GB | 升级 Netlify Pro，或本地 build 后用 `netlify deploy --prod --dir=.next` |
| 字体 404（Console 报错） | §2.1 分包未执行 | 执行 `bun run scripts/split-fonts.ts` 并提交产物 |

### 6.2 运行时问题

| 症状 | 原因 | 解决 |
|------|------|------|
| 中文显示为方块/系统字体 | 字体分包未加载 | 检查 `/fonts/source-han-serif-sc/result.css` 是否 200 |
| Settings 页 API Key 为空 | 未配 env 或未重新部署 | Dashboard 配 Key 后触发新部署 |
| `/api/feedback` 504 | Netlify Free 10s 超时 | 升级 Pro，或不配 Key（降级） |
| 编译页一直转圈 | （showcase 不应发生）若发生说明 mode 配错为 production | 确认 `NEXT_PUBLIC_APP_MODE=showcase` |
| 刷新后数据全丢 | 浏览器禁用 LocalStorage / 隐私模式 | 提示用户关闭隐私模式 |

### 6.3 模式混淆排查

`.env.local` 设了 `NEXT_PUBLIC_APP_MODE=production` 会导致本地 dev 跑 production 模式（`v1.1.0-plan.md` §2.4 红线 #10）。若本地正常但 Netlify 异常，检查：

```bash
# 确认 Netlify 构建用的 env
# Dashboard → Site settings → Environment variables
# 必须有 NEXT_PUBLIC_APP_MODE=showcase（或留空走默认）
# 绝不能是 production（除非配了外部数据库，当前版本不支持）
```

## 7. 持续部署流程

配置完成后，Netlify 的自动部署流程：

| 事件 | 触发 |
|------|------|
| `git push origin main` | 生产部署（Production deploy） |
| Pull Request 创建/更新 | 部署预览（Deploy Preview），获得独立 URL |
| Netlify Dashboard → Trigger deploy | 重新部署指定 commit |

**环境变量生效**：修改 Dashboard 环境变量后，需手动 Trigger deploy 或推送新 commit，旧部署不会自动应用新变量。

**回滚**：Dashboard → Deploys → 找到历史 deploy → **Publish deploy**（一键回滚到任意历史版本）。
