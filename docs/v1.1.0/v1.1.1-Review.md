## 2. 架构决策记录

### 2.1 D-1：skip-ignored 放在 ConceptView 而非 progress-store.advance

**选择**：在 ConceptView 加 useEffect 检测当前 slotQuiz.ignored，answering 阶段自动调 advance() 跳过；**不修改** progress-store.advance 的 quizIndex 推进逻辑。

**理由**：progress-store.advance（concept case）用 `quizCount = concept.quizSeries.quizzes.length` 作原始数组索引，被 StaircaseProgress 进度条、reviewSlots 注入、跨概念复习槽位等多处不变量依赖。若在状态机层改"跳过 ignored"，quizIndex 的"逻辑位置"与"数组下标"会分叉，连锁破坏上述不变量。ConceptView 层跳过是最低风险方案——advance() 始终前向推进，跳过链必然终止（找到非忽略题或 concept/module 结束）。

### 2.2 D-2：F41 用软删除（ignored flag）而非硬删除

**选择**：`quiz.ignored = true`，不从 quizzes[] 物理移除。继承自计划 §D-1。

**理由**：① 可逆（误判可撤销）；② 数据完整性（quizId 被 attempts/mastery/progress 引用，硬删除产生孤儿）；③ 审计性（保留原题便于分析"哪些题易被忽略"指导 Prompt 迭代）。

### 2.3 D-3：reevaluateLastAttempt 返回 FeedbackRuntime

**选择**：action 签名从 `Promise<void>` 改为 `Promise<FeedbackRuntime>`，内部已计算的 result 直接 return。

**理由**：ConceptView 修正答案后需更新 feedback 显示。若 action 不返回结果，组件须独立再调一次 evaluateAnswerAsync——对 fill_blank 语义判分意味着两次 LLM 调用。返回结果使一次调用同时满足"持久化 score 更新"与"显示反馈更新"。

### 2.4 D-4：showcase origin 题库不可修正/忽略

**选择**：`canCorrect = currentModule?.origin !== 'showcase'`，FeedbackPanel 在 canCorrect=false 时不渲染修正/忽略按钮。

**理由**：展示题库是精选内容，运行时用户级修正会破坏跨用户一致性。展示题库确有错误应走"修源文件 → 重新发布"流程，非运行时改。与 F07 重命名对 showcase 的保护一致。

### 2.5 D-5：二次确认用 inline confirm（非 modal）

**选择**：ConfirmInline 组件——点触发按钮后原地变为"[确认标签] [确认] [取消]"，Escape 取消。

**理由**：与现有"蒙对标注"交互风格一致（FeedbackPanel 内轻量操作），不打断学习心流。两步操作已满足"高危操作需二次确认"要求。
