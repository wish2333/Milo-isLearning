## 2. 架构决策记录

### 2.1 D-1：F04 checkpoint 粒度 = 按 stage（非按 concept）

选择 stage 4（module 组装完成）和 stage 6.5（challenge 完成）两个粗粒度 checkpoint，而非 per-concept 细粒度。理由：quiz stage 使用 concurrency=3 并行 + 就地变异 partialModule，per-concept checkpoint 需重构并行逻辑，风险远大于收益。

### 2.2 D-2：F08 token 追踪通过 runAgent() 返回值变更

选择 `runAgent<T>()` 返回 `Promise<{ data: T; usage: TokenUsage }>` 而非回调或包装器。仅 12 个调用点（全部在 pipeline.ts + 3 个 API route + 1 个 eval script），改动面可控。

### 2.3 D-3：F11 双轨策略 = production SQLite / showcase LocalStorage

Production 模式事件写 SQLite events 表；showcase 模式继续 LocalStorage 落盘。Showcase 零后端原则不变。

### 2.4 D-4：F07 重命名不改 moduleId

仅更新 Module.title 字段。moduleId 被 7 个关联 key 引用（attempts/mastery/progress/feynman/quality 等），改 ID 等于局部迁移，风险远大于收益。所有 UI 从存储的 Module 对象实时读取 .title（约 20 处），改值即自动传播。

### 2.5 D-5：F22 主题进度恢复用独立 storage key

新增 `alc:topic-progress:{topicId}` 存储已完成模块快照，不复用 TopicSession（运行时会话状态）。独立 key 可被 TopicCard 等组件读取展示"上次进度"。

### 2.6 PB.2 Pipeline checkpoint 回调解耦

Pipeline 不直接导入 `server-only` 和 persistence 层。通过 CompileOptions.writeCheckpoint 回调注入，保持 pipeline 模块纯粹。checkpoint 用 string key（如 `'module-concepts'`）而非 CompileStage 枚举，支持复合 checkpoint。

### 2.7 PB.2 Resume stage 跳过策略

每个 stage 开头检查 `isStageCompleted(stage)`：若 true 则从 checkpointData 恢复 artifact，跳过 runAgent 调用。Stage 4 resume 有特殊处理——yield 一个带"从断点恢复"消息的 progress event。

## 4. 已知限制

### 4.1 Schema 版本 meta 不更新

`INSERT OR IGNORE` 保证幂等，但 v1/v2 数据库升级到 v3 后 `meta.schema_version` 值不更新（仍为创建时的值）。events/compile_session/compile_staging 表通过 `CREATE TABLE IF NOT EXISTS` 自动创建，功能正常。如需精确版本检测，可查 `sqlite_master` 表存在性。

### 4.2 OG 图片为 SVG 格式

`public/og-image.svg` 是 SVG 格式。部分社交平台（Facebook/LinkedIn）不支持 SVG OG 图片。生产部署前应转换为 PNG（1200x630, <300KB）。

### 4.3 字体分包需手动执行

`scripts/split-fonts.ts` 已就位但需手动运行，且需要 Source Han Serif SC 原始字体文件（项目仓库不含）。首次部署前需执行分包构建。

### 4.4 F04 checkpoint 仅 production 模式

Showcase 模式编译是 mock 回放，无 LLM 调用，不需要 checkpoint。所有 checkpoint 逻辑通过 `sessionId` 存在性守卫，showcase 模式自动跳过。

### 4.5 E2E 基线待回归对比

v1.0.1 建立的 51 tests E2E 基线尚未在 v1.1.0 上重跑。Phase B 的 pipeline 改造（PB.2）和 storage schema 升级可能影响 showcase 回归。建议合入 main 前运行 `bun run e2e`。

### 4.6 PC.2 英文模板仅 PoC 2 个

只翻译了 import.md + quiz-batch.md 两个高优先级模板。其余 8 个模板仍为中文。完整英文化需后续迭代。

## 7. 功能价值与用户意义（补遗）

> 本节面向不熟悉实施细节的读者，用"用户能感知到什么"的视角重述 v1.1.0 的价值。技术细节见 §1-§2。

### 7.1 一句话概括

v1.1.0 是一个**"看不见的体验升级"**版本——它几乎没有改变用户能看见的功能入口，但大幅提升了**编译的可靠性、学习的连贯性、性能的体感、以及开发者对系统的可观测性**。如果把 v1.0.0 定义为"能用"，v1.1.0 的目标是"**好用且可运维**"。

### 7.2 按用户价值分类的功能解读

#### A. 编译更可靠 —— 不再因刷新/断网前功尽弃（F04 断点续编）

**用户痛点**：编译一份长 Markdown（如整章教材）需要 30-60 秒、8 个 LLM 阶段。中途刷新页面、网络波动、误关标签页，整个编译从头再来，消耗的 token 费用白花。

**v1.1.0 做了什么**：在编译的第 4 阶段（Module 骨架完成）和第 6.5 阶段（综合题完成）写入 SQLite 断点。中断后重新进入编译页，系统检测到未完成的会话，提示"检测到未完成的编译，是否从第 X 阶段继续？"。

**意义**：长文档编译的成本和挫败感显著降低。这是 production 模式独有的能力（showcase 是 mock 回放，无中断风险）。

#### B. 学习更连贯 —— 主题进度不再"清零"（F22 + F23）

**用户痛点**：一个主题（Topic）包含多个题库（Module）。学到第 3 个题库时退出，下次回来发现进度没保存，不知道学到哪了，也不知道整体掌握得怎样。

**v1.1.0 做了什么**：
- **F22**：退出主题会话时自动保存"已完成哪些题库"的快照，重新进入时恢复标记。
- **F23**：主题过渡页和完成页展示**聚合掌握度**——把各题库的掌握度按题量加权平均，给出一个"主题整体掌握 X%"的直观数字。

**意义**：主题学习从"一锤子买卖"变成"可中断、可回看"的连续体验。聚合掌握度让用户知道自己离"学完整个主题"还差多远。

#### C. 题库可维护 —— 能改名字了（F07 重命名）

**用户痛点**：LLM 生成的题库标题常常不准（如"马克思主义基本原理概论第一章"被命名为"资本"）。v1.0.0 标题一旦生成就不可改，只能删了重编。

**v1.1.0 做了什么**：题库列表和切换器加了内联重命名（铅笔图标）。改标题只改 `Module.title` 字段，不动 moduleId（避免级联迁移风险），所有页面实时读取新标题。

**意义**：题库管理从"只读"变成"可整理"。展示模式的官方题库（origin=showcase）保持不可改，避免污染精选内容。

#### D. 性能体感 —— 中文加载更快（F10 字体分包）

**用户痛点**：思源宋体整字体约 17MB，首屏一次性加载，弱网下白屏数秒。

**v1.1.0 做了什么**：用 `cn-font-split` 把字体按字符频次拆成数十个 woff2 小包，首屏只加载可见字符对应的包（<2MB），其余按需懒加载。

**意义**：首屏文字渲染速度显著提升，尤其移动端。showcase 和 production 模式均受益。（注：分包产物需手动执行 `scripts/split-fonts.ts` 生成，见 [v1.1.0-guide.md](./v1.1.0-guide.md) §2.1）。

#### E. 学习数据更丰富 —— 知道花了多少时间（F13 学习时长）

**用户痛点**：重刷错题时只知道对错，不知道自己在这道题上磨了多久，无法判断"是粗心还是真不会"。

**v1.1.0 做了什么**：每条作答记录新增 `answeredAt`（提交时间戳）和 `timeSpentMs`（本题耗时）。重刷页展示"本次复习总时长"和每题耗时。

**意义**：为后续的自适应学习（按耗时调整难度）和用户自我反思提供数据基础。

#### F. 可观测性 —— 开发者能看到"花了多少钱、LLM 修了几次"（F08 + F11 + F33 + F20 + D5）

这是 v1.1.0 最重的板块，但对**终端用户几乎不可见**——它的价值面向**运维和产品决策**：

| 功能 | 回答的问题 | 意义 |
|------|-----------|------|
| F08 成本估算 | "编译一份教材花了多少 token、多少钱？" | 让成本透明，避免无意中超预算 |
| F11 埋点云端 | "用户实际编译成功率多少？哪一步最容易失败？" | production 模式事件入 SQLite，可查询分析 |
| F33 Mapper 修复统计 | "LLM 生成的题里有多少被代码自动修正了？" | 衡量 LLM 输出质量，指导 Prompt 迭代 |
| F20 app_mode 标记 | "这个事件来自 showcase 还是 production？" | 区分演示流量与真实使用 |
| D5 env fallback 埋点 | "有多少用户在用免费的服务端 Key 兜底？" | 决定是否需要收费/限流 |

**意义**：从"盲飞"到"有仪表盘"。v1.0.0 验证了"能不能用"，v1.1.0 补齐了"用得怎么样"的度量能力。这是产品从 demo 走向可持续运营的前提。

#### G. 工程基建 —— Prompt 可迭代、可中英对照、mock 更真实（F30 + F31 + F32）

面向**开发流程**的改进，对用户间接生效：

- **F30 Prompt 版本管理**：通过 `PROMPT_VERSION` 环境变量切换不同版本的 Prompt 模板，可安全 A/B 测试新 Prompt 而不覆盖旧版。
- **F31 英文模板**：新增英文 Prompt 目录，为国际化用户和多语言编译质量对比打基础（当前 PoC 翻译了 2 个高优先级模板）。
- **F32 LLM 响应录制**：可以用真实 LLM 调用录制事件流，替代手写 mock，让 showcase 的模拟编译更贴近真实表现。

**意义**：让"改 Prompt"这个高频迭代动作从"改完即覆盖、无法回滚"变成"版本化、可对比、可回退"。

#### H. 体验打磨 —— 无障碍、分享卡、偏好重置、容错（D1-D4）

| 功能 | 用户感知 |
|------|---------|
| D1 a11y 审计 | 读屏软件用户、键盘用户能正常使用；focus 有清晰视觉提示 |
| D2 OG 图片 | 分享链接到社交平台有预览卡片（不再是一行光秃秃的 URL） |
| D3 resetPreferences | "重置学习偏好"和"清除 API Key"分离——重置间隔重复开关不会误删 Key |
| D4 openai-compat 容错 | 配了 OpenAI 兼容端点但忘了填 baseURL 时不再崩溃，而是清晰报错 |

### 7.3 版本定位总结

| 维度 | v1.0.0 | v1.1.0 |
|------|--------|--------|
| 用户可见新功能入口 | 多（完整学习闭环 + 双模式） | 少（重命名、进度恢复、时长显示） |
| 可靠性/连贯性 | 基础（刷新不丢，但编译中断=重来） | 显著提升（断点续编 + 主题进度） |
| 性能体感 | 基础（字体一次性加载） | 提升（分包加载） |
| 可观测性 | 几乎为零 | 完整（成本 + 事件 + 修复统计） |
| 开发可维护性 | 基础 | 提升（Prompt 版本化 + 录制回放） |
| 合规/无障碍 | 基础 | 达标（axe-core 0 critical） |

**一句话**：v1.1.0 是把 v1.0.0 的"能用"打磨成"经得起真实使用和持续迭代的工程产品"。它不追求新功能数量，而追求**已有体验的鲁棒性、可度量性、可演进性**。
