## 第一章 设计哲学与原则

> 视觉与交互不是规格书的装饰，而是规格书的最后一公里表达。如果视觉与交互违反了规格书的第一章五条原则，那么无论多"好看"，这个 UI 都是失败的。

### 1.1 一句话设计哲学

> **让界面消失，让学习发生。**

更具体地：

- 用户不应意识到"我在用一个 UI"，而应只意识到"我在学一件事"
- 视觉**永远服务于降低表达成本**，从不服务于装饰
- 任何让用户分心于"如何操作"的视觉元素，都是设计 bug

这条哲学是本文档所有决策的最终裁决依据。任何章节冲突时，回到这条。

### 1.2 五条产品原则的 UI 翻译

继承规格书 [§1.3 五条产品原则](../Product-Specification.md#13-产品原则)，每条原则翻译为可执行的 UI 决策（共 18 条 UI 准则）。

#### Principle 1 — Quiz 永远不是目的

**UI 翻译**：所有 Quiz 视觉元素必须服务于"为最终的费曼表达铺垫"，而不能让用户感到 Quiz 本身就是终点。

| 编号 | UI 准则 | 反模式（禁止） |
|------|---------|----------------|
| P1-UI-1 | 进度条始终显示"距离费曼还有 X 步"，让费曼是可见的灯塔 | 进度条只显示"Quiz X / N"，让用户陷入题海感 |
| P1-UI-2 | Concept 完成时不显示"恭喜完成 Concept"庆祝弹窗，仅做轻微阶梯上升动效 | Concept 完成时弹出庆祝动画、撒花、奖杯 |
| P1-UI-3 | Module Feynman 阶段的视觉**仪式感必须明显强于** Concept 阶段（背景细微变化、字号上升、阶梯顶点指示） | Feynman 与 Concept 视觉权重相同 |
| P1-UI-4 | 完成页强调"你从选择题走到了完整费曼"的成长叙事，而非"你答对了 X 题" | 完成页只显示答对率、统计图表 |

#### Principle 2 — 永远降低表达成本

**UI 翻译**：视觉权重必须始终偏向"用户只需再点一下"，任何输入框、文本域的出现都必须有充分的视觉过渡。

| 编号 | UI 准则 | 反模式（禁止） |
|------|---------|----------------|
| P2-UI-1 | Choice 选项为全宽卡片、点击热区 ≥ 48px 高，保证"一次点击"的物理低成本 | 紧凑列表式选项，点击热区 < 40px |
| P2-UI-2 | Fill Blank 输入框出现前，必须有视觉过渡（如 Choice 选项淡出 → 输入框淡入），而非突兀切换 | 直接从选项切到输入框，无过渡 |
| P2-UI-3 | Feynman Step 6 大文本框配字数下限提示（100 字），但**视觉上弱化字数压力**（小字号、低对比） | 大字号倒计数显示字数 |
| P2-UI-4 | "下一题"按钮永远位于视线焦点（移动端底部固定 / 桌面端右下角主按钮位） | 用户需要滚动寻找下一步按钮 |

#### Principle 3 — 每一道题都应该让用户成功

**UI 翻译**：错误反馈必须**克制、温暖、无惩罚感**；正确反馈必须**简洁、不夸张**。

| 编号 | UI 准则 | 反模式（禁止） |
|------|---------|----------------|
| P3-UI-1 | 答错用**柔和陶土橙**（`#C99A6E`）标注错误选项，**绝不使用鲜红、绝不使用 ✗ 符号** | 任何含红色（`#FF0000` 系）或叉号元素的反馈 |
| P3-UI-2 | 答错时同时高亮**正确选项的柔和鼠尾草绿**（`#7FA88C`），让用户看到"原来是这样"而非"我错了" | 只标注错误，不显式指出正确 |
| P3-UI-3 | 答错解释文案用衬线正文体，**不使用警示框、不使用感叹号、不使用 emoji** | "⚠️ 错误！你应该选 B" 这类警示框 |
| P3-UI-4 | 答对反馈仅做 240ms 的选项底色淡绿渐入 + 一根 1px 细线从左划过，**不缩放、不弹跳、不撒花、不播放音效** | 任何弹跳、缩放、撒花、🎉 emoji、音效 |
| P3-UI-5 | 答错时不切换页面、不震动、不闪烁；解释**就地展开**于选项下方 | 切换到独立错误页；任何震动/闪烁动效 |

#### Principle 4 — 不让用户思考"如何回答"

**UI 翻译**：题干视觉权重必须显著高于选项；交互方式必须不言自明。

| 编号 | UI 准则 | 反模式（禁止） |
|------|---------|----------------|
| P4-UI-1 | 题干使用 22-28px 衬线字体，选项使用 16-17px 衬线字体，**字号差 ≥ 6px** | 题干与选项同字号 |
| P4-UI-2 | 排序题（Sorting）桌面端用拖拽手柄 + 移动端用上下箭头，**两种方式同时可用** | 仅提供拖拽（移动端难以操作） |
| P4-UI-3 | 填空题（Fill Blank）输入框 placeholder 用同句中的"……"占位，而非提示文字"请输入答案" | 输入框提示"在此输入" |
| P4-UI-4 | 当前 Quiz 的交互类型（Choice / Sorting / Fill Blank）**不在 UI 上显式标注**，让用户自然跟随交互 | 显示标签"选择题 / 排序题 / 填空题" |

#### Principle 5 — 输出自由度逐渐增加

**UI 翻译**：视觉必须**让用户在每一步都感受到表达自由度的连续上升**，而不出现"突然要写一大段"的视觉断层。

| 编号 | UI 准则 | 反模式（禁止） |
|------|---------|----------------|
| P5-UI-1 | 阶梯进度条在每个 Concept 内呈"小阶梯"，Concept 之间呈"大阶梯"，费曼为顶 — **视觉上连续无断层** | 用进度百分比条（无层级感） |
| P5-UI-2 | Feynman Step 1-4（Choice）→ Step 5（Fill Blank）→ Step 6（完整输出）的视觉权重**逐步增加**：选项缩小、输入区域扩大、提交按钮文案从"下一题"变为"写下我的解释" | Step 5 与 Step 6 视觉权重相同 |
| P5-UI-3 | 完成页展示"表达自由度成长曲线"可视化（阶梯已攀登的回顾） | 完成页仅显示百分比完成度 |
| P5-UI-4 | 提交按钮文案随表达自由度上升而变化：`下一题` → `继续` → `补全这句话` → `写下我的解释` | 全程统一文案"提交" |

### 1.3 三条 UI 元原则（meta-principles）

五条产品原则之上，本 UI 设计另外引入三条"元原则"——它们是从五条原则派生出来的、但在视觉决策时更直接生效的指导。

#### M1 — 静默优先（Silence First）

UI 应当像一本好书——大部分时间是安静的，只在关键时刻发声。这意味着：

- **默认状态极简**：背景、容器、装饰元素不与内容争夺注意力
- **动效极少触发**：只在阶段切换、答题反馈、费曼高潮三类时刻触发
- **色彩极克制**：80% 屏幕是中性色（深墨绿黑 / 暖象牙白），20% 是强调色

#### M2 — 进度即方位（Progress as Orientation）

PRD [§9.1 第 3 条](../PRD.md#91-全局交互原则)要求"进度可见"。本 UI 将其升级为：**进度不只是数字，而是用户的方位感**。

- 阶梯进度条同时回答三个问题：我在哪？我走过哪？还要走多远？
- Concept / Challenge / Feynman 三阶段视觉上**必须可区分**，让用户知道"我在山的哪一侧"

#### M3 — 错误即学习（Mistakes as Learning）

这是规格书 [§9.3 Quiz as Scaffold](../Product-Specification.md#93-quiz-as-scaffoldquiz-是脚手架)与 PRD [§9.1 第 2 条不惩罚](../PRD.md#91-全局交互原则)的视觉化：

- 错误反馈的视觉权重**必须低于**正确反馈
- 错误不是"你错了"，而是"系统给你看正确答案"
- 错误后的新题替换动效必须**温和**（淡入淡出，非滑入），不引起视觉冲击

### 1.4 反向参考：不学谁

明确"不学"的产品美学，帮助避免偏航：

| 产品 | 不学的理由 |
|------|-----------|
| 多邻国（Duolingo） | 卡通美学、过度游戏化、奖励刺眼——违反"学术沉静"基调 |
| Khan Academy | 视觉过于朴素、缺乏学术沉静感、依赖视频而非沉浸文字 |
| Quizlet / Anki | 卡片式抽认卡的孤立感——违反"阶梯连续"原则 |
| Brilliant.org | 接近正确但配色过于"科技蓝紫"——违反"温暖学术"基调 |
| Notion / Linear | 视觉系统可借鉴（克制、字体），但**不适合直接套用**——它们是生产力工具，本产品是学习产品，需要更强的"旅程感" |

### 1.5 一句话总结

> **本 UI 的目标是：当一个进阶学习者打开这个产品，他不会觉得自己在"用一个 App"，而会觉得自己在"安静地、被引导着、一步步攀登一座小小的知识山丘"。**

### 10.5 风险与开放问题

#### 10.5.1 字体加载

**风险**：Fraunces 是可变字体，文件较大（~100KB）。首屏加载可能影响 NFR-P4（FCP ≤ 1.5s）。

**缓解**：
- 使用 `font-display: swap`，先用 Georgia 衬线回退
- 仅加载 400/500/600 三个字重 + opsz 默认值
- 中文思源宋体按需加载（用 `unicode-range` 切片）

**CJK 字体子集化强化方案（V1.1 修订 · 关键工程）**：

V1.0 仅提到"按 unicode-range 切片"是不够的——`Source Han Serif` 完整版超 20MB，即使用 `unicode-range`，浏览器仍可能下载完整文件。FCP ≤ 1.5s 在加载 20MB 字体时几乎不可能达成。

**强制三层子集化方案**：

##### 第一层：静态子集化（构建期）

使用 `fonttools` / `subset-font` / `glyphhanger` 在构建期把字体切成多个**真正独立的子集文件**：

```bash
# 使用 glyphhanger（PyPI 包）做静态子集化
glyphhanger \
  --subset=SourceHanSerifSC-Regular.otf \
  --LATIN \
  --US_ASCII \
  --output=SourceHanSerifSC-Regular  # 输出多个 .woff2 文件

# 或者用 cn-font-split（中国社区开源工具，更懂 CJK）
npx cn-font-split ./SourceHanSerifSC-Regular.otf -o ./output/
```

`cn-font-split` 会自动按 GB2312 / GBK / 通用字符集把字体切成约 **200-500 个 woff2 包**，每个 5-30KB。

##### 第二层：首屏精简包（3500 字）

> **首屏 3500 常用汉字**覆盖中文互联网 99.8% 的内容，是事实上的"黄金子集"。

```css
@font-face {
  font-family: "Source Han Serif SC";
  src: url("/fonts/shs-sc-3500-regular.woff2") format("woff2");
  font-weight: 400;
  font-display: swap;
  unicode-range: U+4E00-9FFF;  /* 仅 CJK 区间 */
  /* 首屏精简包仅 ~150-300KB，对 FCP 影响可控 */
}
```

**首屏 3500 字来源**：
- 中华人民共和国国家标准《现代汉语常用字表》（1988）定义的 3500 常用字
- GitHub：`parsimonhi/animated_INTER_HEIGHT` 或 `jayknoxlee/chinese-3500` 等开源字表
- 用 `fonttools` + 字表生成精简字体文件

##### 第三层：按需懒加载剩余字符

对 3500 字外的字符（生僻字、专有名词）：
- `unicode-range` 切片到独立 woff2 文件
- 浏览器只在页面真实出现该字符时才下载对应切片
- 若用户 Markdown 含罕见字，**自动回退到系统宋体**，体验降级但功能不丢

```css
/* 第二批：3500-7000 字 · 约 200KB · 按需加载 */
@font-face {
  font-family: "Source Han Serif SC";
  src: url("/fonts/shs-sc-7000-regular.woff2") format("woff2");
  font-weight: 400;
  font-display: swap;
  unicode-range: U+4E00-9FFF;
}

/* 第N批：剩余字符 · 按需 */
@font-face {
  font-family: "Source Han Serif SC";
  src: url("/fonts/shs-sc-ext-regular.woff2") format("woff2");
  font-weight: 400;
  font-display: swap;
  unicode-range: U+20000-2FA1F, U+3000-303F;
}
```

**系统宋体回退兜底**：

```css
:root {
  --font-serif: "Fraunces", "Source Han Serif SC", "Noto Serif SC",
                "Songti SC", "SimSun",           /* macOS / Windows 系统宋体 */
                "LXGW WenKai", Georgia, serif;
}
```

## 第八章 可访问性

本章落实 PRD [§6.2 NFR-U4](../PRD.md#62-可用性nfr-u) 的要求：**关键交互支持键盘导航；颜色对比度 ≥ AA（WCAG 2.1）**。

### 8.1 可访问性设计原则

本产品的可访问性服务于三类用户：
1. **视障/色弱用户**——通过对比度、形状冗余、屏幕阅读器支持
2. **键盘用户**——通过完整键盘导航、可见焦点
3. **对动效敏感用户**——通过 `prefers-reduced-motion` 降级

设计原则：**可访问性不是"额外功能"，而是质量门槛**。任何不满足本章的设计都视为未完成。

### 8.2 颜色对比度（WCAG 2.1 AA）

详见 [§2.1.4 对比度验证](#214-对比度验证wcag-21-aa)。

**已验证**：所有正文文字配色 ≥ AA，大部分 ≥ AAA。

#### 8.2.1 不依赖颜色传达信息

本产品有多个"颜色 + 形状"双重编码场景，确保色弱用户可识别：

| 场景 | 颜色 | 形状冗余 |
|------|------|---------|
| 答对反馈 | 鼠尾草绿 | 1px 细线划过（视觉确认） |
| 答错反馈 | 陶土橙 | 解释文案明确说明"差一点" |
| Rubric 命中 | 实心绿点 ● | 形状本身可识别 |
| Rubric 部分命中 | 半填充 ◐ | 形状本身可识别 |
| Rubric 未提及 | 空心 ○ | 形状本身可识别 |
| 阶梯已完成 | 鼠尾草绿 | 实心格 vs 空心格 |
| 阶梯进行中 | 金棕 | 半填充 |

**结论**：本产品**绝不**仅靠颜色传达"对/错"——形状、文字、布局都是冗余编码。

### 8.3 键盘导航

#### 8.3.1 完整键盘可操作

所有交互元素必须可通过键盘操作：

| 元素 | 键盘操作 |
|------|---------|
| 选项（Choice） | `Tab` 聚焦；`1-4` / `Space` / `Enter` 选择 |
| 排序项（Sorting） | `Tab` 聚焦；`↑` `↓` 调整顺序 |
| 填空输入 | `Tab` 聚焦；直接输入 |
| 主按钮 | `Tab` 聚焦；`Enter` 触发 |
| 返回链接 | `Tab` 聚焦；`Enter` 触发 |
| Concept 内 Quiz | `1-4` 选择；`Enter` 提交 |
| Concept 切换 | 当前 Concept 完成后 `Enter` 进入下一个 |
| Feynman Step 切换 | `Enter` 进入下一步 |

#### 8.3.2 可见焦点（Focus Visible）

```css
/* 所有可聚焦元素必须有可见焦点 */
*:focus-visible {
  outline: 2px solid var(--accent-primary);
  outline-offset: 4px;
  border-radius: var(--radius-sm);
}

/* 按钮 focus 用 box-shadow 替代 outline（更美观） */
.btn-primary:focus-visible,
.btn-secondary:focus-visible {
  outline: none;
  box-shadow: 0 0 0 4px var(--accent-primary-soft);
}
```

**关键约束**：
- **绝不** `outline: none` 而不提供替代焦点
- focus-visible 仅在键盘聚焦时显示（鼠标点击不显示，避免视觉污染）

#### 8.3.3 逻辑 Tab 顺序

#### 7.3.6 移动端键盘弹出的视口适配（V1.1 新增 · 关键工程）

**问题**：iOS Safari 与 Android Chrome 在键盘弹出时对 `vh` / `position: sticky` 的处理存在历史性 bug：
- iOS Safari 旧版 `100vh` 包含地址栏 + 键盘高度，导致底部 sticky 按钮被键盘遮挡
- iOS Safari 偶发键盘弹出后页面 `scroll-into-view` 失效，输入框被键盘盖住
- Android Chrome 切换地址栏显隐时 `100vh` 抖动

**三层防御方案**：

##### 第一层：使用 `100dvh` 替代 `100vh`

```css
:root {
  --app-height: 100vh;       /* 兜底 */
}

@supports (height: 100dvh) {
  :root {
    --app-height: 100dvh;    /* 动态视口高度，跟随键盘/地址栏变化 */
  }
}

.app-shell {
  min-height: var(--app-height);
}
```

**关键**：所有依赖"满屏高度"的元素（`.app-shell`、`.nav-top` sticky、底部 sticky 按钮）必须用 `--app-height`，禁止直接 `100vh`。

##### 第二层：监听 `visualViewport` API

```js
function setupKeyboardAdaptation() {
  if (!window.visualViewport) return;
  
  const viewport = window.visualViewport;
  
  const onResize = () => {
    // 键盘弹出时 visualViewport.height 显著缩小
    const keyboardHeight = window.innerHeight - viewport.height;
    
    if (keyboardHeight > 100) {
      // 键盘弹出
      document.body.classList.add('keyboard-open');
      document.documentElement.style.setProperty('--keyboard-height', `${keyboardHeight}px`);
      
      // 把聚焦元素 scroll 进可视区
      const activeEl = document.activeElement;
      if (activeEl && typeof activeEl.scrollIntoView === 'function') {
        activeEl.scrollIntoView({ behavior: 'smooth', block: 'center' });
      }
    } else {
      // 键盘收起
      document.body.classList.remove('keyboard-open');
      document.documentElement.style.style.removeProperty('--keyboard-height');
    }
  };
  
  viewport.addEventListener('resize', onResize);
  viewport.addEventListener('scroll', onResize);
}
```

##### 第三层：sticky 按钮贴键盘上沿

```css
/* 默认底部 sticky */
.submit-bar {
  position: sticky;
  bottom: 0;
  padding: var(--space-4) var(--space-5);
  background: var(--bg-base);
  border-top: 1px solid var(--border-default);
  transition: transform var(--duration-base) var(--ease-standard);
}

/* 键盘弹出时，向上让出键盘空间 */
.keyboard-open .submit-bar {
  bottom: var(--keyboard-height, 0);
}
```

##### 第四层（聚焦元素自动滚动）

Fill Blank 与 textarea-feynman 聚焦时：

```js
document.addEventListener('focusin', (e) => {
  if (!e.target.matches('input, textarea')) return;
  
  // 延迟一帧，等键盘完全弹出
  requestAnimationFrame(() => {
    setTimeout(() => {
      e.target.scrollIntoView({
        behavior: 'smooth',
        block: 'center'  // 让聚焦元素居于键盘上方可见区域中央
      });
    }, 300);
  });
});
```

## 第六章 微交互与动效详设

本章定义所有页面共享的微交互时序、状态过渡、仪式动效。第二章给出 token，本章给出**用 token 拼装的具体场景**。

### 6.1 答题反馈动效（核心微交互）

#### 6.1.1 答对动效时序

```
用户点击 [提交]
   │
   ▼  T+0ms      提交按钮瞬间下沉 1px（160ms `--ease-standard`）
   │              选项视觉无变化
   │
   ▼  T+200ms    [模拟 Feedback Agent 延迟 · 让反馈不显得"预设"]
   │
   ▼  T+200ms    正确选项开始过渡：
   │              - border-color: --border-default → --success（240ms `--ease-standard`）
   │              - background: transparent → --success-soft（240ms `--ease-standard`）
   │
   ▼  T+240ms    选项底部 1px 细线从左到右划过（240ms `--ease-emphasized`）
   │              - 细线颜色：--success
   │
   ▼  T+280ms    反馈短语（"很好。"）淡入 + 上移 4px（400ms `--ease-standard`）
   │
   ▼  T+440ms    解释面板（panel-explanation）从 max-height:0 → auto 展开（400ms）
   │              - opacity: 0 → 1 同步过渡
   │
   ▼  T+840ms    按钮文案从"提交"渐变为"下一题"（240ms 渐变交叉）
   │
   ▼  T+1080ms   全部就绪，等待用户点击"下一题"

总反馈时长：约 1.08 秒（其中用户感知动效主要在 200-840ms 这 640ms）
```

**关键约束**：
- 全程**无任何缩放、弹跳、粒子、撒花**
- 细线划过是唯一的"肯定性"动效——克制但可感知
- 解释面板的展开节奏要慢，让用户感觉"系统在为我准备解释"

**1px 细线划过的工程实现（V1.1 修订 · 重要）**：

V1.0 文档没有给出具体实现，工程实现时容易踩两个坑：
1. ❌ 用 `border-bottom-width: 0 → 1px` 动画——触发 Layout，性能差
2. ❌ 用 `width: 0 → 100%` 动画——同样触发 Layout

**正确做法**：用 `::after` 伪元素 + `transform: scaleX(0 → 1)`，完全在合成层（GPU）执行：

```css
.option-choice--correct {
  position: relative;  /* 为 ::after 定位锚 */
  border-color: var(--success);
  background: var(--success-soft);
}

.option-choice--correct::after {
  content: "";
  position: absolute;
  left: 0;
  right: 0;
  bottom: -1px;          /* 紧贴边框底部 */
  height: 1px;
  background: var(--success);
  
  transform: scaleX(0);
  transform-origin: left center;  /* 从左到右划过 */
  transition: transform var(--duration-base) var(--ease-emphasized);
  transition-delay: 80ms;  /* 与 border-color 反馈错开 80ms，制造"先亮底再划线"的层次 */
}

.option-choice--correct.is-line-drawn::after {
  transform: scaleX(1);
}
```

**JS 触发（伪代码）**：

```js
function markCorrect(optionEl) {
  optionEl.classList.add('option-choice--correct');
  // 下一帧再添加 is-line-drawn，触发 scaleX 过渡
  requestAnimationFrame(() => {
    optionEl.classList.add('is-line-drawn');
  });
}
```

**优势**：
- **零 Layout**——`transform` 不影响布局，浏览器直接合成
- **GPU 加速**——`transform` 自动提升到合成层
- **从左到右方向明确**——`transform-origin: left center` 呼应"阅读方向"
- **80ms 延迟错峰**——`border-color` 先反应（80ms），细线随后（240ms），制造层次

**禁忌（再次强调）**：
- ❌ `border-bottom-width` 动画
- ❌ `width` 动画
- ❌ `left` 动画（替代方案，仍触发 Layout）

#### 6.1.2 答错动效时序

```
用户点击 [提交]
   │
   ▼  T+0ms      提交按钮下沉 1px（160ms）
   │
   ▼  T+200ms    反馈 Agent 延迟
   │
   ▼  T+200ms    用户所选选项过渡：
   │              - border-color → --warning（240ms）
   │              - background → --warning-soft（240ms）
   │              ↑ 无任何 ✗ 符号、无任何震动
   │
   ▼  T+240ms    正确选项过渡（与错误选项同步）：
   │              - border-color → --success（240ms）
   │              - background → --success-soft（240ms）
   │
   ▼  T+280ms    反馈短语（"差一点。关键在于……"）淡入（400ms）
   │              ↑ 文案中包含正向引导，非纯负面
   │
   ▼  T+440ms    解释面板展开（400ms）
   │              - 解释内容包含"为什么对 + 为什么错"（PRD FR-04）
   │

   │
   ▼  T+1080ms   全部就绪
```

**关键约束**：
- 错误选项与正确选项**同步**反馈，让用户第一时间看到"正确答案是什么"，而非"我错了"
- 全程**无任何闪烁、无任何震动、无任何震动动效**


### 6.2 选项替换动效（答错新题）

#### 6.2.1 时序

```

   │              - opacity: 1 → 0（400ms `--ease-standard`）
```

- **布局线框**（ASCII 框图，桌面端为主）
- **状态变体**（空 / 加载 / 错误 / 成功）
- **关键交互**（与第四章组件的映射）

> 所有页面共享：顶部 `.nav-top` + 暗色 `--bg-base` 背景 + 衬线字体。页面切换**不**做横向滑动，仅做内容淡入淡出（400ms）。

```
│              ──  ──  ──  ──  ──                                   │  阶梯式 · 5 阶
│              ✓   ✓   ◐   ○   ○                                    │  状态点
│              清洗  切分  概念  练习  费曼                          │  12px 等宽标签
│                                                                    │
│              约 18 秒                                              │  18px 衬线 · 次要色
│                                                                    │
│              当前阶段：从知识块提取概念...                          │  14px 衬线 · 三级色
│                                                                    │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
```

#### 5.2.3 状态变体

| 状态 | 视觉 | 时长 |
|------|------|------|
| 阶段 1：清洗（Import Agent） | 第 1 阶点亮，文字"清洗 Markdown..." | 0-15% |
| 阶段 2：切分（Chunk Agent） | 1-2 阶段亮，文字"语义切分..." | 15-40% |
| 阶段 3：概念（Concept Agent） | 1-3 阶段亮，文字"提取概念..." | 40-55% |
| 阶段 4：练习（Mission + Quiz Agent） | 1-4 阶段亮，文字"生成练习..." | 55-80% |
| 阶段 5：费曼（Feynman Agent） | 全部亮，文字"设计费曼任务..." | 80-100% |
| 完成态 | 全部点亮（鼠尾草绿），1 秒后跳转概览页 | — |
| 失败态 | 当前阶段点变橙色，下方显示错误信息 + "返回修改" 按钮 | — |

#### 5.2.4 关键交互

- **进度不可手动推进**：完全由编译 Agent 驱动
- **预计剩余时间**：基于编译进度，每 2 秒更新
- **取消**：点击"取消编译"二次确认（"已编译的内容将丢失"）→ 返回首页
- **失败重试**：失败态显示"重新尝试"按钮

#### 5.2.5 设计决策

- **用阶梯式进度**而非百分比条：呼应全局"阶梯"隐喻
- **预计时间用衬线大字**："约 18 秒"——降低等待焦虑
- **当前阶段文字**：让用户"看见"系统在做什么，避免"它卡了吗"的疑虑

#### 5.2.6 Agent 超时抚慰态（V1.1 新增）

**问题**：编译依赖 LLM，单 Agent 实际耗时受网络/模型负载影响，可能远超预估的 18 秒。若用户长时间看到"约 18 秒"却卡在 60% 不动，会判定页面卡死。

**两层抚慰阈值**：

| 触发条件 | 视觉变化 | 文案 |
|---------|---------|------|
| 当前阶段持续 **8 秒** 无进度更新 | 当前阶段点保持 active，文字行**就地替换**（淡入 240ms） | `正在深度思考复杂概念，这需要一点时间…` |
| 当前阶段持续 **15 秒** 无进度更新 | 文字行二次替换；底部新增一行小字（`--fg-tertiary`，14px） | `如果 30 秒后仍未完成，可以取消后重试` + `[取消编译]`（btn-text） |

**视觉示例**（15 秒态）：

```
              正在编译                                              36px

              ────────────
              ──  ──  ──  ──  ──
              ✓   ✓   ◐   ○   ○
              清洗  切分  概念  练习  费曼

              约 18 秒（已 32 秒）                                  18px 衬线 · --fg-secondary
                                                            ↑ 时间声明后追加实际耗时，不掩饰

              正在深度思考复杂概念，这需要一点时间…                14px 衬线 · --fg-tertiary

              如果 30 秒后仍未完成，可以取消后重试  [取消编译]      12px 等宽 · --fg-tertiary
```

**关键决策**：
- **绝不**移除"约 18 秒"原始估算——保留它并在括号内追加实际耗时，**透明优于粉饰**
- 抚慰文字用 `--fg-tertiary`（低对比），**不喧哗**
- 第二阈值（15 秒）才提供"取消"操作，给用户**控制感而非压迫感**
- 取消操作不需要二次确认（编译中产物本就不可用）

**禁止反模式**：
- ❌ 显示 spinner 加载圈（暗示卡死）
- ❌ 抚慰文字闪烁、抖动（制造焦虑）
- ❌ 自动取消（抢夺用户控制权）

#### 5.5.5 状态变体

| 状态 | 视觉 |
|------|------|
| 未答 | 所有选项默认态；提交按钮 disabled |
| 已选未提交 | 选项 `--selected` 态；提交按钮启用 |
| 提交后（答对） | 选项 `--correct`；下方展开 panel-explanation；按钮文案变"下一题" |
| 提交后（答错） | 用户选项 `--wrong`；正确选项 `--correct`；展开解释；按钮文案"再看一题" |
| 同题连续答错 3 次 | 解释下方显示"我们继续下一个吧" + 按钮"进入下一题"（强制 advance） |
| Concept 完成 | 整页淡出，Module 阶梯对应格"上升"，新 Concept 名淡入 |

#### 5.5.6 关键交互

1. **选项点击即选中**（不需要二次确认），点击"提交"才锁定
2. **键盘操作**：`1-4` 选项数字键选择，`Enter` 提交
3. **反馈延迟感**：提交后**等 200ms**（FR-03 NFR-P2 模拟）再显示反馈，避免"太快像预设"
4. **答错新题替换**：解释面板保留，旧选项淡出（400ms）→ 200ms 间隔 → 新选项淡入（400ms）

#### 5.5.7 设计决策

- **题干字号 28px > 选项字号 17px**：P4-UI-1 强制
- **Concept 名 36px 高于题干**：建立"我在学什么"的方位感
- **阶梯进度条永远在顶部**：M2 进度即方位
- **提交按钮右下角而非居中**：符合 F 型视线，降低寻找成本（P2-UI-4）

#### 5.10.5 被跳过概念的复习入口（V1.1 新增）

**问题**：PRD FR-04 允许"同 Concept 内同题连续答错 3 次强制 advance"，但极较真的用户会因"我没拿下这个概念"产生隐性挫败。完成页必须给这类用户**可见的弥补路径**，而非把跳过隐藏起来。

**视觉设计**：

仅当用户存在被强制 advance 的 Concept 时，完成页底部（按钮上方）淡入一张复习卡：

```
              ─────

  ┌─── 可以再看看（1 个概念）──────────────────────┐
  │                                                │
  │  Concept B · 什么是 Embedding                  │  ← 28px 衬线
  │                                                │
  │  你在「应用层级」连续答错 3 次后跳过。          │  ← 16px 衬线 · --fg-secondary
  │  现在可以单独回去看一下。                       │
  │                                                │
  │                          [回到 Concept B]      │  ← btn-secondary
  └────────────────────────────────────────────────┘

              ─────
```

**关键决策**：
- **标题用"可以再看看"**（中性偏温暖），不写"未掌握"、"跳过项"、"失败"
- **明确告知跳过原因**（"应用层级连续答错 3 次"），透明不掩饰
- **使用 `--bg-surface`**（与背景同色系），不抢眼，但可点击
- **点击进入 Concept B 的失败题位**（直接定位到当时卡住的 Quiz，而非整个 Concept 重做）
- 若用户有多个被跳过 Concept，则卡片列示（最多 3 个；超过 3 个折叠为"还有 N 个概念可以再看看"链接）
- 若用户**没有**任何跳过，**完全不显示**这张卡——避免给已掌握用户制造"我可能漏了什么"的焦虑

**与 §6.1.3 的呼应**：
- §6.1.3 在用户跳过当下显示温和文案"我们继续下一个吧"
- §5.10.5 在完成页给出**回看入口**——既不卡死当下，也不放弃弥补

**禁忌**：
- ❌ 用红色、感叹号、警示图标
- ❌ 用"你失败了 N 次"、"弱点概念"等负面词
- ❌ 强制用户必须回看（仅作为可选入口）

放弃 V1.0 的"移动端纯箭头"方案——4-5 个选项时把第 5 项移到第 1 项需要点击 4 次，体验成本过高。改为**双轨支持**：

| 桌面（≥ 1024px） | 鼠标拖拽（手柄 ⋮⋮） | 上下箭头（聚焦后 ↑↓） |

**长按拖拽工程要求**：
- **使用 `dnd-kit`（推荐）或 `SortableJS`**：已成熟处理移动端长按、滚动手势冲突、键盘 a11y
- **长按延迟 200ms**：避免与滚动冲突；200ms 后触觉反馈（`navigator.vibrate(10)` 轻震）+ 选项微亮（`--bg-elevated`）
- **拖拽幻影**：跟随手指的"幻影"用 `--bg-elevated` + `--shadow-md`
- **原位占位**：原位置保留半透明轮廓（`opacity: 0.3`），让用户感知"我从哪里来"

**上下箭头**作为 a11y 兜底（屏幕阅读器、键盘用户、不愿长按的用户）：

```
移动端选项布局：
┌────────────────────────────────────────┐
│ 1   用户提问                    ↑  ↓   │  ← 每项右侧固定箭头
```

- ❌ 仅提供上下箭头（违反 §7 触摸优化，4-5 项时点击成本过高）

**关键 CSS**：

```css

  color: var(--fg-primary);

  border-bottom-width: 2px;  /* 视觉聚焦反馈 */
}
```

**关键决策**：填空输入框**嵌入题干**而非独立显示——降低"我要输入"的视觉压力（P2 永远降低表达成本）。

**宽度自适应（V1.1 新增）**：

V1.0 设 `min-width: 120px` 是定宽，但中文语境下填空词通常较短（2-4 字），定宽会留下巨大空白，破坏句子连贯性。

**自适应方案**（按优先级降序）：

```css
/* 方案 1：原生 CSS · 现代浏览器（Chrome 123+ · Safari 17.4+ · Firefox 134+） */
.option-fill-blank {
  field-sizing: content;  /* 输入框随内容自动撑开 */
  min-width: 2ch;         /* 至少 2 个字符宽 */
  max-width: 320px;       /* 防止撑爆题干 */
}

/* 方案 2：JS 兜底 · 老浏览器 */
/* 用隐藏的 mirror span 测量内容宽度，反向设置 input 宽度 */
```

**JS 兜底逻辑**：

```js
function autoGrowInput(inputEl) {
  const mirror = getOrCreateMirror(inputEl);  // 隐藏 span，font 与 input 一致
  mirror.textContent = inputEl.value || inputEl.placeholder;
  const width = mirror.getBoundingClientRect().width;
  inputEl.style.width = `${Math.min(Math.max(width + 16, 32), 320)}px`;
}
```

**关键约束**：
- **最小宽度 2ch**（约 32px）——保证点击热区
- **最大宽度 320px**——防止超长答案撑爆题干
- **过渡 120ms `--ease-standard`**——宽度变化时丝滑
- **placeholder 与实际值都参与测量**
- **空内容时回退到 `min-width`**（2ch）+ 占位符宽度

**浏览器兼容矩阵**：

| 浏览器 | 方案 |
|-------|------|
| Chrome 123+, Edge 123+, Safari 17.4+, Firefox 134+ | 原生 `field-sizing: content` |
| 其他 | JS mirror span 兜底 |

通过 `@supports` 优雅降级：

```css
@supports (field-sizing: content) {
  /* 原生方案，无需 JS */
  .option-fill-blank { field-sizing: content; }
}

@supports not (field-sizing: content) {
  /* JS 兜底，class hook */
  .option-fill-blank { /* min-width 保留 */ }
}
```

#### 4.4.2 字数计数器的渐进式提示（V1.1 新增）

**问题**：V1.0 的字数计数器仅有"输入中（达标 / 未达标 / 超限）"三态，颜色从 `--fg-tertiary`（灰）直接跳到 `--warning`（橙），用户在接近阈值时缺乏**温和的方位感**。

**渐进式四态**（适用于 textarea-feynman 与 textarea-import）：

| 输入进度 | 计数器颜色 | 视觉示例（Feynman，下限 100 / 上限 500） | 心理意图 |
|---------|-----------|-------------------------------------------|---------|
| **极弱**（< 80% 下限） | `--fg-tertiary` (#7A7466) | `47 / 100-500` | 用户刚开始写，不打扰 |
| **预提示**（80-100% 下限） | **`--accent-primary`** (#D4A574) | `85 / 100-500` | 接近下限，可提交——给用户"快到了"的方位感 |
| **达标**（100-90% 上限） | `--fg-tertiary` (#7A7466) | `234 / 100-500` | 已达标、未超限，回到极弱 |
| **超限预警**（≥ 90% 上限） | `--warning` (#C99A6E) | `467 / 100-500` | 接近超限，温和提醒 |

**关键设计**：
- **80% 是预提示阈值**——比"达标后才反馈"更早，但比"超出后才警告"更柔
- **颜色过渡 240ms `--ease-standard`**——避免突变
- **绝不闪烁、绝不放大、绝不变粗**——只是颜色变化
- **不显示"还差 N 字"**——避免制造追赶感
- **不显示进度条**——字数计数器就是计数器，不是进度条

**CSS 实现**：

```css
.textarea-counter {
  /* 默认极弱 */
  color: var(--fg-tertiary);
  transition: color var(--duration-base) var(--ease-standard);
}

.textarea-counter--approaching-min,  /* 80-100% 下限 */
.textarea-counter--approaching-max { /* ≥ 90% 上限 */
  /* 由 JS 切换 class */
}

.textarea-counter--approaching-min {
  color: var(--accent-primary);  /* 金棕 · 方位感 */
}

.textarea-counter--approaching-max {
  color: var(--warning);          /* 陶土橙 · 温和预警 */
}
```

**JS 逻辑（伪代码）**：

```js
function updateCounterClass(counterEl, length, min, max) {
  counterEl.classList.remove(
    'textarea-counter--approaching-min',
    'textarea-counter--approaching-max'
  );
  if (length >= max * 0.9) {
    counterEl.classList.add('textarea-counter--approaching-max');
  } else if (length >= min * 0.8 && length < min) {
    counterEl.classList.add('textarea-counter--approaching-min');
  }
  // 其他情况默认状态，无 class
}
```

**禁忌**：
- ❌ 颜色直接从灰跳到橙（无过渡）
- ❌ 显示"还差 N 字 / 已超 N 字"
- ❌ 接近阈值时计数器字号变大、加粗、闪烁


**关键决策**：
- 解释用**左侧 2px 强调线**（金棕色），不用整框边框——保持视觉轻量
- 文字颜色 `--fg-secondary`（非 primary），弱化"评判"感

**展开动效实现（V1.1 修订 · 重要）**：

V1.0 使用 `max-height: 0 → 600px` 的方案有两个问题：
1. **触发 Layout**（重排），性能不如合成层动画
2. 实际内容若远小于 600px，动画速率感不均匀（240ms 走完 600px ≠ 240ms 走完实际 200px）

**改为 `grid-template-rows: 0fr → 1fr`** 方案，丝滑且性能更优：

```css
/* 外层 wrapper 控制 grid 行高 */
.panel-explanation-wrapper {
  display: grid;
  grid-template-rows: 0fr;          /* 收起：行高 0 */
  transition: grid-template-rows var(--duration-slow) var(--ease-standard);
}

.panel-explanation-wrapper--open {
  grid-template-rows: 1fr;          /* 展开：行高自动 */
}

/* 内层 overflow: hidden 是关键 */
.panel-explanation {
  overflow: hidden;
  min-height: 0;                    /* grid item 必须，否则 1fr 不生效 */
  
  /* opacity 同步过渡 */
  opacity: 0;
  transition: opacity var(--duration-slow) var(--ease-standard);
}

.panel-explanation-wrapper--open .panel-explanation {
  opacity: 1;
}
```

**HTML 结构**：

```html
<div class="panel-explanation-wrapper">
  <div class="panel-explanation">
    <p>Few-shot 的核心是提供示例...</p>
  </div>
</div>
```

**优势**：
- **真实高度展开**——无论内容 100px 还是 500px，动画速率均匀
- **GPU 加速**——浏览器优化 grid-template-rows 过渡
- **无需 JS 计算实际高度**

**浏览器兼容**：
- Chrome 117+, Edge 117+, Safari 17.2+, Firefox 122+ 支持 grid 过渡
- 不支持的浏览器会瞬间展开（无动画但仍可用），可接受降级

**降级方案**（@supports 检测）：

```css
@supports not (transition: grid-template-rows 0.3s) {
  /* 老浏览器回退到 opacity 单独过渡 */
  .panel-explanation-wrapper {
    display: block;
  }
  .panel-explanation-wrapper:not(.panel-explanation-wrapper--open) .panel-explanation {
    display: none;
  }
}
```

## 第三章 核心视觉隐喻：阶梯

> 阶梯是本产品**唯一**贯穿全局的视觉隐喻。它的存在不是为了装饰，而是为了让用户在视觉上**感知到表达自由度的连续上升**——这是规格书 [§2.5 Expression Freedom Curve](../Product-Specification.md#25-expression-freedom-curve表达自由度曲线)的视觉化载体。

### 3.1 为什么是"阶梯"

候选隐喻对比：

| 隐喻 | 优势 | 劣势 | 采纳 |
|------|------|------|------|
| **阶梯 / 台阶** | 直接对应 Quiz Ladder 与表达自由度曲线；克制不喧哗；天然承载"上升"语义；视觉重量轻 | 在移动端横向布局略受限（需做响应式变体） | **采纳** |
| 路径 / 旅程 | 叙事感强 | 横向占用空间大；移动端难承载；与"学术沉静"基调偏离（偏游戏化） | 不采纳 |
| 编译管线 | 呼应 Knowledge Compiler 名 | 仅适合"编译"阶段展示，不适合贯穿全程；技术隐喻对学习者不友好 | 不采纳 |
| 脚手架 | 对应 Vygotsky ZPD 理论 | 视觉表达抽象，用户难以直观理解；脚手架"撤除"语义负面 | 不采纳 |
| 进度百分比条 | 通用 | 无层级感、无方位感；违反 P5-UI-1（不显示连续无断层的层级） | 不采纳 |

**结论**：阶梯是唯一同时满足"视觉克制 + 表达自由度上升语义 + 全程贯穿"的隐喻。

### 3.2 阶梯的三层结构

本产品的学习路径在数据上有三层粒度，对应阶梯的三种视觉变体：

```
Module
  ├── Concept A      ←── 小阶梯（一阶）
  │     └── Quiz ×N  ←── 单格（Quiz 颗粒度，可选展示）
  ├── Concept B      ←── 小阶梯（一阶）
  ├── Concept C      ←── 小阶梯（一阶）
  ├── Challenge      ←── 中阶梯（一阶，色加深）
  └── Feynman        ←── 顶点（最高阶，色提亮）
```

视觉上，**Module 阶梯** = 全局进度条，是用户最常看到的元素；**Concept 内 Quiz 进度**仅在题干上方做辅助指示，**不**渲染为完整阶梯。

#### 3.2.1 Module 阶梯（全局进度条 · 主视图）

**视觉描述**：

```
桌面端（默认）：
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   ▁▁  ▁▁  ▂▂  ▃▃  ▄▄▄▄  ▆▆▆▆  ▇▇▇▇  ████  ████                  │
│   C1   C1   C2   C2   C3    C3    Chl   F1-5   F6                │
│  └──── Concept ────┘└── Concept ──┘└─Challenge┘└── Feynman ──┘    │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘

简化呈现：水平排布的小方格，按"上升"语义逐渐增高
- Concept A 两格（Recognition + Application 层级）
- Concept B 两格
- Concept C 两格
- Challenge 两格（高度更高一阶，色更深）
- Feynman 两格（顶点，色更亮）
```

**关键视觉决策**：
- 每个 Concept 占据**等宽的两格**（不展示 Quiz 数 N，避免"题海感"）
- Challenge 与 Feynman 的格子高度**略高于** Concept（视觉上"更上一阶"）
- 格子高度差异**克制**（最高/最低比例 ≤ 1.6），不喧哗

#### 3.2.2 Concept 内 Quiz 进度（辅助指示 · 次视图）

Concept 内的 Quiz 阶梯**不**渲染为完整阶梯，仅做：
- **当前位置指示**：`Concept A · 第 3 步 / 共 8 步` （小字号、等宽字体）
- **细线进度条**：1px 高度的水平细线，宽度按当前 Quiz 索引填充

```
Concept A · 第 3 步 / 共 8 步
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ (1px 灰色底线)
█████████████                       (1px 金棕色填充，约 37.5%)
```

**理由**：用户对 Concept 内的颗粒度感知是"我还有多远"，而非"我要爬多少阶"——细线比阶梯更适合这种感知。

#### 3.2.3 Feynman 顶点指示（仪式性 · 仅 Feynman 阶段）

进入 Feynman 阶段时，Module 阶梯的 Feynman 部分高亮：

```
┌──────────────────────────────────────────────────────────────────┐
│                                                                  │
│   ▁▁  ▂▂  ▃▃  ▄▄  ▆▆  ▇▇  ████  ████ ← Feynman 顶点，金色高光    │
│   └已完成─┘└已完成┘└─Challenge─┘└── 你在这里 ──┘                 │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

加上文字标签"你已到达 Module 顶点"（小字号、低对比），强化 P1-UI-3 的"费曼仪式感"。

### 3.3 阶梯的状态变体

每个阶梯格子有 4 种状态：

| 状态 | 视觉 | 颜色 | 动效 |
|------|------|------|------|
| **已完成（completed）** | 实心格 + 鼠尾草绿圆点 | `--state-completed` 底 + `--success` 边框 | 完成瞬间 240ms 渐入 |
| **进行中（active）** | 半填充 + 金棕色 1px 细线 | `--state-active` 底（半透明） | 每答一题，填充比例更新（240ms 过渡） |
| **待解锁（locked）** | 仅边框，无填充 | `--state-locked` 边框 | 无 |
| **失败但跳过（skipped）** | 实心格 + 极淡橙点（不显眼） | `--warning-soft` 底 + `--fg-tertiary` 边框 | 完成瞬间 240ms 渐入（与 completed 同） |

**关键设计**：
- "失败但跳过"（FR-04 中"连续答错 3 次强制 advance"）与"已完成"在阶梯上**几乎无差异**——避免用户产生"我留下了瑕疵"的负面感（M3 错误即学习）

### 3.4 阶梯在不同页面的呈现

| 页面 | 阶梯呈现 | 说明 |
|------|---------|------|
| 首页 / 编译中页 / 课程概览页 / Module 导言页 | **不显示** | 进入学习前，阶梯不应过早暴露 |
| 学习页（Concept 阶段） | 顶部完整 Module 阶梯 + Concept 内细线进度 | 主视图 |
| 反馈态 | 阶梯不动，仅 Concept 细线进度更新 | 反馈不破坏方位感 |
| Challenge 页 | 阶梯 Challenge 格高亮，文字标签切换为"综合挑战" | 强化阶段感 |
| Feynman Step 1-5 | 阶梯 Feynman 部分高亮 + 步骤指示器（1/6 ... 6/6） | 双重方位感 |
| Feynman Step 6 | 阶梯 Feynman 顶点提亮 + 仪式光晕（仅 0.06 opacity） | 强化顶点感 |
| 完成页 | 阶梯**全部点亮** + 慢动作回放（1200ms 依次填充） | "你已攀登完毕"叙事 |

### 3.5 阶梯的响应式变体

| 断点 | 阶梯呈现 |
|------|---------|
| 桌面（≥ 1024px） | 水平完整阶梯（10 格左右） |
| 平板（768-1023px） | 水平压缩阶梯（格子宽度减小，高度不变） |
| 手机（< 768px） | **简化为 3 段**：`Concept ─ Challenge ─ Feynman`，下方文字标注当前阶段 |

详见 [§7 响应式适配](#第七章-响应式适配)。

#### 3.6.3 阶梯动效

- **格子状态切换**：`240ms` 渐变（颜色 + 背景）
- **格子完成时的"上升感"**：完成后 `transform: translateY(-2px)` 一次（240ms），暗示"登顶"
- **完成页全阶梯点亮**：从左到右，每格延迟 `120ms` 依次点亮（共约 1200ms）

### 3.7 阶梯的禁忌

| 反模式 | 理由 |
|--------|------|
| 用百分比文字代替阶梯 | 违反 M2 进度即方位（百分比无层级） |
| 用连续曲线代替阶梯 | 违反 P5-UI-1（必须显示层级，不能模糊） |
| 阶梯格子里显示 Quiz 数 N | 制造题海感，违反 P1-UI-1 |
| 阶梯上叠加 emoji（✓、★） | 违反 M1 静默优先 + §2.6.1 禁用 emoji |
| 阶梯格子上方/下方加阴影 | 破坏沉静感 |
| 阶梯做粒子/光晕动画 | 仅完成页允许 0.06 opacity 的微光晕 |

### 2.1 色彩系统

#### 2.1.1 设计意图

色彩服务于三个目标，按优先级排序：
1. **降低表达焦虑**（P3-UI-1/2）—— 错误反馈必须柔和，绝不刺眼
2. **构建学术沉静感**（M1 静默优先）—— 80% 屏幕为中性色，20% 为强调色
3. **方位感**（M2 进度即方位）—— 不同阶段（Concept / Challenge / Feynman）通过细微色温变化区分

色彩**不服务于**：吸引眼球、制造兴奋、奖励刺激。

### 2.2 字体系统

#### 2.2.1 设计意图

字体服务于三个目标：
1. **学术沉静感** —— 衬线字体的"墨迹感"和"书卷气"
2. **降低阅读疲劳** —— 长时间学习，衬线字体重音节清晰
3. **中英混排和谐** —— 中文宋体的横竖笔画与英文衬线的笔画粗细对比呼应

**禁止使用**：Inter / Roboto / Arial / system-ui（学术沉静感缺失）；紫色渐变文字（违反 M1 静默优先）。

#### 2.2.2 字体家族

```css
/* 主字体 · 衬线 · 标题与正文 */
--font-serif: "Fraunces", "Source Han Serif SC", "Noto Serif SC",
              "LXGW WenKai", Georgia, serif;

/* 等宽字体 · 标签、计数、代码、字数 */
--font-mono: "JetBrains Mono", "IBM Plex Mono", "Source Han Mono SC",
             "Cascadia Mono", Consolas, monospace;

/* 旧版数字风格（仅费曼完成页使用，营造仪式感） */
--font-display: "Fraunces", "Source Han Serif SC", serif;
```

##### 为什么是 Fraunces

- **可变字重（Variable Font）**：100-900 字重任意调节，无需加载多份字体文件
- **可变光学尺寸（opsz）**：大字号自动收紧字距、小字号自动展开，天然适配从题干到标签的多场景
- **温暖衬线感**：与冷的 Times New Roman 不同，Fraunces 的弧度更柔和，匹配"学术灯下的羊皮纸"基调
- **古典而又现代**：常用于 Stripe Press、Lemonada 等高设计感出版物

##### 为什么是思源宋体（备选：霞鹜文楷）

- **开源、可商用**
- **横竖笔画对比呼应 Fraunces**（思源宋体的横竖比与 Fraunces 的衬线粗细比相近，混排不违和）
- **LXGW WenKai 作为备选**：手写楷体感更柔和，适合更"亲切"场景；但 MVP 默认用思源宋体保证可读性

#### 2.2.3 字号阶梯（Type Scale）

采用 **1.250 大三度音阶（Major Third）**——比 1.200 小三度更克制，但比 1.333 完美四度更有层次。

```css
/* 字号阶梯 · 桌面端基准 */
--text-xs:    12px;   /* 极小标签 · 字数计数器角落 */
--text-sm:    14px;   /* 小标签 · 元信息、阶段标签 */
--text-base:  16px;   /* 选项、辅助说明 */

--text-lg:    22px;   /* 正文重点、Module 导言 */
--text-xl:    28px;   /* 题干（核心字号） */
--text-2xl:   36px;   /* 二级标题、Concept 名 */
--text-3xl:   48px;   /* 一级标题、首页标语 */
--text-4xl:   64px;   /* 完成页主标语、费曼顶点 */

/* 字重 */
--weight-regular:   400;
--weight-medium:    500;
--weight-semibold:  600;
--weight-bold:      700;

/* 行高 */
--leading-tight:    1.15;   /* 大标题 */
--leading-snug:     1.30;   /* 小标题 */
```

#### 2.2.5 中英混排规则

中文与英文/数字之间**自动添加 1/4 全角空格**（约 4px）。CSS 实现：

```css
/* 中英文自动间距（中文之间、中文与英文/数字之间） */
.mix-cn-en {
  font-feature-settings: "halt";
  /* 或使用 padding 微调 */
}
```

字体加载顺序：英文 Fraunces 优先（保证英文衬线感），中文回退思源宋体（保证中文笔画清晰）。

**中英混排的自动间距工程化（V1.1 修订）**：

V1.0 仅描述了"自动添加 1/4 全角空格"的需求，但未给出工程方案。中英文混排时若不自动添加间距，"RAG（Retrieval-Augmented Generation）通过检索外部文档"这类句子会显得拥挤。三层方案，按优先级降序：

##### 方案 1：CSS `text-spacing-trim`（W3C 草案，推荐）

```css
:root {
  /* 自动处理 CJK 与拉丁字符间距 · 自动去除 CJK 标点前后的多余空白 */
  text-spacing-trim: space-all;     /* 或 trim-adjacent + space-start + space-end 组合 */
}

/* 浏览器兼容（截至 2026-07）：
   Chrome 123+ 已支持（behind flag 默认开启）
   Safari TP 已支持
   Firefox 仍在实现中
*/
```

**优势**：纯 CSS、零 JS 开销、对 LLM 动态输出友好。

##### 方案 2：`pangu.js`（兜底 · 兼容老浏览器）

```html
<script src="https://cdn.jsdelivr.net/npm/pangu@4.0.7/dist/browser/pangu.min.js" defer></script>
<script defer>
  document.addEventListener('DOMContentLoaded', () => {
    pangu.spacingElement('body');  // 对整页文本自动加间距
  });
</script>
```

或在 React 渲染时统一处理：

```js
import pangu from 'pangu';

// 在 Quiz stem / option / explanation 等字段渲染前预处理
function preprocessText(text) {
  return pangu.spacing(text);
}
```

**优势**：兼容所有浏览器；**劣势**：JS 运行时开销，需在 hydration 后再处理一次（动态文本）。

##### 方案 3：Prompt 工程兜底

在 Quiz Agent / Feynman Agent 的 Prompt 中明确：

```
约束：输出题干、选项、解释时，中文与英文/数字之间自动加一个半角空格。
正确：Few-shot 是一种 Prompt 技术，需要 3-5 个示例。
错误：Few-shot是一种Prompt技术，需要3-5个示例。
```

让 LLM 输出时已正确混排，前端无需后处理。

##### 推荐组合

**MVP 阶段**：方案 1（`text-spacing-trim`）+ 方案 3（Prompt 约束）双保险。
- 方案 1 处理 LLM 偶发的混排瑕疵
- 方案 3 让大部分输出在源头就符合规范
- 不引入 pangu.js（减少 JS bundle 体积）

**老浏览器降级**：若用户在不支持 `text-spacing-trim` 的浏览器上，体验略有损失（中英字符紧贴），但功能不受影响。

##### 验证清单

- [ ] 题干含 `RAG（Retrieval-Augmented Generation）` 显示时中英自动间距
- [ ] 选项含数字（如 `3-5 个示例`）显示时间距正确
- [ ] Feynman 用户输入混排时，**用户输入不强制处理**（尊重用户书写习惯），但 AI 范文处理
- [ ] 解释文案含中文标点（，。；：）时，标点后无多余空白

### 2.4 圆角、阴影、边框

#### 2.4.1 圆角

```css
--radius-none:   0;
--radius-sm:     4px;     /* 输入框、小标签 */
```

**使用规则**：
- 选项卡用 `--radius-md` (8px)
- 卡片用 `--radius-lg` (12px)
- 圆角**不**叠加阴影外发光（违反 M1 静默优先）

#### 2.4.2 阴影

学术沉静基调下，阴影使用极克制——绝大多数场景**用边框替代阴影**构建空间感。

```css
/* 阴影仅在"浮起交互"（hover、modal）时使用 */
--shadow-sm:   0 1px 2px rgba(0, 0, 0, 0.3);
--shadow-md:   0 4px 12px rgba(0, 0, 0, 0.4);
--shadow-lg:   0 12px 32px rgba(0, 0, 0, 0.5);

/* 阶梯进度的内阴影（模拟雕刻感） */
--shadow-inset-stair: inset 0 1px 0 rgba(236, 230, 212, 0.06);
```

**禁止**：
- 大范围弥散阴影（`blur > 32px`）—— 破坏沉静感
- 彩色阴影（如 `0 0 20px rgba(212, 165, 116, 0.5)`）—— 违反 M1 静默优先

#### 2.4.3 边框

边框是构建空间感的主要手段：

```css
/* 默认边框 · 1px 实线 · 12% 透明度 */

/* 强调边框 · 1px 实线 · 22% 透明度 · 用于活动状态 */
--border-strong: 1px solid rgba(236, 230, 212, 0.22);

/* 装饰边框 · 1px 虚线 · 用于"待解锁"区域 */
--border-dashed: 1px dashed rgba(236, 230, 212, 0.16);
```

### 2.5 动效系统

#### 2.5.1 设计意图

动效服务于：
1. **反馈**（240ms）—— 答题后的即时视觉反馈
2. **过渡**（400ms）—— 阶段切换、选项替换
3. **仪式**（600ms-1200ms）—— 费曼高潮、完成页

动效**绝不服务于**：吸引眼球、制造惊喜、奖励刺激。

#### 2.5.2 时长阶梯

```css
--duration-instant:  80ms;     /* 即时反馈 · hover 颜色变化 */
--duration-fast:     160ms;    /* 快速过渡 · 按钮按下 */
--duration-base:     240ms;    /* 标准反馈 · 答题反馈 */
--duration-slow:     400ms;    /* 内容过渡 · 选项替换、阶段切换 */
--duration-ceremony: 800ms;    /* 仪式动效 · 费曼顶点、完成页主标题 */
--duration-epic:     1200ms;   /* 史诗动效 · 完成页掌握度环满格 */
```

#### 2.5.3 缓动曲线

```css
/* 标准缓动 · 柔和加速减速 */
--ease-standard: cubic-bezier(0.2, 0.8, 0.2, 1);

/* 强调缓动 · 慢入快出（用于选项高亮） */
--ease-emphasized: cubic-bezier(0.16, 1, 0.3, 1);

/* 仪式缓动 · 极慢入（用于费曼顶点） */
--ease-ceremony: cubic-bezier(0.7, 0, 0.3, 1);

/* 标准缓出 · 用于退出/隐藏 */
--ease-out: cubic-bezier(0.0, 0.0, 0.2, 1);

/* 标准缓入 · 用于进入 */
--ease-in: cubic-bezier(0.4, 0.0, 1, 1);
```

**禁止使用** `linear`（机械感）和 `ease-in-out`（俗套）。

#### 2.5.4 动效规则（强制）

| 场景 | 时长 | 缓动 | 备注 |
|------|------|------|------|
| 答对选项底色渐入 | 240ms | `--ease-standard` | 仅渐入，无缩放 |
| 答对细线划过 | 240ms | `--ease-emphasized` | 1px 实线从左到右 |
| 答错选项底色渐入 | 240ms | `--ease-standard` | 与答对相同时长，颜色不同 |
| 解释文案展开 | 400ms | `--ease-standard` | 高度从 0 到 auto |
| 答错后新题替换旧题 | 400ms 淡出 + 200ms 间隔 + 400ms 淡入 | `--ease-standard` | **不允许滑入/滑出** |
| Concept → Concept 阶梯上升 | 600ms | `--ease-emphasized` | 进度条阶梯格填充 |
| 进入 Challenge 阶段 | 800ms | `--ease-ceremony` | 背景色细微变化（金棕加深） |
| 进入 Feynman 阶段 | 800ms | `--ease-ceremony` | 阶梯进度顶点亮起 |
| Feynman Step 6 提交后 Rubric 展开 | 600ms | `--ease-standard` | 每条 rubric 评分卡 200ms 错峰展开 |
| 完成页掌握度环填充 | 1200ms | `--ease-ceremony` | 满格时刻 0.6s 延迟后微小光晕（不撒花） |
| 完成页主标题入场 | 800ms | `--ease-ceremony` | 字符级 stagger（仅大标题） |

#### 2.5.5 动效禁用（prefers-reduced-motion）

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
  }
}
```

详见 [§8 可访问性](#第八章-可访问性)。

### 2.6 图标与装饰

#### 2.6.1 图标原则

**极少使用图标**——本产品的视觉语言以排版为主，图标仅在以下场景使用：

| 场景 | 图标 | 风格 |
|------|------|------|
| 排序题拖拽手柄 | ⋮⋮ 或 ⠿ | 1.5px 线条，16px |
| 上下箭头（移动端排序） | ↑ ↓ | 1.5px 线条，20px |
| 上传文件 | ⬆ | 1.5px 线条，20px |
| 关闭/返回 | ← | 1.5px 线条，20px |
| 已完成 Concept 标记 | ●（实心圆点） | 8px 直径，无图标 |

**禁止使用**：
- 任何 emoji（包括 ✅ ❌ ⚠️ 🎉 📚）
- 任何填充式（filled）图标（破坏衬线美学）
- 任何彩色图标（除默认 `--fg-secondary`）

#### 2.6.2 装饰元素

| 元素 | 用途 | 实现 |
|------|------|------|
| 1px 细线（金棕色） | 答对反馈的视觉强调 | CSS border-bottom 从左到右动画 |
| 阶梯进度格 | 全局进度 | CSS grid + 渐变填充（详见第三章） |
| 1/3 屏渐变光晕 | 完成页背景仪式感 | radial-gradient，opacity 0.06 |
| 网格底纹 | 首页背景 | SVG grid pattern，opacity 0.02 |

**禁止装饰**：
- 粒子动画、流光、波纹
- 任何 hover 时的缩放/弹跳
- 任何"奖杯"、"勋章"、"星星"图形

### 2.7 视觉语言系统总结

```
色彩：80% 中性 + 20% 强调；错误柔和（橙）、成功柔和（绿）、仪式金棕
字体：Fraunces × 思源宋体（衬线学术）；JetBrains Mono（等宽标签）
间距：8px 基线；题干与选项间留 48px；选项全宽，热区 ≥ 48px
圆角：选项 8px、卡片 12px、输入框 16px；圆角不配外发光
阴影：极克制；用边框替代阴影；仅在 hover/modal 用
动效：80ms 即时 / 240ms 反馈 / 400ms 过渡 / 800ms 仪式；cubic-bezier 柔和曲线
装饰：极简；细线、阶梯、网格底纹；禁 emoji / 粒子 / 奖杯
```
