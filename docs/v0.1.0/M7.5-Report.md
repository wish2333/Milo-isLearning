# M7.5 Report — 从 Product Specification 到 M7 的专项审查

> 状态：Draft | 日期：2026-07-08  
> 审查范围：`Product-Specification.md` / PRD / Technical Spec / Prompt Engineering / UI Design / M1-M7 文档 / 当前前端与编译管线实现  
> 重点问题：题目生成质量与 Agent 工程、题目保存与导出、作答回退、UI Design 落地、刷新后的无损恢复

---

## 0. 结论

M1-M7 已经把 MVP 的核心链路做到了“能编译、能学习、能完成”：Markdown 输入、LLM 编译、Quiz / Challenge / Feynman、LocalStorage 进度、完成页与本地判分都已具备。M7 的实际状态不是“内测完成”，而是“进入内测前的工程收口完成”：修复了编译页 Strict Mode abort、settings 水合、schema 过紧、运行时判分成本等关键问题。

但从产品规格的“编译训练系统”目标看，当前还缺一层 M7.5 必须补齐的基础设施：**编译产物的可管理性、可恢复性、可复用性与 UI 一致性**。如果不补，M8 公测会暴露 5 类高概率问题：

1. 用户生成一次题目后无法明确保存、命名、恢复、导出或复用，LLM 成本不可控。
2. 编译中刷新仍会丢失正在进行的编译状态，不能“无损继续”。
3. 学习中只能前进或重试，不能回退到上一题 / 上一阶段。
4. 当前 React UI 基本没有落地 `docs/ui-design` 的学术沉静风格与阶梯隐喻。
5. 题目生成 pipeline 已可用，但质量闭环还偏弱：schema 放宽后成功率提高，质量门禁、版本记录、产物审查与缓存策略还没跟上。

建议把 M7.5 定位为：**Compiled Module Workspace + Recovery + UI Alignment**。它不是新增学习功能，而是让已生成的学习路径成为可保存、可复用、可继续、可审查的资产。

## 1. Findings / 主要问题

### P0 — 生成产物没有成为“可管理资产”

当前编译完成后会写入 `alc:module:{moduleId}` 和 `alc:source:{sourceId}`，完成页也能列出历史 Module 标题。但它不是用户可操作的题库 / 课程资产：

- 历史 Module 只能看标题，不能打开继续、重新学习、导出、复制 JSON、重命名或删除单项。
- 编译产物没有显式“保存成功”的用户承诺，也没有“从历史恢复”的稳定入口。
- 没有导出格式，无法把生成题目保存到文件、提交到版本库、人工编辑或二次导入。
- `ensureCapacity()` 可能淘汰历史 Module，但 UI 不提示用户哪些内容被淘汰。
- `removeModule()` 使用 `StorageKeys.source(moduleId)` 删除 source，而保存 source 时用的是 `StorageKeys.source(compiledModule.sourceId)`；这会导致淘汰 Module 时原文 source 残留。

影响：用户会重复编译同一份材料，浪费 LLM 费用；内测中也难以复现“某次生成质量不好”的样本。

### P0 — 编译中刷新不能无损继续

`compile-store` 明确是不持久化的临时状态；`/learn/compiling` 的源文本来自 `sessionStorage`，SSE 流来自当前浏览器页面。刷新后：

- 前端 SSE 连接断开。
- 后端 `/api/compile` 无服务端任务 ID、无状态快照、无 resume endpoint。
- 已完成的 Agent 中间产物没有保存。
- 如果旧请求仍在服务端继续消耗 LLM，前端也无法重新绑定它。

M7-Plan W6 把“编译中页刷新后状态丢失”列为预期行为，但用户提出的目标是“无损继续并恢复到对应前端界面”。两者冲突，需要在 M7.5 改设计。

### P1 — 学习作答没有回退机制

`progress-store` 只有 `advance()` / `retry()` / `setStage()`，没有正式的 `goBack()`、`jumpToSlot()` 或状态历史栈。完成页的“重新练习”能跳到某个 Concept 第一题，但作答中没有：

- 上一题
- 上一 Concept
- 回到概览
- 重新查看已答题解释
- 撤销一次错误进入下一题

这会削弱低摩擦体验。尤其是用户误点“继续”或想回看 explanation 时，会被迫前进。

### P1 — React 前端未应用 ui-design 的核心风格

`docs/ui-design/DESIGN-SPEC.md` 定义了明确的视觉系统：深墨绿黑 `#0E1311`、暖象牙文字 `#ECE6D4`、金棕强调 `#D4A574`、衬线字体 Fraunces / 思源宋体、阶梯进度、柔和陶土橙错误反馈、鼠尾草绿成功反馈。

当前实现主要是 Tailwind `neutral-*` 灰黑系统：

- `globals.css` 只有 Tailwind 三行，没有设计 token。
- `layout.tsx` 未加载字体或设置 body 字体。
- 学习页进度是文字 “概念 X/Y · 题目 N”，不是阶梯方位。
- 题干字号偏小，未达到设计稿“题干显著高于选项”的要求。
- 完成页使用 `✓` 符号，设计稿明确禁止 emoji / 奖杯 / 强庆祝符号，也强调完成页应展示表达自由度成长曲线。
- Challenge 用 amber 区分，但不是设计 token 中的阶段色系统。

影响：产品哲学没有通过 UI 传达出来，内测用户很可能把它感知为普通 Quiz App，而非“表达自由度逐渐上升”的训练系统。

### P1 — 题目生成质量从“能过 schema”退到了“需要后置质量门禁”

M7 放宽了多个 schema，这是正确方向：Zod 不应把教学建议当作硬约束。但放宽后需要补上质量门禁，否则会出现“合法但教学质量差”的题目：

- Mission 的 expressionLevel 单调性已从硬校验移除，但没有后置质量报告说明曲线是否符合预期。
- Quiz `distractors` min 从 3 降到 1，缺少 distractor 质量评分。
- `options[0] !== answer` 改为 mapper 自动修复是对的，但没有记录修复次数，无法观察模型偏差。
- Challenge schema 保留“涉及 ≥ 2 Concept”，但 mapper 丢弃了 `involvedConceptIds`，运行时和报告页无法解释一道 Challenge 连接了哪些概念。
- `Feedback Agent` 降级后，运行时成本下降；但开放输入、Feynman 评分和生成后审查仍需要 LLM 质量策略。

建议：M7.5 不要重新收紧 schema，而是新增“编译产物质量报告”和“可导出样本”，让人和后续自动评估能看见质量。

## 2. 重点问题审查

### 2.1 生成题目的过程还有哪些优化空间

#### LLM 请求模块

当前优点：

- Provider 抽象已经成型，支持 DeepSeek / GLM / SenseNova。
- `runAgent()` 有 JSON parse、Zod 校验、重试提示。
- Pipeline stage 有日志，M7 增强后更容易定位卡点。
- 运行时 Choice / Sorting / Fill Blank 已改成本地判分，显著降低费用与不确定性。

可优化点：

1. **引入 compile job id**
   每次编译创建 `compileJobId`，所有 stage 事件、产物、错误、LLM usage、schema 修复记录都挂到该 id 下。刷新页面后以 job id 恢复。

2. **记录 usage 与成本估算**
   Provider 返回的 `usage` 目前没有进入用户可见或调试报告。建议在 `CompileEvent` 或 `CompileJobMeta` 中累计 prompt/completion tokens、模型、耗时、重试次数，完成页或 Module 管理页显示“本次生成成本估算”。

3. **保留 Agent 中间产物**
   Import / Chunk / Concept / Module / Mission / Quiz / Challenge / Feynman 的输出应作为 debug artifact 保存。至少开发模式和内测模式保存，方便定位“为什么这道题怪”。

4. **区分结构重试与质量重试**
   现在重试主要针对 empty / invalid JSON / schema violation。建议新增生成后质量检查：如题干自包含、正确答案是否唯一、干扰项是否相似但错误、Challenge 是否跨概念、Feynman rubric 是否覆盖核心概念。质量不达标时只重试对应 quiz batch，而不是全流程。

5. **Quiz batch 并发策略需显式化**
   目前按 concept 分批串行生成，每 concept 一次 batch。这降低请求数，但长 concept 会拖慢。建议配置化：`quizBatchConcurrency=1..3`，遇 429 自动降级。

#### Prompt 设计

当前 Prompt 文档非常完整，但实现阶段需要从“规则很多”转成“可评估闭环”：

- Quiz Agent 可继续采用 overgenerate-and-rank 思路，但 M7.5 可先做轻量版：让模型输出 `misconception` 字段，再映射到 distractor，便于审查。
- Challenge Agent 应保留 `involvedConceptIds` 到 domain，方便 UI 和质量报告展示“这题连接了 A+B”。
- Feynman Agent 的 Step 6 作为占位存在，容易造成 domain 注释和实际渲染心智负担。建议 V1.1 把 `steps` 改成 5 个脚手架步骤，`finalPrompt` 单独表达开放输出；M7.5 先在报告中标注兼容状态即可。

#### 数据验证

建议分三层：

1. **Schema 层：只保证可运行**
   字段存在、类型正确、数组长度不崩 UI、answer 存在于 options。

2. **Mapper 层：可无损修复**
   answer 移到 options[0]、缺省 explanation 兜底、排序题 answer 归一化。

3. **Quality 层：生成报告，不一定阻断**
   记录 expression curve、distractor 数量、题干长度、跨概念覆盖、重复题比例、mapper 修复次数。内测阶段先作为报告，达到阈值后再阻断。

#### Agent 工程

建议新增 `CompileArtifact`：

```ts
interface CompileArtifact {
  jobId: string
  moduleId?: string
  sourceId: string
  status: 'running' | 'complete' | 'error' | 'aborted'
  stage: CompileStage | null
  percent: number
  createdAt: number
  updatedAt: number
  provider: string
  model: string
  usage?: { promptTokens: number; completionTokens: number }
  stageOutputs: Partial<Record<CompileStage, unknown>>
  qualityReport?: CompileQualityReport
  error?: unknown
}
```

MVP 可先存 LocalStorage / IndexedDB，不必上后端。

### 2.2 生成题目保存 / 导出功能

当前已有“隐式保存”，但没有“产品化保存”：

- 编译完成自动保存 Module JSON。
- 原始 Markdown 也保存。
- 完成页列出历史 Module，但不能操作。

M7.5 应补齐：

1. **Module Library 页面**
   路由建议：`/learn/library`。显示所有已保存 Module：标题、创建时间、概念数、题目数、完成状态、最近学习时间、估算生成成本。

2. **历史 Module 可恢复**
   点击历史项后：
   - `useModuleStore.setModule(module)`
   - 恢复该 module 对应 progress / attempts（需要从当前全局 store 改为 module-scoped 或加载时筛选）
   - 跳到 overview 或继续学习页

3. **导出 JSON**
   导出完整对象：`KnowledgeSource + Module + qualityReport + version + generatedAt + provider/model metadata`。

4. **导入已生成 Module**
   用户可导入 JSON，跳过 LLM 编译。这是降低成本的关键闭环。

5. **导出 Markdown / CSV（可选）**
   JSON 是 P0；Markdown 题目清单和 CSV 可作为人工校对友好格式。

6. **单 Module 删除 / 重命名**
   不要只提供“清空全部数据”。用户需要管理资产。

7. **修复 source 淘汰 bug**
   `removeModule()` 应先读取 module，拿到 `module.sourceId` 后删除对应 source。

### 2.3 前端作答后的回退方式

建议把“回退”分成三个层级，避免一次做复杂：

#### M7.5 P0：只读回看上一题

在 Concept / Challenge / Feynman Step 页面增加“上一题”入口，回到上一个 slot 的只读结果态：

- 展示题干、用户答案、正确答案、解释。
- 不允许改分数。
- 入口文案用低权重，避免打断主流程。

#### M7.5 P1：可重新练习当前 Concept

当前完成页已有待复习入口。建议在学习中也提供“重新开始本 Concept”，清除该 Concept 的 attempts，并把 stage 回到该 Concept 第一题。

#### V1.1：真正撤销 / 分支

如果允许用户撤销已提交答案，需要处理 attempts、mastery、retry replacement、feynmanAttempt 的一致性。建议先不做，等 Module-scoped attempt 存储完成后再做。

状态层建议新增：

```ts
goBack(): void
jumpToConcept(conceptIndex: number, mode: 'review' | 'restart'): void
jumpToChallenge(quizIndex: number, mode: 'review' | 'restart'): void
```

并把“查看历史答案”与“改变学习进度”分开。

### 2.4 UI Design 风格未应用

M7.5 建议做一次 UI token 对齐，不必重写全部页面，但要先把视觉底座换掉：

1. **建立设计 token**
   在 `globals.css` 添加 `:root` 变量：背景、文字、金棕、成功、warning、radius、duration、font family。

2. **加载字体与 body 基础样式**
   `layout.tsx` 或 CSS 中设置 serif 字体栈。MVP 可先使用系统宋体 / Georgia fallback，不必立刻做完整字体子集化。

3. **抽出基础组件 class**
   如 `.alc-page`, `.alc-button-primary`, `.alc-option`, `.alc-label`, `.alc-panel`，避免每个页面手写 `neutral-*`。

4. **实现阶梯进度**
   先在 Concept / Challenge / Feynman 三个学习页替换顶部文字进度。最小版本可以用 CSS grid 阶梯格，不必做复杂动画。

5. **反馈色替换**
   成功用鼠尾草绿，retry/差一点用陶土橙，避免红色清空按钮以外的警告视觉。

6. **完成页改成成长叙事**
   去掉强符号 `✓`，加入“从选择到费曼”的表达自由度回顾。

### 2.5 生成过程中刷新是否能无损继续

结论：**当前不能**。如果要无损继续，M7.5 有两个可选方案：

#### 方案 A：Local-first checkpoint（推荐 M7.5）

仍使用当前 API Routes，不引入数据库。

流程：

1. 用户点击编译时生成 `compileJobId`，把 rawMarkdown、config 摘要、status 写入 LocalStorage 或 IndexedDB。
2. 每收到一个 SSE event 就把 stage/percent/message 写入 job。
3. 服务端每完成一个 stage，就把该 stage output 作为 SSE event 发回前端。
4. 前端收到 stage output 后 checkpoint 到本地。
5. 刷新后读取 job：
   - 如果已有 complete module，直接进 overview。
   - 如果停在某 stage，显示“从 X 阶段继续”。
   - 继续时把 checkpoint outputs 发给 `/api/compile/resume`，服务端从下一个 stage 开始。

优点：符合 MVP 的 LocalStorage / BYOK 思路，不需要服务端持久化。  
缺点：刷新瞬间正在服务端跑的请求无法接回，只能从最近 checkpoint 继续，可能重复最后一个 stage 的 LLM 成本。

#### 方案 B：Server job registry

服务端持久化 job，刷新后前端通过 jobId 重新订阅 SSE。

优点：真正不断点。  
缺点：需要服务端存储，和 MVP “无服务端状态”冲突；Vercel serverless 也不适合长生命周期内存任务。

建议 M7.5 采用方案 A，并把产品文案定义为：**“刷新后从最近完成的编译阶段继续，最多重跑当前阶段。”** 这比“完全无损”更诚实，也更符合当前架构。

## 3. M7.5 建议范围

### Must

1. Module Library：列表、恢复、删除单项。
2. Export / Import compiled Module JSON。
3. Compile Job checkpoint：至少保存 rawMarkdown、stage、percent、complete module；刷新后能恢复界面。
4. Source 淘汰 bug 修复。
5. UI token 落地：颜色、字体栈、基础按钮/选项/页面背景。
6. 学习中“上一题回看”。
7. Compile Quality Report：记录修复次数、重试次数、题目数、Challenge 跨概念覆盖、distractor 数量概览。

### Should

1. Module 重命名。
2. 导出 Markdown 题目清单。
3. Module 生成成本估算。
4. 当前 Concept 重新练习。
5. 阶梯进度条首版。

### Not Now

1. 云端账号 / 同步。
2. 真正服务端 job resume。
3. 完整 Prompt A/B 平台。
4. 自适应学习。
5. 多 Module 课程。

## 4. 验收标准

| 项 | 验收 |
|---|---|
| 产物保存 | 编译完成后在 Library 可见；刷新浏览器仍可见 |
| 产物恢复 | 从 Library 打开历史 Module，可进入 overview / 继续学习 |
| 导出导入 | 导出的 JSON 重新导入后无需 LLM 即可学习 |
| 成本复用 | 同一导入 JSON 不触发 `/api/compile` |
| 编译刷新恢复 | 编译中刷新后能回到 compiling 页，显示最近 stage，并可继续或重新开始 |
| 学习刷新恢复 | Concept / Challenge / Feynman 刷新后回到正确界面 |
| 回退 | 作答后可查看上一题与解释，不破坏当前进度 |
| UI 对齐 | 页面背景、文字、主按钮、选项、反馈色不再使用裸 `neutral` 系统作为主视觉 |
| 质量报告 | 每次编译生成一份可查看的 quality summary |
| 数据清理 | 删除 Module 时 source/module/progress/feynman 关联数据一致删除 |

## 5. 建议实施顺序

1. **数据资产层**
   修复 source 删除；定义 `CompiledModulePackage`；实现 export/import JSON；实现 Library 恢复。

2. **编译恢复层**
   增加 `compileJobId` 与本地 checkpoint；刷新后恢复 compiling UI；先不做跨请求续跑，至少不丢用户上下文。

3. **学习导航层**
   增加上一题回看；补 URL moduleId 与 store moduleId 校验；Library 打开历史 Module 时正确加载进度。

4. **UI token 层**
   添加 design tokens 与基础 class；改 import / overview / module / quiz / done 的主色、字体和按钮。

5. **质量报告层**
   在 mapper / pipeline 中累计 schema 修复、重试、salvage、distractor 概览、Challenge involvedConceptIds，并在 Library / debug 面板展示。

---

## 6. 需要确认的产品决策

1. 导出 JSON 是否视为用户可编辑格式？如果是，需要版本号和迁移策略。
2. 编译刷新恢复的承诺是否可以表述为“从最近完成阶段继续”，而不是“正在跑的请求完全不断”？
3. 历史 Module 上限仍是“当前 + 最近 3 个”吗？如果提供导出，是否允许本地保留更多？
4. 统一 Key 内测模式下，是否允许用户导出包含 provider/model metadata 但不包含 API Key 的生成报告？
5. “上一题”是只读回看，还是允许重新作答并覆盖掌握度？建议 M7.5 先做只读回看。

---

## 7. 最终判断

当前项目已经越过了“能不能跑通”的阶段，M7.5 不应继续堆新题型或新 Agent，而应把已生成的 Module 当成产品资产来经营。保存、导出、恢复、回看、风格对齐，是进入 M8 内测/公测前最值得补的一层。

一句话：**M7 做到了生成学习路径；M7.5 应该让学习路径可拥有、可复用、可继续。**
