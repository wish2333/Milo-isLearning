## 3. 问题修复与根因

### 3.1 `removeModule()` source 残留

**现象**：Module 淘汰或删除时，Module 本体可删除，但 source 可能残留。

**根因**：source 保存时使用 `StorageKeys.source(module.sourceId)`，旧删除逻辑按 `StorageKeys.source(moduleId)` 清理，key 不一致。

**修复**：`removeModule()` 先读取 stored Module，使用 `module.sourceId` 删除 source，并同步清理 progress / feynman / attemptsModule / qualityReport。

### 3.2 导入产物无法作为可复用资产

**现象**：编译结果虽已写入 LocalStorage，但用户无法导出、跨浏览器迁移或复用，重复编译会浪费 LLM 成本。

**修复**：新增 `CompiledModulePackage` V1 格式：包含 source、module、version、exportedAt、qualityReport 和 provider/model metadata。导入时生成新的 moduleId/sourceId，并跳过 `/api/compile`。

### 3.3 编译刷新后上下文丢失

**现象**：`compile-store` 是页面内临时状态，刷新 compiling 页后用户只看到空状态或重新开始入口，源文本和阶段提示不稳定。

**修复**：新增 LocalStorage compile job。点击编译即写入 sourceContent / provider / model / running status；SSE event 到达时更新 stage / percent / status / moduleId / errorMessage。刷新后若 job 存在，显示恢复界面和诚实文案：

> 已恢复上次编译上下文。M7.5 会保留源文本和进度提示；继续时会重新发起编译请求。

### 3.4 学习中误点后无法回看解释

**现象**：学习流程只有 forward / retry，用户进入下一题后无法回看上一题解析。

**修复**：新增只读 `ReviewPanel`。Concept / Challenge 从 attempts-store 读取上一 slot 最后一次作答；Feynman Step 从 `feynmanAttempt.stepResults` 读取上一步得分。回看面板不调用 addAttempt、advance、retry，也不修改 progress。

### 3.5 Route id 与 currentModule 不一致

**现象**：直接访问 `/learn/module/{id}` 或刷新后，如果 Zustand currentModule 丢失或不匹配，页面会误回导入页。

**修复**：module route 使用 URL id 从 `StorageKeys.module(id)` 恢复 stored Module；找不到时回 `/learn/library`。这让 Library 打开历史 Module 和浏览器刷新都更稳定。

### 3.6 旧 smoke 断言与当前产品行为不一致

**现象**：M7 将 Concept / Challenge 判分改成本地 `evaluateAnswer()` 后，旧 E2E 仍期待 `/api/feedback` mock 文案；T7 完成页也把 H1 改为 Module 标题。

**修复**：更新 `e2e/smoke.spec.ts`，断言当前真实文案：`答对！继续保持这个节奏。`、`再试一题，重点看解析里的关键关系。`，完成页断言 `学习完成` 标签 + Module 标题。

## 5. 关键设计决策回顾

### 5.1 Local-first Library vs 服务端资产库

**决策**：继续使用 LocalStorage repository，不引入账号、数据库或云同步。

**理由**：MVP 仍是 BYOK + local-first。M7.5 的目标是让生成产物成为用户本地可管理资产，而不是引入服务端产品面。导出 / 导入 JSON 已能满足内测复现、成本复用和人工审查。

### 5.2 导入时 re-id vs 保留原 id

**决策**：导入 package 时生成新的 sourceId / moduleId，并同步更新 concept.moduleId / feynmanTask.moduleId。

**理由**：避免导入覆盖现有 Module；同一个 package 可多次导入做对比学习或复测。原 package 不被 mutate，测试覆盖 immutability。

### 5.3 API Key 安全边界

**决策**：导出 package 不包含 API Key；parse 阶段发现 JSON 中包含 `"apiKey"` 即拒绝。

**理由**：M7.5 导出文件可能被用户分享、提交或用于 bug 复现。provider/model metadata 可以保留，密钥绝不能进入导出文件、质量报告、URL 或日志。

### 5.4 Compile Recovery V1 的诚实边界

**决策**：M7.5 只保留源文本、配置摘要、stage、percent 和恢复入口；继续时重新发起 `/api/compile`。

**理由**：当前 pipeline 没有把 stage output 建模为可恢复输入。若直接承诺“从中断 stage 继续”，会牵动 compiler 架构并产生虚假产品承诺。M7.6 再做 IndexedDB checkpoint 与真正 stage resume。

### 5.5 只读回看 vs 撤销重答

**决策**：M7.5 只做上一题只读回看，不允许修改分数或覆盖 progress。

**理由**：撤销/重答会影响 attempts、mastery、retry replacement、feynmanAttempt 的一致性，需要 module-scoped attempts 与状态历史栈配套。只读回看能解决“想看解释”的高频需求，同时不增加状态复杂度。

### 5.6 Quality Report 观察优先

**决策**：质量报告只观察，不阻断编译。

**理由**：M7 已正确放宽 schema，M7.5 不应重新把教学质量建议变成硬校验。先让报告可见，内测积累样本后再决定哪些指标进入阻断门禁。
