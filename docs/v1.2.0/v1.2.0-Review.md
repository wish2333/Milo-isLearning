## 2. 架构决策记录

### 2.1 D-1：F24 跳过用新 `'skipped'` 状态（非复用 pending）

**选择**：新增 `'skipped'` 作为 `ModuleTopicStatus` 第 4 态，与 `'pending'` / `'in_progress'` / `'done'` 互斥。

**理由**：① 语义清晰——skipped = "用户主动放弃但已处理"，区别于 pending"从未进入"；② allDone 逻辑明确——`done + skipped === total`，skipped 计为"已处理"不阻塞主题完成；③ 可回看补做——skipped module 可点击重新进入（回退为 in_progress）；④ 持久化——`TopicProgress.skippedModuleIds` 记录跳过的 module，exit→resume 后 skipped 标记保留。

### 2.2 D-2：F42 编辑扩展现有 AnswerCorrector（非新建组件）

**选择**：扩展现有 `AnswerCorrector` 的 3 个 editor 子组件加可编辑字段，而非新建 `QuizEditor`。

**理由**：① 复用分发逻辑——AnswerCorrector 已按 `interactionType` 分发，编辑只需在 editor 内加输入区；② 复用持久化链路——`correctQuizAnswer` + `updateQuizInModule` 已支持 patch，仅需 widening patch 类型；③ 避免双入口——合并为单入口"编辑此题"更自然。

### 2.3 D-3：F40 扩展 Challenge 修正是纯接线

**选择**：ChallengeView 按 ConceptView 模式接入修正/忽略 props，无新组件。

**理由**：AnswerCorrector 已支持 3 题型；ChallengeView 复用 QuizRenderer + FeedbackPanel，只是缺了 5 个 props 接线。slotId 约定 = `slotQuiz.id`（与 concept 一致，无 `challenge:` 前缀），R-M2（slotId 不一致风险）已排除。

### 2.4 D-4：F43 用题库页内嵌分区（非独立路由）

**选择**：`IgnoredQuizSection` 组件嵌入 `/learn/library` 页底部，而非新建 `/learn/ignored` 路由。

**理由**：① 信息架构一致——忽略的题属于某个 Module，按 module 聚合展示最自然；② 减少路由膨胀——项目已 9 个 learn 页；③ 批量操作——分区支持全选恢复/按 module 恢复。

### 2.5 D-5：showcase origin 保护延续

**选择**：F42 编辑正文、F40 扩展 Challenge 修正、F43 忽略管理——对 `module.origin === 'showcase'` 题库一律禁用。

**理由**：展示题库是精选内容，运行时用户级编辑破坏跨用户一致性。与 v1.1.1 D-4、F07 重命名一致。

## 4. 已知限制

### 4.1 E2E 回归推迟

Playwright Chromium 二进制（v1228 / Chrome 149）已安装，但 `bun run e2e` 运行中 Next.js dev server 在前 2 个测试通过后 crash（后续全部 `ERR_CONNECTION_REFUSED`），系 Playwright webServer 管理层面问题，非代码回归。建议：① 干净环境（无端口冲突）重跑；② 检查 playwright.config.ts webServer timeout 配置；③ 或改为 preview server（`bun run build && bun run start`）模式。部署前必须完成 E2E 回归。

### 4.2 新增 E2E 覆盖未写

plan 要求补 `e2e/challenge-flow.spec.ts`（Challenge 流程 ~3-4 tests）和扩展 `e2e/a11y.spec.ts`（学习中页 + 重刷页 axe 扫描）。随 P1.1 一起推迟。

### 4.3 无新增单元测试

F42 QuizEditPatch 类型 widening、F24 skipCurrentModule/reenterModule 逻辑、IgnoredQuizSection 扫描/分组逻辑均建议后续补单测覆盖。当前由 typecheck + 既有 639 测试间接保障。

### 4.4 OG PNG 渲染工具链

`sharp` 作为 devDependency 引入（用于 SVG→PNG 转换）。`scripts/render-og.ts` 可通过 `bun run og:render` 重渲染。若未来 OG 设计变更，编辑 `public/og-image.svg` 后重新渲染即可。
