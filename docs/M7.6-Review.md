## 0. 结论

M7.6 已完成，并在最终验证中通过。项目从“能把 Module 存起来、找回来、复用起来”进一步推进到“能按预期顺序学习、能在学习和题库之间稳定切换、能让首页/设置/输入页真正进入 ui-design 视觉系统、能避免新编译覆盖旧题库”。

这轮最关键的变化不是单点功能，而是三类偏差被收住了：

1. **答题流重新对齐顺序状态机**：学习主线不再让题号提示和实际题目脱节，错题反馈保留，但推进逻辑回到 `progress-store.advance()` 的顺序路径。
2. **ui-design 真正覆盖到入口页面**：首页、设置页、输入页从旧的裸 `neutral-*` 视觉迁移到 `alc-*` token 和统一页壳，不再只是学习页局部换肤。
3. **题库不再静默消失**：容量策略改成“接近上限提示、删除必须显式确认”，编译时不再用自动淘汰把旧题库悄悄清掉。

**核心成果**：

1. Learning flow 回到顺序推进，题目提示与当前阶段一致
2. 错题保留解析与“下一步安排”，但不再改写主线题号
3. `ensureCapacity()` 退化为兼容性钩子，不再自动删除题库
4. 首页、设置页、导入页换用 ui-design token、页壳和通用按钮/表单类
5. Library 维持多 Module 可见，容量接近时明确提示用户导出或删除
6. 题库与学习页的 Playwright 回归通过，验证顺序流、旧题库可见性和页面入口都正常
7. `bun run typecheck`、`bun run lint`、`bun run test`、`bun run e2e` 全部通过

## 3. 问题修复与根因

### 3.1 答题过程中题号与实际题目脱节

**现象**：学习时上方提示“第几题”，但点“继续”后题目有时会跳到非顺序的 slot，导致界面语义和学习进度不对齐。

**根因**：上一轮为了照顾自适应重排，`ConceptView` 曾把推进逻辑交给 `buildAdaptiveQueue()`，直接写入任意 `quizIndex`，从而绕开了 `progress-store.advance()` 的顺序状态机。

**修复**：恢复 `ConceptView` 的主推进路径，统一调用 `advance()`。错题仍会显示解析和“下一步安排”，但这只是反馈层，不再驱动题号跳转。自适应面板保留为解释性提示，而不是路由控制器。

### 3.2 首页、设置页、输入页仍停留在旧视觉系统

**现象**：学习页和 Library 已进入 token 风格，但首页、设置页、Markdown 输入页仍大量使用裸 `neutral-*`，整体产品像是拼接出来的两套界面。

**根因**：M7.6 早期的视觉迁移主要集中在学习和 Library，入口页还没被真正收口到统一的 `alc-*` 基础类。

**修复**：为首页、设置页、输入页补齐 `alc-page`、`alc-app-header`、`alc-button-*`、`alc-input`、`alc-textarea`、`alc-pill-button` 等基础 class，并把三页主布局切换到 token 化结构。

### 3.3 新编译后旧题库“还是不见了”

**现象**：用户新编译一套题后，旧题库在库里消失，体验上像是资产被系统替换掉了。

**根因**：`ensureCapacity()` 仍保留自动淘汰逻辑。只要估算容量接近阈值，就会在编译保存阶段主动删掉旧 Module。这个策略和 M7.6 的产品目标冲突。

**修复**：把 `ensureCapacity()` 改成兼容性钩子，保留签名但不执行删除；容量接近时由 Library UI 提示用户导出或手动删除，真正删除只走显式二次确认。

### 3.4 quiz-batch 编译阶段重试率过高

**现象**：编译 pipeline 的 quiz-batch 阶段频繁因 schema 校验失败触发整批重试，导致单次 compile 耗时远超预期。日志中反复出现三类错误：(1) `extendedKnowledge` 字段内容过短（<20 字符，如 "Good"）；(2) distractor 的 text 与 answer 完全相同；(3) LLM 返回空 JSON。

**根因**：三个层面叠加导致：(1) Prompt 与 schema 约束不对齐——prompt 说"尽量输出"但 schema 有 `minLength` 硬约束；(2) quiz-batch schema 的 explanation 示例与真实 schema 漂移（示例写 20/200 但实际是 40/1200）；(3) 整批重试粒度过粗——单条 quiz 的一个字段不合规就导致整批 6-16 条全部重新生成；(4) 串行执行 concept，4 个 concept 逐个等待。

**修复（三层防御）**：

| 层级 | 改动 | 对应问题 |
|------|------|---------|
| Prompt 引导 | 追加字段长度硬约束表 + distractor 硬约束 + 逐题自检指令；"尽量输出"改为"要么符合要求，要么省略" | extendedKnowledge 短串、distractor=answer |
| Schema 容错 | `extendedKnowledge` 用 `.transform()` 自动将 <20 字符转为 undefined；`_runner.ts` 新增 `autoFix` 回调在 schema 校验失败后尝试修复再重试 | 可修复问题触发全量重试 |
| Mapper 兜底 | `assembleQuiz` + `assembleChallengeQuiz` 过滤 `distractor.text === answer`（双重保险） | 最终产物中干扰项等于答案 |
| 性能优化 | concept 级 chunked 并行（并发度 3）；quiz-batch max 从 16 降到 10 | 串行执行慢、batch tail degradation |

### 3.5 排序题排序正确但判错

**现象**：排序题用户拖拽出正确顺序后提交，系统仍判定为错误。

**根因**：Prompt 要求 LLM 生成的 `quiz.answer` 格式为 `→` 拼接的自由描述（如 `"检索→Embedding→生成"`），但 `SortingQuiz` 组件提交的用户答案是 `items.join('\n')`（换行拼接）。`evaluateAnswer` 做的是 `userAnswer === quiz.answer` 字符串精确比较，两者格式不匹配导致排序正确也永远判错。

**修复**：排序题正确性改为比较 `userAnswer === (quiz.options ?? []).join('\n')`。`quiz.options` 数组本身就是正解顺序（prompt 明确要求 options 按正确顺序排列），SortingQuiz 打乱后用户重排回去即为正确。同时修复了排序题答错时的 gaps，从显示无关 distractors 改为显示 `→` 拼接的正确顺序。

### 3.6 题库列表显示"学习中"但说明仍显示"从未学习"

**现象**：已作答过一些题的 Module，badge 正确显示"学习中"，但下方说明文字仍显示"从未学习"，两者矛盾。

**根因**：`listStoredModules` 从 per-module key `alc:progress:{moduleId}` 读取进度，但 progress-store 的 Zustand persist 只写到全局 key `alc:state:progress`。两者在代码中从不连通，生产环境从不写 per-module key，导致 `updatedAt` 永远为 0 → `formatDate` 返回"从未学习"。同时 badge 逻辑只有 `completed` 二态（已完成/学习中），对未开始的 Module 也显示"学习中"，加剧了矛盾感。

**修复**：
1. `progress-store.ts` 新增 `subscribe` 和 `onRehydrateStorage`，每次状态变更将 `ProgressState` 同步到 per-module 存储 key。切换 Module 时保存前一个 Module 的最终进度。reset 时不写（避免 clearAll 后又写入）。
2. `ModuleLibraryList.tsx` badge 改为三态：`updatedAt > 0 && !completed` → "学习中"，`updatedAt === 0 && !completed` → "未开始"，`completed` → "已完成"。`formatDate` 中"从未学习"改为"未开始"。主按钮文案也随状态变化（开始学习/继续/查看）。

### 3.7 学习中只能回看上一题，无法回到任意已做题

**现象**：ConceptView 和 ChallengeView 中只有"回看上一题"按钮，仅支持查看紧邻的上一道题，无法回看更早的题或已完成概念的题。

**修复**：新建 `AnswerHistoryList` 组件，遍历 Module 全部 Concept quiz + Challenge quiz，匹配 attempts-store 展示所有已作答题。每条可折叠展开查看：你的作答、参考答案、解析、常见误区、延伸知识。ConceptView 和 ChallengeView 中的"回看上一题"按钮替换为"答题历史"按钮，展开后可查看当前 Module 全部已答题（跨概念），当前题高亮标记。

### 3.8 题库列表缺少作答记录入口

**现象**：用户在题库列表中无法查看某个 Module 的作答情况，必须先进入学习流才能看到。

**修复**：`ModuleLibraryList` 每个 Module 卡片新增"作答记录"按钮，导航到新建的 `/learn/history/[id]` 页面。该页面读取 Module 本体和 attempts-store，通过 `AnswerHistoryList` 组件展示全部答题历史。空态显示"暂无作答记录"引导。

### 3.9 作答过程导航栏/背景与首页主题不一致，整体字号偏小

**现象**：作答过程（Concept / Challenge / Feynman）的导航栏和背景色使用 Tailwind 默认 `neutral-*` 中性灰，与首页/设置页/题库页的暖琥珀暗底主题割裂；导航栏各元素样式基本没按 ui-design 规范设置；整体字号偏小（实际用 Tailwind 默认值而非规范值），小字体（label / 计数器）对比度低看不清。

**根因**：三层断层叠加——

1. `globals.css` 只迁移了颜色 token，缺失完整字号 token（`--text-xs..4xl`）、字重、行高、间距、动效 token；body 也缺少设计规范要求的暖色纹理背景。
2. `tailwind.config.ts` 没有 extend `fontSize`，导致 `text-lg/xl/2xl/3xl/4xl` 用 Tailwind 默认值（18/20/24/30/36px）而非规范值（22/28/36/48/64px），整体偏小约 30%。
3. `LearnShell` 和 `LearnNavTop` 用 `neutral-950/800/400/500` 中性灰而非暖色 token，与"学术沉静"基调不符；`LearnNavTop` 没有套用 `.nav-top` 设计规范（rgba 半透明 + backdrop-blur）。

**修复**：

| 层级 | 改动 | 对应问题 |
|------|------|---------|
| Token 层 | `globals.css` 重写：补齐完整字号/字重/行高/间距/动效 token（与 `docs/ui-design/styles.css` 完全对齐），body 添加暖色纹理背景，新增核心组件类（`.alc-nav-top` / `.alc-stage-badge` / `.alc-button-primary` 等） | 缺失设计 token、无统一组件类 |
| Tailwind 暴露 | `tailwind.config.ts` extend `fontSize` / `fontWeight` / `letterSpacing`：把 `text-xs..text-4xl` 全部映射到 `--text-*` CSS 变量，让所有 Tailwind 字号类自动套用规范值（`text-lg` 18→22px、`text-xl` 20→28px、`text-2xl` 24→36px、`text-3xl` 30→48px、`text-4xl` 36→64px，整体放大约 30%） | 字号偏小 |
| 容器修复 | `LearnShell.tsx`：`bg-neutral-950 text-neutral-100` → `bg-bg-base text-fg-primary` | 作答过程背景与首页不一致 |
| 导航栏重构 | `LearnNavTop.tsx` 全面重构：套用 `.alc-nav-top`（暖色半透明 + blur 8px），导航链接用 `.alc-nav-top__link`，stageLabel 改为 `.alc-stage-badge` 强调色徽章 | 导航栏样式未按规范设置 |
| 批量 token 化 | 15 个文件 110 处 `neutral-*` 替换为暖色 token：`neutral-100/200` → `fg-primary`，`neutral-300/400` → `fg-secondary`，`neutral-500/600` → `fg-tertiary/quaternary`，`neutral-800` → `border-default`，`neutral-900` → `bg-surface`，`neutral-950` → `bg-base`，`bg-neutral-100/white`（按钮）→ `bg-accent-primary` | 全局中性灰 → 暖色 token |

涉及文件：11 个 quiz/learn 组件（`FillBlankQuiz` / `FeedbackPanel` / `BackgroundPanel` / `AdaptivePlanPanel` / `ConceptView` / `ChallengeView` / `ModuleIntroView` / `FeynmanIntroView` / `FeynmanStepView` / `FeynmanFinalView` / `ReviewPanel`）+ 4 个 learn 页面（`overview` / `done` / `compiling` / `module/[id]`）。

### 3.10 删除 ConceptView 的"现在再练一题"按钮

**现象**：用户要求删除 ConceptView 中答错后出现的"现在再练一题"按钮及其相关逻辑。

**修复**：移除 `ConceptView.tsx` 中的"现在再练一题"按钮 UI、`handleRetry` 函数（含 `/api/regenerate` 调用链）、`regenerating` phase、`Phase` 类型中的 `'regenerating'` 成员、以及删除 handler 后变未使用的 `replaceCurrentQuiz` / `retry` 引用。反馈阶段按钮简化为单一"继续"按钮（文案按 `isAdvancing` 切换：答对→"继续"，答错→"继续下一步"）。`AdaptivePlanPanel`（答错提示）保留——删除 retry 按钮后它的存在变得更必要，向用户解释"答错的题已记录为薄弱点，主线会继续推进"。`progress-store.retry()` 和 `module-store.replaceCurrentQuiz()` 仍被 `ChallengeView` 的"换一道题"按钮使用，未动。

### 3.11 题库数据隔离 bug（调研记录）

**现象**：(1) 生成新题库后，旧题库的作答记录出现在新题库的作答历史中；(2) 用户感知"生成新题库导致旧题库被删除"。

**已确认根因（作答记录混淆）**：

- `quiz.id` 被 Zod schema（`src/lib/compiler/schemas/quiz.ts:20`、`challenge-batch.ts:26`）强制为 `concept-N:slot-M` / `challenge-N` 格式，不含 moduleId，**跨 module 必然碰撞**——每个 module 的 concept-1 的 slot-1 的 quiz.id 都是 `concept-1:slot-1`。
- `attempts-store` 是全局单例（持久化到 `alc:state:attempts`），`attemptsBySlot` 按 `originalQuizId` 索引，无 module 隔离。
- `AnswerHistoryList` 按 `quiz.id` 查询全局 attempts → 显示其他 module 的记录。
- `removeModule` 删除的是从未被写入的遗留 key（`alc:attempts-module:*`），真正的 attempts 数据残留。

**后续确认（题库被删除）**：用户确认"题库被删除"的具体表现是：编译 A → 作答 A → 编译 B 后，`/learn/library` 列表里完全看不到 A，且稳定复现。该确认推翻了"只是作答记录混淆的衍生感知"假设，说明还存在独立的 Module 覆盖/删除路径。

**状态**：本问题已在后续修复中闭环，见 3.12。

### 3.12 题库数据隔离 bug 修复闭环

**最终根因**：Bug 1 不是静默删除，而是持久化 key 覆盖。编译产物的 `Module.id` 来自 LLM schema，通常稳定输出 `module-1`；编译完成页直接用 `StorageKeys.module(compiledModule.id)` 写入 localStorage，因此第二次编译会再次写入 `alc:module:module-1`，把第一次编译的 Module 覆盖掉。Library 从 `alc:module:*` 枚举，自然只剩新题库，看起来像旧题库被删除。

**修复**：

| 改动 | 文件 | 说明 |
|------|------|------|
| 本地身份分配 | `src/lib/persistence/module-package.ts` | 新增 `assignLocalModuleIdentity()`，把 LLM/导入包里的局部 id 转换为本浏览器持久化用的唯一 id：`module-${nanoid()}` / `source-${nanoid()}` |
| 编译路径接入 | `src/app/learn/compiling/page.tsx` | 编译完成后、写入 storage 前调用 `assignLocalModuleIdentity(parsed.module)`，避免继续用 `module-1` 覆盖旧题库 |
| quiz slot 全局唯一 | `src/lib/persistence/module-package.ts` | 将 Concept quiz / Challenge quiz 的 `id` 改为 `${moduleId}:${oldQuizId}`，例如 `module-abc:concept-1:slot-1`、`module-abc:challenge-1`，从源头切断 attempts 跨 module 碰撞 |
| 导入路径统一 | `src/lib/persistence/module-package.ts` | `.alc-module.json` 导入也复用同一身份分配逻辑，避免导入题库继续产生无前缀 quiz id |
| quality report 同步 | `src/app/learn/compiling/page.tsx`、`src/lib/persistence/module-package.ts` | 写入 quality report 时同步改写 `moduleId` 为本地新 module id |
| 删除清理补齐 | `src/lib/persistence/quota.ts` | `removeModule()` 除了删除遗留 `alc:attempts-module:*`，还会清理 `alc:state:attempts` 中以 `${moduleId}:` 开头的 slot，避免新数据残留膨胀 |

**数据兼容影响**：

- 新编译/新导入的 Module 不再覆盖旧题库，作答记录也不会再跨题库串库。
- 已经被覆盖的旧 `alc:module:module-1` 本体无法从 localStorage 自动恢复，除非用户此前导出过 `.alc-module.json` 或有浏览器备份。
- 旧的无前缀 attempts（如 `concept-1:slot-1`）不会被强行迁移或删除；它们不会匹配新 Module 的前缀化 slot id，因此不会继续污染新数据。

## 5. 关键设计决策回顾

### 5.1 顺序流优先于“看起来聪明”的跳转

**决策**：主学习流保持顺序推进，不把 adaptive queue 直接接管 `quizIndex`。

**理由**：用户界面上明确展示了题号和阶段，主线跳转必须与这层提示一致。自适应内容可以存在，但只能作为建议或提示，不该破坏学习进度的可预测性。

### 5.2 题库容量以显式用户动作收口

**决策**：题库接近上限时提示，不静默删除。

**理由**：旧题库在用户心里是资产，不是可无声抹掉的缓存。导出 / 删除必须被看见，才能建立信任。

**补充**：`ensureCapacity()` 现在只保留兼容性入口，不再承担自动清理职责。这个职责被移交回 Library 的显式删除对话框和容量提示。

### 5.3 入口页先统一骨架，再谈更细的视觉差异

**决策**：首页、设置页、输入页先全部接入 `alc-*` token 和统一页壳。

**理由**：M7.6 的问题不是少一个颜色，而是产品入口和学习入口像两个系统。先统一骨架，才能继续细化风格。

### 5.4 兼容性钩子保留，不让历史实现误导当前语义

**决策**：`ensureCapacity()` 保持函数存在，但不再做自动删除。

**理由**：这样可以稳住调用方和测试面，避免把“容量检查”误解成“自动清理策略”。真正的清理动作现在只由用户显式触发。

### 5.5 自适应能力保留为轻提示而非主路由

**决策**：保留 adaptive sequencer 和提示面板，但不让它接管顺序推进。

**理由**：M7.6 的 ui 和题号提示已经向顺序学习做了明确承诺。若主路由仍可被重排，用户会持续感到“看板与实际题目不一致”。把自适应能力降级为提示，可以保留教学意图，又不破坏可预测性。

## 6. 与 M7.7 / M8 的衔接

M7.6 完成后，下一阶段更适合处理这些事情：

1. **真正的 Stage Resume**：把编译阶段输出做成可恢复 checkpoint，而不是只保留上下文提示。
2. **更细的自适应学习**：如果未来要恢复题目重排，应该先把题号展示、当前 slot、revisit 节奏和状态机契约设计清楚，再让自适应逻辑接管。
3. **Library 的重命名 / 导出清单**：让题库资产管理更完整。
4. **Prompt 质量报告扩展**：把背景覆盖、解析长度、Fill Blank 的答案变体这些信号做得更细。
5. **UI 收尾**：学习页、完成页、Library 的细节色彩和排版还可以继续往 ui-design 原型靠。
