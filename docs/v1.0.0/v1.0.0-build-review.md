## 2. 架构决策

### 2.1 why better-sqlite3 not bun:sqlite

**原决策**：用 `bun:sqlite`，理由是"Bun 运行时原生支持"。

**实际失败**：`next dev` / `next build` / Vercel serverless 都跑在 Node 上——即使 `bun run dev`，Next CLI 内部仍 spawn Node 子进程。`bun:sqlite` 是 Bun 虚拟模块，Node 永远 `Cannot find module`。

**修正决策**：改用 `better-sqlite3`（Node 原生 C++ addon，npm 包），并通过 `SqliteAdapter` 提供 `query()` / `run()` 兼容 API（保留原 schema/repository/migration 代码风格），单点适配层比改 5 个文件简单且更安全。

### 2.2 StorageRepository 同步语义保留

**约束**：评审红线 #2 要求 `StorageRepository` 接口保持同步（`get/set/remove` 直接返回值）。

**实现**：客户端 `ClientFetchStorageRepository` 内部维护 `Map<string, string>` cache，`set()` 立即更新 cache（同步返回），同时入队 `WriteQueue` 异步落盘。读操作直接走 cache 不打 server，符合同步语义；写操作的异步性对调用方透明，失败由 retry 机制 + UI toast 兜底。

### 2.3 WriteQueue 的 per-key 合并

**问题**：连续 `set('alc:state:module', v1)` `set('alc:state:module', v2)` 会产生两个 PUT，浪费带宽且可能竞态。

**方案**：queue 内部按 key 维护 pending map，新 enqueue 同 key 直接覆盖旧任务（按 operationId 比较跳过过时任务）。同一时刻每个 key 至多 1 个 in-flight 请求，避免竞态。

### 2.4 Chrome keepalive fetch 64 KiB 限制

**问题**：客户端 `sendPut` 用 `keepalive: value.length < 64 * 1024` 判断是否启用 keepalive，但 Chrome 限制是 64 KiB **字节**。中文富数据 30K 字符 ≈ 90K UTF-8 字节，`value.length < 65536` 误判为 true 启用 keepalive，Chrome 立即抛 `TypeError: Failed to fetch`（无 HTTP 状态码）。

**修正**：用 `TextEncoder.encode(value).length` 精确算 UTF-8 字节，超限时禁用 keepalive（让请求走正常流程，页面卸载可能丢失但有 retry 兜底）。

### 2.5 migration.ts chunkBySize 字节估算

**问题**：`chunkBySize` 用 `entry.key.length + entry.valueRaw.length`（字符数）算批次大小，目标 ≤ 512 KiB。中文富数据字符数 ≤ 512K 但 UTF-8 字节可能 ~1.5 MiB，触发服务端 Content-Length 检查 413。

**修正**：用 `TextEncoder` 精确算单条 entry 的 JSON 序列化字节数（含 `{"key":"","value":""}` 包装 + 逗号分隔符），`MAX_BATCH_BYTES` 从 512 KiB 降到 384 KiB（留 25% 余量给 sessionId + 整体 JSON 结构）。

### 2.6 app-mode 默认 showcase

**问题**：v1.0.0 之前 app-mode 默认 production，但 server 端 `isProductionMode` 默认 fail-closed（未设 `ALC_STORAGE_BACKEND` 时 false），导致 client 误以为 production、server 误以为 showcase，`/api/data/*` 404。

**修正**：默认改 showcase，与 server fail-closed 一致。用户必须显式设 `NEXT_PUBLIC_APP_MODE=production` + `ALC_STORAGE_BACKEND=sqlite` 才进入 production 模式。

## 5. 迁移说明

### 5.1 showcase 模式（默认，无需操作）

不设 `NEXT_PUBLIC_APP_MODE` 或设为 `showcase` → 行为同 v0.1.0，零回归。LocalStorage 仍生效，无 SQLite。

### 5.2 production 模式（首次启用）

`.env.local` 加：

```
NEXT_PUBLIC_APP_MODE=production
ALC_STORAGE_BACKEND=sqlite
```

首次访问时客户端检测到 LS 有数据 + SQLite 未迁移 → 弹出迁移向导。迁移流程：

1. 扫描 LS（legacy-local-storage-scanner）
2. 上传来源快照（POST `/api/migrate/source-snapshot`）
3. 创建 migration session（POST `/api/migrate/session`）
4. 分批上传 staging（POST `/api/migrate/staging`，每批 ≤ 384 KiB / 100 条）
5. 提交（POST `/api/migrate/commit`）
6. 客户端重新加载 cache + rehydrate
7. 写 client marker（markMigrated）

迁移失败可点取消（POST `/api/migrate/cancel`），session 状态置 cancelled。

### 5.3 SQLite 数据位置

- `data/alc.db` — 主数据库
- `data/alc.db-shm` / `data/alc.db-wal` — WAL 模式辅助文件（自动管理）
- `data/backup/alc-snapshot-*.db` — VACUUM INTO 自动备份（保留最近 10 个）
- `data/backup/alc-ls-snapshot-*.json` — LS 来源快照（迁移时上传）
- `data/migration-logs/*.jsonl` — 迁移日志（保留最近 10 个）

`.gitignore` 已更新忽略 `data/` 目录。

## 附：本会话暴露的系统性问题

**类别**：JavaScript string `.length` 算字符数，HTTP / SQLite / Chrome 限制按字节数。

**影响位置（已修复 3 处）**：

| 位置 | 之前（bug） | 修复后 |
|------|------------|--------|
| `db-singleton.ts` bun:sqlite | `nodeRequire('bun:sqlite')` Node 不可用 | `import Database from 'better-sqlite3'` + SqliteAdapter |
| `migration.ts` chunkBySize | `entry.key.length + entry.valueRaw.length` | `textEncoder.encode(JSON.stringify(...)).length` |
| `client-fetch-storage.ts` sendPut | `value.length < 64 * 1024` | `utf8Bytes(value) < 64 * 1024` |

**建议**：项目内全局搜索 `.length <` 和 `.length >` 找剩余字符数当字节数用的地方，统一改 `TextEncoder`。
