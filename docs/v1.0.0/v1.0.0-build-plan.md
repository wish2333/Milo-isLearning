## 3. 架构总览

### 3.1 目标状态

```
showcase 模式（Vercel，用户行为零回归）:
  Client → LocalStorageRepository → window.localStorage
  （保留 5MB / 12 题上限）
  /api/data/* → 404（fail-closed 双开关均不满足）

production 模式（本地 dev，需 NEXT_PUBLIC_APP_MODE=production AND ALC_STORAGE_BACKEND=sqlite）:
  Client (Browser)                              Server (Next.js dev)
    └─ AppShell                                   └─ /api/data/*（双重开关守卫）
         └─ StorageInitializer (状态机)                ├─ GET /api/data/bulk → 全量拉取
              ├─ idle → loading → ready | error       ├─ GET /api/data/[...key] → sqlite.get()
              ├─ 加载 cache                            ├─ PUT /api/data/[...key] → sqlite.set()
              ├─ 检查迁移                              ├─ DELETE /api/data/[...key] → sqlite.remove()
              ├─ rehydrate 6 stores                    ├─ POST /api/data/flush → 强制落盘
              └─ 渲染主应用                             ├─ POST /api/data/clear → 清库
                                                      ├─ GET /api/data/status → 健康检查
         └─ Zustand store (skipHydration)             ├─ POST /api/migrate/* → staging 迁移
              └─ zustandStorageAdapter                 ├─ GET /api/data/export → BackupPackage
                   └─ ClientFetchStorageRepository     ├─ POST /api/data/import → 全库恢复
                        ├─ Map<string, string> cache        └─ ServerSQLiteRepository
                        └─ FIFO WriteQueue                      └─ bun:sqlite → data/alc.db
                             ├─ per-key 合并
                             ├─ operationId 排序
                             ├─ 自动重试 3x (250ms/1s/4s)
                             ├─ failed 状态追踪
                             └─ visibilitychange flush
```

### 3.2 核心设计决策（评审定案）

以下决策均来自评审 §二（D1-D11 定案）与 §三（开放问题定案），为不可变更的实施约束。

#### 3.2.1 Fail-closed 双开关

```
NEXT_PUBLIC_APP_MODE=production  (client + server 可见)
ALC_STORAGE_BACKEND=sqlite       (server-only，不暴露给 client)
```

只有两个条件**同时成立**，server 才启用 SQLite。`ALC_STORAGE_BACKEND` 未设置时 `/api/data/*` 一律 404。

- 启动配置检查：server 初始化时验证两个环境变量组合合法性
- `/api/data/status`：返回 `{ enabled: boolean, schemaVersion: number, stats }`
- client 端检测到 `enabled=false` 时显示「本地数据库未启用」错误页，不静默用空 cache 继续
- production API 仅接受 `alc:*` key，校验 key 长度上限、请求体大小上限、同源请求（`Origin` header 检查）

#### 3.2.2 Hydration 时序

production 模式下 6 个 store **全部设置** `skipHydration: true`。

启动流程固定为：

```
1. AppShell mount
2. StorageInitializer 状态机：idle → loading
3. fetch /api/data/bulk → 加载全量 cache 到 Map<string, string>
4. 检查旧 LocalStorage 迁移（详见 Phase 5）
   - 若需迁移且用户确认 → 执行迁移 → 重新加载 cache
   - 若需迁移且用户「稍后」→ 用当前 cache 继续
5. 依次调用 6 个 store 的 persist.rehydrate()
6. 全部成功 → 状态机 → ready → 渲染主应用
7. 任一步骤失败 → 状态机 → error → 显示错误页 + 「重试」按钮
```

- `initClientStorage()` 必须幂等，多次调用共享同一个 Promise
- 初始化失败时**不得用空数据继续渲染**

#### 3.2.3 FIFO 写队列

```
写入流程：
  store.set(key, value)
    → 立即更新内存 cache（同步）
    → 入队 WriteQueue（operationId 递增）
    → 同 key 的 pending 写入合并为最新值
    → FIFO 单线程发送（不并发）

重试策略：
  自动重试 3 次：250ms → 1s → 4s
  3 次失败后 → 任务进入 failed 状态（不丢弃）
  UI 显示「有未保存数据 / 保存失败」→ 允许手动重试

flush 时机：
  visibilitychange → hidden 时自动 flush
  sendBeacon 仅发送小批量 POST
  fetch(..., { keepalive: true }) 作为补充
  beforeunload 不作为主要保障

强制 await flushNow() 的操作：
  - 编译结果正式完成
  - Quiz 作答提交
  - Feynman 提交
  - Module 删除
  - 导入完成
  - 迁移完成
  - 恢复完成

普通阅读进度：
  不逐次 await，按 key 约 300ms 合并，页面隐藏时 flush
```

#### 3.2.4 Staging 迁移机制

```
客户端扫描 LS → 生成 sourceFingerprint
→ POST /api/migrate/session 创建 migration session
→ 分批 POST /api/migrate/staging 上传到 staging 表
→ server 校验条数、hash、业务结构
→ POST /api/migrate/commit 单次 SQLite transaction：staging → kv
→ 写 migration meta → 清理 staging
→ 客户端重新加载 cache + rehydrate
```

- 上传阶段可取消（最终 commit 前）
- server 重启后可续传
- 最终 commit 前正式数据不变
- 最终提交是一个很短的不可取消事务（UI 短暂禁止关闭）
- **不再使用「全屏 modal 永远禁止关闭」**

#### 3.2.5 BackupPackage 格式

```typescript
interface BackupEntryV1 {
  key: string
  valueRaw: string       // 原始 JSON 字符串，不重新序列化
  namespace: string      // key.split(':')[1]
  updatedAt: number      // Date.now()
}

interface BackupPackageV1 {
  version: 1
  exportedBy: 'ai-learning-compiler'
  exportedAt: number
  appMode: 'production' | 'showcase'
  schemaVersion: number
  entries: BackupEntryV1[]
  meta: {
    moduleCount: number
    totalBytes: number
    checksum: string     // 见下方算法
    appVersion: string
  }
}
```

Checksum 算法（长度前缀编码，避免对象字段顺序问题）：

```
checksum = sha256(
  entries.sort(by key)
    .map(e => e.keyLength + e.key + e.valueLength + e.valueRaw + e.updatedAt)
    .join('')
)
```

v1.0.0 不实现「按 updatedAt 自动 merge」。导入默认策略为**跳过冲突**；灾难恢复为独立操作（见 §3.2.6）。

#### 3.2.6 导入与灾难恢复

**普通导入**：
- 默认冲突策略：**跳过**（同 key 不覆盖）
- 不自动备份

**灾难恢复（独立操作）**：
- 自动备份当前 SQLite 库（SQLite 一致性快照，非文件拷贝）
- 清空 kv 表
- 导入备份文件
- 全程单事务
- 需二次确认

> **文档一致性修正**：D8 否决「默认覆盖」。普通导入默认跳过；灾难恢复作为独立操作。

#### 3.2.7 容量限制

```typescript
// 不使用 Number.MAX_SAFE_INTEGER 代表「无上限」
interface StorageCapacity {
  maxModules: number | null  // showcase: 12, production: null
  // ...
}
```

> **文档一致性修正**：production 容量统计不塞进同步的 `getStorageCapacitySummary()`。拆成异步 `useStorageStats()`（从 `/api/data/status` 拉取）；showcase 保留现有同步 quota。

### 3.3 约束红线

1. **showcase 用户行为和数据语义零回归**（共享代码会改动，但行为不变）
2. **`StorageRepository` 接口保持同步**（加 `setRaw` 方法，不改异步）
3. **不自动删除 LocalStorage**
4. **不引入 `as any` / `@ts-ignore`**
5. **不裸用 `localStorage`**——唯一例外：`legacy-local-storage-scanner.ts`（只读访问旧 LS，用于迁移扫描）
6. **不生成 emoji**
7. **中文优先**
8. **server-only / client-only 强制隔离**——server 代码 `import 'server-only'`，client 代码 `import 'client-only'`

> **文档一致性修正**：「禁止裸用 localStorage」增加唯一例外：`legacy-local-storage-scanner.ts` 可只读访问旧 LocalStorage，用于迁移。production storage 已指向 SQLite，不读旧 LS 就无法扫描。

function sanitizeForExport(entries: BackupEntryV1[]): BackupEntryV1[] {
  return entries.map(entry => {
    if (entry.key === 'alc:settings') {
      // 按 schema 剔除已知敏感字段
      const parsed = JSON.parse(entry.valueRaw)
      delete parsed.config?.apiKey
      delete parsed.config?.apiConfig?.apiKey
      return { ...entry, valueRaw: JSON.stringify(parsed) }
    }
    return entry
  })
}

// 辅助：关键词扫描仅警告，不阻断
function scanSensitiveKeywords(pkg: string): string[] {

**迁移流程详解**：

```
1. StorageInitializer 检查迁移状态
   - 查 client marker `localStorage['alc:migrated-at']`
   - 查 server meta `migration_source_fingerprint`
   - 若两者均存在 → 跳过迁移
   - 若 client marker 不存在 → 扫描 LS

2. 扫描 LS（legacy-local-storage-scanner.ts）
   - 精确匹配静态 key：'alc:settings'、'alc:events'、'alc:ratings'、'alc:topic-index'
   - 前缀匹配动态 key：'alc:source:'、'alc:module:'、'alc:mastery:' 等
   - 排除：'alc:runtime-mode'（sessionStorage，不扫）
   - 生成 sourceFingerprint = sha256(sorted keys + values)

3. 扫描结果为空 → 写 client marker → 跳过

4. 扫描结果非空 → 显示 MigrationPrompt
   「检测到旧版数据（X 条，Y MiB）」
   [立即迁移] [稍后再说] [查看详情]

5. 用户点「立即迁移」→ MigrationDetail
   - 默认勾选：排除 showcase origin Module
   - 默认勾选：迁移前自动备份（强制开启，不可关）
   - [开始迁移]

6. 开始迁移 → MigrationProgress
   Phase A（可取消）：
     a. 备份当前 SQLite 库（VACUUM INTO）
     b. 上传 LS 来源快照到 server 保存
     c. 创建 migration session
     d. 分批上传到 staging（100 条或 512 KiB/批）
     e. server 校验：条数 + SHA-256 + 业务结构（每个 Module 可 loadStoredModule）
   Phase B（短暂禁止关闭）：
     f. POST /api/migrate/commit → 单事务 staging → kv
     g. 写 migration meta
     h. 清 staging
   Phase C：
     i. 客户端重新加载 cache
     j. rehydrate 6 stores
     k. 写 client marker + server meta

7. 迁移成功 → MigrationResult
   - 显示：迁移 N 条，跳过 M 条（损坏 key），耗时 Xs
   - LocalStorage 不自动删除

8. 用户点「稍后」→ 记录 dismissed-at，7 天内不再提示
```

## 8. 风险登记簿

### 8.1 高优先级风险（Phase 1-5 全程关注）

| 编号 | 风险 | 缓解 |
| --- | --- | --- |
| R-H1 | **Hydration 时序错误**：store 在 cache 就绪前 hydrate | `skipHydration: true` + 状态机 + 显式 rehydrate 序列 |
| R-H2 | **`bun:sqlite` 打入 showcase bundle** | `server-only` 标记 + showcase 构建门禁 |
| R-H3 | **ALC_STORAGE_BACKEND 误配置** | fail-closed 双开关；默认关闭；配置错误启动失败 |
| R-H4 | **写队列乱序覆盖** | FIFO 单队列 + per-key 合并 + operationId |
| R-H5 | **迁移半完成**（网络中断） | staging 机制 + 最终单事务 commit |
| R-H6 | **迁移完成后旧 LS 覆盖 SQLite** | server meta sourceFingerprint 双重标记 |
| R-H7 | **WAL 数据遗漏**（备份时直接拷贝 db 文件） | `VACUUM INTO` 一致性快照 |
| R-H8 | **sendBeacon 64KB 限制** | 小批量发送 + `fetch keepalive` 补充 |

### 8.2 中优先级风险

| 编号 | 风险 | 缓解 |
| --- | --- | --- |
| R-M1 | dev server HMR 连接泄漏 | `globalThis.__alcDb` 守卫 |
| R-M2 | progress-store 双写断裂 | 迁移覆盖两个 key；保留双写行为 |
| R-M3 | `alc:state:attempts` 全局表迁移 | 按 slotId 前缀拆分；关联校验单独处理 |
| R-M4 | 无 Zustand version/migrate | SQLite schema 独立版本管理（meta 表） |
| R-M5 | production API 输入校验不足 | key 前缀/长度/body 大小/Origin 同源四重校验 |
