## 1. 背景与目的

v0.1.0 → v1.0.0 的 MVP 阶段已完成核心编译流水线、学习旅程、双模式架构与 LLM 接入。当前作者本人使用本地 `bun run dev` 时遇到两个具体痛点：

1. **LocalStorage 易失**：浏览器清缓存/重装即全丢，长期积累的学习历史不可靠
2. **12 题上限**：5MB 容量约束的产物，长期使用不够

本报告固化解决这两个痛点的技术方案。**核心约束**：现有 showcase 模式（Vercel 部署）必须**零改动**保留，所有升级仅作用于 production 模式（本地 dev）。

## 4. 技术方案选型

### 4.1 后端：Bun 内置 SQLite（仅 production 模式）

**为什么**：

| 候选 | 评价 |
| --- | --- |
| **`bun:sqlite`（选定）** | Bun 原生内置 0 依赖；同步 API 与现有 `StorageRepository` 接口完全对齐；本地文件持久化；性能优秀（毫秒级）；dev server 长进程持有连接稳定 |
| `better-sqlite3` | 同步 API 但需 native binding，Windows 偶有编译问题；Bun 已内置 SQLite 没必要 |
| IndexedDB | 浏览器内虚拟 FS，清缓存即丢；不满足 R1 |
| sql.js / OPFS | 同上 + 包体积 + 兼容性问题 |

**关键决策**：SQLite 仅在 production 模式下启用。showcase 模式继续用 LocalStorage，0 行代码改动。

### 4.2 双模式存储适配

**核心机制**：`storage` 单例按 `APP_MODE` 在初始化时选择实现。

```typescript
// src/lib/persistence/index.ts（新增，替换原 local-storage.ts 末尾的单例导出）
import { isShowcaseMode } from '@/lib/runtime/app-mode'
import type { StorageRepository } from './repository'
import { LocalStorageRepository } from './local-storage'

// showcase 模式：保持现状
// production 模式：用 client-fetch 适配器（内部走 /api/data → server SQLite）
export const storage: StorageRepository = isShowcaseMode
  ? new LocalStorageRepository()
  : new ClientFetchStorageRepository()
```

**为什么 client 端用 `ClientFetchStorageRepository` 而不是直接调 SQLite**：
- SQLite 是 server-side only（Bun 进程内），浏览器无法直接访问
- 必须通过 fetch 调用 Next.js API Route
- Client 端需要保留**同步**接口（与现有 Zustand store 调用约定一致），所以需要内存 cache

**整体架构**：

```
showcase 模式（Vercel，0 改动）:
  Client → LocalStorageRepository → window.localStorage
                                           ↓
                                    （仅 5MB / 12 题上限）

production 模式（本地 dev）:
  Client (Browser)                         Server (Next.js dev)
    └─ Zustand store                         └─ /api/data/*（仅 production 注册）
         └─ ClientFetchStorageRepository          ├─ GET    → sqlite.get()
              ├─ 内存 cache (全量)                ├─ PUT    → sqlite.set()
              ├─ 读：内存命中                     └─ DELETE → sqlite.remove()
              └─ 写：内存 + write-behind POST          └─ ServerSQLiteRepository
                                                          └─ bun:sqlite → data/alc.db
```

### 4.3 SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS kv (
  key        TEXT PRIMARY KEY,        -- "alc:module:abc123" 等，与现有 StorageKeys 1:1
  value      TEXT NOT NULL,           -- JSON.stringify 后的字符串
  namespace  TEXT NOT NULL,           -- 从 key 解析：module/attempts/progress/...
  updated_at INTEGER NOT NULL         -- Date.now()
);

CREATE INDEX IF NOT EXISTS idx_kv_namespace ON kv(namespace);

CREATE TABLE IF NOT EXISTS meta (
  name  TEXT PRIMARY KEY,
  value TEXT NOT NULL
);
```

**设计要点**：
- 单一 KV 表对齐现有 `StorageKeys` 命名空间，**不引入 relational schema**
- `namespace` 字段从 key 前缀解析，替代当前 `keys().filter(isAlcKey)` 模式
- `updated_at` 字段为未来可能的同步逻辑预留（本次不使用）
- Schema 简单，迁移成本最低

### 4.4 接口适配：同步接口 + 异步 fetch 的桥接

**问题**：当前 `StorageRepository` 是同步接口，但 fetch 是异步的。

**方案**：Client-side **in-memory cache + write-behind**

```
启动流程：
1. AppShell 渲染前 await /api/data/bulk 全量拉取到内存 cache
2. cache 加载期间显示 loading UI（骨架屏）
3. cache 就绪后渲染主应用，后续读全部走内存

读流程（同步）:
  store.get(key) → cache.get(key) → 内存命中，同步返回
  与 LocalStorage 行为完全一致

写流程（同步返回 + 异步落盘）:
  store.set(key, value)
    → 立即更新内存 cache（同步）
    → 微任务队列 POST /api/data/{key}（fire-and-forget）
    → 失败时 console.error + 重试 3 次（指数退避）
```

**取舍**：
- **优点**：5 个 Zustand store 内部 0 改动；`StorageRepository` 接口保持同步；UX 与 LocalStorage 一致
- **缺点**：极端情况下浏览器在 fetch 未完成时关闭，最后一次写可能丢失
- **缓解**：
  - `beforeunload` + `navigator.sendBeacon` 强制 flush
  - 关键操作（compile 完成、progress 更新）立即 await flush
  - server 收到后 bun:sqlite 同步落盘

**为什么不让接口变异步**：影响面巨大（5 个 store 内部所有 set 加 await、quota/library/topic 等纯函数库全部要改签名），收益不明显。

### 4.5 突破 12 题限制（仅 production 模式）

直接在 `quota.ts` 按 `APP_MODE` 条件分支：

| 位置 | showcase 模式（保留） | production 模式（移除） |
| --- | --- | --- |
| `keys.ts:62` `STORAGE_MAX_HISTORY_MODULES` | 保留导出，值 = 12 | 同一常量，值改为 `Number.MAX_SAFE_INTEGER` |
| `quota.ts:23` `MAX_STORED_MODULES` | 保留 12 | 改为基于 `isShowcaseMode` 的条件值 |
| `getStorageCapacitySummary()` | `maxModules: 12` + 容量预警 | `maxModules: null`（无上限） + 基于实际 SQLite 文件大小展示 |
| UI 文案 | "已达上限"警告保留 | 改为纯展示「磁盘占用 X MB」 |

**实现策略**：常量值改为按 `isShowcaseMode` 选择，避免散落的 `if` 分支。

### 4.6 数据迁移：LocalStorage → SQLite（仅 production 模式）

#### 4.6.1 UX 流程

1. v1.0.0 升级后首次进入 `/learn`（production 模式下），弹层：
   ```
   检测到旧版 LocalStorage 数据（X 个 Module、Y 条作答记录，共 Z MB）
   [立即迁移到本地数据库]  [稍后再说]  [查看详情]
   ```
2. 详情页展示：扫描结果（按 namespace 分类）、风险说明、自动备份承诺
3. 点击迁移：分批进度条（已处理 N / 总 M），实时显示当前 namespace
4. 成功后：自动触发一次全库 JSON 导出作为兜底备份，落到 `data/backup/alc-pre-migration-YYYYMMDD.json`
5. 后续启动：`alc:migrated=true` 标记位阻止重复扫描提示

**用户决策点**（详情页可配置）：
- [x] 默认勾选：仅迁移 `module.origin !== 'showcase'` 的数据（避免访客残留混入）
- [ ] 可选：包含 sessionStorage 数据（默认不迁，因为 runtime-mode 是会话级）
- [ ] 可选：迁移前自动全库导出 JSON 备份（默认强制开启，不可关）

#### 4.6.2 完整风险清单与缓解

**A. 数据完整性风险**

| 风险 | 详情 | 缓解 |
| --- | --- | --- |
| A1 损坏的 key（JSON.parse 失败） | LocalStorage 中存在但反序列化失败 | 跳过该 key + 写入 `data/migration.log` + 在结果页逐条列出，不阻断整体迁移 |
| A2 单 value 超大（>1MB） | 极端 Module 含大量 quiz/attempts | 分批迁移，每批 ≤ 100 条或 ≤ 1MB（先达到即提交）；不依赖 Next.js 默认 body 限制 |
| A3 Unicode/emoji | 中文标题/内容、特殊符号 | SQLite TEXT 默认 UTF-8 与 LocalStorage 一致；测试套件覆盖 emoji、零宽字符、CJK 标点 |
| A4 null 语义对齐 | LS `getItem` 不存在返回 null；Repository 已统一为 null | 沿用现有 `StorageRepository.get` 语义，迁移工具调用 `getRaw` 拿原始字符串后直接存 SQLite，避免重复 parse/stringify |
| A5 数值精度 | Date.now() 13 位整数、概率小数 | 存 TEXT（JSON 字符串），反序列化时 JS number 自动恢复；不依赖 SQLite INTEGER 类型 |
| A6 已废弃 key 格式 | M7.5 之前的旧 schema key | namespace 无法识别的 key 跳过 + 日志；不强制兼容 |
| A7 关联数据断裂 | Module 迁了但 attempts/feynman/progress 没迁 | 按 namespace 整批扫描，强制迁移 `{module, source, attempts, feynman, progress, mastery, quality}` 七个 namespace 配套；事务边界按 namespace 划分 |

**B. 流程风险**

| 风险 | 详情 | 缓解 |
| --- | --- | --- |
| B1 迁移期间用户操作 | 用户在弹层下继续编译/答题 | 迁移期间全屏 modal + 进度条，禁止关闭；底层 UI 不接收交互 |
| B2 网络中断 | POST 中途断网 | 分批提交，每批独立事务；断点续传（记录已成功 namespace + offset）；UI 提供「重试失败批次」按钮 |
| B3 dev server 热重载 | 用户改代码触发重启 | bun:sqlite 单事务原子性保护；client 检测 fetch 失败后提示「server 重启中，请重试」 |
| B4 进度假死 | UI 卡在某个百分比 | 心跳机制：每批完成更新进度；超过 5 秒无响应提示「检查 dev server 状态」 |
| B5 空 LocalStorage | 全新浏览器无数据 | 扫描结果为空时直接跳过弹层，不骚扰用户 |
| B6 部分失败 | 1000 条中 N 条失败 | 失败列表显示（key + 错误原因）；用户决定「重试失败批次」或「忽略完成」 |

**C. 业务风险**

| 风险 | 详情 | 缓解 |
| --- | --- | --- |
| C1 sessionStorage vs localStorage 混淆 | `alc:runtime-mode` 在 sessionStorage（`runtime-mode-store.ts:37`） | 迁移工具只扫 `window.localStorage`；明确文档说明「sessionStorage 数据不迁，因为是会话级临时状态」 |
| C2 Zustand persist version 字段 | 每个 store 有 `version`，migrate 函数依赖此值 | 迁移时保留原始 value 字符串不变（不重新 stringify），version 字段天然保留 |
| C3 showcase 数据混入 | 用户在 showcase 模式访问过，浏览器残留 showcase module | 详情页默认勾选「过滤 showcase origin」（§4.6.1 决策点） |
| C4 attempts 全局表残留 | `alc:state:attempts` 内含跨 module 的 slot（见 `quota.ts:116`） | namespace `state` 单独处理，按 moduleId 前缀过滤，匹配已选 namespace 集合 |

**D. Server 风险**

| 风险 | 详情 | 缓解 |
| --- | --- | --- |
| D1 迁移接口幂等性 | 同批数据 POST 多次 | `INSERT OR REPLACE INTO kv`，同 key 覆盖；幂等 |
| D2 迁移日志 | 出问题无据可查 | server 追加写 `data/migration.log`：每批 `[timestamp] batch=N, namespace=X, count=M, status=ok/fail, sha256=...` |
| D3 事务边界 | 单批失败时已写入数据 | 每批一个 SQLite 事务（`BEGIN ... COMMIT`），失败 `ROLLBACK` 该批；不影响其他批 |
| D4 数据库连接竞争 | 迁移写入期其他 API 调用 | bun:sqlite 同步 + dev server 单进程，无竞争 |

**E. 后续风险**

| 风险 | 详情 | 缓解 |
| --- | --- | --- |
| E1 LS 与 SQLite 共存的困惑 | 「永不删除 LS」但每次启动扫 LS 烦人 | 迁移成功后写 `localStorage['alc:migrated-at'] = Date.now()`；client 启动时该标记存在则跳过扫描 |
| E2 重复提示骚扰 | 用户点「稍后」后反复提示 | `localStorage['alc:migration-prompt-dismissed-at']` 记录；7 天内不再提示，7 天后再次提示 |
| E3 「迁移成功」判定模糊 | 条数对 ≠ 业务可用 | 三层校验：①条数对齐 ②所有 key+value SHA-256 总和对齐 ③业务校验（每个 Module 通过 `loadStoredModule` 能正常加载且 concepts 非空） |
| E4 撤销迁移 | 用户后悔想回 LocalStorage | 迁移前自动全库导出（§4.6.1 决策点）+ 设置页提供「从备份恢复到 LocalStorage」入口（仅当备份存在时显示） |
| E5 APP_MODE 切换混乱 | 用户从 production 切到 showcase 部署，LocalStorage 数据还在 | 文档说明「两种模式数据独立」；不自动同步 |

#### 4.6.3 安全网（汇总）

1. **LocalStorage 永不自动删除**（避免任何"自动清理导致丢失"）
2. **迁移前自动全库 JSON 备份**（落到 `data/backup/`）
3. **每批事务原子性**（失败可回滚该批，不影响其他批）
4. **幂等性**（同 key 覆盖，可反复触发）
5. **三层校验**（条数 + SHA-256 + 业务加载）
6. **完整日志**（`data/migration.log` 每批记录）

### 4.7 数据库文件位置

**位置**：`<project-root>/data/alc.db`

### 4.8 API Route 守卫

`/api/data/*` 与 `/api/migrate` **仅在 production APP_MODE 下注册**：

```typescript
// src/app/api/data/[...key]/route.ts
import { isShowcaseMode } from '@/lib/runtime/app-mode'

export async function GET() {
  if (isShowcaseMode) {
    return new Response('Not Found', { status: 404 })
  }
  // ... 实际逻辑
}
```

**为什么不用文件级跳过**（如 conditional route）：Next.js App Router 不原生支持「按环境变量注册路由」，最干净的方式是每个 handler 内部守卫。Vercel 部署时这些路由的 bundle 仍存在，但调用永远返回 404，等于功能等价。

### 4.9 JSON 导出/导入完整格式设计

现有 `module-package.ts` 已定义 `CompiledModulePackage v1` 格式，用于单 Module 分享。本次升级必须**保留向后兼容**，同时新增全库备份格式。

#### 4.9.1 三种导出场景

| 场景 | 格式 | 触发位置 | 范围 |
| --- | --- | --- | --- |
| 单 Module 分享 | `CompiledModulePackage v1`（现有，保留） | Library 页 → 单 Module 菜单 | source + module + qualityReport + 可选 attempts/feynman/progress（v2 增强） |
| 全库备份 | `BackupPackage v1`（新增） | 设置页 → 数据管理 → 备份；迁移前自动触发 | 所有 `alc:*` localStorage 数据 |
| 调试导出（可选） | `DebugDump`（新增，开发期） | 隐藏入口或环境变量触发 | 全库 + 运行时元数据 + 最近 N 条事件 |

#### 4.9.2 BackupPackage v1 格式规范

```typescript
interface BackupPackage {
  /** 格式版本，未来变更时识别 */
  version: 1
  /** 导出方标识，防止误读其他工具 JSON */
  exportedBy: 'ai-learning-compiler'
  /** 导出时间戳（Date.now()） */
  exportedAt: number
  /** 来源 APP_MODE，标识是 production 模式导出 */
  appMode: 'production' | 'showcase'
  /** SQLite schema 版本（迁移版本管理用） */
  schemaVersion: number
  /** 数据条目数组 */
  entries: Array<{
    /** 完整 key，如 "alc:module:abc123" */
    key: string
    /** JSON.parse 后的原始对象（不重新序列化，避免精度漂移） */
    value: unknown
    /** namespace（从 key 解析） */
    namespace: string
  }>
  /** 元数据 */
  meta: {
    /** Module 数量 */
    moduleCount: number
    /** 总字节数（UTF-8 编码后） */
    totalBytes: number
    /** SHA-256 of entries sort by key 后拼接的 key+value 字符串 */
    checksum: string
    /** 导出工具版本（package.json version） */
    appVersion: string
  }
}
```

**文件命名规范**：
- 自动备份：`data/backup/alc-backup-YYYYMMDD-HHmm.json`
- 用户触发：`alc-backup-YYYYMMDD-HHmm.json`（浏览器下载）
- 迁移前兜底：`data/backup/alc-pre-migration-YYYYMMDD-HHmm.json`

#### 4.9.3 单 Module 导出 v2 增强（向后兼容）

现有 `CompiledModulePackage v1` 只含 `source + module + qualityReport`，**丢失了学习历史**。本次升级到 v2（保留 v1 解析能力）：

```typescript
interface CompiledModulePackageV2 {
  version: 2
  exportedBy: 'ai-learning-compiler'
  exportedAt: number
  source: KnowledgeSource
  module: Module
  qualityReport?: unknown
  generatedBy?: { provider?: string; model?: string; generatedAt?: number }
  /** v2 新增：学习历史（可选，老 v1 文件没有这些字段） */
  history?: {
    attempts?: unknown      // StorageKeys.attemptsModule(moduleId)
    feynman?: unknown       // StorageKeys.feynman(moduleId)
    progress?: unknown      // StorageKeys.progress(moduleId)
    mastery?: unknown       // StorageKeys.mastery(moduleId)
  }
}
```

**解析策略**：
- `version === 1` → 用 v1 schema 解析（兼容现有导出文件）
- `version === 2` → 用 v2 schema 解析
- 其他版本 → 拒绝并提示「不支持的导出版本」

#### 4.9.4 导入校验三层

任何导入（单 Module 或全库）必须通过：

1. **结构校验**（Zod schema）：字段齐全、类型正确
2. **完整性校验**（checksum）：SHA-256 比对（仅 BackupPackage）
3. **安全校验**（沿用现有）：JSON 文本不含 `"apiKey"` 字符串
4. **业务校验**：每个 Module 能通过 `loadStoredModule` 加载且 concepts 数组非空

任一层失败：返回具体错误，不写入任何数据（事务性）。

#### 4.9.5 导入冲突策略

**单 Module 导入**（沿用现有）：ID 重分配（`assignLocalModuleIdentity`），不冲突。

**全库导入**（新增 BackupPackage）：

| 模式 | 行为 | 适用场景 |
| --- | --- | --- |
| 覆盖（默认） | 同 key 直接覆盖 | 灾难恢复（用备份替换当前数据） |
| 合并 | 同 key 时取 `updated_at` 较新者 | 跨设备合并（v1.0.0 不做但保留接口） |
| 跳过 | 同 key 不动 | 仅导入新增数据 |

UI 提供 3 个 radio 选项，默认「覆盖」。

#### 4.9.6 大文件导出处理

| 文件大小 | 策略 |
| --- | --- |
| < 50MB | 单次响应，浏览器直接下载 |
| 50MB - 500MB | 分页 API（`/api/data/export?cursor=xxx`）+ 客户端流式组装 |
| > 500MB | NDJSON 流式响应（每行一个 entry）；UI 提示「数据量极大，建议直接备份 `data/alc.db` 文件」 |

**v1.0.0 实现范围**：仅做 < 50MB 单次响应；分页与 NDJSON 列入 v1.1.0 候选。

#### 4.9.7 安全过滤清单

导出前必须移除的字段（防止泄漏）：

| 字段 | 来源 | 处理 |
| --- | --- | --- |
| `apiKey` | settings-store | 直接拒绝导出（沿用现有 `parseModulePackage` 安全检查） |
| `DEEPSEEK_API_KEY` / `GLM_API_KEY` / `SENSENOVA_API_KEY` | env-config 缓存 | 同上 |
| 浏览器 cookie / session | （不应出现在 alc:* 数据中） | 防御性扫描，发现即拒绝 |

**实现**：导出前对 `JSON.stringify(pkg)` 整体扫描敏感关键词列表，命中即拒绝并提示用户检查数据。

#### 4.9.8 schema 版本演进策略

```
BackupPackage / CompiledModulePackage
    ├─ v1（现有 / 本次新增）
    ├─ v2（未来字段扩展）
    └─ vN+1（破坏性变更）
```

**演进规则**：
- 字段新增 → minor 版本号递增（v1 → v2 可向后兼容）
- 字段删除/语义变更 → major 版本号递增（v2 → v3 不兼容）
- 解析时识别版本号，< 当前版本走升级函数链（`migrateV1toV2(pkg)`）
- 旧版本 JSON 在 N 个版本内必须能解析（N = 3）

## 6. 关键技术风险与权衡

| 风险 | 影响 | 缓解策略 |
| --- | --- | --- |
| R1：write-behind 在浏览器异常关闭时数据丢失 | production 模式最近一次写丢失（毫秒-秒级） | `beforeunload` + `sendBeacon` flush；关键操作 await flush；bun:sqlite server 端同步落盘 |
| R2：首次启动加载延迟（全量拉取到内存） | < 5MB < 200ms；< 50MB 约 1-2s | 骨架屏 UI；分批拉取（settings → progress → module → attempts） |
| R3：内存占用上升（全量 cache） | 大数据量常驻内存 | 监控阈值；超 100MB 改 LRU 策略（按 namespace 懒加载） |
| R4：dev server 重启时数据库连接断开 | 重启期间 fetch 报错 | 重连逻辑 + UI 网络错误提示（dev 下用户直接看到 server 状态） |
| R5：迁移工具误删 LocalStorage | 用户历史丢失 | 设计上**永不自动删**；只提供「确认后手动清空」入口 |
| R6：APP_MODE 守卫被遗漏（某处 import 没 branch） | showcase 模式行为意外变化 | 单元测试覆盖两模式；e2e 在 showcase 模式下跑 smoke 测试确保 0 回归 |
| R7：内存 cache 与 server 数据不一致（外部修改 SQLite） | 用户手动改了 `data/alc.db` | 提供「重新加载」按钮清空 cache；文档提示不要在 dev server 运行时手动改 db |
| R8：bun:sqlite 在 Next.js dev 热重载下连接泄漏 | 句柄累积 | 全局单例 + `globalThis.__alcDb` 守卫（参考 prisma 在 dev 模式标准模式） |
| R9：showcase 部署误把 `/api/data/*` 调用成功 | 安全漏洞 | 每个路由 handler 入口 `if (isShowcaseMode) return 404`；CI 测试覆盖 |
| R10：迁移期间损坏 key 污染 SQLite | 数据完整性破坏 | 跳过 JSON.parse 失败的 key + 日志记录；不强行搬运损坏数据（§4.6.2 A1） |
| R11：单批迁移体量超 Next.js body 限制 | 迁移中途失败 | 每批 ≤ 100 条或 ≤ 1MB（先达到即提交），不依赖默认 body 限制（§4.6.2 A2） |
| R12：Unicode/emoji 在 SQLite 中行为差异 | 中文/特殊符号乱码 | 测试套件覆盖 emoji、零宽字符、CJK 标点；SQLite TEXT 默认 UTF-8 与 LS 一致（§4.6.2 A3） |
| R13：关联数据迁移断裂（Module 迁了 attempts 没迁） | 学习历史丢失 | 按 namespace 整批扫描，强制七关联配套；事务边界按 namespace（§4.6.2 A7） |
| R14：迁移期间用户继续操作 | 数据状态不一致 | 全屏 modal + 进度条锁住 UI；底层不接收交互（§4.6.2 B1） |
| R15：网络中断导致迁移半完成 | 部分数据已写入 SQLite | 分批提交 + 断点续传 + 「重试失败批次」UI（§4.6.2 B2） |
| R16：showcase 数据误迁入 production SQLite | 数据污染 | 默认勾选「过滤 showcase origin」；用户可选关（§4.6.1 决策点 + C3） |
| R17：sessionStorage 数据误迁 | runtime-mode 状态混乱 | 迁移工具只扫 `window.localStorage`；明确文档说明（§4.6.2 C1） |
| R18：Zustand persist version 字段被覆盖 | store hydrate 异常 | 直接搬运原始字符串，不重新 stringify（§4.6.2 C2） |
| R19：导出文件含 apiKey 等敏感字段 | 隐私泄漏 | 沿用现有 `"apiKey"` 检查 + 扩展敏感关键词列表（§4.9.7） |
| R20：旧版本 v1 导出文件无法导入 | 用户历史数据不可用 | v1 + v2 解析能力并存；版本演进策略（§4.9.8） |
| R21：单 Module 导出丢失 attempts/feynman 历史 | 学习进度无法迁移 | module-package 升级 v2 含 history 字段（§4.9.3） |
| R22：全库导入冲突覆盖现有数据 | 误操作丢数据 | UI 强制选择冲突策略（覆盖/合并/跳过）；默认覆盖但需二次确认（§4.9.5） |
| R23：导入部分失败时已写入数据 | 数据状态不一致 | 整个导入单事务，部分失败 ROLLBACK（§4.9.4） |
| R24：迁移完成后 LS 与 SQLite 共存导致混乱 | 用户重复迁移 | `localStorage['alc:migrated-at']` 标记位；启动时跳过扫描（§4.6.2 E1） |
| R25：用户点「稍后」后被反复提示骚扰 | 体验问题 | `alc:migration-prompt-dismissed-at` 标记；7 天内不再提示（§4.6.2 E2） |
| R26：迁移「成功」但业务数据不可用 | 假阳性成功 | 三层校验：条数 + SHA-256 + 业务加载（§4.6.2 E3） |
| R27：用户后悔想撤销迁移 | 不可逆操作 | 迁移前自动全库备份；设置页提供「从备份恢复到 LS」入口（§4.6.2 E4） |
| R28：导出大文件 OOM | server 崩溃 | < 50MB 单次响应；> 50MB 分页；> 500MB NDJSON + 提示直接拷贝 db 文件（§4.9.6） |

## 9. 待后续 Spec 确认的开放问题

**架构层**：
1. 内存 cache 数据结构：`Map<string, unknown>` 还是 `Map<string, string>`（延迟反序列化）
2. write-behind 的 flush 时机清单（哪些事件触发强制 await flush）
3. Bulk 拉取分批策略（一次拉 vs 按 namespace 拉）
4. 启动加载 UI 形态（全屏骨架 vs AppShell 内 spinner）
5. `sendBeacon` 失败时的兜底
6. 是否需要「数据库诊断」页面（开发期调试）
7. APP_MODE 切换时的状态处理（用户从 production 切到 showcase 部署时，LocalStorage 与 SQLite 数据如何同步）

**迁移层**：
8. 迁移分批大小阈值：100 条/批 vs 1MB/批，先达到即提交的策略是否最优
9. 迁移失败批次的「最大重试次数」（避免无限重试）
10. 迁移日志保留策略（永久 vs 滚动保留最近 N 次）
11. 「从备份恢复到 LocalStorage」是否真有必要做（迁移撤销场景），还是仅保留备份 JSON 文件作为冷备
12. 迁移工具的「过滤 showcase origin」默认值是否合理（用户可能想保留某些 showcase 示例）

**导出/导入层**：
13. BackupPackage checksum 算法：`entries.sort(by key).map(e => sha256(key+value)).join()` 还是 merkle 树
14. 单 Module v2 的 history 字段是否对现有 v1 导入流程产生回归（导入 v1 文件时 history 字段如何填充）
15. 全库导入的默认冲突策略（覆盖 / 合并 / 跳过）：覆盖风险高但简单，合并需要 `updated_at` 比对
16. 大文件导出阈值：50MB / 500MB 是否合理，需要实测
17. 导入文件大小上限：是否设硬限制（避免用户误导入超大文件导致 OOM）
18. 导出/导入是否需要进度 cancel 按钮（中断长操作）

**安全层**：
19. 敏感字段扫描的关键词列表：`apiKey` / `api_key` / `secret` / `password` / `token` 哪些必须拦截
20. 备份 JSON 是否需要加密（用户本地落盘的备份文件，明文存储是否可接受）

**运维层**：
21. `db:backup` 保留策略：永久 vs 滚动保留最近 N 个 vs 按时间窗口（保留近 7 天 + 近 4 周）
22. APP_MODE 误配置的处理（用户 dev 时忘了设 production，或 Vercel 误设 production）

## 10. 结论与下一步

**核心结论**：

> 升级 production 模式（本地 dev）的持久化到 Bun 内置 SQLite，**showcase 模式（Vercel 部署）零改动**。
>
> 关键设计：
> - `StorageRepository` 单例按 `isShowcaseMode` 选择实现：showcase 用 LocalStorage，production 用 fetch+SQLite
> - 所有新 API 路由（`/api/data/*`、`/api/migrate`）入口按 `isShowcaseMode` 守卫，showcase 调用永远返回 404
> - 12 题限制常量按 `isShowcaseMode` 取值，showcase 保留 12，production 取消上限
> - Client 端通过内存 cache + write-behind 桥接同步接口与异步 fetch，**5 个 Zustand store 内部 0 改动**
> - LocalStorage 旧数据一键迁移到 SQLite，永不自动删除（安全网）
