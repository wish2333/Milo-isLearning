## 2. 架构决策

### 2.1 origin 标记 + 运行时模式过滤

**问题**：展示模式和实用模式共享同一个 `alc:module:*` LocalStorage 池。展示加载的题库与用户编译的题库混在一起，无法区分。

**方案**：Module 和 Topic 新增 `origin?: ContentOrigin` 字段（`'showcase' | 'user'`）。展示加载时通过 `importModulePackage(storage, pkg, { origin: 'showcase' })` 打标。`listStoredModules` / `getStorageCapacitySummary` / 库页 topic 列表全部按 `isShowcaseMode && !studioMode` 过滤——展示模式只看 showcase，实用模式只看 user。

**向后兼容**：旧数据无 `origin` 字段，视为 `undefined`，在实用模式可见、展示模式不可见。编译产物不经 `importModulePackage`，天然为 undefined。

### 2.2 展示主题直接加载 .alc-topic.json（废弃手动拆分）

**问题**：M8.1 的 showcase-loader 设计为主题条目引用多个独立 `.alc-module.json` 文件（`modulePackages` 方案）。实际上主题包是一个完整的 `.alc-topic.json` 文件，手动拆分到 manifest 既繁琐又容易出错（本会话初期只提取了 modules[0]，丢失了另外 2 个模块）。

**方案**：manifest topic 条目改为 `package: "showcase-x.alc-topic.json"` + `moduleCount: N`。showcase-loader 直接 fetch 完整 `.alc-topic.json`，用 `parseTopicPackage` 校验，`importTopicPackage` 导入。用户可以直接把 studio 模式导出的 `.alc-topic.json` 复制到 `public/showcase-modules/` 并注册到 manifest。

### 2.3 去重机制

**问题**：每次点击展示卡片都调用 `importModulePackage`（分配新 nanoid）+ `createTopic`（创建新主题），导致重复加载、副本堆积、容量浪费。

**方案**：加载前检查已存储的展示数据——模块按 `title + origin === 'showcase'` 匹配，主题按 `name + origin === 'showcase'` 匹配（用包内的 `pkg.topic.name`）。已有则直接返回旧副本，不产生新 ID、不消耗容量。

### 2.4 Studio 路由：运行时模式覆盖

**问题**：`APP_MODE` 是构建时常量。在 showcase 部署中，所有页面的 `isShowcaseMode` 恒为 `true`。`/studio` 只是孤岛首页——从它导航到 `/settings`、`/learn/library` 全部走 showcase 逻辑。

**方案**：引入 `runtime-mode-store`（Zustand + sessionStorage）。`/studio` mount 时 `enterStudio()` 设置 `studioMode = true`。所有模式判断从 `isShowcaseMode` 升级为 `isShowcaseMode && !studioMode`。

**为什么用 sessionStorage 而非 localStorage**：studioMode 只在当前 tab 有效。新 tab 从 showcase 首页开始，避免跨 tab 污染。同一 tab 内导航（/studio → /learn/library → /settings）保持 studio 上下文。

**为什么不自动退出**：访问 `/` 时如果自动 `exitStudio()`，会导致 ShowcaseHome → ProductionHome → ShowcaseHome 的渲染闪烁。用户通过 ProductionHome 上的「返回展示首页」按钮显式退出。

### 2.5 页面级 hydration 守卫

`/` 和 `/settings` 转为 Client Component 后，SSR 期间 Zustand sessionStorage 尚未 hydrate。使用 `useHydrated()` 守卫：hydration 前 `effectiveShowcase = isShowcaseMode`（构建时默认值，避免 SSR mismatch），hydration 后 `effectiveShowcase = isShowcaseMode && !studioMode`（运行时值）。

## 5. 迁移说明

### 5.1 manifest v2 格式变更

**旧的 topic 条目格式（modulePackages）已废弃**：

```json
// 旧格式（不再支持）
{
  "modulePackages": [{ "package": "x.alc-module.json", "title": "..." }]
}
```

**新格式（package + moduleCount）**：

```json
{
  "package": "showcase-x.alc-topic.json",
  "moduleCount": 3
}
```

### 5.2 origin 字段

Module 和 Topic 类型新增 `origin?: ContentOrigin`。已有代码无需修改——字段可选，undefined 在实用模式可见。仅在以下场景需显式传递：

- `importModulePackage(repo, pkg, { origin: 'showcase' })` — 展示加载
- `importTopicPackage(repo, pkg, { origin: 'showcase' })` — 展示主题加载
- `createTopic(name, desc, moduleIds, 'showcase')` — 展示主题创建

### 5.3 页面转 Client Component

`/`（page.tsx）和 `/settings`（settings/page.tsx）从 Server Component 转为 Client Component。这意味着这两个页面不再 SSR 预渲染（但 ShowcaseHome / ProductionHome / ShowcaseSettings / ProductionSettings 内部本身已是 Client Component，实际影响极小）。
