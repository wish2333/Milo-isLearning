## 0. 结论

M8.1 已完成，全部静态验证与单元测试通过。项目从「扁平题库列表 + 单库独立学习」扩展为「主题分组 + 跨库顺序编排 + 聚合错题重刷」的三层能力体系。

核心变化：

1. **数据层扩展**：新增 `Topic` 领域模型，一对多关联题库，全量存储在 `alc:topic-index`；新增 `TopicSession` 持久化会话，支持刷新恢复。
2. **编排层无侵入**：主题层在 `ModuleStage` 状态机**之上**编排——每个题库仍走完整的 module_intro → concept → challenge → feynman → done 流程，主题层只在 done 阶段拦截，决定跳过渡页还是 `/learn/done`。
3. **聚合重刷**：review-store 从单库 `Quiz[]` 队列扩展为多库 `ReviewQueueItem[]`（带 moduleId），新增 `startTopicSession(topicId, filter)` 支持跨库错题/蒙对题聚合。
4. **独立蒙对重刷**：新增 `ReviewFilter`（all / wrong / guessed），review 页筛选 tab + URL query param 驱动。
5. **主题导出/导入**：`.alc-topic.json` 格式，内含 topic 元数据 + 有序 `CompiledModulePackage[]`，导入时重新分配所有 ID + 创建本地 Topic。
6. **展示模式兼容**：manifest v2 新增可选 `topics` 数组，v1 向后兼容（无 topics → 空数组）。

## 2. 架构决策

### 2.1 主题层不侵入 ModuleStage 状态机

主题编排严格在状态机**之上**运行。每个题库仍走完整的 7-state discriminated union 流程（module_intro → concept → challenge → feynman_* → done）。主题层唯一的介入点是 `module/[id]/page.tsx` 的 done 重定向 useEffect——检测 `useTopicSessionStore.getState().session` 决定跳过渡页还是 `/learn/done`。

这意味着：
- 不创建主题时，所有 M7.8~M8 行为完全不变
- 主题功能是纯增量，可以安全回退（删除主题相关文件 + 恢复 done 重定向即可）

### 2.2 一对多归属 + 全量存储

一个题库最多归入一个主题（`enforceExclusiveMembership` 在 createTopic / addModuleToTopic 时自动从其他主题移除）。全量 `Topic[]` 存储在单个 `alc:topic-index` key 中，12 个题库上限下单次读写完整列表，无需索引/实体分离。

### 2.3 会话 moduleIds 快照

`startSession` 时快照 `topic.moduleIds` 到 `session.moduleIds`。用户中途在库页调整主题模块顺序不影响进行中的会话——已完成的模块状态不受影响，修改顺序下次会话生效。

### 2.4 ReviewQueueItem 替代裸 Quiz[]

review-store 的 `queue` 从 `Quiz[]` 改为 `ReviewQueueItem[]`（`{ quiz, moduleId, slotId }`），使跨库聚合成为可能。每个 queue item 自带 moduleId，答题后记录 attempts 时能正确路由到对应模块。recordResult / nextQuestion 签名不变，向后兼容。

### 2.5 topic-package 复用 module-package 校验

`CompiledTopicPackage` 的每个模块通过 `parseModulePackage(JSON.stringify(moduleObj))` 逐个校验（parseModulePackage 接受 string 参数）。导入时逐个 `importModulePackage` 分配新 ID，再用 `createTopic` 关联。安全规范与 `.alc-module.json` 一致（拒绝 apiKey）。

### 2.6 展示模式 manifest v2 向后兼容

`ShowcaseManifest.topics` 为可选字段。v1 manifest（无 topics）→ `listShowcaseTopics()` 返回 `[]`，ShowcaseHome 不渲染主题区块，行为与 M8 一致。

## 4. 已知限制

| # | 限制 | 影响 | 缓解 |
|---|------|------|------|
| 1 | **E2E / Playwright 未覆盖主题流程** | 主题创建/刷题/导出/导入的端到端路径未被自动化测试覆盖 | 运维团队手动验证或补充 e2e 用例 |
| 2 | **展示模式无示例主题内容** | manifest v2 `topics: []` 为空（基础设施就绪，无示例内容） | 后续编译 2-3 个题库组成示例主题（如「前端基础」）填入 manifest |
| 3 | **主题进度不跨会话恢复** | exitSession 后再次「开始主题学习」重新初始化 all pending（Must 项不恢复） | Should 项可基于 attempts-store 判断哪些模块已 done |
| 4 | **无拖拽排序** | 模块在主题中的排序用上/下箭头按钮 | 12 个题库上限内箭头足够；V1.2+ 可加拖拽 |
| 5 | **无多级主题嵌套** | 只有一级主题，不支持主题树 | V1.2+ 评估需求 |
| 6 | **LocalStorage 无事务** | 极端情况下两个 tab 同时操作 topic-index 可能不一致 | UI 层单次操作原子读写全量列表；概率极低 |
| 7 | **过渡页 mastery 为粗粒度** | Must 项只显示完成/未完成状态（✓/●/○），不显示 mastery 百分比 | Should 项可通过 computeMastery 聚合展示 |
| 8 | **lint 修复：`module` 变量名** | Next.js `no-assign-module-variable` 规则禁止变量名 `module`，已统一改用 `mod` | 无功能影响 |

## 5. V1.2 衔接建议

1. **主题进度恢复**：基于 attempts-store 判断哪些模块已完成，startSession 时自动标记 done 模块、从下一个 pending 模块开始。
2. **聚合 mastery 展示**：过渡页和主题完成页展示各模块 mastery 的加权平均。
3. **过渡页「跳过此题库」选项**：允许用户不学习当前模块直接标记 done 进入下一个。
4. **展示模式示例主题**：编译 2-3 个高质量题库（如 HTML/CSS/JS 入门），登记到 manifest v2 的 topics 数组。
5. **拖拽排序**：如果题库数量增长，评估引入 drag-and-drop reorder。
6. **主题级 LLM 定制**：为整个主题生成综合挑战题（跨模块知识整合）。
