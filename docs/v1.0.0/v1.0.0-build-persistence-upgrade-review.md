## 2. 高优先级问题（建议合入前修复）

### H1. `write-queue.ts` 的 `sendingKey` 是 dead code

**位置**：`src/lib/persistence/client/write-queue.ts:38, 136-138, 162`

字段 `private sendingKey: string | null = null` 在 line 38 定义，line 162 finally 设 `this.sendingKey = null`，但**整个文件没有任何位置赋值** `this.sendingKey = task.key`。因此 line 136-138 的 `if (this.sendingKey !== null && ...)` 检查永远 false。

**影响**：低（只是冗余代码，不影响功能）。
**修复**：删除 sendingKey 字段和 line 136-138 死代码块；或者补全赋值（在 line 123 `task.status = 'sending'` 后加 `this.sendingKey = task.key`），让 in-flight 合并真正生效。

### H2. `client-fetch-storage.ts` 的 `clearAll()` 效率低

**位置**：`src/lib/persistence/client/client-fetch-storage.ts:106-112`

```typescript
clearAll(): void {
  for (const key of this.cache.keys()) {
    this.queue.enqueue(key, null)  // 入队 N 个 DELETE
  }
  this.cache.clear()
}
```

逐 key 入队 delete，N 条数据会发 N 个 HTTP DELETE 请求，占用写队列。Phase 1.B 的 `/api/data/clear` route 已实现「单 SQL `DELETE FROM kv` 一次性清空」。

**影响**：中。production 模式清空大库时显著慢。
**修复**：直接调用 `/api/data/clear` 一次性清空。可扩展 WriteQueue 支持「全局清空」语义，或拆出独立 async 方法 `serverClearAll()`。

### H3. `restore/route.ts` 的 updatedAt 丢失

**位置**：`src/app/api/data/restore/route.ts:87-90`

```typescript
for (const entry of pkg.entries) {
  repo.setRaw(entry.key, entry.valueRaw)  // setRaw 内部用 Date.now() 覆盖 updated_at
  restored++
}
```

`ServerSQLiteRepository.setRaw`（line 34-41）写 kv 时用 `Date.now()` 作为 `updated_at`，覆盖了 BackupPackage entry 自带的 `updatedAt`。这导致灾难恢复后所有数据的 updated_at 全部变成「恢复时刻」。

**影响**：中。v1.0.0 评审 §6.2 把「按 updatedAt 自动 merge」延后到 v1.0.1，所以 v1.0.0 不依赖 updatedAt 排序——可接受。但是数据保真度有损，未来 v1.0.1 实现 merge 时会回填错误的 updatedAt。

**修复**：扩展 `setRaw` 签名支持可选 `updatedAt`，或在 restore route 用直接 SQL `INSERT OR REPLACE ... VALUES (?, ?, ?, ?)` 显式指定 updated_at。

## 3. 中优先级问题（v1.0.1 跟进）

### M1. `[...key]/route.ts` 的 `isSameOrigin` 安全漏洞

**位置**：`src/app/api/data/[...key]/route.ts:39`

```typescript
function isSameOrigin(req: NextRequest): boolean {
  const origin = req.headers.get('origin')
  if (!origin) return true  // ← 无 Origin 时放行
  ...
}
```

`Origin` header 不是浏览器必发的（同源 GET 通常不发；某些 redirect 场景不发）。攻击者用 curl、server-side fetch、modified fetch 都可以构造无 Origin 的请求绕过校验。

**实际影响**：v1.0.0 production 模式仅本地 dev，dev server 监听 localhost，物理隔离。但是代码层面应该更严格。

**修复**：
- 写操作（PUT/DELETE/POST）必须有 Origin 且同源，否则 403
- 或加 `Sec-Fetch-Site: same-origin` 检查（现代浏览器都发）
- 或加 CSRF token

### M2. `write-queue.ts` 的 `flushNow` 用 10ms 轮询

**位置**：`src/lib/persistence/client/write-queue.ts:69-80`

```typescript
async flushNow(): Promise<void> {
  while (this.processing || this.pending.size > 0) {
    if (this.processing) {
      await new Promise((r) => setTimeout(r, 10))  // ← 10ms 轮询
    } else if (this.pending.size > 0) {
      await this.processNext()
    }
  }
}
```

10ms 轮询精度低，CPU 浪费。当 server 慢（每次请求 1s+）时，flushNow 会浪费 100 次空轮询。

**修复**：用 Promise resolver 替代轮询。在 WriteQueue 加 `private resolveFlushWaiters: Array<() => void> = []`，enqueue 和 processNext 的 finally 检查 idle 时通知所有 waiter。

### M3. `app-mode.ts` 的默认值与「fail-closed」精神不符

**位置**：`src/lib/runtime/app-mode.ts:13-14`

```typescript
export const APP_MODE: AppMode =
  process.env.NEXT_PUBLIC_APP_MODE === 'showcase' ? 'showcase' : 'production'
```

注释说「未设 → production」，代码也是。但是评审 §0.2 fail-closed 双开关的精神是「默认不启用危险功能」。

实际影响：
- 开发者 clone 项目后直接 `bun run dev`（不设任何 env），看到 ProductionHome
- ProductionHome 调 `/api/data/status` → 因 ALC_STORAGE_BACKEND 未设 → 404 → 显示「数据库未启用」错误
- 用户体验差

**这是 pre-existing 行为**，不是 v1.0.0 引入。但是 v1.0.0 让这个默认值的后果变得更明显（之前 production 模式 = user mode 不依赖 server，现在依赖 server）。

**建议**（v1.0.1 跟进）：默认改为 showcase：

```typescript
export const APP_MODE: AppMode =
  process.env.NEXT_PUBLIC_APP_MODE === 'production' ? 'production' : 'showcase'
```

但这会改变现有 e2e 行为，需要同步调整 playwright.config.ts。

### M4. `progress-store.ts` 双写在 production 模式的额外开销

**位置**：`src/lib/state/progress-store.ts:347-366`（subscribe 双写）

Phase 0 的 P0.4 加注释说 progress-store 同时写：
- `alc:state:progress`（Zustand persist blob，6 store 之一）
- `alc:progress:{moduleId}`（per-module 副本，给 listStoredModules 用）

在 showcase 模式，两者都写 LocalStorage，OK。
在 production 模式，两者都通过 `ClientFetchStorageRepository` 入队 WriteQueue，每次 progress 变更发**两次** PUT（key 不同），多一倍 SQLite 写入。

**影响**：中。production 模式下频繁 progress 更新会产生 2x 写入。
**修复**（v1.0.1）：评估 listStoredModules 是否真的需要 per-module 副本——如果 SQLite 查询足够快，可以从 `alc:state:progress` 直接反序列化，不需要双写。

## 4. 低优先级 / 备注

### L1. `encodeURIComponent(key)` 多余地编码 `:`

`alc:module:m1` 编码后是 `alc%3Amodule%3Am1`。`:` 是 URL path 的合法字符，没必要编码。功能正确但 suboptimal。

### L2. `storage-initializer.ts:79-81` 的 TODO 已过时

```typescript
// 2. Phase 5 占位：检查 LocalStorage 迁移
// TODO(phase-5): 调用 checkAndMigrate()，如有迁移数据则触发迁移流程
// 本任务先不实现，避免阻塞 Phase 3
```

Phase 5.B 实际是通过 AppShell 挂载 `<MigrationOrchestrator>` 实现的，**没有**在 storage-initializer 内部调用。这个 TODO 注释应该删除或改为「迁移 UI 由 AppShell 的 MigrationOrchestrator 处理」。

### L3. `vitest-shims/bun-sqlite.ts` 是测试专用 hack

为了在 vitest（Node ESM loader）下用 better-sqlite3 替身跑 bun:sqlite 测试，加了：
- `query()` 方法（bun:sqlite 有，better-sqlite3 用 prepare）
- `run()` 方法（bun:sqlite 有，better-sqlite3 用 exec + prepare.run）
- `get()` 归一化（bun 返回 null，better-sqlite3 返回 undefined）

这个 shim 隐藏了两个库的真实差异。如果未来源码用了 shim 没覆盖的 bun:sqlite 特性（比如 `db.transaction()`、`db.prepare().iterate()`），测试会通过但生产可能炸。

**建议**：在 shim 文件顶部加明确警告注释，并在 CI 加 bun runtime 的 e2e 兜底。

### L4. MigrationOrchestrator 状态机的 cancelRequested 信号时机

`MigrationOrchestrator.tsx` 用 `cancelRequested` state 通知 `runMigration` 取消。但 React state 更新是 async 的——用户点「取消」后，state 还没传到 `runMigration` 的 `shouldCancel` 闭包，下一批上传可能已经开始。

**影响**：用户感知有 ~100ms 延迟才真正停止上传。可接受，但可改进用 `useRef` 替代 state。

## 10. 结论
