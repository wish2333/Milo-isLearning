## 2. MVP 收尾阶段适配性评估

### 2.1 评估标准

| 维度 | 权重 | 说明 |
|------|------|------|
| 核心闭环贡献 | 高 | 是否直接服务于 PRD 定义的核心体验闭环验证（Markdown -> 编译 -> Quiz -> 费曼） |
| 工作量 | 高 | 是否可在收尾阶段（1-3 天）完成 |
| 风险 | 高 | 是否可能引入新 bug 或破坏现有功能 |
| 用户感知 | 中 | 用户是否能直接感知到改善 |
| 上线阻塞 | 中 | 是否影响上线或核心指标衡量 |

### 2.2 适合 MVP 收尾阶段添加的功能（推荐）

> 以下 4 项功能满足"低风险、低工作量、直接提升上线质量"的标准，建议在 v1.0.0 最终收尾阶段实施。

#### R1: SEO metadata（OpenGraph） -- F05

| 维度 | 评估 |
|------|------|
| 工作量 | **小**（~30 分钟）—— 仅改 `src/app/layout.tsx` 的 `metadata` 导出 |
| 风险 | **极低** —— 纯静态 metadata，不影响运行时行为 |
| 用户感知 | **中** —— 分享链接时显示标题/描述/图片预览 |
| 上线阻塞 | 否，但影响分享传播 |
| 实现方案 | 在 `layout.tsx` 的 `metadata` 中添加 `openGraph`、`twitter`、`icons` 字段 |

```typescript
// 示例实现
export const metadata: Metadata = {
  title: 'AI Learning Compiler',
  description: '将任何知识自动编译为一条低摩擦、高掌握度的学习路径',
  openGraph: {
    title: 'AI Learning Compiler',
    description: '将任何知识自动编译为一条低摩擦、高掌握度的学习路径',
    type: 'website',
    locale: 'zh_CN',
  },
  twitter: {
    card: 'summary_large_image',
  },
}
```

#### R2: "蒙对"标注可撤销 -- F34

| 维度 | 评估 |
|------|------|
| 工作量 | **小**（~1 小时）—— FeedbackPanel 按钮状态管理 + attempts-store `unmarkGuessed` action |
| 风险 | **低** —— 仅 UI 交互层变更，不触及状态机或数据模型 |
| 用户感知 | **高** —— 减少误操作焦虑（PRD §15 Q6 明确建议"提供一次撤销机会"） |
| 上线阻塞 | 否，但 PRD 开放问题待闭环 |
| 实现方案 | FeedbackPanel 中标记后显示"撤销"链接（5 秒内可点击），调用 `attempts-store.unmarkGuessed(slotId)` |
| 前提条件 | 无 |

#### R3: 间隔重复配置开关 -- F12

| 维度 | 评估 |
|------|------|
| 工作量 | **小**（~2 小时）—— settings-store 新增 `confirmReviewEnabled: boolean` + ConceptView 条件渲染 |
| 风险 | **低** —— 复习题注入逻辑已有，仅加开关控制"确认掌握题"通道 |
| 用户感知 | **中** —— 部分用户可能觉得"确认掌握题"打扰，允许关闭提升体验 |
| 上线阻塞 | 否 |
| 实现方案 | `settings-store` 新增 `confirmReviewEnabled`（默认 true）；`adaptive-sequencer.collectConfirmSlots()` 检查该设置 |
| 前提条件 | 无 |

#### R4: API 路由服务端 env fallback -- F15

| 维度 | 评估 |
|------|------|
| 工作量 | **中**（~3 小时）—— `/api/feedback` 和 `/api/feynman-eval` 路由在 `llmConfig=null` 时 fallback 到 `process.env` |
| 风险 | **低** —— 仅在 config=null 时走 fallback 路径，不影响正常 BYOK 流程 |
| 用户感知 | **高**（展示模式）—— 展示模式下 FillBlank 语义兜底和费曼评分恢复正常 |
| 上线阻塞 | **是**（展示模式）—— 展示部署下 LLM 增强反馈完全不可用 |
| 实现方案 | 两个路由在 `llmConfig` 为 null/undefined 时，从 `process.env.DEEPSEEK_API_KEY` 等环境变量构造默认 config |
| 前提条件 | Vercel 环境变量已配置 `DEEPSEEK_API_KEY` |

> **注意**：F15 被标记为 M8 Review 的 P0 项。虽然它主要影响展示模式，但考虑到展示模式是 V1.0.0 公测发布的核心载体，建议在收尾阶段修复。

## 5. 风险评估

### 5.1 MVP 收尾阶段推荐功能的风险分析

| 功能 | 风险等级 | 风险描述 | 缓解措施 |
|------|---------|---------|---------|
| R1 SEO metadata | 极低 | 纯静态字段，不影响运行时 | 无需特殊缓解 |
| R2 蒙对撤销 | 低 | UI 状态管理可能引入边缘 case | 添加单测覆盖撤销逻辑 |
| R3 间隔重复开关 | 低 | 修改 adaptive-sequencer 逻辑 | 现有 20 个单测覆盖；新增开关默认 true 保持向后兼容 |
| R4 API env fallback | 低-中 | 需确保 fallback 不泄露 key | fallback 仅在 config=null 时触发；key 从 `process.env` 读取不落盘 |

### 5.2 不引入高风险功能的理由

> V1.0.0 正在收尾，331 个测试全绿、Vercel 构建通过。此时引入大工作量功能（如 True Stage Resume、主题进度恢复）的风险远大于收益：
>
> 1. **测试回归风险**：新功能可能破坏现有 331 个测试
> 2. **时间风险**：收尾阶段应聚焦验证和修复，不是开发
> 3. **范围蔓延风险**：一旦开始添加新功能，很难控制边界
> 4. **上线延迟风险**：每项新功能都需设计、实现、测试、验证，可能延迟上线
