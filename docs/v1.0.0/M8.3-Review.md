## 2. 架构决策

### 2.1 为什么 tsc 通过而 Vercel 构建失败

**问题**：`![a!, b!] = [b!, a!]` 这类在解构模式左侧使用非 null 断言 `!` 的写法，`tsc` 在宽松解析下放行，但 Next.js 15 的生产构建走 SWC/webpack 转译，对赋值模式（assignment pattern）的语法校验更严格，直接报 `Not a pattern`（Syntax Error）。

**方案**：解构赋值的目标必须是合法绑定模式（binding pattern），`!` 不属于绑定模式的一部分，只能出现在表达式位置（右侧）。因此把交换改成显式临时变量：

```typescript
const j = Math.floor(Math.random() * (i + 1))
const temp = result[i]!
result[i] = result[j]!
result[j] = temp
```

- 右侧 `result[j]!` 是合法表达式位置，`!` 仍用于压制 `noUncheckedIndexedAccess` 推导出的 `T | undefined`，类型安全未丢失。
- 临时变量交换与原始解构交换的运行时语义等价（均为两两互换），洗牌结果不受影响。

**为什么不复用 `Math.random()` 排序或第三方库**：Fisher–Yates 是现有实现，仅修正语法形态，不引入新依赖、不改动算法，影响面最小。
