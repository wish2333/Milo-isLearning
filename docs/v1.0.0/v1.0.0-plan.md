## 1. Task 1 — API 路由服务端 env fallback（R4 / F15）

### 1.1 问题

展示模式（`NEXT_PUBLIC_APP_MODE=showcase`）部署下，客户端无 `alc:settings` LocalStorage 配置。`EnvConfigLoader` 在展示模式下跳过 fetch `/api/env-config`（M8 设计决策），导致客户端 `settings-store.config` 为 `null`。

当学习展示题库时：
- FillBlank 语义兜底调用 `/api/feedback`，请求体 `llmConfig` 为 `null`
- 费曼 Step 6 评分调用 `/api/feynman-eval`，请求体 `llmConfig` 为 `null`

两个路由当前在 `llmConfig` 为 falsy 时直接返回 400 错误：

```typescript
// src/app/api/feedback/route.ts:53
if (!quiz || typeof userAnswer !== 'string' || !llmConfig) {
  return Response.json({ error: 'Missing required fields: ...' }, { status: 400 })
}
```

**影响**：展示模式下 FillBlank 语义兜底和费曼评分完全不可用。本地判分（Choice/Sorting/FillBlank 标准化匹配）正常工作，核心学习闭环不受影响，但 LLM 增强反馈缺失。

### 1.2 方案

在 `/api/feedback` 和 `/api/feynman-eval` 路由中，当 `llmConfig` 为 `null/undefined` 时，从服务端环境变量构造默认 config。复用 `/api/env-config/route.ts` 中已有的环境变量读取逻辑，提取为共享 helper。

### 1.3 实现步骤

#### Step 1.1：新建 `src/lib/providers/env-fallback.ts`

- [ ] 新建 `src/lib/providers/env-fallback.ts`，提取环境变量 fallback 逻辑

```typescript
/**
 * 服务端环境变量 fallback — 当客户端未携带 llmConfig 时构造默认配置
 *
 * 用于展示模式部署：客户端无 alc:settings，API 路由从 process.env 读取。
 *
 * 复用 /api/env-config/route.ts 的环境变量映射逻辑。
 * 安全约束：apiKey 仅从 process.env 读取，不落盘、不日志。
 */

import type { LLMConfig, ProviderKind } from './types'

const PROVIDER_DEFAULT_BASE_URL: Record<ProviderKind, string> = {
  deepseek: 'https://api.deepseek.com',
  glm: 'https://open.bigmodel.cn/api/coding/paas/v4',
  'openai-compat': '',
}

/**
 * 从服务端环境变量构造 LLMConfig。
 * 返回 null 表示环境变量未配置（无法 fallback）。
 */
export function getEnvLLMConfig(): LLMConfig | null {
  const provider = (process.env.DEFAULT_LLM_PROVIDER ?? 'deepseek') as ProviderKind
  const model = process.env.DEFAULT_LLM_MODEL ?? 'deepseek-chat'

  const apiKeyMap: Record<ProviderKind, string | null> = {
    deepseek: process.env.DEEPSEEK_API_KEY ?? null,
    glm: process.env.GLM_API_KEY ?? null,
    'openai-compat': process.env.OPENAI_COMPAT_API_KEY ?? null,
  }

  const apiKey = apiKeyMap[provider]
  if (!apiKey) return null

  const baseURLByProvider: Record<ProviderKind, string | undefined> = {
    deepseek: process.env.DEEPSEEK_BASE_URL,
    glm: process.env.GLM_BASE_URL,
    'openai-compat': process.env.OPENAI_COMPAT_BASE_URL,
  }
  const baseURL = baseURLByProvider[provider] ?? PROVIDER_DEFAULT_BASE_URL[provider]

  return { provider, apiKey, model, baseURL }
}
```

## 3. Task 3 — "蒙对"标注可撤销（R2 / F34）

### 3.1 问题

用户答对后可点击"其实是蒙对的"按钮标注 `guessed=true`。标注后按钮变为"✓ 已标记蒙对"（`disabled`），不可撤销。

PRD §15 Q6 建议"提供一次撤销机会，减少误操作焦虑"。

### 3.2 方案

1. `attempts-store` 新增 `unmarkGuessed(originalQuizId: string)` action，将最后一条 attempt 的 `guessed` 改回 `undefined`
2. `FeedbackPanel` 新增 `onUnmarkGuessed` prop；标注后显示"撤销"链接
3. `ConceptView` 和 `ChallengeView` 传入 `onUnmarkGuessed` 回调

### 3.3 数据流

```
用户答对 → FeedbackPanel 显示"其实是蒙对的"按钮
  → 点击 → markGuessed(slotId) + setIsGuessed(true)
  → 按钮变为"✓ 已标记蒙对" + "撤销"链接
  → 点击"撤销" → unmarkGuessed(slotId) + setIsGuessed(false)
  → 回到"其实是蒙对的"按钮
```

### 3.4 实现步骤

#### Step 3.1：修改 `src/lib/state/attempts-store.ts`

- [ ] 新增 `unmarkGuessed` action

在 `AttemptsStoreState` interface 中新增：
```typescript
interface AttemptsStoreState {
  // ...existing fields...
  markGuessed: (originalQuizId: string) => void
  /** 撤销最后一次的蒙对标注 */
  unmarkGuessed: (originalQuizId: string) => void
  // ...existing fields...
}
```

在 store 实现中新增（紧接 `markGuessed` 之后）：
```typescript
unmarkGuessed: (originalQuizId) =>
  set((state) => {
    const attempts = state.attemptsBySlot[originalQuizId]
    if (!attempts || attempts.length === 0) return state
    const last = attempts[attempts.length - 1]
    if (!last || !last.guessed) return state
    const { guessed: _guessed, ...rest } = last
    const updated: AttemptRecord = rest
    return {
      attemptsBySlot: {
        ...state.attemptsBySlot,
        [originalQuizId]: [...attempts.slice(0, -1), updated],
      },
    }
  }),
```

**实现说明**：
- `unmarkGuessed` 通过解构移除 `guessed` 字段（`{ guessed: _, ...rest }`），而非设为 `false`
- 这确保 `guessed === undefined` 与"从未标注"行为一致（`computeMastery` 检查 `guessed === true`）
- 如果最后一条 attempt 的 `guessed` 不是 `true`，直接返回 `state`（no-op）

## 4. Task 4 — 间隔重复配置开关（R3 / F12）

### 4.1 问题

`collectConfirmSlots`（`src/lib/runtime/adaptive-sequencer.ts:186`）在 `progress-store.advance()` 的 concept 切换分支中被无条件调用（`progress-store.ts:163-167`）。它将 Concept N-1 中首次答对的题在 Concept N+1 重现一次（"确认掌握题"）。

M7.8 Review 已知限制 3 指出：此机制固定开启，无配置开关。部分用户可能觉得打扰。

### 4.2 方案

1. `settings-store` 新增 `confirmReviewEnabled: boolean`（默认 `true`，向后兼容）
2. `progress-store.advance()` 在调用 `collectConfirmSlots` 前检查此设置
3. `ProductionSettings` 新增 toggle 开关 UI

### 4.3 数据流

```
用户在 Settings 页切换"确认掌握题"开关
  → settings-store.confirmReviewEnabled = false
  → 下次 advance() 到 concept 切换时
  → collectConfirmSlots 返回 []（被跳过）
  → nextReviewSlots 仅包含 wrongSlots + carriedSlots（无确认题）
```

### 4.4 实现步骤

#### Step 4.1：修改 `src/lib/state/settings-store.ts`

- [ ] 新增 `confirmReviewEnabled` 字段和 setter

在 `SettingsState` interface 中新增：
```typescript
interface SettingsState {
  config: LLMConfig | null
  availableKeys: ApiKeyMap | null

  /** 跨概念间隔重复：是否注入"确认掌握题"。默认 true。 */
  confirmReviewEnabled: boolean

  setConfig: (config: LLMConfig) => void
  setAvailableKeys: (keys: ApiKeyMap) => void
  updateConfig: (partial: Partial<LLMConfig>) => void
  /** 设置是否启用"确认掌握题" */
  setConfirmReviewEnabled: (enabled: boolean) => void
  clear: () => void
}
```

在 store 实现中新增初始值和 setter：
```typescript
export const useSettingsStore = create<SettingsState>()(
  persist(
    (set) => ({
      config: null,
      availableKeys: null,
      confirmReviewEnabled: true, // 默认开启，向后兼容

      setConfig: (config) => set({ config }),
      setAvailableKeys: (keys) => set({ availableKeys: keys }),
      updateConfig: (partial) =>
        set((state) => {
          if (!state.config) return state
          return { config: { ...state.config, ...partial } }
        }),
      setConfirmReviewEnabled: (enabled) => set({ confirmReviewEnabled: enabled }),
      clear: () => set({ config: null }),
    }),
    {
      name: 'alc:settings',
      storage: createJSONStorage(() => localStorage),
    },
  ),
)
```

**向后兼容说明**：`persist` middleware 在反序列化时，如果旧数据中没有 `confirmReviewEnabled` 字段，Zustand 会使用初始值 `true`。无需迁移。

## 6. 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| env fallback 泄露 API Key | 低-中 | `getEnvLLMConfig()` 仅从 `process.env` 读取，不写日志、不落盘；与现有 `/api/env-config` 安全级别一致 |
| `unmarkGuessed` 解构移除字段可能影响类型推断 | 低 | TypeScript 解构 `{ guessed: _, ...rest }` 是合法的 rest pattern；`AttemptRecord.guessed` 已是 optional |
| `confirmReviewEnabled` 旧数据无此字段 | 无 | Zustand `persist` 反序列化时使用初始值 `true`；零迁移成本 |
| `progress-store` 导入 `settings-store` 可能循环依赖 | 低 | `settings-store` 不导入 `progress-store`；依赖方向单向 |
| ProductionSettings toggle 样式不一致 | 低 | 复用 `alc-*` token + Tailwind 原生 class；参照现有按钮风格 |
