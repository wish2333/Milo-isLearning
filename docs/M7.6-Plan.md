# M7.6 Implementation Plan — Pedagogy, Navigation & UI Application

> **For agentic workers:** REQUIRED SUB-SKILL: Use `subagent-driven-development` or `executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 解决 M7.5 后暴露出的题目质量、答题流、题库资产、导航逃生口与 ui-design 未落地问题，让 M7.6 从“能保存和学习 Module”推进到“能被引导地学、能看懂为什么、能管理多个题库、界面与产品哲学一致”。

**Architecture:** M7.6 继续保持 local-first 和 BYOK，不引入账号、云同步或服务端任务持久化。题目生成质量通过“schema 向后兼容扩展 + prompt 强约束 + 质量报告观察”提升；运行时通过“本地确定性判分优先，LLM 语义判分兜底”控制成本；学习流通过自适应重排当前已生成题目替代答错即重新生成。

**Tech Stack:** Next.js App Router, React 19, TypeScript strict, Zustand persist, LocalStorage repository, Zod, Vitest, Playwright, Tailwind CSS.

## Global Constraints

- 不引入服务端持久化，不增加账号系统，不做云同步。
- 不把 API Key 写入导出文件、质量报告、URL、日志或 package metadata。
- 不收紧 M7 已放宽的 LLM schema；M7.6 新字段必须是 optional，旧 Module 必须可继续学习和导入。
- 不把“多做一题 / 重新生成一题”作为答错后的主流程；它只能是低优先级附加动作。
- Fill Blank 和开放回答允许 LLM 语义判分；Choice / Sorting 默认仍走本地确定性判分。
- LLM 语义判分必须有本地 exact/normalized match 前置、结果缓存、加载态和失败降级。
- 正确与错误都必须展示答案解析；答对时解析可更克制，但不能隐藏。
- ui-design 落地以 `docs/ui-design/DESIGN-SPEC.md` 和 HTML/CSS 原型为准，React 端不得继续扩大裸 `neutral-*` 主视觉。
- 所有导航逃生口必须保留当前学习进度，不得隐式重置 Module。
- 所有删除题库 / 清理资产操作必须二次确认。

## 0. 审查修订摘要

4. **Product Shell / 导航、题库、UI**：全局学习导航、多 Module 共存、完整应用 ui-design 主题与阶梯隐喻。

## 2. Milestone Scope

### 2.1 Must（M7.6）

1. Quiz domain 和 Zod schema 向后兼容扩展：`background`、`answerHint`、`acceptableAnswers`、`extendedKnowledge`、`misconception`、`evaluationMode`。
2. 重写 Quiz / Quiz Batch / Challenge Prompt：题前背景、详尽解析、基础知识、延伸知识、Fill Blank 限自由度。
3. Fill Blank 运行时重构：本地归一化匹配 + 多接受答案 + LLM 语义兜底 + 缓存 + 失败降级。
4. 正确和错误都显示解析，并把解析从单段文本升级为“核心解析 / 常见误区 / 延伸知识”的渐进展示。
5. 答错后的主流程改为“解析 → 标记薄弱点 → 重排剩余题目 / 稍后复现”，不再默认重新生成题目。
6. 新增自适应重排引擎，基于 attempts、mastery、题目 ladder/expression 和错因对当前 Module 内已生成题目重新排序。
7. 学习页增加全局 nav-top：回首页、回题库、回概览、当前 Module 切换，不丢进度。
8. 多 Module 共存体验：放宽本地保存容量、容量提示、Library 入口显著化、生成新题库后旧题库仍可见。
9. React 前端系统性迁移 ui-design：字体、色彩、阶梯进度、题型版式、反馈面板、完成页成长叙事。
10. 质量报告扩展：Prompt 字段覆盖率、mapper fix stats、LLM eval 命中率、语义判分成本估算。
11. Typecheck、lint、unit tests、Playwright smoke/library/learn-flow 全部通过。

1. Module 重命名并同步导出文件名。
2. Markdown / CSV 题目清单导出，便于人工审题。
3. IndexedDB 保存 compile stage outputs，为 true stage resume 铺路。
4. 当前 Concept 重新练习，但不覆盖原 attempts，只追加新 session。
5. Prompt evaluation 脚本增加一组固定 fixture，对“背景引导”和“解析详尽度”评分。

### 2.3 Not Now

- 云端账号、云同步、多人课程。
- 服务端 job registry 或长生命周期 SSE 任务恢复。
- Prompt A/B 平台。

- [ ] **Step 1: Replace Fill Blank rules**

In `_shared/expression-level-explanation.md`, replace the current “1-3 个关键词” rule with:

```md
#### Expression 3 Fill Blank

- 只允许 1 个空白。
- 空白答案必须是一个明确短语，而不是开放观点。
- `answer` 是最推荐标准答案。
- `acceptableAnswers` 必须包含 2-6 个可接受变体，并且包含 `answer`。
- `answerHint` 必须给出语境提示，帮助用户理解要补的是哪类概念，而不是背题。
- `evaluationMode` 默认使用 `semantic`，除非答案是唯一术语。
- 题干必须包含足够背景，让用户可通过理解推断答案。
- 禁止问题：`____ 是什么？`、`请填入概念名 ____`、要求背诵原句的题。
- 推荐问题：给出一个小场景或因果链，让用户补全关键关系。
```

- [ ] **Step 3: Add async evaluation**

```ts
export async function evaluateAnswerAsync(args: {
  quiz: Quiz
  userAnswer: string
  llmConfig?: LlmConfig
}): Promise<FeedbackRuntime> {
  const local = evaluateAnswerSync(args.quiz, args.userAnswer)
  if (local.score === 100 || !shouldUseSemanticEval(args.quiz, args.userAnswer)) return local
  if (!args.llmConfig) return local

  const response = await fetch('/api/semantic-eval', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      quiz: args.quiz,
      userAnswer: args.userAnswer,
      acceptableAnswers: args.quiz.acceptableAnswers ?? [args.quiz.answer],
    }),
  })

  if (!response.ok) return local
  const data = (await response.json()) as FeedbackRuntime
  return data
}
```

|---|---|---|
| LLM semantic eval raises latency | Fill Blank submit feels slow | Local match first; cache; show calm evaluating state; provider failure falls back to local |
| Prompt new fields reduce schema pass rate | Compile success may regress | Fields optional in Zod; prompt eval before full compile; mapper keeps fallback |
| Adaptive sequencing complicates progress | User may feel route jumps unexpectedly | Keep sequence within current Module; show `AdaptivePlanPanel` rationale; do not persist opaque queue unless needed |
| UI migration too large | M7.6 becomes long milestone | Implement in visual waves: tokens/nav/staircase first, quiz components second, pages third |
| Capacity policy increases LocalStorage pressure | Browser quota risk | Keep package export prominent; show capacity summary; consider IndexedDB in M7.7 |
| Correct-answer explanation adds visual noise | Fast learners feel slowed down | Explanation visible but compact; extended knowledge in details disclosure |
| Existing tests assume sync evaluation | Async path may break views | Keep `evaluateAnswerSync`; async only where view opts in |

## 7. 待确认的产品决策

1. Fill Blank 语义判分是否只用于 `evaluationMode: 'semantic'`，还是所有 Fill Blank 本地判错后都可兜底？建议：默认 semantic，但允许 exact。
