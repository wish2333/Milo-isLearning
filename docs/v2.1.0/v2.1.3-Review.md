## 0. TL;DR

V2.1.3 处理了连续答错触发 force-advance 时编辑入口被隐藏的问题，并在用户编辑题目后增加一次性 amnesty 窗口：该题下一次答对时删除旧作答历史，仅保留新的 score=100 attempt；下一次答错则消费窗口并按普通失败追加记录。

本版本新增 20 个单元测试与 4 个浏览器 E2E 场景。根代理验证结果为：typecheck、lint、format 全绿；849/849 单元测试通过；showcase/production 双模式 build 通过；全量 E2E 53 passed / 8 skipped / 0 failed。

### 1.5 Production build storage consistency hotfix（Done）

构建后实践发现，production 编译路径曾把 Module 写入 browser LocalStorage，而编辑路径从 SQLite-backed client cache 读取，导致 `correctQuizAnswer` 抛出 `Module ... not found`。本次 hotfix 完成两层修复：

- `src/app/learn/compiling/page.tsx`：编译完成及 compile job 统一使用 `getStorage()`，不再绕过 production repository
- `src/app/learn/history/[id]/page.tsx`、`src/app/learn/module/[id]/page.tsx`、两个 review 页面：Module/Topic 读取统一使用 `getStorage()`
- `src/lib/state/module-store.ts`：兼容旧数据；若 currentModule 已存在但 repository 缺少 Module 记录，先补写当前状态再执行题目更新
- `src/lib/state/__tests__/module-store.test.ts`：新增缺失 Module 记录自愈回归测试

### 1.6 Legacy LocalStorage module recovery hotfix（Done）

进一步验证发现，首个 hotfix 后，V2.1.3 之前已保存、仅存在浏览器 LocalStorage 的旧题库，在 production history/module 页面会被误判为“Module 不存在或已被删除”。本次补充兼容读取与迁移：

- `src/lib/persistence/client/storage.ts`：新增 `getStorageValueWithLegacyFallback`；production repository 查不到时回读旧 LocalStorage，并通过当前 write queue 回填 production storage
- `src/app/learn/history/[id]/page.tsx`、`src/app/learn/module/[id]/page.tsx`：使用 fallback 读取 Module；history 恢复题库后同步注入 `module-store`，确保编辑保存链路可继续工作
- `src/lib/persistence/client/__tests__/storage.test.ts`：覆盖 legacy 回退、production 优先及双侧缺失三种场景

### 1.7 Review correction persistence hotfix（Done）

实测发现错题重刷中的编辑只更新了当前反馈和最后一条 attempt，没有稳定写入所属 Module，也没有清理旧错题历史。本次补充：

- `src/app/learn/review/[moduleId]/page.tsx`：读取 Module 时使用 legacy fallback，并在启动 review 前注入 `module-store`
- `src/app/learn/review/topic/[topicId]/page.tsx`：跨 Module 复习切题时同步注入当前所属 Module
- `src/lib/state/review-store.ts`：review 会话统一读取当前 production storage，兼容旧 LocalStorage Module/Topic
- `src/components/learn/AnswerHistoryList.tsx`：保存前显式同步当前 Module，避免独立作答记录页编辑 no-op
- `src/lib/state/attempts-store.ts`：纠题重评为正确时只保留已纠正 attempt，并清理旧 schedule，错题筛选立即归零
- `e2e/v2.1.3-amnesty.spec.ts`：新增 review 编辑持久化、attempt 收敛及移出错题队列回归

## 2. 决策记录

### 2.1 D-1：force-advance 时保留编辑入口

删除 `!forceAdvance` 条件，仅保留 `!quiz.ignored`。force-advance 正是用户最需要修正 LLM 标错正解的场景；编辑保存后既有重评逻辑仍负责刷新反馈与 schedule。

### 2.2 D-2：amnesty 的数据语义

采用“删除该 slot 全部历史 attempt”的彻底遗忘语义。答对后的新 attempt 仍保留为一条普通 score=100 记录，因此不会丢失 streak、analytics 或后续 FSRS 重放所需的首答事实。

### 2.3 D-3：amnesty 边界与消费规则

amnesty 对任何答题入口生效，但本版 token 只由 `AnswerHistoryList` 产生。编辑后下一次 attempt 无论对错都消费 token：score≥80 清空旧历史并保留当前记录；score<80 走普通追加流程。再次编辑会重新生成窗口。

### 2.4 D-4：schedule 重建时序

amnesty 答对时先移除旧 schedule，避免旧历史继续影响派生缓存；Concept/Challenge/Today/Review 等调用方随后以当前 attempts 调用 `synchronizeScheduleForSlot`，重建出只基于新正确 attempt 的 schedule。

### 2.5 D-5：入口范围

本版只扩展 `AnswerHistoryList`，覆盖 ConceptView、ChallengeView 与 history 页。复习页独立编辑入口、多设备同步与 token 过期属于后续版本，避免扩大本次发布边界。

## 4. 已知不满足项 / 非阻塞观察

1. **复习页独立编辑入口未纳入本版**：`/learn/review/[moduleId]` 与 topic review 仍沿用既有流程，后续可复用同一 amnesty API。
2. **amnesty 视觉提示未增加**：当前编辑后仅通过数据层 token 生效，没有额外“下次答对将清空历史”的持久化状态提示；编辑器内已有说明文字。
3. **`clearAll` 未显式清理 pending token**：清空 attempts 后残留 token 无历史可清除，下一次作答会消费该 token；不影响数据正确性，但可作为后续小补丁。
4. **多设备同步与 token 过期未测试/未实现**：当前 production 方案是单机 localhost；token 暂无时间窗。
5. **人工视觉走查未执行**：自动化已覆盖入口可见、编辑保存、答案提交与持久化结果；颜色、字体和动画仍建议用户在浏览器中做一次快速确认。
