# V2.1.6 实施计划 — 跨模块错题穿插（主题学习流统一）

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在主题学习流中，学某模块时于每个 concept 转移处自动穿插同主题其他模块的 due/错题作为复习槽位，让"主题"从"模块顺序列表"升级为"统一的知识复习场"。

**Architecture:** 复用 V2.0.0 已建的 `reviewSlots` 注入机制（`progress-store.advance()` 跨概念转移处是唯一注入点）与 `collectReviewItemsForModules` 跨模块聚合能力（V2.0.0 主题重刷已在用）。新增 `collectCrossModuleReviewSlots` 收集器 + `findQuizInTopic` 跨模块查找；attempt/schedule 归属原模块（保持 attempts 真值一致，FSRS 协调层零改动）。**不复活已废弃的 `buildAdaptiveQueue`**（M7.6 已判定其绕开顺序状态机，生产代码零调用）。

**Tech Stack:** Next.js 15 + React 19 + Zustand + ts-fsrs@5.4.1 + Vitest + Playwright（无新增依赖）。

**Baseline:** V2.1.5（879 tests / 84 files 全绿，typecheck/lint/format/双模式 build 通过；E2E 56 passed 基线，B5 补丁后 E2E 待回归）。V2.1.5 已修复 progress-store 跨模块同步缺陷（B5：`enterModule` 统一入口 + `markModuleDone`），为本版的"跨模块数据访问"提供了正确的基础。

## 0. 访谈结论（spec-driven-lifecycle 驱动，2026-07-23）

通过 `spec-driven-lifecycle` 深度访谈确认的 4 个核心决策 + 5 条推导决策，为本 plan 的设计依据：

| 维度 | 决策 |
|---|---|
| 触发节奏 | 每个 concept 转移时注入（复用 `reviewSlots` 唯一注入点，`progress-store.advance()` L377-412） |
| 数量上限 | 动态：due 优先 + 硬上限 3-5 道（FSRS 关闭时 fallback 最近答错） |
| 作答归属 | attempt + schedule 归**原模块**（`originalQuizId` 指向原 quiz，schedule 写原 slot） |
| 入口范围 | **仅主题学习流**（题库单独学某模块不穿插） |

**5 条推导决策（从上述选择逻辑推出，已获用户确认）：**

1. `reviewSlots: string[]` 数据结构不变 — quiz ID 全局唯一，跨模块 ID 直接存入；向后兼容，无 schema 迁移。
2. `findQuizInModule` 扩展为 `findQuizInTopic` — ConceptView 渲染跨模块 quiz 的唯一必须改的数据流。
3. 穿插题答错**不 carry** — 不靠 `collectCarriedReviewSlots` 在模块2 concept 间流转；靠 FSRS due（开启时）或主题重刷页（关闭时）自然回来。保持穿插轻量。
4. 模块2 进度（`computeModuleProgress` conceptPercent）**不体现穿插题** — 只算当前 concept 新学题；`positionLabel` 的 totalSlots 含穿插题（反映真实做题位置），但 conceptPercent 不受影响。
5. 降级：主题内仅 1 模块 / 其他模块无 due 无错题 → 不穿插，零行为变化。showcase 模式同源支持。

## 1. 范围与约束

### 1.1 本版目标

- 在主题学习流的 `progress-store.advance()` 跨概念转移处，除现有模块内 `collectReviewSlots` 外，新增 `collectCrossModuleReviewSlots`，从主题内**其他模块**收集 due/错题注入下一概念的 `reviewSlots`。
- 提供 `findQuizInTopic` 跨模块 quiz 查找，让 ConceptView 能渲染穿插的跨模块 quiz（现有 `findQuizInModule` 限单模块，会因找不到而自动跳过）。
- 穿插题作答的 attempt + FSRS schedule 归属原模块（通过现有 `synchronizeScheduleForSlot` 协调层，按 slotId 自然归属，零改动）。
- 穿插题答错后不进入 `collectCarriedReviewSlots`（不跨模块 carry）。
- 每次 concept 转移注入上限默认 5 道（`cap` 参数可调），due 优先排序（FSRS 开启）/ 最近答错时间排序（FSRS 关闭）。
- 配套单测（收集器 + 注入回归 + carry 排除）与 E2E（主题学习流穿插场景）。

### 1.2 非目标

- **不复活 `buildAdaptiveQueue`** — M7.6 已判定其绕开 `progress-store.advance()` 顺序状态机，生产代码零调用；本版不引入任何基于它的重排逻辑。
- **不改 `reviewSlots` 数据结构** — 保持 `string[]`（quiz ID 全局唯一），不引入 `{quizId, sourceModuleId}[]` 对象数组（避免持久化 schema 变更）。
- **不在题库单独学某模块时穿插** — 仅主题学习流；题库入口保持单模块纯粹（与"让主题概念更统一"目标一致）。
- **不做穿插题的全库范围扩展** — 仅"主题内其他模块"，不扩展到全题库（全库穿插会破坏主题聚焦）。
- **不做 settings 开关** — 主题入口默认开启，不增加用户配置项（YAGNI；如需可后续版本加）。
- **不改 FSRS 协调层 / schedule repository** — 穿插题 schedule 走现有 slotId 路径，归属原模块是自然的。
- **不改 `computeModuleProgress` / mastery 计算** — 这些只看当前 module 的 concept，穿插题（属于其他模块）天然不参与。
- **不做 showcase 专项适配** — 代码同源，showcase 主题（若有）自动支持；不为 showcase 加特殊路径。

### P1.2：`collectCrossModuleReviewSlots` 收集器

**Files:**

1. 在 `adaptive-sequencer.ts` 新增导出函数：
   ```typescript
   /**
    * 收集主题内"其他模块"的 due/错题 slotId，用于注入当前模块学习流的 reviewSlots。
    *
    * - FSRS 开启：filter='due'，按 due 紧迫度升序（最紧迫在前）
    * - FSRS 关闭：filter='wrong'，按最近答错时间降序（最近在前）
    * - 排除 currentModuleId 的 slot（避免与模块内 collectReviewSlots 重复）
    * - 硬上限 cap 截断（默认 5）
    */
   export function collectCrossModuleReviewSlots(
     topicModules: Module[],
     currentModuleId: string,
     attemptsBySlot: Record<string, AttemptRecord[]>,
     options?: SequencerOptions & { cap?: number },
   ): string[] {
     const cap = options?.cap ?? 5
     const otherModules = topicModules.filter((m) => m.id !== currentModuleId)
     if (otherModules.length === 0) return []

     const filter: ReviewFilter = readFsrsEnabled(options?.fsrsEnabled) ? 'due' : 'wrong'
     // 复用 topic-review.ts 的 collectReviewItemsForModules 聚合 + matchesFilter
     const items = collectReviewItemsForModules(otherModules, attemptsBySlot, filter, {
       timezone: options?.timezone,
       now: options?.now,
     })
     if (items.length === 0) return []

     // 排序：due 按紧迫度（FSRS）；wrong 按最近答错时间
     const sorted = sortReviewItems(items, attemptsBySlot, options)
     return sorted.slice(0, cap).map((item) => item.slotId)
   }
   ```
2. 实现 `sortReviewItems` 辅助（FSRS 开启读 `scheduleLibrary.get(slotId).due` 升序；关闭读 `attemptsBySlot[slotId]` 最后一条 timestamp 降序）。
3. 写单测覆盖：(a) FSRS 开启 + due 题 → 按 due 排序截断；(b) FSRS 关闭 + wrong 题 → 按最近答错排序；(c) 排除 currentModuleId；(d) cap 截断；(e) 其他模块无 due/wrong → 返回空；(f) FSRS 关闭但无 wrong → 空；(g) ignored 题 excluded（`matchesFilter` 已处理，验证不回归）。
4. `bun run test adaptive-sequencer` 全绿。

**Acceptance:** 收集器返回 `slotId[]`；cap 默认 5；FSRS 双模式正确；6+ 边界用例覆盖；复用 `collectReviewItemsForModules` 不重写过滤逻辑。

     currentReviewSlots: string[] | undefined,
     attemptsBySlot: Record<string, AttemptRecord[]>,
     currentModule: Module,  // 新增参数
   ): string[] {
     if (!currentReviewSlots || currentReviewSlots.length === 0) return []
     const carried: string[] = []
     for (const slotId of currentReviewSlots) {
       // 跨模块 slot 不 carry（靠 FSRS due / 主题重刷回来）
       if (!findQuizInModule(currentModule, slotId)) continue

       if (latest && latest.score < PASS_THRESHOLD) carried.push(slotId)
     }
     return carried
   }

## 6. 交付顺序、风险与完成定义

### 6.1 顺序

```text
P0 Challenge 答错对齐（前置 bugfix，独立可先行验证）
  └─ P1 数据层（findQuizInTopic + collectCrossModuleReviewSlots）
              └─ P2 集成层（advance 注入 + ConceptView 渲染 + carry 排除）
                          └─ P3 门禁 + E2E + Review
```

P1.1 与 P1.2 可并行（无依赖）。P2.1 依赖 P1.2（收集器）；P2.2 依赖 P1.1（查找器）。P2.1 与 P2.2 可部分并行（P2.2 的 carry 排除独立于 P2.1 的注入）。

### 6.2 主要风险

| 风险 | 应对 |
|---|---|
| **progress-store 获取 topic 上下文的性能** | `getTopicContext()` 每次 advance 调用，若主题模块多则重复读 storage。MVP 接受（主题通常 ≤10 模块）；若 benchmark 证明慢，加模块级 memo 缓存（key=topicId）。 |
| **跨模块 slot 在 ConceptView 的 currentQuiz retry 替换冲突** | ConceptView 现有 currentQuiz 守卫（L96-118）已按 conceptId 匹配。跨模块 slot 的 conceptId 不同，需确认守卫不会误清。P2.2 组件测试覆盖。 |
| **positionLabel 分母膨胀** | totalSlots 含跨模块 slot 会让"题目 n/总数"分母变大。MVP 接受（反映真实位置）；若 UX 反馈混乱，P2.2 Step 4 的 label 区分方案落地。 |
| **showcase 主题数据不足** | showcase 预置主题（`showcase-das-kapital.alc-topic.json`）可能单模块或少 attempts。E2E 用 production 隔离实例（`ALC_DB_PATH`）构造多模块 + 错题数据，不依赖 showcase。 |
| **FSRS 关闭时的穿插力度** | FSRS 关闭只能用 wrong（最近答错），若主题内其他模块无错题则不穿插。属预期降级，Review §4 记录。 |
| **attempt 归属原模块的统计副作用** | 在模块2 作答模块1 的题，模块1 的 mastery/stats 会变化。这是预期（复习就该影响原模块），但需在 Review 明确，避免误判为 bug。 |
| **E2E B5 待回归叠加** | V2.1.5 B5 补丁后 E2E 未重跑。P3.1 必须先确认 V2.1.5 E2E 基线（56 passed）仍稳，再加新 E2E。 |
