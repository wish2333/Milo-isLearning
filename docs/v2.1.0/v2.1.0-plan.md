# V2.1.0 实施计划 — 发布验证与学习闭环强化

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.
> **V1.1 (2026-07-19)：** 基于 V2.0.1 修复实际开发结果更新基线、P0 范围与 P1 集成约束。详见 §8 修订记录。

**Goal:** 在 **V2.0.1** 代码基线上完成真实 localhost production 验证（含 V2.0.0/V2.0.1 暴露的 pre-existing E2E 修复），补齐可发布门禁，并将个人学习闭环扩展到多 Module/Topic 批量流程。

**Architecture:** 先以发布验证和回归为 P0，确认 SQLite、自动备份、FSRS replay、真实 LLM expand、双模式搜索和 **V2.0.1 主题入口恢复进度**可靠；再以 Module 为最小检查点实现 X2 批量扩充，避免把多个模型任务绑成不可恢复的单一 SSE；统计与搜索只在数据规模证明需要时增加复杂度。

**Tech Stack:** Next.js 15 + React 19 + Zustand + SQLite(localhost) + ts-fsrs@5.4.1 + Vitest + Playwright + uv 管理 Python 浏览器验证脚本。

**Baseline:** V2.0.1 (772 tests / 73 files 全绿，typecheck/lint/双模式 build 通过，5/5 re-entry E2E 通过)。V2.0.0 P0 数据丢失 bug 已在 V2.0.1 修复（`resumeModule` + `enterModule` + `computeModuleProgress`）。

## 1. 范围与约束

### 1.1 本版目标

- 完成 V2.0.1 的人工发布验证，**修复 V2.0.0/V2.0.1 暴露的 2 个 pre-existing E2E 失败**（Topic learning flow / Showcase Home clicking module card）。
- 跑 V2.0.1 fix-plan §B4 定义的 6 场景手动 production 回归矩阵，确认主题入口进度恢复在真实环境下不丢。
- 对 V2.0.1 5 项已知不满足项（`feynmanAttempt` 清空 / `handleReenter` 不区分继续与重做 / "已答题数"不展示 / per-module storage 二分 / today review 入口未加 E2E）做出处理决策（修/推迟到 V2.0.2/接受现状）。
- 让多 Module/Topic AI 扩充支持逐 Module checkpoint、暂停、恢复和取消；批量入口必须复用 V2.0.1 的 `enterModule`（禁止直接调 `startModule`，否则触发已修的 P0 bug）。
- 提供足够的 7/30 日学习趋势数据（在 V2.0.0 已有的 `stats-compute.ts` 基础上扩展，不重写），帮助判断复习闭环是否有效。
- 保持客户端搜索在当前规模下简单、可解释、跨 showcase/production 一致。

### 1.2 非目标

- 不增加账号、登录、云同步或多设备冲突解决。
- 不把 localhost production 变成公网多租户服务。
- 不在没有规模指标前引入 SQLite FTS5、中文分词服务或增量双写索引。
- 不重写 attempts 为新的学习真值；FSRS schedule 继续是可重放派生缓存。
- 不在趋势指标未被实际使用前引入重量级图表依赖。
- **不在 V2.1.0 内统一 per-module storage 二分架构债**（V2.0.1 fix-report §9 第 4 项，候选 V2.0.2）；但 P1 必须显式兼容当前 LS + 全局 blob 双写路径。

### 1.3 全局验收门禁

每个 Phase 完成后执行：

```bash
UV_CACHE_DIR=/tmp/uv-cache uv run --with playwright python <validation-script.py>
bun run typecheck
bun run lint
bun run test
NEXT_PUBLIC_APP_MODE=showcase bun run build
NEXT_PUBLIC_APP_MODE=production ALC_STORAGE_BACKEND=sqlite bun run build
```

禁止违反仓库既有约束：不使用 `any`、不直接访问 `localStorage`、不新增 `console.log`、不混用 showcase 与 production 数据、**Topic 入口必须用 `enterModule`（V2.0.1 约定）**；每批由根代理 review 后统一提交，子代理不单独提交。

## 3. Phase 1 — X2 多 Module/Topic 扩充（预计 1-2 周）

> **V2.0.1 集成约束：**
> - 批量生成的 module 完成后，从 Topic 入口进入学习必须调 `enterModule({ moduleId, allowResume: true })`，**禁止**直接调 `startModule`（V2.0.1 约定，违反会触发已修的 P0 bug）。

> - **per-module storage 二分架构债（V2.0.1 fix-report §9 第 4 项）**：批量扩充生成的 module 会同时写 LS per-module key + 全局 Zustand blob。P1 实现时必须显式兼容当前双写路径（与 V2.0.1 `resumeModule` 的三级 fallback 一致），不能假设两者 backend 相同。统一到 `getStorage()` 的清理留作 V2.0.2 candidate。

## 6. 交付顺序、风险与完成定义

### 6.1 顺序

```text
P0 发布验证
  ├─ 未通过 → 修复并重新跑门禁
  └─ 通过 → P1 批量扩充
              └─ P2 趋势统计
                          └─ P3 按规模评估搜索
```

### 6.2 主要风险

| 风险 | 应对 |
|------|------|
| 真实 provider 输出不稳定 | 固定 schema、保留 checkpoint、单 Module 失败可重试，不让批量任务原子化成一大块 |
| 批量任务重复导入 | 保存 source hash 与 moduleId，恢复前做幂等检查 |
| **批量入口误用 `startModule` 触发 V2.0.1 已修的 P0 bug** | P1.2 Step 5 强制用 `enterModule`；code review 时 grep 批量 pipeline 确认零残留 `startModule` |
| **per-module storage 二分（V2.0.1 架构债）** | P1 必须显式兼容 LS per-module + 全局 blob 双写路径；统一 backend 推迟到 V2.0.2 |
| FSRS 统计口径漂移 | 继续从 attempts/schedule 派生，所有边界用固定时钟单测 |
| 搜索复杂度过早增加 | 先 benchmark，规模未达到阈值就保持客户端索引 |
| 用户数据被测试污染 | 所有验证使用隔离 `data/`/storage，禁止读取真实 `.env.local` 内容 |
| pre-existing E2E 修复影响面超出预期 | P0.4 用 `--headed` 观察后再改；若涉及应用代码改动需重新跑全量 E2E + 双模式 build |

### 6.3 V2.1.0 完成定义

- **P0**：真实 production 验证（6 场景矩阵）通过；2 个 pre-existing E2E 修复；V2.0.1 5 项已知不满足项均有明确决策；正式 E2E + typecheck/lint/test/双模式 build 全绿。
