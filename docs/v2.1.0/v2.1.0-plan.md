# V2.1.0 实施计划 — 发布验证与学习闭环强化

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在 V2.0.0 代码基线上完成真实 localhost production 验证，补齐可发布门禁，并将个人学习闭环扩展到多 Module/Topic 批量流程。

**Architecture:** 先以发布验证和回归为 P0，确认 SQLite、自动备份、FSRS replay、真实 LLM expand 和双模式搜索可靠；再以 Module 为最小检查点实现 X2 批量扩充，避免把多个模型任务绑成不可恢复的单一 SSE；统计与搜索只在数据规模证明需要时增加复杂度。

**Tech Stack:** Next.js 15 + React 19 + Zustand + SQLite(localhost) + ts-fsrs@5.4.1 + Vitest + Playwright + uv 管理 Python 浏览器验证脚本。

---

## 1. 范围与约束

### 1.1 本版目标

- 完成 V2.0.0 的人工发布验证并修正实际发现的问题。
- 让多 Module/Topic AI 扩充支持逐 Module checkpoint、暂停、恢复和取消。
- 提供足够的 7/30 日学习趋势数据，帮助判断复习闭环是否有效。
- 保持客户端搜索在当前规模下简单、可解释、跨 showcase/production 一致。

### 1.2 非目标

- 不增加账号、登录、云同步或多设备冲突解决。
- 不把 localhost production 变成公网多租户服务。
- 不在没有规模指标前引入 SQLite FTS5、中文分词服务或增量双写索引。
- 不重写 attempts 为新的学习真值；FSRS schedule 继续是可重放派生缓存。
- 不在趋势指标未被实际使用前引入重量级图表依赖。

### 1.3 全局验收门禁

每个 Phase 完成后执行：

```bash
UV_CACHE_DIR=/tmp/uv-cache uv run --with playwright python <validation-script.py>
bun run typecheck
bun run lint
bun run test
NEXT_PUBLIC_APP_MODE=showcase bun run build
NEXT_PUBLIC_APP_MODE=production ALC_STORAGE_BACKEND=sqlite bun run build
```

禁止违反仓库既有约束：不使用 `any`、不直接访问 `localStorage`、不新增 `console.log`、不混用 showcase 与 production 数据；每批由根代理 review 后统一提交，子代理不单独提交。

## 3. Phase 1 — X2 多 Module/Topic 扩充（预计 1-2 周）

### P1.1：定义批量任务与 Module checkpoint

**Files:**

- Modify: `src/types/domain.ts`
- Create: `src/types/expand-job.ts`
- Create: `src/lib/persistence/expand-job-library.ts`
- Create: `src/lib/persistence/expand-job-library.test.ts`

**Steps:**

1. 定义 `ExpandJob`、`ExpandJobItem`、状态、attempt、error、checkpoint 和 source hash。
2. 以 Module 为最小可恢复单元，保存顺序、topicId、当前 item 和最终 moduleId。
3. repository 必须经过既有 storage 抽象；showcase 不写入批量任务。
4. 为新建、推进、失败、取消、恢复和清理写单测。

**Acceptance:** 刷新页面或重启后可以从最后一个 Module checkpoint 恢复；重复恢复不会重复插入 Module。

### P1.2：批量 pipeline 与取消/恢复

**Files:**

- Create: `src/lib/compiler/pipeline/compile-topic-with-expand.ts`
- Modify: `src/lib/compiler/pipeline/types.ts`
- Modify: `src/app/api/compile/route.ts`
- Create: `src/lib/compiler/pipeline/__tests__/compile-topic-with-expand.test.ts`

**Steps:**

1. 为每个 Module 复用 P2 的 expand pipeline，不把整个 Topic 拼成一个巨大 prompt。
2. 每完成一个 Module 就持久化 checkpoint，并发出明确的 item-level SSE event。
3. 支持 cancel；恢复时校验 source hash、options/config revision 和已保存 moduleId。
4. 限制并发和 token 预算；provider 失败只阻断当前 item，并保留可重试状态。

**Acceptance:** 单个 Module 失败不丢失已完成 Module；暂停后恢复只继续未完成项；取消不会删除已完成数据。

### P1.3：Topic UI 与导入结果

**Files:**

- Modify: `src/app/learn/import/page.tsx`（以实际路径为准）
- Create: `src/components/learn/ExpandJobView.tsx`
- Modify: `src/lib/state/compile-job-store.ts`
- Create: `src/components/learn/__tests__/ExpandJobView.test.tsx`

**Steps:**

1. 增加 Topic/Module 顺序、约束、暂停、恢复、取消和重试操作。
2. 展示每个 Module 的 queued/running/done/failed/cancelled 状态。
3. 完成项接入既有 topic-library，失败项保留 source 和错误，不自动删除。
4. showcase 继续使用静态内容，不出现 production 批量任务入口。

**Acceptance:** 用户能看懂批量进度，并可从失败 Module 单独重试。

## 5. Phase 3 — 搜索规模化评估（按指标触发，预计 2-4 天）

### P3.1：先测量再决定 FTS5

**Files:**

- Create: `scripts/search-benchmark.ts`
- Modify: `src/lib/runtime/search-client.test.ts`
- Modify: `docs/v2.1.0/v2.1.0-plan.md`（记录基准结果）

**Steps:**

1. 用 10/50/100/500 Module 的合成数据测量 rebuild 时间、内存和首结果延迟。
2. 在低端机器上记录结果，不把开发机单点性能当作门槛。
3. 只有超过既定阈值（例如 100 Module 或 50MB 内容且打开延迟明显）才设计 FTS5/增量索引方案。
4. 若未超过阈值，仅优化字段拼接、结果上限和空查询行为。

**Acceptance:** 有基准数据支撑“不变更”或“引入 FTS5”的决定；不为假设的规模提前增加双写复杂度。

## 6. 交付顺序、风险与完成定义

### 6.1 顺序

```text
P0 发布验证
  ├─ 未通过 → 修复并重新跑门禁
  └─ 通过 → P1 批量扩充
              └─ P2 趋势统计
                          └─ P3 按规模评估搜索
```

### 6.2 主要风险

| 风险 | 应对 |
|------|------|
| 真实 provider 输出不稳定 | 固定 schema、保留 checkpoint、单 Module 失败可重试，不让批量任务原子化成一大块 |
| 批量任务重复导入 | 保存 source hash 与 moduleId，恢复前做幂等检查 |
| FSRS 统计口径漂移 | 继续从 attempts/schedule 派生，所有边界用固定时钟单测 |
| 搜索复杂度过早增加 | 先 benchmark，规模未达到阈值就保持客户端索引 |
| 用户数据被测试污染 | 所有验证使用隔离 `data/`/storage，禁止读取真实 `.env.local` 内容 |

### 6.3 V2.1.0 完成定义

- P0 真实 production 验证和正式 E2E 通过。
- X2 批量扩充具备 Module checkpoint、暂停、恢复、取消和单项重试。
- 7/30 日统计有固定时钟单测，且不引入重复真值。
- 搜索是否升级由 benchmark 结果决定，并记录在文档中。
- 更新 Review，完成 typecheck/lint/test/双模式 build 后才允许 PR/tag。
