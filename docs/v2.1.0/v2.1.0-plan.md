## 1. 范围与约束

### 1.1 本版目标

- 完成 V2.0.1 的人工发布验证，**修复 V2.0.0/V2.0.1 暴露的 2 个 pre-existing E2E 失败**（Topic learning flow / Showcase Home clicking module card）。
- 跑 V2.0.1 fix-plan §B4 定义的 6 场景手动 production 回归矩阵，确认主题入口进度恢复在真实环境下不丢。
- 对 V2.0.1 5 项已知不满足项（`feynmanAttempt` 清空 / `handleReenter` 不区分继续与重做 / "已答题数"不展示 / per-module storage 二分 / today review 入口未加 E2E）做出处理决策（修/推迟到 V2.0.2/接受现状）。
- 让多 Module/Topic AI 扩充支持逐 Module checkpoint、暂停、恢复和取消；批量入口必须复用 V2.0.1 的 `enterModule`（禁止直接调 `startModule`，否则触发已修的 P0 bug）。
- 提供足够的 7/30 日学习趋势数据（在 V2.0.0 已有的 `stats-compute.ts` 基础上扩展，不重写），帮助判断复习闭环是否有效。
- 保持客户端搜索在当前规模下简单、可解释、跨 showcase/production 一致。

### 1.2 非目标

- 不增加账号、登录、云同步或多设备冲突解决。
- 不把 localhost production 变成公网多租户服务。
- 不在没有规模指标前引入 SQLite FTS5、中文分词服务或增量双写索引。
- 不重写 attempts 为新的学习真值；FSRS schedule 继续是可重放派生缓存。
- 不在趋势指标未被实际使用前引入重量级图表依赖。
- **不在 V2.1.0 内统一 per-module storage 二分架构债**（V2.0.1 fix-report §9 第 4 项，候选 V2.0.2）；但 P1 必须显式兼容当前 LS + 全局 blob 双写路径。

### 1.3 全局验收门禁

每个 Phase 完成后**必须全部通过**以下命令（无占位符，均为可直接执行的命令）：

```bash
# 静态检查
bun run typecheck
bun run lint

# 单元测试
bun run test

# 双模式 build
NEXT_PUBLIC_APP_MODE=showcase bun run build
NEXT_PUBLIC_APP_MODE=production ALC_STORAGE_BACKEND=sqlite bun run build

# 正式 E2E（V2.1.0 P0 修复后必须全绿）
bun run e2e
```

**P0.5 手动 production 验证矩阵**不使用脚本，由人工在浏览器中执行 6 场景（P0.5 定义），结果记录到 `docs/v2.1.0/v2.1.0-Review.md`（**不修改已发布的 v2.0.1-Review.md**）。

**P3.1 搜索 benchmark** 使用 `scripts/search-benchmark.ts`（P3.1 创建），用 Bun 运行合成数据测量 rebuild 时间/内存/首结果延迟。

禁止违反仓库既有约束：不使用 `any`、不直接访问 `localStorage`、不新增 `console.log`、不混用 showcase 与 production 数据、**Topic 入口必须用 `enterModule`（V2.0.1 约定）**；每批由根代理 review 后统一提交，子代理不单独提交。

### P1.2：批量 pipeline 与取消/恢复协议

> **⚠ 审计纠偏（V1.2）：** V1.1 只说"支持 cancel"，未定义状态持久化、取消 API、幂等规则和 SSE 事件契约。现有 V2.0.0 checkpoint 机制（`compile-checkpoint.ts`）只支持单 Module compile 的 resume，不证明"暂停后只恢复未完成项"。本 task 必须显式定义协议。

**Files:**

- Create: `src/lib/compiler/pipeline/compile-topic-with-expand.ts`
- Modify: `src/lib/compiler/pipeline/types.ts`（新增 TopicExpand SSE 事件类型）
- Modify: `src/app/api/compile/route.ts`（接入批量 pipeline + cancel 端点）
- Create: `src/app/api/compile/cancel/route.ts`（cancel API）
- Create: `src/lib/compiler/pipeline/__tests__/compile-topic-with-expand.test.ts`

**协议规范（必须在实现前锁定）：**

#### 1. ExpandJob 状态机（P1.1 定义，P1.2 消费）

```
created → running ↔ paused → cancelled
                ↓
              failed（单 item 失败，可重试）
                ↓
            completed（所有 item done）
```

- `created`：job 已创建，未开始
- `running`：至少一个 item 正在 compile
- `paused`：用户主动暂停；不取消已 in-flight 的 item，但不再启动新 item
- `failed`：某 item 多次重试后仍失败；其他已完成 item 保留
- `completed`：所有 item done
- `cancelled`：用户主动取消；保留已完成 item，未完成 item 标记 cancelled

#### 2. Checkpoint 持久化

每个 item 完成后**必须**写入 `ExpandJobCheckpoint`：
```ts
interface ExpandJobCheckpoint {
  jobId: string
  itemId: string                    // 单 Module 扩充任务 ID
  status: 'queued' | 'running' | 'done' | 'failed' | 'cancelled'
  moduleId?: string                 // 成功时：生成的 module ID
  sourceHash: string                // 源 Markdown hash，resume 时校验
  error?: { code: string; message: string; retryable: boolean }
  attempts: number                  // 已重试次数
  updatedAt: number
}
```

存储位置：`StorageKeys.expandJobCheckpoint(jobId, itemId)`，经 `storage` 抽象（showcase 不写入）。

#### 3. Cancel API 契约

`POST /api/compile/cancel` body：`{ jobId: string }`
- 幂等：重复 cancel 同一 jobId 返回 200 + 当前状态，不报错
- 已 `completed/cancelled` 的 job：返回 200 + `{ alreadyTerminal: true }`
- `running` 的 job：设置 cancel 标记，让 pipeline 在下一个 item 边界停止；in-flight item 完成后才返回 200 + `{ cancelled: true, completedItems: N }`
- 返回体：`{ jobId, status, cancelledItemCount, completedItemCount }`

#### 4. Pause/Resume 语义

- **Pause**：客户端调 `expandJobLibrary.updateJob(jobId, { status: 'paused' })`；pipeline 在下一个 item 边界检查 job.status，不再启动新 item；in-flight item 完成。
- **Resume**：客户端调 `updateJob(jobId, { status: 'running' })` 后重新 POST `/api/compile` 带 `{ jobId, resume: true }`；pipeline 从 checkpoint 列表加载，**只启动 status !== 'done' 的 item**。
- **幂等校验**：resume 时比较 job.sourceHash 与请求体 sourceHash；不一致返回 409 `source_changed`。

#### 5. SSE 事件契约（在现有 compile SSE 事件基础上新增）

| 事件 | 触发时机 | payload |
|------|---------|---------|
| `topic_expand_started` | 批量任务开始 | `{ jobId, totalItems }` |
| `item_started` | 单 item 开始 | `{ jobId, itemId, moduleIndex }` |
| `item_completed` | 单 item done | `{ jobId, itemId, moduleId }` |
| `item_failed` | 单 item 失败 | `{ jobId, itemId, error, retryable }` |
| `topic_expand_paused` | 暂停生效 | `{ jobId, completedItems }` |
| `topic_expand_cancelled` | 取消生效 | `{ jobId, completedItems, cancelledItems }` |
| `topic_expand_completed` | 所有 item done | `{ jobId, moduleIds }` |

**Steps:**

1. 为每个 Module 复用 **V2.0.0 的 `compile-with-expand.ts`**（expand pipeline），不把整个 Topic 拼成一个巨大 prompt。
2. 每完成一个 Module 就持久化 checkpoint（按协议 §2），并发出 `item_completed` SSE 事件（按协议 §5）。
3. 实现 cancel API（`/api/compile/cancel`，按协议 §3），pipeline 在 item 边界检查 cancel 标记。
4. 实现 pause/resume（按协议 §4）：resume 时从 checkpoint 列表加载，只启动非 done item；幂等校验 sourceHash。
5. 限制并发（单 job 内 item 串行，避免 provider rate limit）和 token 预算；provider 失败只阻断当前 item，保留可重试状态（`error.retryable`）。
6. **Module 完成后的 Topic 入口必须用 `enterModule`**（调用 `setModule` + `resumeModule`），而不是 `startModule`。

**Acceptance:**
- 单个 Module 失败不丢失已完成 Module（checkpoint 持久化验证）
- 暂停后恢复只继续未完成项（resume 时 status !== 'done' 过滤验证）
- 取消不会删除已完成数据（cancel 后 checkpoint 仍在）
- cancel 幂等（重复调 cancel 同一 jobId 不报错）
- sourceHash 不一致时 resume 返回 409（幂等校验验证）
- 协议 §5 的 7 个 SSE 事件全部有单测覆盖

## 6. 交付顺序、风险与完成定义

### 6.1 顺序

```text
P0 发布验证
  ├─ 未通过 → 修复并重新跑门禁
  └─ 通过 → P1 批量扩充
              └─ P2 趋势统计
                          └─ P3 按规模评估搜索
```

### 6.2 主要风险

| 风险 | 应对 |
|------|------|
| 真实 provider 输出不稳定 | 固定 schema、保留 checkpoint、单 Module 失败可重试，不让批量任务原子化成一大块 |
| 批量任务重复导入 | 保存 source hash 与 moduleId，恢复前做幂等检查 |
| **批量入口误用 `startModule` 触发 V2.0.1 已修的 P0 bug** | P1.2 Step 5 强制用 `enterModule`；code review 时 grep 批量 pipeline 确认零残留 `startModule` |
| **per-module storage 二分（V2.0.1 架构债）** | P1 必须显式兼容 LS per-module + 全局 blob 双写路径；统一 backend 推迟到 V2.0.2 |
| FSRS 统计口径漂移 | 继续从 attempts/schedule 派生，所有边界用固定时钟单测 |
| 搜索复杂度过早增加 | 先 benchmark，规模未达到阈值就保持客户端索引 |
| 用户数据被测试污染 | 所有验证使用隔离 `data/`/storage，禁止读取真实 `.env.local` 内容 |
| pre-existing E2E 修复影响面超出预期 | P0.4 用 `--headed` 观察后再改；若涉及应用代码改动需重新跑全量 E2E + 双模式 build |
