## 0. TL;DR

V2.1.2 修复了 V2.1.0 发布后用户反馈的 2 类答题环节 UX 缺陷，按 fix-report 拆为 3 个修复点 / 8 个任务（T1–T8）：

1. **F1（P0 三态渲染）**：`ChoiceQuiz` 答错后未标记正解 → 扩展 `submittedAnswer` prop，在 disabled 状态按"正解 success + ✓ / 错选 warning + ✗ / 未选 opacity-50"三态渲染；通过 `QuizRenderer` 透传给 5 处调用方
2. **F2（P0 降噪）**：`FeedbackPanel` 解析前的"两段无 label 复述文字"溯源为 LLM `feedback.gaps` 字段 → 直接隐藏 UI 渲染（保留 AttemptRecord 持久化以兼容未来分析）
3. **F3（P0 精简）**：`AnswerCorrector` 把"选正解"和"编辑文本"混在一起 → 精简为"只指定正确答案"；Choice 走 radio 列表，FillBlank 保留正解+同义答案，Sorting 保持；移除 stem/explanation 字段；496→321 行（-35%）

scope 严格收敛在 3 个核心组件 + 1 个透传层 + 5 处调用方 + 3 个新增测试；不动 feedback agent prompt/schema、不动 AttemptRecord 数据形状、不动 QuizEditPatch 类型、不动 Sorting/FillBlank 题型 disabled 视觉。

## 2. 决策记录

### 2.1 D-1：F2 选"隐藏 gaps"而非"加 label"

**选择**：方案 A（直接隐藏 `feedback.gaps` UI 渲染），不采用方案 B（加 label "容易忽略的关键点" + 改 prompt 禁止复述选项）。

**理由**：
- F1 实施后，正解选项已在选项层视觉高亮，gaps 的"复述选项"信息完全冗余
- 即使加 label，信息仍与选项高亮 + 解析重复
- 修改 prompt 需要重跑 prompt eval（`scripts/prompt-eval.ts`），scope 放大
- 用户已明确反馈"这部分 UI 没标是啥我就是看到了" → 隐藏比加 label 更彻底降噪
- 数据层保留 `feedback.gaps` 写入 AttemptRecord，未来做学习分析仍可用

### 2.2 D-2：F3 FillBlank 保留 acceptableAnswers（覆盖 fix-report §4.3.3 默认）

**选择**：§6 #2 用户明确选 B — FillBlank editor 保留"正解 + 同义答案（acceptableAnswers）"编辑，仅去掉"答案提示（answerHint）"。

**理由**：
- FillBlank 题型中同义答案（如"RAG" / "检索增强生成"）是常见需求，去掉后用户无法在 UI 上补充
- 用户原话"完全去掉文本编辑功能"严格解读应一并去掉，但实际产品语义需要同义答案
- fix-report §4.3.3 默认是方案 a（仅保留正解），本版按用户二次确认实施 b
- 决策已记录在 fix-report §6 #2 用户批注

### 2.3 D-3：调用方透传分两种模式

**选择**：ConceptView / ChallengeView / TodayReviewView 用 `useState` 在 handleAnswer 时设置；review/[moduleId] / review/topic/[topicId] 直接从 attempts-store `getAttempts(slotId).at(-1)` 派生。

**理由**：
- 前 3 处是答题中场景，userAnswer 是临时态（切题即重置），适合 useState
- 后 2 处是复习页场景，attempt 数据已在 store 中（可能跨会话），适合直接派生
- 两种模式都通过 `phase !== 'answering' ? submittedAnswer ?? undefined : undefined` 守卫，避免答题中显示标记

### 2.4 D-4：未改 QuizEditPatch 类型，靠 patch 字段范围收敛

**选择**：`QuizEditPatch` 类型保持 widening 不变（仍含 stem/explanation/answerHint 等），但实际 `handleSave` 中按题型决定 patch 字段：choice 只 `{answer}`、sorting `{answer,options}`、fill_blank `{answer,acceptableAnswers}`。

**理由**：
- fix-report §4.5 / §7 明确"不改类型定义保持向后兼容"
- TypeScript widening 类型允许 patch 不含所有字段，调用方 `correctQuizAnswer(quizId, patch)` 不受影响
- 若未来恢复 stem/explanation 编辑功能，类型层无需改动

### 2.5 D-5：测试用 jsdom + 原生 createRoot，不引入 testing-library

**选择**：组件测试沿用项目现有 `ExpandJobView.test.tsx` 模式 — `// @vitest-environment jsdom` + `react-dom/client.createRoot` + `act`，不引入 `@testing-library/react`。

**理由**：
- 项目无 `@testing-library/*` 依赖，引入需改 package.json
- 原生 createRoot 模式足以覆盖 className/textContent/aria 属性断言
- 显式 `import { describe, it, expect, vi } from 'vitest'` 符合项目 `globals: false` 约定

## 4. 已知不满足项 / 非阻塞观察

明确写为**本版不满足**或**非阻塞观察**：

1. **人工视觉验证未执行**：UI 改动的视觉效果（颜色搭配、✓/✗ 字符可读性、radio 交互流畅度）自动化测试无法完全替代。fix-report §5.4 列出 8 项人工验证边界（答错后正解高亮、答对后其他项 opacity-50、Sorting/FillBlank 视觉不变、gaps 已隐藏、编辑器保存后 quiz.answer 更新等），需用户在浏览器实际验证。

2. **SortingQuiz / FillBlankQuiz 的 disabled 状态视觉未动**：fix-report §2.5 明确本次不动，用户未反馈这两个题型的视觉问题。若未来反馈，可参照 ChoiceQuiz 的三态模式扩展。

3. **AnswerCorrector 行数未达 ~200 目标**：fix-report §4.4 预估改造后 ~180 行，实际 321 行。差距主因：SortingEditor 完整保留（40+ 行）+ FillBlankEditor 保留 acceptableAnswers 编辑（增加 input + blur handler）+ 主组件 validation/save 逻辑保留。功能完整优先于行数指标。

4. **gaps 字段在数据层的长期保留策略**：UI 不渲染，但 AttemptRecord 仍写入。若 6 个月内无分析需求落地，考虑 V2.3.0 一并清理数据层（fix-report §6 #4 选项 b）。

5. **Resume API 在 failed 场景下仍不校验 retryable**（继承自 V2.1.1 非阻塞项）：与本次无关，留作 V2.2.0 候选。
