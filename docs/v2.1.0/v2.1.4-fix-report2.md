# V2.1.4 Fix Report 2 — 作答进度系统联动故障

> 状态：Proposal（待评审，未实施） | 日期：2026-07-22
> 对应基线：[V2.1.4](./v2.1.4-Review.md)（854 tests 全绿，费曼练习 UX 与答案反馈统一）
> 文档类型：联动缺陷根因分析 + 统一修复设计
> 关闭边界：本报告覆盖 4 个用户反馈缺陷的根因主线、联动关系与修复方案；实施后产出 `v2.1.5-Review.md` 并合入 `main`

## 0. TL;DR

V2.1.4 发布后用户在 production 模式实际使用中反馈 4 个缺陷。溯源后确认它们**不是孤立 bug，而是同一条"作答进度系统"链条上的多点断裂**，共享三个叠加的架构层根因：

| # | 缺陷 | 用户原话 | 所属根因层 |
|---|------|---------|-----------|
| **B1** | 费曼完成后 `/learn/done` 客户端异常 | "费曼流程全完成之后点击完成作答跳转 /learn/done 显示 Application error: a client-side exception" | 待定（疑似层 ① 数据时序） |
| **B2.1** | 首页"继续学习"取的不是最近作答的模块 | "继续学习好像是继续题库中最上面的而不是最近作答的" | 层 ① 存储二分 |
| **B2.2** | 题库页主题平铺，切换主题需翻过前一主题全部模块 | "选择题库（主题）学习时，如果不想学第一个，得在题库页翻过第一个主题的所有模块才到第二个主题，不太直观" | UX（非数据层） |
| **B3** | 主题内模块完成后重新"开始主题学习"会重开已完成模块 | "一个主题内，模块1作答完之后，再点击开始主题学习，会导致模块1重开……而不是推进到模块2" | 层 ② 主题会话持久化缺口 |
| **B4** | 主题内模块进度呈现不反映真实作答状态 | "一个主题内，如果模块1没完成，对模块2的进度呈现仍是未开始" | 层 ③ 双真相源未同步 |

**核心判断**：用户观察到的"作答进度系统失效"是真实状态。4 个缺陷在"作答 → 落盘 → 恢复 → 展示"生命周期上相互触发，修任何一个都无法单独闭合，必须作为一次联动修复。

**实施周期预估**：1.5–2.5 天（含存储入口统一 + 主题会话持久化 + 进度呈现真相源切换 + UX 重构 + 单测/E2E 回归 + 双模式 build）。

## 1. 背景：用户反馈与联动判断

### 1.1 用户反馈时间线

1. 首轮：反馈 B1（done 崩溃）+ B2（继续学习取错 + 题库导航不直观，要求出方案）。
2. 二轮：反馈 B3（主题模块完成后重开），并报告"误操作了重开"，询问能否恢复。
3. 三轮：反馈 B4（主题内模块进度呈现失真），明确指出"bug3 和 bug4 还有对应的联动 bug 导致更多问题，导致作答进度系统失效"，要求出具联动修复报告。

### 1.2 联动判断的依据

4 个缺陷表面分散在不同页面（done 页 / 首页 / 题库页 / 主题页 / 模块页），但都作用于同一组进度数据。逐层溯源后，它们收敛到三个互相叠加的根因层（详见 §2）。其中 B3 是最关键的放大器——它的持久化缺口会让主题会话状态在"重新进入"时回退，进而触发 B4 的呈现失真和 B2.1 的"最近作答"误判。

## 2. 根因主线：三层叠加

### 2.1 进度数据的完整生命周期

一个模块的进度从产生到展示，经过 5 个环节：

```
作答(attempts-store) → stage 转移(progress-store) → 双写{全局blob, per-module key}
    → 恢复入口(enterModule/resumeModule) → 展示层(首页/题库/主题/done)
```

本次 4 个缺陷分别打断不同环节，但都让"展示层看到的进度"与"用户实际作答"不一致。

### 2.2 层 ① — Production 存储二分

**事实**：production 模式下，进度相关数据被拆到两个存储后端：

| 数据 | 写入端 | 存储后端 | 读取端 |
|------|--------|---------|--------|
| `alc:module:{id}`（Module 本体） | module-store persist → `getStorage()` | **SQLite** | 多处 |
| `alc:state:progress`（全局 blob） | progress-store persist → `getStorage()` | **SQLite** | progress-store rehydrate |
| `alc:state:attempts`（作答记录） | attempts-store persist → `getStorage()` | **SQLite** | attempts-store rehydrate |
| `alc:progress:{id}`（per-module 快照） | progress-store subscribe → `storage` | **LocalStorage** | `listStoredModules` / `resumeModule` |
| `alc:topicProgress:{id}`（主题进度） | `exitSession` → `storage` | **LocalStorage** | `startSession` |
| `alc:feynman:{id}`（费曼记录） | `storage` | **LocalStorage** | mastery 计算 |

**断裂点**：读取入口混用 `storage`（LS 单例）和 `getStorage()`（production 返回 SQLite Repository）。AGENTS.md 已把此列为 V2.0.2 候选架构债（"per-module progress storage 二分"），但至今未还债，B2.1 就是它的一次具体爆发。

### 2.3 层 ② — 主题会话持久化缺口

**事实**：`useTopicSessionStore` 的完成态写入与落盘不对称：

- `markCurrentModuleDone`（`topic-session-store.ts:76`）与 `skipCurrentModule`（`:113`）**只更新内存 `session.moduleStatus`，不写 `alc:topicProgress:{id}`**。
- 唯一把主题进度落盘的是 `exitSession`（`:167`，显式"退出主题"才触发）。
- `startSession`（`:33`）重建 session 时**只读磁盘 `topicProgress`**，不参考内存 session。

**后果**：任何不经过"退出主题"按钮的重新进入（首页继续学习、题库页重开主题、刷新后重进），都会让 `startSession` 读到空的主题进度 → 从第一个模块重来 → `resumeModule` 见 `stage==='done'` 走 `startModule` 语义（重置）→ **已完成模块被重开**。这就是 B3。

### 2.4 层 ③ — 双真相源未同步

**事实**：主题内模块的"完成状态"存在两套独立真相：

- **主题维度**：`session.moduleStatus[mid]`（`pending` / `in_progress` / `done` / `skipped`）
- **模块维度**：`alc:progress:{id}.stage.kind === 'done'`（per-module 快照）

两者没有任何同步机制。`TopicTransitionView`（主题学习页）的进度呈现以 `session.moduleStatus` 为准（`TopicTransitionView.tsx:244`），`computeModuleProgress` 只提供 label/百分比点缀。当层 ② 导致 `moduleStatus` 失真时，层 ③ 让失真直接传导到 UI——已完成模块不被标记 done、后续模块永远 pending。这就是 B4。

### 2.5 联动图

```
                    层 ① 存储二分（production 专属）
                    ┌──────────────────────────────────────────────┐
                    │  per-module/topicProgress 在 LS              │
                    │  module/全局blob/attempts 在 SQLite          │
                    │  读取入口混用 → listStoredModules(getStorage)│
                    │  读不到 per-module → updatedAt 全为 0        │
                    └───────────────┬──────────────────────────────┘
                                    │ 放大
                                    ▼
   层 ② 主题会话持久化缺口          层 ③ 双真相源未同步
   ┌─────────────────────────┐     ┌─────────────────────────────┐
   │ markDone/skip 不落盘     │────▶│ moduleStatus ≠ per-module   │
   │ 只有 exitSession 落盘    │     │ TopicTransitionView 采信     │
   │ 重新 startSession 丢状态 │     │ 失真的 moduleStatus          │
   │ → done 模块被 startModule│     │ → B4 后续模块恒显"未开始"    │
   │ → B3 已完成模块重开      │     └─────────────────────────────┘
   └─────────────────────────┘
                                    │
                                    ▼
                         B2.1 继续学习取错模块（首页 getStorage 读 SQLite）
                         B1  done 页异常（疑似数据时序，待堆栈定位）
```

## 3. B1 — `/learn/done` 客户端异常（production）

### 3.1 复现路径

production 模式 → 单模块（非主题）完成费曼最终步 → 点击"完成学习" → `submitFeynman` 置 `stage='done'` → `module/[id]/page.tsx:100` `router.replace('/learn/done')` → done 页报 "Application error: a client-side exception has occurred"。

### 3.2 已排查路径（均排除了确定性崩溃）

| 排查点 | 文件:行 | 结论 |
|--------|---------|------|
| `computeMastery` 空值防护 | `runtime/mastery.ts:66-164` | 全链路有 `??` / `|| 1` / `!== undefined` 防护，不会抛 |
| `RatingStars` | `components/learn/RatingStars.tsx:13` | `existingRating?.score ?? 0`，安全 |
| done 页 reduce/除零 | `done/page.tsx:59-64` | `conceptMastery.length || 1`，安全 |
| Feynman 数据形状 | `progress-store.ts:402-421` | `finalGaps` 可能为 undefined，但 computeMastery 未访问 `.gaps.length` |

### 3.3 当前结论与所需信息

**静态分析未能定位确定性崩溃点。** B1 最可能与层 ① 的 production 数据时序相关——client-side navigation 后某些 store 在特定时序下状态短暂不一致，触发渲染期异常。

**需要用户提供**：复现时浏览器控制台的**红色错误堆栈**（文件名 + 行号 + 调用栈）。有了具体抛错位置可一次定位。本报告的统一修复方案（§7）会顺带收紧 done 页对 `feynmanAttempt` / `attemptsBySlot` 的空值边界，作为防御层。

## 4. B2 — 继续学习取错模块 + 题库导航不直观

### 4.1 B2.1 继续学习（层 ① 根因，确定性）

**现状**（`components/home/ProductionHome.tsx:39-48`）：

```tsx
const modules = listStoredModules(getStorage())   // ← getStorage() = SQLite
if (modules.length === 0) return
const recent = modules[0]!
if (!recent.completed) {
  setButtonLabel('继续学习')
  setTargetHref(`/learn/module/${recent.id}`)
}
```

`listStoredModules`（`module-library.ts:39-77`）内部用传入的 repo 读 `alc:progress:{id}` 计算 `updatedAt`，再按 `updatedAt` 降序排序。production 模式下 `getStorage()` 返回 SQLite Repository，而 per-module progress 写在 **LS**——SQLite 里读不到 → 所有模块 `updatedAt === 0` → 排序失效 → `modules[0]` 退化为 `storage.keys()` 的首项，即"题库中最上面的"。

**对比**：题库页 `library/page.tsx:64` 用的是 `listStoredModules(storage)`（LS），能读到 per-module progress，所以题库页排序正常。**`ProductionHome` 是唯一用 `getStorage()` 读 listStoredModules 的调用方**（`grep "listStoredModules(getStorage())"` 仅此一处）。

**根因**：层 ① 存储二分 + 读取入口选错。

### 4.2 B2.2 题库主题平铺（UX 缺陷，非数据层）

**现状**（`app/learn/library/page.tsx:237-245`）：

```tsx
{topics.map((t) => (
  <TopicSection key={t.id} topic={t} modules={getTopicModules(t, allModules)} ... />
))}
```

每个 `TopicSection` 纵向展开 `TopicCard`（含主题内全部模块的进度行）+ 模块行列表。多主题时，用户必须**滚动过第一个主题的完整内容**才能到达第二个主题的入口。主题越多、模块越多，越不直观。

**这是纯布局问题，与层 ①②③ 无关，但与 B3/B4 的修复一起做能一次性提升主题学习体验。**

### 4.3 B2.2 修复方案（三选一，待用户定）

| 方案 | 描述 | 改动量 | 移动端 | 推荐 |
|------|------|--------|--------|------|
| **A. 折叠/展开** | TopicSection 默认折叠（只显示主题标题 + 进度条 + 展开箭头），点击展开模块行 | 小 | 友好 | ⭐ 推荐 |
| **B. 主题切换器** | 顶部横向主题 Tab/分段控件，内容区只渲染当前选中主题的模块 | 中 | 友好 | |
| **C. 两栏布局** | 左栏主题列表，右栏选中主题的模块详情 | 大 | 不友好 | 不推荐 |

推荐 **A**：最小改动直接解决"翻过第一个主题"痛点，保留现有单栏阅读流，与移动端兼容。实施时给"有进行中模块的主题"默认展开、其余默认折叠，兼顾发现性与简洁。

## 5. B3 — 主题模块完成后重开（层 ② 根因，确定性）

### 5.1 复现路径

主题内学完模块1 → `module/[id]/page.tsx:96-98` 检测 `stage='done'` 且在主题会话 → `markCurrentModuleDone()` → 跳转 `/learn/topic/{id}`。用户不点"继续"而是回到题库页 → `TopicCard.handleStartTopic`（`TopicCard.tsx:61`）→ `startSession` 重建 session → 读磁盘 `topicProgress`（空）→ `firstIncompleteIndex=0` → 从模块1重来 → `enterModule({allowResume:true})` → `resumeModule` 见 `stage='done'` → 走 `startModule`（重置）→ **模块1被重开**。

### 5.2 根因（精确）

`topic-session-store.ts` 的完成态写入与落盘不对称：

```typescript
// markCurrentModuleDone (:76) — 只改内存，不落盘
markCurrentModuleDone: () => {
  set((state) => state.session ? {
    session: { ...state.session, moduleStatus: { ...state.session.moduleStatus, [currentId]: 'done' } }
  } : state)
  // ❌ 缺少：storage.set(StorageKeys.topicProgress(topicId), ...)
},
```

`skipCurrentModule`（`:113`）有同样问题。唯一落盘的 `exitSession`（`:167`）只在显式退出时触发。

### 5.3 误操作恢复说明（对应用户二轮提问）

用户误点重开后，模块1 的 `stage` 已被 `startModule` 重置为 `{kind:'module_intro'}`（全局 blob + per-module key 均被覆盖）。**未丢失**的是：`alc:state:attempts`（作答记录）、`alc:feynman:{id}`（费曼记录）。**已丢失**的是：`stage='done'` 这个终态标记。

- **无一键恢复**：`submitFeynman` 时触发过一次 `force` 自动备份（服务器 `.db` 快照），但从快照恢复单字段需直接操作 SQLite，**不推荐**。
- **务实建议**：接受重做。因 attempts 保留，题目层面的作答历史不丢；但需重新走完概念→挑战→费曼流程才能回到 done。若用户坚持恢复 done 态，可在确认后提供一段控制台手动改 `alc:progress:{id}` 的指引，属于"动数据"操作，风险自担。

本报告的修复（§7）会从根上杜绝此类误重开，但无法回滚已发生的重置。

## 6. B4 — 主题内模块进度呈现失真（层 ③ 根因，与 B3 联动）

### 6.1 现状

`TopicTransitionView.tsx:242-312` 渲染主题模块列表时：

```tsx
const status = session.moduleStatus[moduleId] ?? 'pending'  // ← 真相源是 session
const info = mod ? computeModuleProgress(mod, progressMap[moduleId] ?? null) : null
...
<span>{info?.label ?? '未开始'}</span>   // label 来自 per-module，但颜色/图标由 status 决定
```

进度条颜色、状态图标、文案主调全部由 `session.moduleStatus` 决定；`computeModuleProgress` 只提供 label/百分比的点缀。

### 6.2 与 B3 的联动

当 B3 发作（`moduleStatus` 因持久化缺口失真）时：

- 已完成模块1 的 `moduleStatus` 回退为 `pending` 或 `in_progress`
- 模块2 的 `moduleStatus` 恒为 `pending` → 即使模块2 有 per-module progress，主题页也呈现"未开始"
- 用户看到的主题进度条停滞、后续模块永远"未开始"

即使没有 B3，层 ③ 本身也有独立问题：用户若在主题外（题库页直接点开模块）对某模块做了题，`session.moduleStatus` 不会更新，主题页同样不反映。

### 6.3 根因

`session.moduleStatus` 与 `alc:progress:{id}` 是两套独立真相，无双向同步；`TopicTransitionView` 单方面采信 `moduleStatus`。

## 7. 统一修复方案

按根因层组织，层间有依赖关系，必须按序实施。

### 7.1 层 ② — 主题会话持久化（修 B3，B4 的前置）

**改动**：`topic-session-store.ts`

1. 抽出内部 helper `persistTopicProgress(session)`，把 `exitSession` 里的落盘逻辑（`:170-183`）复用。
2. `markCurrentModuleDone` / `skipCurrentModule` / `reenterModule` 末尾调用 `persistTopicProgress(get().session)`。
3. `advanceToNextModule` 也会改变 status，同样补落盘。

**效果**：任何路径重新 `startSession` 都能读到最新的完成/跳过集合，B3 闭合。

**增强（可选，推荐）**：`startSession` 入口先检查内存中是否已有同 `topicId` 的有效 session，若有则直接复用而非无条件重建。降低"误点重开"的破坏面。这是防御层，不是根因修复。

### 7.2 层 ③ — 进度呈现真相源切换（修 B4）

**改动**：`TopicTransitionView.tsx` 的模块状态推导

把 `status` 的计算从"单方面采信 session.moduleStatus"改为"session 优先，per-module progress 兜底校正"：

```tsx
const perModuleDone = progressMap[moduleId]?.stage.kind === 'done'
const effectiveStatus: ModuleTopicStatus =
  status === 'done' ? 'done'
  : perModuleDone ? 'done'                      // per-module 已完成但 session 没标 → 校正
  : (progressMap[moduleId] && status === 'pending') ? 'in_progress'  // 有进度但 session 没标
  : status
```

**效果**：即使 `moduleStatus` 因任何原因失真，主题页也能从 per-module progress 恢复正确呈现。B4 闭合，且对 B3 形成第二道防线。

**边界**：这只校正"呈现"，不回写 `session.moduleStatus`（避免引入新的同步复杂度）。session 的真值维护仍由层 ② 负责。

### 7.3 层 ① — 存储入口统一（修 B2.1，并还技术债）

**最小修复**：`ProductionHome.tsx:39`

```tsx
// 改前
const modules = listStoredModules(getStorage())
// 改后
const modules = listStoredModules(storage)
```

把读取入口从 `getStorage()`（production=SQLite）切到 `storage`（LS 单例），与 `library/page.tsx`、`TopicTransitionView` 等保持一致。一行改动闭合 B2.1。

**根本修复（架构债还清，推荐同期做）**：把 `alc:progress:{id}` 和 `alc:topicProgress:{id}` 的写入/读取统一纳入 `getStorage()`，消除 LS/SQLite 二分。这涉及：

- progress-store 的 subscribe 监听器（`progress-store.ts:474` 起）改写 per-module key 时用 `getStorage()` 而非 `storage`
- `topic-session-store` 的 `storage` 全部替换为 `getStorage()`
- 一次性迁移：首启动时把 LS 里的 per-module/topicProgress 搬到 SQLite

**取舍**：最小修复（一行）能立即闭合 B2.1 且风险极低；根本修复消除整个层 ①，但改动面大、需要迁移逻辑。**建议本次先做最小修复 + 把根本修复列为 V2.0.2 架构债专项**（AGENTS.md 已有记录），不混入本次联动修复。

### 7.4 B2.2 — 题库导航 UX（待用户选方案）

按 §4.3 实施选定方案（推荐 A 折叠/展开）。纯布局改动，不碰数据层。

### 7.5 B1 — done 页防御 + 待堆栈

1. **防御层**（本次必做）：done 页对 `feynmanAttempt`、`attemptsBySlot`、`mastery` 增加更严格的空值边界（即便 computeMastery 已防护，在渲染入口再加一层 optional chaining + 早返回）。

| 风险 | 缓解 |
