## 0. TL;DR

V2.1.5 发布后用户在主题学习场景下反馈：手动从题库点进主题内的模块2 作答，模块2 进度不记录（仍显示"未开始 · 已答 14 题"），主题进度仍停在模块1。用真实 `data/alc.db` 溯源后定位为**两个独立缺陷叠加**：

| # | 缺陷 | 根因层 |
|---|------|--------|
| **B5.1** | 手动跳转进非当前模块时，作答 stage 被错写到旧模块 per-module key，新模块永远显示未开始 | 进度双写同步链路 — 进入入口未同步 progress-store |
| **B5.2** | 主题内非 currentIndex 模块学完后 done 拦截失效，主题进度永不推进 | 主题编排 — done 拦截只认 currentIndex |

修复后全量单测 **879 passed**（+3 回归测试），typecheck/lint/format/双模式构建均通过。

## 2. 根因

### 2.1 B5.1：进入入口未同步 progress-store

进度双写机制（progress-store subscribe 监听器）按 `state.moduleId` 决定 per-module key 写入目标。作答 `advance()` 基于 `module-store.currentModule` 计算 stage，但写入目标由 `progress-store.moduleId` 决定。

**故障路径**：用户从题库页 `ModuleLibraryList.handleOpen` 点进模块2：

1. `handleOpen` 只调 `setModule(m2)`（module-store.currentModule=m2），不调 `resumeModule`（progress-store.moduleId 保持旧值 m1）
2. `router.push('/learn/module/m2')`，module page 挂载
3. module page 兜底 useEffect 的早返回条件 `currentModule?.id === routeModuleId`（m2===m2）命中 → **跳过 enterModule**
4. 用户作答 → `advance` 基于 m2 结构推进 stage → subscribe 监听器发现 `moduleId` 没变（还是 m1）→ 把 stage 写进 `progress(m1)`
5. `progress(m2)` 永不被写，停留在 `module_intro` → UI 显示"未开始 · 已答 14 题"

这违反 V2.0.1 建立的"所有进入模块的入口必须经 enterModule"约定。`TopicTransitionView.handleReenter` 正确调了 enterModule，但 module page 兜底与 `handleOpen` 漏了。

### 2.2 B5.2：done 拦截只认 currentIndex

`module page` 的 done 重定向（原 92-102 行）仅当 `topicSession.moduleIds[currentIndex] === routeModuleId` 时才走主题拦截（`markCurrentModuleDone`）。用户偏离主题编排学完非当前模块时，条件为 false → 走 `/learn/done`，模块在主题层面永不标记完成，`TopicProgress.completedModuleIds` 不更新。

## 3. 修复

| 文件 | 改动 | 对应缺陷 |
|---|---|---|
| `src/app/learn/module/[id]/page.tsx` | 兜底 useEffect 早返回条件增加 `progressModuleId === routeModuleId` 校验；不满足时调 `enterModule` 同步两个 store | B5.1 |
| `src/components/library/ModuleLibraryList.tsx` | `handleOpen` 有进度分支从裸 `setModule` 改为 `enterModule({ allowResume: true })`；已完成/无进度分支保留 `setModule`（避免对 done 模块触发 startModule 重置） | B5.1 |
| `src/lib/state/topic-session-store.ts` | 新增 `markModuleDone(moduleId)`：标记主题内任意模块完成并落盘 `TopicProgress` | B5.2 |
| `src/app/learn/module/[id]/page.tsx` | done 拦截增强：`routeModuleId` 在活跃 session 的 moduleIds 内即标记完成（currentIndex 走 `markCurrentModuleDone`，其他走 `markModuleDone`），统一回主题页 | B5.2 |

### 3.1 设计决策

- **module page 作为最后防线**：不只依赖入口合规，早返回条件同时校验 module-store 与 progress-store 两个真相源。即使未来出现新的裸 `setModule` 入口，module page 仍能纠正。
- **done 拦截分层**：保留"模块进度不反向改写主题编排"的 V2.1.5 分层原则——只在模块真正学完（stage=done）时才回写主题，作答中途不改 session。
- **已完成模块不走 enterModule**：`resumeModule` 对 `stage.kind==='done'` 的快照会走 `startModule` 语义（重置），故 `handleOpen` 对 completed 模块保留裸 `setModule` + 跳概览页。

## 5. 已知边界 / 后续

1. **模块1 历史数据回退**：用户模块1 的 per-module progress 因历史修 bug 回退到 `concept(1,2)`，但 attempts 完整（42 slot / 97 条，覆盖全部 concept + challenge）。本次修复只防止未来再发生，已错乱的存量数据需要用户确认后单独处理（推进 stage 到 feynman 或 done）。attempts 是真值，主题重刷与 Today due 不受影响。
2. **跨模块错题不自动穿插**：学习流程内的复习槽位（`collectReviewSlots`）是模块内的，不会把模块1 错题自动插入模块2 学习序列。跨模块复习走主题重刷页（`collectReviewItemsForModules`）与 Today due（`collectDueQueue`）。若期望"学新时自动穿插旧错题"，需另行评估 `advance` 的跨模块收集设计。
3. **topic-session.currentIndex 不主动跟随手动跳转**：用户从题库直进模块2 时，session.currentIndex 仍保持原值（V2.1.5 分层）。done 拦截增强已保证学完能标记完成；作答中途的主题页显示靠 `mergeModuleTopicStatus` 兜底提升为 in_progress。
