## 0. 背景

V2.1.2 阶段交付后用户反馈两个相关问题：

1. **Bug**：连续答错多次触发 force advance 时（UI 显示"已尝试多次，自动进入下一题"），"编辑此题"按钮被隐藏（`FeedbackPanel.tsx:172` 含 `!forceAdvance` 条件）。用户最需要修正 LLM 标错正解的场景反而进不去编辑器。
2. **Feature 需求**：在错题集里能够编辑一道题，编辑完成后下一次复习如果答对了，就直接记录为"从来没有错误过"。仅适用于编辑完之后第一次答题答对的情况。

---

## 1. 核心决策（brainstorming 锁定）

### 1.1 Force-advance 编辑入口 bug 修复

| 决策 | 选择 | 理由 |
|------|------|------|
| 修复方式 | 删除 `!forceAdvance` 条件 | force advance 时用户最需要修正 LLM 标错的正解；编辑保存后 `reevaluateLastAttempt` 可能让 nextAction 变 advance 通过 |
| 保留 `!quiz.ignored` | 是 | ignored 题用户主动忽略，编辑入口无意义 |

### 1.2 Amnesty 机制（4 主决策 + 4 次决策）

**主决策（一次性确认）**：

| # | 决策点 | 选择 |
|---|--------|------|
| D1 | 编辑入口位置 | **AnswerHistoryList 展开后加编辑按钮**（覆盖 ConceptView/ChallengeView/history 页；复习页后续扩展） |
| D2 | "从来没有错误过"数据语义 | **a. 删除该 slot 全部历史 attempt**（彻底忘记错过） |
| D3 | "编辑后第一次答对"边界 | **a. 任何场景，一次性**（学习流/Today/复习都算；答对消费 / 答错作废） |
| D4 | amnesty 触发后 attempt 处理 | **a. 保留 1 条 score=100** attempt（既符合"首答正确"语义，又不丢 streak/analytics） |

**次决策（推荐选项采纳）**：

| # | 决策点 | 选择 |
|---|--------|------|
| D5 | AnswerHistoryList 中哪些题显示编辑入口 | **a. 所有题**（与 AnswerCorrector 语义一致；用户也可能想修正答对的题） |
| D6 | 用户编辑后答错（amnesty 被消费），再次编辑 | **a. 重新生成 token**（允许用户多次修改答案后获得新 amnesty 窗口） |
| D7 | amnesty 复习页覆盖 | **a. 仅 AnswerHistoryList**（符合用户选项；review/* 后续扩展） |
| D8 | amnesty 后是否需要 attempt 特殊标记 | **不需要**（既然"从来没错过"，attempt 应看起来像普通首答正确） |

## 2. 实施方案

### 2.1 数据层（attempts-store）

新增 state + 方法：

```typescript
interface AttemptsStoreState {
  // 现有字段...
  pendingAmnesty: Record<string, string>  // key=slotId, value=token（存在性检查）

  markPendingAmnesty: (slotId: string) => void      // 编辑时调用，生成新 token 覆盖
  hasPendingAmnesty: (slotId: string) => boolean    // 查询
  clearPendingAmnesty: (slotId: string) => void     // 清除（不消费）
  // 现有方法...
}
```

`addAttempt` 改造（amnesty 消费）：

```typescript
addAttempt: (attempt) => {
  const slotId = attempt.originalQuizId
  const existingAmnesty = get().pendingAmnesty[slotId]

  if (existingAmnesty) {
    get().clearPendingAmnesty(slotId)  // 无论对错都消费 token
    if (attempt.score >= 80) {
      // 答对：清空历史，仅保留这次
      set((state) => ({
        attemptsBySlot: { ...state.attemptsBySlot, [slotId]: [attempt] }
      }))
      scheduleLibrary.remove(slotId)  // 清空 FSRS schedule（调用方会重建）
    } else {
      // 答错：amnesty 作废，按正常流程 append
      // (走 default 分支)
    }
  }
  // default: append to existing
}
```

### 2.2 UI 层（AnswerHistoryList）

- 展开后详情区底部加"编辑此题"按钮（`module.origin !== 'showcase'` 守卫）
- 点击切换到内嵌 `AnswerCorrector`（V2.1.2 已精简的组件）
- `onSave`：调 `correctQuizAnswer(quizId, patch)` + `markPendingAmnesty(slotId)` + 关闭编辑模式
- `onCancel`：仅关闭编辑模式
- 单 `editingSlotId` state（同时只能编辑一道题）

### 3.2 关键技术点记录

- **`clearSlot` 已存在**（`attempts-store.ts:93`）—— 删除 slot 全部 attempt + schedule，但 amnesty 实现没用它（因为 amnesty 保留 1 条新 attempt），而是直接 set + scheduleLibrary.remove
- **schedule 重建时序**：amnesty 触发后 `addAttempt` 内部 `scheduleLibrary.remove(slotId)` 清空旧 schedule；调用方（ConceptView/ChallengeView/TodayReviewView/review-page）的 handleAnswer 在 addAttempt 后调 `synchronizeScheduleForSlot`，基于剩余 attempts（仅 1 条 score=100）重建 schedule。语义正确
- **AnswerHistoryList.tsx 加 `import React from 'react'`**：组件用 JSX 但原文件未显式 import React；Next.js/SWC 自动 JSX runtime 下能跑，但 vitest esbuild 转换需要 React 在模块作用域。这是测试环境限制，非代码质量问题
- **`clearAll` 不清 `pendingAmnesty`**：现有 `clearAll` 只清 `attemptsBySlot` + `scheduleLibrary.clearAll()`。残留 token 无害（下次 addAttempt 时无 attempt 历史所以清空操作无副作用），暂不修

## 5. 已知风险 / 未解决问题

1. **Amnesty + Topic 跨 module 复习**：Topic 复习跨多个 module，用户在 module A 的 AnswerHistoryList 编辑了某题，下次在 Topic 复习中答对——amnesty 是否正确触发？理论上应该（pendingAmnesty 按 slotId 索引，跨 module 共享 attempts-store），但需 E2E 验证。
2. **Amnesty + 多设备同步**：production 模式下 attempts-store 持久化到 SQLite，pendingAmnesty 字段会跟随持久化。多设备同步场景未测试（当前 production 是单机 localhost 方案，无多设备）。
3. **AnswerHistoryList editingSlotId 状态保持**：折叠/切换题目后 state 不清空，再次展开同一题编辑模式恢复。子代理认为可接受 UX，待用户实际验证。
4. **Force-advance 修复未单独测试**：删除 `!forceAdvance` 条件的修复未加 E2E 测试覆盖，仅依赖手动验证。T5 中可考虑补一个 force-advance + 编辑入口可见的 E2E 用例。
