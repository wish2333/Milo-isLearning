## 0. TL;DR

V2.1.1 修复了 V2.1.0 收尾审计中识别的 1 项 P1 UI 缺陷并补全对应的 UX 软引导：

scope 严格收敛在 1 个组件文件 + 1 个测试文件；不动批量协议、不动 pipeline、不动 cancel API、不动 globals.css。

## 2. 决策记录

### 2.1 D-1：UI 层禁用按钮（选项 A）而非 API 层加 retryable 校验（选项 B）

**选择**：UI 层在 `job.status === 'failed'` 时不渲染"恢复"按钮（fix-report §2.5 选项 A），让用户只能走"重试此项"（已有 retryable 校验）。

**理由**：
- fail-fast 状态机下同一时刻 job 内最多 1 个 failed item，禁用"恢复"对用户操作成本影响极小（最多 1 次额外点击）
- 选项 B 会让 resume API 语义复杂化（需扫描所有 failed item、判定全部 retryable），与 fail-fast 状态机重复
- 改动最小（1 行 JSX + 1 个测试），与 V2.1.1 最小 scope 原则一致
- API 层"resume 不校验 retryable"留作 V2.2.0 候选（fix-report §7.2），未来协议变更前 UI 防护足够

### 2.2 D-2：软引导而非硬阻断

**选择**：`attempts >= 3` 时显示诊断提示，**不**禁用"重试此项"按钮。

**理由**：高可靠 ≠ 高限制。强制阻断会剥夺用户判断权（"我知道 source 没问题，就是 provider 偶发抖动"）。attempts 高意味着系统性问题，UI 引导用户进入诊断路径（检查 source / constraints / provider），而不是默默让用户点 100 次。硬上限（如 attempts ≥ 10 禁用按钮 + acknowledge 流程）留作 V2.2.0 候选（fix-report §3.4）。

### 2.3 D-3：JSX 追加而非包裹

**选择**：在现有 `<div>尝试 {item.attempts} 次</div>` 之后追加新 `<p>`，**不**把现有 attempts span 包裹进 `item.status === 'failed'` 条件。

**理由**：现有展示让所有 status（done/running/queued）的 item 都显示 attempts，便于用户看到"已完成的 Module 也曾尝试 N 次"。包裹会回归此展示。fix-report §3.3.2 V1.1 校正了 V1.0 的包裹错误。

### 2.4 D-4：阈值 3/5 是产品 UX 阈值

**选择**：3 与 5 作为软引导阈值，常量化（非配置化）。

**理由**：fix-report §3.3.1 V1.1 校正了 V1.0 把阈值关联到 `MAX_ATTEMPTS=5` 的错误（该常量是 agent runner 的 schema 校验重试上限，与批量扩充 attempts 字段无直接对应关系）。3/5 依据是"极低吞吐量"场景下的内测观察（典型用户 retry 不超过 2 次；超过 3 次几乎可断定系统性问题）。阈值为常量便于后续根据真实使用反馈调整。

## 4. 已知不满足项 / 非阻塞观察

明确写为**本版不满足**或**非阻塞观察**（沿用 fix-report §3.4 + §5.3 + §6）：

1. **视觉堆叠（非阻塞）**：当 failed item 的 `attempts >= 5` 且 `error` 存在时，新追加的诊断 `<p>`（danger 色）与原 error `<p>`（`bg-danger/10`）会上下堆叠。两段语义不同（操作建议 vs 技术细节 `[code] message`），堆叠可接受；fix-report §5.3 列为"中概率/低影响"。V2.2.0 候选：合并到同一容器或重组视觉层级。

2. **resume API 在 failed 场景下仍不校验 retryable（协议层防护）**：V2.1.1 仅在 UI 层禁用按钮，API 层 `action=resume` 在 job.status='failed' 时仍会绕过 retryable。若未来出现新的失败恢复入口（如外部脚本、其他客户端），UI 防护不足以兜底。fix-report §3.4 列为 V2.2.0 候选。

3. **attempts 硬上限**：当前无上限，理论上可累积到任意值。fix-report §3.4 列为 V2.2.0 候选（如 attempts ≥ 10 禁用重试 + acknowledge 流程）。

4. **失败诊断结构化**：`ExpandJobError` 仅 `code/message/retryable`，未透传 `pipeline/errors.ts` `ERROR_TABLE` 的可操作诊断信息。fix-report §3.4 列为 V2.2.0 候选。

5. **失败 item source 就地编辑**：当前失败后必须放弃整个 job 重新发起。fix-report §3.4 列为 V2.2.0 候选。

6. **`retry-all-failed` 批量入口**：在 fail-fast 状态机下同时刻最多 1 个 failed item，无批量对象可操作（fix-report §4.2）。若未来产品语义转向"高吞吐量"再考虑。

### 5.2 V2.2.0 候选（基于本次评估衍生）

- **失败诊断结构化**：`ExpandJobError` 增加 `hint` / `suggestedAction` 字段

- **失败 item source 就地编辑**

### 5.3 V2.3.0+ 候选（产品语义若变化）

- 引入 `retry-all-failed` 入口
- 批量协议增加 "skip-failed-and-continue" 模式（与当前 fail-fast 互斥）
- 增量索引/FTS5 重新评估
