## 2. #1 单 item 失败即停止 — 确认设计正确，不改

### 2.1 现状

`compile-topic-with-expand.ts` 在 item 失败时的处理（L256-286 与 L298-322）：

```ts
if (itemError || !compiledModule) {
  // ...构造 error payload...
  library.updateItem(
    options.jobId,
    currentItem.itemId,
    { status: 'failed', attempts, error: { ... } },
    options.repository,
  )
  library.update(options.jobId, { status: 'failed' }, options.repository)
  yield { kind: 'item_failed', jobId, itemId, error, retryable }
  return  // ← 单 item 失败 → 整个 job 标记 failed → 立即停止，后续 item 不执行
}
```

后续 item 不会被启动。

### 2.2 证据：设计是有意的，不是 bug

| 证据类型 | 位置 | 内容 |
|---------|------|------|
| plan 协议原文 | `docs/v2.1.0/v2.1.0-plan.md` §3 P1.2 协议状态机 | "failed：某 item 多次重试后仍失败；**其他已完成 item 保留**" — 显式只承诺保留已完成 item，未承诺"后续 item 继续" |
| 状态迁移表 | `src/lib/persistence/expand-job-library.ts:88-95` | `failed → ['running', 'cancelled']` — 允许 failed→running（用户 retry 后继续），符合 fail-fast 后由用户驱动恢复的语义 |
| 单测覆盖 | `src/lib/compiler/pipeline/__tests__/compile-topic-with-expand.test.ts:164` | 用例 "单 item 失败时保留已完成 Module 与 done checkpoint" 显式验证"失败前的 done item 仍保留" |
| 单测覆盖（cancel 类比） | 同文件 L242 | "cancel 在边界停止后续 item，保留已完成 Module" — cancel 与 failed 都采用"边界停止"语义，协议一致 |
| Review 记录 | `docs/v2.1.0/v2.1.0-Review.md` §1.10 P1.2 验收 | "暂停恢复只执行未完成项"（验证 sourceHash + status filter）但未承诺"失败时跳过该项继续" |

### 2.3 在"高可靠 + 极低吞吐"语义下的判断

**结论：✅ 设计正确，不改。**

理由：

1. **fail-fast 符合高可靠性**：单 item 失败可能是系统性问题（source 不适合、constraints 不当、provider 限流、prompt 不适配）。继续跑后面的 item 大概率也会失败，浪费 token 与用户时间。
2. **极低吞吐量降低早停代价**：典型批量 3-10 个主题，第 3 个失败时最多影响后续 item 的延迟。用户本来就不会一次批量几十个，"早停 + 逐项 retry"的操作成本可接受。
3. **"已完成 item 保留"已兑现承诺**：plan §3 协议只承诺"已完成 item 保留"，未承诺"失败后续 item 继续"。实现与协议对齐。
4. **用户驱动恢复符合"可靠性"语义**：失败后由用户判断"是 source 的问题还是偶发" → 决定 retry 或放弃 → 比系统盲目继续更可靠。
5. **若改成"跳过失败项继续"反而是反模式**：在极低吞吐量下，用户看到"N 个失败"会困惑（"是不是我的 source 都不对？"），不如"早停 + 单项 retry"的诊断路径清晰。

### 2.4 失败恢复的两条路径（V1.1 新增，对应审计 P1-1）

V1.0 报告只描述了"重试此项"路径，遗漏了 `ExpandJobView` 的"恢复"按钮在失败时也会显示。完整路径如下：

**路径 A：点"重试此项"（item 级，正确路径）**

- 触发：`ExpandJobView.tsx:621-629` 失败 item 卡片上的按钮，仅当 `item.status === 'failed' && item.error?.retryable` 时渲染
- API：`POST /api/compile/expand-job action=retry`（`route.ts:93-107`）
- 校验：
  - `item.status === 'failed'`（否则 409 `expand_job_item_not_failed`）
  - `item.error.retryable === true`（否则 409 `expand_job_item_not_retryable`）
- 效果：item 重置为 `queued` + error 清空；job 从 `failed` 置回 `running`；客户端触发新 SSE pipeline

**路径 B：点"恢复"（job 级，问题路径）**

- 触发：`ExpandJobView.tsx:547-556` `{(job.status === 'paused' || job.status === 'failed') && (...)}` —— **失败时也渲染**
- API：`POST /api/compile/expand-job action=resume`（`route.ts:89-92`）
- 校验：**仅检查 `job.status` 是否属于 `{paused, failed, created}`**，不校验任何 item 的 retryable
- 效果：job 置为 `running`；客户端 `ExpandJobView.tsx:440` 的 `if (action === 'resume' || action === 'retry')` 分支触发新 SSE pipeline；pipeline 重启后 `getTopicExpandRunnableItems` 过滤 `status !== 'done'`（`compile-topic-with-expand.ts:60-62`），**会再次启动 failed item，包括 `error.retryable === false` 的项**

### 2.5 P1 缺陷判断与修复方案（V1.1 新增）

**缺陷定性**：路径 B 在失败场景下绕过 retryable 校验，与"高可靠"语义直接冲突——用户可能对"不可重试"的错误反复点"恢复"，每次都触发 LLM 调用、每次都失败、无任何引导。

**两个修复选项**：

| 选项 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| A（推荐）失败时禁用"恢复"按钮 | UI 改 1 行：`{job.status === 'paused' && (...)}`（移除 `failed` 分支） | 语义清晰："暂停后恢复" ≠ "失败后继续"；强制用户走"重试此项"（已有 retryable 校验）；改动最小 | 用户若有多项 retryable 失败（fail-fast 下不可能同时多项 failed），仍需逐项重试 |
| B 让 resume action 校验 retryable | API 加逻辑：若 job 有 failed item，校验所有 failed item 的 retryable，全部可重试才允许 resume；UI 加错误提示 | 保留"恢复"按钮的便利性 | API 语义复杂化；部分可重试 + 部分不可重试的边界需要新文案；与 fail-fast 状态机（最多 1 个 failed item）的设计重复 |

**推荐选项 A**，理由：

1. 在 fail-fast 状态机下，同一时刻 job 内最多只有 1 个 failed item（其他 item 要么 done，要么因 job 失败而未触达）。失败时禁用"恢复"对用户操作成本的影响极小（最多 1 次额外点击：点"重试此项"而非"恢复"）。
2. "恢复"的语义本就是"暂停 → 继续"，而不是"失败 → 重启"。失败时让该按钮消失，能让 UI 更准确反映可用的操作集。
3. 选项 A 改动最小（1 行 JSX 条件 + 1 个测试），与 V2.1.1 "最小 scope" 原则一致。

**T1 实现**（`src/components/learn/ExpandJobView.tsx:547`）：

```diff
-              {(job.status === 'paused' || job.status === 'failed') && (
+              {job.status === 'paused' && (
```

**T4 测试**：新增用例 "job.status='failed' 时不渲染'恢复'按钮"。

## 3. #2 attempts 无上限 — 中优先级 UI 软引导

### 3.1 现状

`attempts` 字段生命周期：

| 阶段 | 位置 | 行为 |
|------|------|------|
| 创建 | `expand-job-library.ts:133` | `attempts: 0` |
| 进入 running | `compile-topic-with-expand.ts:219` | `const attempts = currentItem.attempts + 1`，写入 `updateItem(..., { attempts })` |
| LLM/compile 失败 | `compile-topic-with-expand.ts:264-276` | `attempts` 透传到 checkpoint，未被清零 |
| 持久化失败 | `compile-topic-with-expand.ts:298-322` | 同样 `attempts` 累积 — **不涉及 LLM 调用** |
| 用户 retry | `expand-job/route.ts:104` | `updateItem(jobId, itemId, { status: 'queued', error: null })` — **只清 error，attempts 保留** |
| UI 展示 | `ExpandJobView.tsx:607` | `<span>尝试 {item.attempts} 次</span>`，所有 item 都显示 |

**问题**：没有任何地方对 `attempts` 数值做判断或上限保护。用户可以对一个持续失败的 item 无限点击"重试此项"，每次 attempts +1，UI 只显示"尝试 N 次"而无引导。

### 3.2 在"高可靠 + 极低吞吐"语义下的判断

**结论：🟡 保留为中优先级。改 UI（软引导），不改协议/不改 pipeline/不强制阻断。**

为什么"软引导"而不是"硬上限"：

- 高可靠 ≠ 高限制。强制阻断会剥夺用户判断权（"我知道 source 没问题，就是 provider 偶发抖动，让我再试一次"）。
- 高可靠 = 强诊断。attempts 高意味着系统性问题，UI 必须引导用户进入诊断路径（检查 source / constraints / provider 状态），而不是默默让用户点 100 次。
- "极低吞吐量"下，attempts 应该是低值（典型 1-3 次）。如果超过，几乎可以断定系统性问题。

### 3.3 修复设计（V2.1.1 scope）

#### 3.3.1 阈值与文案（V1.1 校正）

| `attempts` 值 | UI 表现 | 文案 |
|--------------|---------|------|
| 1-2 | 现状不变 | "尝试 N 次" |
| 3-4 | 黄色提示 + 诊断建议 | "已尝试 N 次，建议检查源 Markdown 与约束文本" |
| ≥5 | 红色警告 + 强诊断建议 | "已尝试 N 次仍失败，强烈建议编辑源 Markdown 后重新提交；或确认 provider 配置正常" |

**阈值依据（V1.1 校正）**：

- V1.0 报告把阈值与 `MAX_ATTEMPTS=5`/`25 次 LLM 调用`关联是**错误的**。事实是：
  - `MAX_ATTEMPTS = 5` 位于 `src/lib/compiler/agents/_runner.ts:29`，是 **agent runner** 的 schema 校验重试上限
  - 主 pipeline 的瞬时错误重试位于 `pipeline.ts:448` `MAX_RETRIES = 3`（即最多 4 次调用）
  - `attempts` 字段还会在**持久化失败**时累积（`compile-topic-with-expand.ts:298-322`），此时**不涉及任何 LLM 调用**
  - 因此 `attempts = 5` **不能**等同于"至少 25 次 LLM 失败"
- 正确表述：3 与 5 是**产品 UX 阈值**，依据是"极低吞吐量"场景下的内测观察（典型用户 retry 不超过 2 次；超过 3 次几乎可断定系统性问题）。阈值为常量，便于后续根据真实使用反馈调整。

## 4. #3 retry-all-failed 批量入口 — 撤销建议（论证校正）

### 4.1 原建议

评估清单建议：增加 `POST /api/compile/expand-job action=retry-all-failed`（无 `itemId`），把所有 `status='failed' && error.retryable` 的 item 一次性置为 `queued`，让用户一键重试所有失败项。

### 4.2 在"高可靠 + 极低吞吐"语义下的判断（V1.1 校正论证）

**结论：❌ 撤销建议，不实现。**

V1.0 给出的理由包含两处不准确表述（"最多 9 次点击"暗示有多个失败项；"批量失败往往是系统性问题"在 fail-fast 下不成立）。V1.1 校正后的论证如下：

1. **协议下没有批量对象**：在 §2.1 的 fail-fast 状态机下，单 item 失败立即 `library.update(...'failed') + return`，job 内**同一时刻最多只有 1 个 failed item**（其他 item 是 done/queued/cancelled 状态）。`retry-all-failed` 在现有协议下没有"批量"目标可操作。
2. **与失败恢复语义重叠**：失败后用户已有两条恢复路径（§2.4 路径 A "重试此项" 与路径 B "恢复"；V2.1.1 通过禁用路径 B 收敛到只剩路径 A）。新增 `retry-all-failed` 会让"失败恢复"的入口数量超过实际需要。
3. **未来扩展空间**：若产品语义转向"高吞吐量"（允许多 item 并发失败），可重新评估。届时需要：
   - 协议层修改 fail-fast 状态机（允许 job 内多 failed item 共存）
   - 配套定义"是否重置 attempts"、"并发与串行策略"等新协议点
   - 与 §3.4 "失败诊断面板"组合提供（汇总所有失败项的 error code 分布）

这些是 V2.2.0+ 候选，不在 V2.1.1 内决策。

### 4.3 替代方案（不在 V2.1.1 内）

若未来产品语义转向"高吞吐量"，再考虑：

- 引入 `retry-all-failed` 但**重置 attempts**（避免 attempts 累积失真）
- 协议层允许"partial-completed"状态（用户可显式放弃失败项，让 job 进入 completed 而非 failed）
- 增加"诊断面板"汇总所有失败项的 error code 分布，帮助用户判断"是不是同一个原因"
