## 0. TL;DR

V2.1.0 发布后用户反馈答题环节存在 2 类 UX 缺陷，溯源后拆为 3 个修复点：

| # | 缺陷 | 用户原话 | 修复方向 |
|---|------|---------|---------|
| **F1** | 选择题答错后选项上没有任何视觉标记正解 | "答错之后正确答案没有显示到选项上，不直观还得去看解析才能知道正确答案是哪一项" | ChoiceQuiz 在 `disabled=true` 时给正解加 success 边框 + ✓，给错选加 warning 边框 + ✗ |
| **F2** | FeedbackPanel 解析区域前出现两段无 label 的复述文字（LLM `gaps` 字段） | "解析里面出现了两个复述选项的文字或者没有（这部分 UI 也没标是啥我就是看到了这个东西在真正的解析前面）" | F1 实施后 gaps 信息冗余，**直接隐藏**；保留 AttemptRecord 持久化以兼容未来分析 |
| **F3** | AnswerCorrector 把"选正解"和"编辑选项文本/题干/解析"混在一起 | "我点击编辑之后让我选选项其实感觉很不直观，每个选项后面跟这个输入框也意义不明" → 用户明确"完全去掉文本编辑功能" | AnswerCorrector 精简为"只指定正确答案"；Choice 走 radio，Sorting 保留箭头，FillBlank 仅留正解输入框；Stem/Explanation 字段移除 |

**实施周期预估**：0.5–1 天（含 3 个组件改造 + 单测 + E2E 回归 + 双模式 build）。

## 1. 背景：用户反馈与产品语义

### 1.1 用户反馈原文（首轮）

> 1. 当前题目编辑功能没有用，而我发现一般答案出错的都是会显示两个正确答案，我点击编辑之后让我选选项其实感觉很不直观，每个选项后面跟这个输入框也意义不明
> 2. 答错之后正确答案没有显示到选项上，不直观还得去看解析才能知道正确答案是哪一项

### 1.2 二次澄清（用户确认）

> 1. 不对，首先 UI 没标正确答案，然后解析里面出现了两个复述选项的文字或者没有（这部分 UI 也没标是啥我就是看到了这个东西在真正的解析前面）
> 2. 完全去掉文本编辑功能
> 3. 实施前先撰写详细报告到 2.1.0 文档文件夹中 2.1.2-fix-report

### 1.3 关键澄清点

- **"两个正确答案"是错觉**：UI 既没标正解选项，又在解析前渲染了 LLM 给的 `gaps`（最多 2 条无 label 文本，常复述选项内容）。两者叠加让用户产生"两个正确答案"的观感。
- **AnswerCorrector 改造方向**：完全去除文本编辑，仅保留"指定正确答案"这一核心动作。
- **实施顺序约束**：必须先产出本报告，等用户评审对齐边界后再动代码。

### 1.4 产品语义

本项目是中文优先的个人学习工具，答题反馈是用户与学习材料互动的最高频触点：

| 维度 | 含义 |
|------|------|
| 反馈即时性 | 答错后用户应在 1 秒内通过视觉锁定正解，不需要阅读文本 |
| 编辑器场景 | 用户的典型场景是"LLM 把答案标错了 / 有两个选项看着都对，想指定真正的正解"；不是改题干/解析/选项文本 |
| 信息降噪 | 解析区域不应出现语义不明的裸文本块，每段信息都要有明确 label 或被吸收到更强的视觉信号里 |

## 2. F1 — ChoiceQuiz 答错后未标记正解选项

### 2.1 现状（`src/components/quiz/ChoiceQuiz.tsx:49-86`）

```tsx
{shuffledOptions.map((option, i) => {
  const isSelected = selected === option
  return (
    <button
      ...
      className={`alc-option w-full text-left text-base ${
        disabled ? 'cursor-default opacity-60' : ''
      }`}
      data-selected={isSelected ? 'true' : undefined}
    >
      <span>{String.fromCharCode(65 + i)}</span>
      {option}
    </button>
  )
})}
```

提交后（`disabled=true`）：
- **所有选项统一变 `opacity-60` + `cursor-default`**
- 只有用户选中的选项有 `data-selected` 标记（仅靠 CSS selector 控制，无明显视觉差异）
- **完全没有标记哪个是正解**（`quiz.answer` 没有传入组件）

### 2.2 根因

1. `ChoiceQuizProps` 接口只接收 `{ quiz, disabled, onAnswer }`，缺少 `userAnswer`（用户上次选择）的传入路径。
2. disabled 状态下的渲染逻辑没有按"正解 / 错选 / 未选"做差异化样式。
3. 现有调用方（5 处）虽然各自持有 `userAnswer`，但没透传到 QuizRenderer。

### 2.3 调用方 blast radius（5 处）

| # | 文件 | 调用形式 | userAnswer 来源 |
|---|------|---------|---------------|
| 1 | `src/components/learn/ConceptView.tsx:323` | `<QuizRenderer quiz={quiz} disabled={phase !== 'answering'} onAnswer={handleAnswer} />` | `handleAnswer(userAnswer)` 已接收，需新增 state 持有 |
| 2 | `src/components/learn/ChallengeView.tsx:336` | 同上 | 同上（L109-131） |
| 3 | `src/components/learn/today/TodayReviewView.tsx:200` | 同上 | 同上（L56-74） |
| 4 | `src/app/learn/review/[moduleId]/page.tsx:399` | `<QuizRenderer ... />` | 复习页面从 attempts-store 取 latestAttempt.userAnswer |
| 5 | `src/app/learn/review/topic/[topicId]/page.tsx:386` | 同上 | 同上 |

**结论**：5 处调用方都已有 `userAnswer` 上下文，改造只是把它从各自组件 state（或 attempts-store）透传到 `QuizRenderer` → `ChoiceQuiz`。

### 2.4 修复方案

#### 2.4.1 接口扩展

```tsx
// src/components/quiz/QuizRenderer.tsx
interface QuizRendererProps {
  quiz: Quiz
  disabled: boolean
  onAnswer: (userAnswer: string) => void
  /** 已提交的答案（disabled=true 时用于显示正解/错选标记），可选 */
  submittedAnswer?: string
}

// src/components/quiz/ChoiceQuiz.tsx
interface ChoiceQuizProps {
  quiz: Quiz
  disabled: boolean
  onAnswer: (userAnswer: string) => void
  /** 已提交答案；disabled=true 时与 quiz.answer 一起决定每项样式 */
  submittedAnswer?: string
}
```

Sorting/FillBlank 不接收 `submittedAnswer`（用户未反馈它们的视觉问题，本次不动）。

#### 2.4.2 渲染逻辑

```tsx
{shuffledOptions.map((option, i) => {
  const isSelected = selected === option           // 未提交时的临时选中
  const isSubmitted = disabled && submittedAnswer  // 已提交
  const isCorrectAnswer = option === quiz.answer
  const isUserWrong = isSubmitted && option === submittedAnswer && !isCorrectAnswer

  let stateClass = ''
  let suffix: string | null = null
  if (isSubmitted && isCorrectAnswer) {
    stateClass = 'border-success/60 bg-success-soft/40 text-success'
    suffix = '✓'
  } else if (isUserWrong) {
    stateClass = 'border-[color:var(--danger)]/50 bg-warning-soft/30 text-warning'
    suffix = '✗'
  } else if (isSubmitted) {
    stateClass = 'opacity-50'  // 未选的其他项
  }

  return (
    <button className={`alc-option ${stateClass}`} ...>
      <span>{String.fromCharCode(65 + i)}</span>
      {option}
      {suffix && <span className="ml-auto">{suffix}</span>}
    </button>
  )
})}
```

#### 2.4.3 视觉规格（ASCII 原型）

**答对时**（submittedAnswer === quiz.answer）：

```
┌──────────────────────────────────────┐
│ A. 选项一                            │
│ B. 选项二 ✓  ← sage success 边框     │  ← 用户选对了
│ C. 选项三                            │
│ D. 选项四                            │  (其他项 opacity-50)
└──────────────────────────────────────┘
```

**答错时**（submittedAnswer !== quiz.answer）：

```
┌──────────────────────────────────────┐
│ A. 选项一                            │
│ B. 选项二 ✗  ← muted terracotta 边框 │  ← 用户选错了
│ C. 选项三 ✓  ← sage success 边框     │  ← 这是正解
│ D. 选项四                            │
└──────────────────────────────────────┘
```

Design tokens 沿用 `globals.css`：
- success: `#7fa88c`（sage）
- warning/danger: muted terracotta（非亮红，遵循项目"no bright red"约定）
- 已有 utility class：`text-success` / `bg-success-soft` / `border-success/60` / `text-warning` / `bg-warning-soft`

### 2.5 边界

- **Sorting / FillBlank 本次不动**：用户未反馈这两个题型的视觉问题，且它们的"已提交状态"展示逻辑独立。
- **Showcase 模式兼容**：showcase 题库的 choice quiz 同样适用本改造（不改 mode-dispatch 逻辑）。
- **答对时不再展示正解以外的项高亮**：避免视觉噪音，未选项统一 `opacity-50`。

## 3. F2 — FeedbackPanel 解析前的"裸文本"是 `gaps`

### 3.1 现状（`src/components/quiz/FeedbackPanel.tsx:89-115`）

```tsx
{/* Feedback text */}
<p>{feedback.feedbackText}</p>

{/* Gaps */}
{feedback.gaps.length > 0 && (
  <ul className="space-y-1">
    {feedback.gaps.map((gap) => (
      <li className="text-xs text-fg-tertiary flex items-start gap-2">
        <span>-</span>
        <span>{gap}</span>
      </li>
    ))}
  </ul>
)}

{/* Explanation */}
{explanation && (
  <div>
    <p>解析</p>
    <p>{explanation}</p>
  </div>
)}
```

UI 渲染顺序：

1. `feedbackText`（LLM 鼓励性反馈，**无 label**）— 不在用户投诉范围
2. **`gaps`**（LLM 给的"用户遗漏关键点"，**无 label**，最多 2 条，每条 ≤30 字）— **这就是用户说的"两段复述选项文字"**
3. `explanation`（`quiz.explanation`，有"解析" label）

### 3.2 用户反馈的"两段复述选项文字"溯源

对照 feedback agent prompt（`src/lib/compiler/prompts/feedback.md`）和 schema（`src/lib/compiler/schemas/feedback.ts`）：

- `gaps` schema：`maxItems: 2`，每条 `maxLength: 30`
- prompt 规则：Choice 答错时 `gaps` = "1-2 条用户忽略的关键点（来自 `quiz.distractors` 或 `concept.keyPoints`）"
- `distractors` 本身就是错误选项文本 → LLM 复述时常常直接抄录

→ 用户看到的"两段无 label 复述文字" = `gaps` 数组，UI 上前面只有 `-` 符号，没有任何说明这是什么。

### 3.3 根因

两层叠加：

1. **ChoiceQuiz 没标正解选项**（F1）→ 用户不知道哪个对
2. **FeedbackPanel 把 LLM 的 `gaps` 渲染为无 label bullet list** → 用户看到"在真正的解析前面的两段文字"，语义不明；且 LLM 倾向于直接复述选项文本，与正解信息重复

### 3.4 修复方案（**选 A：隐藏 gaps 区域**）

F1 实施后，正解选项已有视觉高亮，`gaps` 的"复述选项"信息完全冗余。直接隐藏 `gaps` 的 UI 渲染：

```tsx
// FeedbackPanel.tsx
// {/* Gaps */}            ← 删除整个 gaps 渲染块
// {feedback.gaps.length > 0 && (...)}
```

**理由**：

- `gaps` 本质是把"你忽略了 distractor 1 / distractor 2"换个说法，F1 已在选项层标记，再渲染一次是噪音
- 减少 UI 层级，解析区域只保留 `feedbackText`（鼓励）+ `explanation`（解析）+ `misconception`（易错点）
- 不调 LLM prompt / schema，保留 `gaps` 字段在 API 响应和 AttemptRecord 持久化中（未来可做学习分析）

### 3.5 备选方案（**选 B：加 label**，本次不采用）

> 保留 `gaps` 渲染，但加 label "容易忽略的关键点"（与 `misconception` 的"容易卡住的地方"风格一致），并修改 prompt 禁止复述选项文本。

不采用理由：

- 即使加 label，信息仍与选项高亮 + 解析重复
- 修改 prompt 需要重跑 prompt eval（`scripts/prompt-eval.ts`），scope 放大
- 用户已明确反馈"这部分 UI 没标是啥我就是看到了" → 隐藏比加 label 更彻底降噪

### 3.6 影响

| 维度 | 影响 |
|------|------|
| UI | FeedbackPanel 减少 1 个信息块，解析区域更聚焦 |
| 数据 | `feedback.gaps` 仍由 LLM 生成、仍写入 `AttemptRecord.gaps`，未来分析可用 |
| 兼容 | showcase 题库的 mock feedback 仍包含 `gaps`，不渲染但不报错 |
| 测试 | 现有涉及 `gaps` DOM 的测试可能需要调整（需 grep 确认） |

## 4. F3 — AnswerCorrector 精简

### 4.1 现状（`src/components/quiz/AnswerCorrector.tsx`）

当前 496 行，对 3 种题型分别实现：

| 题型 | 当前编辑器 | 用户反馈 |
|------|----------|---------|
| Choice（`ChoiceEditor` L64-130） | 圆形字母按钮（点击设正解）+ 每项 input 文本框 + 删除按钮 + 添加选项 | "让我选选项其实感觉很不直观，每个选项后面跟这个输入框也意义不明" |
| Sorting（`SortingEditor` L136-177） | 箭头调整顺序 | 未投诉 |
| FillBlank（`FillBlankEditor` L183-247） | 正确答案 input + 其他可接受答案 input + 答案提示 input | 未投诉 |

顶部还有 `StemField`（L30-43，题干编辑）+ `ExplanationField`（L45-58，解析编辑）作为公共字段。

### 4.2 根因

用户的真实场景是"LLM 把答案标错了 / 有两个选项看着都对，想指定真正的正解"。但当前 UI 把所有字段都做成可编辑：

- 主操作（"选正解"）反而不显眼 —— 字母按钮和文本框并排，视觉重心落在 input
- 每项都有 input 暗示"你应该改文本"，与用户意图错位
- 顶部 Stem/Explanation 编辑器占用视觉空间，进一步稀释主操作

### 4.3 修复方案（用户明确："完全去掉文本编辑功能"）

#### 4.3.1 Choice editor → 单选 radio 列表

```
┌─ 编辑此题 ─────────────────────────┐
│                                    │
│ 指定正确答案（点击切换）：         │
│                                    │
│ ○ A. 选项文本一                    │
│ ● B. 选项文本二    ✓ 当前正解      │  ← success 色
│ ○ C. 选项文本三                    │
│ ○ D. 选项文本四                    │
│                                    │
│ [保存]  [取消]                     │
└────────────────────────────────────┘
```

- **整行可点击**（radio 行为）
- 当前正解项用 `border-success/50 bg-success-soft/30` + 实心 `●`
- **去掉**：选项文本 input、"删除选项"按钮、"添加选项"按钮
- **去掉**：顶部 `StemField` 和 `ExplanationField`

#### 4.3.2 Sorting editor → 保持现状

箭头调整顺序属于"指定正确答案"的另一种形态（不是文本编辑），**保留不动**。

#### 4.3.3 FillBlank editor → 仅保留正解输入

FillBlank 题型的正解本质是文字，无法用 radio 操作。保留：

- ✅ **正确答案 input**（核心，无法去掉）

去掉：

- ❌ "其他可接受答案" input（`acceptableAnswers` 编辑）
- ❌ "答案提示" input（`answerHint` 编辑）

> **⚠️ 待用户确认**：FillBlank 去掉 `acceptableAnswers` 编辑后，用户将无法在 UI 上添加同义答案。如果用户大量使用这个功能，需要重新评估。详见 §6 待确认 #2。

#### 4.3.4 公共字段 → 全部移除

- ❌ `StemField`（题干编辑）→ 移除
- ❌ `ExplanationField`（解析编辑）→ 移除

`QuizEditPatch` 类型保持不变（只是 patch 实际不再包含 stem/explanation/acceptableAnswers 等字段），向后兼容。

### 4.4 改造后文件规模预估

| 文件 | 改造前 | 改造后（预估） |
|------|-------|--------------|
| `AnswerCorrector.tsx` | 496 行 | ~180 行 |

主要删减：
- 移除 `StemField`（14 行）
- 移除 `ExplanationField`（14 行）
- 简化 `ChoiceEditor`（67 行 → ~25 行）
- 简化 `FillBlankEditor`（65 行 → ~15 行）
- 主组件 state 减少（stem/explanation/acceptableAnswers/answerHint 相关 useState 全删）

### 4.5 边界与不可逆性

- 删除的功能（题干编辑、解析编辑、选项增删、同义答案编辑）**不可逆**，git history 保留可随时 revert
- `onCorrectAnswer(patch)` 接口签名不变，调用方（FeedbackPanel → module-store）无需改造
- 现有的"纠题后 schedule 全量重放"逻辑（`contentRevision` 变更触发）保持不变
- 不影响 F43 ignored quiz 管理（路径完全独立）

## 6. 待用户确认的边界

本报告基于二次澄清"完全去掉文本编辑功能"展开，但有几处细节需要实施前对齐：

### #1 F2 修复方向确认

- **方案 A（推荐）**：隐藏 `gaps` 区域（保留底层持久化）
- **方案 B**：加 label "容易忽略的关键点" + 改 prompt 禁止复述选项

报告 §3.5 已论证推荐 A。请确认。

【用户：我确认A】

### #2 FillBlank 编辑器是否保留 `acceptableAnswers`（同义答案）编辑

用户原话"完全去掉文本编辑功能"严格解读应一并去掉。但 FillBlank 题型中同义答案（如"RAG" / "检索增强生成"）是常见需求，去掉后用户无法在 UI 上补充。

- **选项 a（严格按用户原话）**：仅保留正解 input，去掉同义答案和答案提示
- **选项 b（实用主义）**：保留正解 + 同义答案，去掉答案提示

请选 a 还是 b。

【用户：我确认B】

### #3 Stem / Explanation 编辑是否真去掉

用户原话"完全去掉文本编辑功能"覆盖 stem 和 explanation。但：
- 去掉 stem 编辑 → 用户无法修正 LLM 生成的题干错别字
- 去掉 explanation 编辑 → 用户无法补充解析

请确认是否真要全去掉（报告 §4.3.4 默认全去）。

【用户：我确认去掉】

### #4 `gaps` 字段在 AttemptRecord 中的持久化是否保留

UI 不再渲染，但 `attempts-store` 仍会写入 `gaps`（`feedback.gaps` 来源不变）。

- **选项 a（推荐）**：保留持久化，未来可做"用户常忽略点"分析
- **选项 b**：一并从 AttemptRecord 移除，彻底清理

请选 a 还是 b。

【用户：我确认A】

## 7. 非目标（明确不做）

- **不改 feedback agent prompt / schema**：`gaps` 字段保留在 LLM 输出契约中，只是不渲染
- **不改 AttemptRecord 数据形状**：`gaps` 字段保留在持久化层
- **不改 SortingQuiz / FillBlankQuiz 的 disabled 状态视觉**：用户未反馈，本次不动
- **不改 `QuizEditPatch` 类型定义**：保持向后兼容
- **不改 `onCorrectAnswer` 调用链**：FeedbackPanel → module-store 路径不变
- **不引入新的 design token**：全部复用现有 `success` / `warning` / `danger` 变量
