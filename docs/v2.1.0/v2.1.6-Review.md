## 2. 关键决策

### 2.1 normalizeReviewSlots 主题感知

**计划 gap 纠正**：原计划伪代码直接将跨模块 slot 塞入 `normalizeReviewSlots(currentModule, [...])`，但该函数会过滤掉不属于 currentModule 的 quiz ID。实测若不修正，跨模块 slot 在注入后被立即剔除，不会进入游标运算。

**修复**：给 `normalizeReviewSlots` 加可选 `extraValidIds` 参数（topic 模块的 quiz ID 集合），在 topic 上下文下传参，使游标运算（normalized）与 ConceptView 渲染（raw `stage.reviewSlots`）对 `reviewSlots.length` 保持一致，避免游标错位。

### 2.2 getTopicContext 实现

`getTopicContext()` 从 `useTopicSessionStore.getState().session` 读取 topic 上下文，通过 `loadStoredModule` 加载模块。每次 `advance()` 调用一次，非主题学习流返回 null（零行为变化）。MVP 未做 memo 缓存（主题 ≤10 模块，benchmark 可接受）。

### 2.3 collectCarriedReviewSlots 向后兼容

新增可选 `currentModule` 参数：提供时跳过不属于当前模块的 slot（穿插题不 carry）；未提供时保持旧行为。现有调用点（progress-store L381）传入 `currentModule`，测试文件保持旧行为。向后兼容，不破坏已有单测。

### 2.4 E2E 顺序无关策略

跨模块 E2E 使用两个 2-concept 模块 fixture（`mockModule-cross` + `mockModule-cross-b`），seed 两个模块的 concept 1 quiz 错题到 localStorage。主题起始模块顺序由库排序决定，但无论哪个先学，另一个都能提供跨模块错题。`addInitScript` 直接 seed 模块（绕过导入的 id 重写，保留已知 quiz id），使 seed 的错题 slot id 与模块 quiz id 一致。

### 2.5 组件测试延后

P2.2 要求的 ConceptView 组件测试（`ConceptView.cross-module.test.tsx`）因无既有组件测试基建（需 mock 5+ stores + storage + quiz rendering），延后到 P3.1 E2E 覆盖。E2E 更可靠（真实浏览器环境），且 carry 排除逻辑已有单元测试。

## 4. 已知边界与后续候选

- **settings 开关**：当前主题入口默认开启穿插，不增加用户配置项（YAGNI）。如需可后续版本加。
- **全库扩展**：当前仅主题内其他模块，不扩展到全题库。后续可考虑全库 due 穿插。
- **positionLabel 细化**：跨模块 slot 计入复习题分母（totalSlots），但 conceptPercent 不受影响。若 UX 反馈混乱，可加 label 区分"跨模块复习"。
- **FSRS 关闭时的穿插力度**：FSRS 关闭只能用 wrong（最近答错），若主题内其他模块无错题则不穿插。属预期降级。
- **attempt 归属原模块**：在模块 A 作答模块 B 的题，模块 B 的 mastery/stats 会变化。这是预期行为（复习就该影响原模块），但用户需知晓。
- **组件测试**：ConceptView 跨模块渲染组件测试因基建不足延后到 E2E 覆盖。若后续增加组件测试框架，可补充。

## 5. 与 V2.1.5 的衔接

V2.1.5 B5 修复（`enterModule` 统一入口 + `markModuleDone` 跨模块同步）为本版的跨模块数据访问提供了正确基础。`progress-store.advance()` 的 `getTopicContext()` 依赖 `topic-session-store` 的 session 状态，而 V2.1.5 确保主题会话的进度持久化正确，使本版的 topic 上下文读取可靠。
