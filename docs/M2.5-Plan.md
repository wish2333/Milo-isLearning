# M2.5 工作计划（M2 → M3 过渡）

> **M2.5 Plan V1.0**
> 版本：1.0 | 状态：Draft | 日期：2026-07-07
> 定位：M2（Prompt 工程闭环，mock 单测通过）与 M3（Knowledge Compiler 串联）之间的**工程过渡里程碑**，非 PRD §14 正式里程碑。

---

## 0. 为什么需要 M2.5

M2 按 Tech Spec §16 的验收口径是「单 Agent 单测（**含 mock LLM**）」，已达标（见 [`M2-Review.md`](./M2-Review.md)）。但有两类工作被 Tech Spec §17.2 / Prompt-Eng §12 / M1-Review §6 明确标为"M2 末验证"或"留待 M2 末"，它们**不阻塞 M2 验收**，却是 M3 编译器串联前的真实质量地基：

1. **mock → 真实 LLM**：M2 全部用 mock provider，真实 DeepSeek/GLM 的 Schema 通过率、Retry 触发率、业务约束达成率（Concept 数 ∈ [2,5] 等）从未实测
2. **三个 A/B 决策悬而未决**（Prompt-Eng §12）：模型选型、Prompt 中英文、enable_thinking 开关——这些直接影响 M3 pipeline 的默认配置

M3（编译流水线串联）如果直接在未经验证的真实 LLM 上跑，出问题难定位是 Prompt 问题还是串联问题。M2.5 把单 Agent 在真实 LLM 上跑稳，再交棒 M3 做串联。

```
M2（mock 闭环）───────► M2.5（真实 LLM 验证 + 工程基建）───────► M3（编译器串联）
  已完成                  本计划                                    下一步
```

**M2.5 不阻塞 M3 启动**：若资源允许，M3 的 `/api/compile` 端点骨架可与 M2.5 并行。但 M3 的集成测试（输入 Markdown → 输出合法 Module）依赖 M2.5 验证过的 Prompt + 模型组合。

## 2. 交付物详情

### W1 fixtures 测试集

目录：`src/lib/compiler/__fixtures__/`

| 文件 | 主题 | 类型 | 长度 | 用途 |
|------|------|------|------|------|
| `rag-medium.md` | 检索增强生成 | 技术（theory/procedure） | ~2000 字 | Import/Chunk/Concept 主测集 |
| `feynman-medium.md` | 费曼学习法 | 理论（theory） | ~2000 字 | 跨主题稳定性验证 |
| `gitflow-medium.md` | Git 工作流 | 流程（procedure） | ~2000 字 | procedure 类 Concept 覆盖 |
| `edge/very-short.md` | 200 字符边界 | — | ~200 字 | Import 下限边界 |
| `edge/code-heavy.md` | 代码块占 >50% | — | ~1500 字 | Import 保留代码块约束验证 |
| `rag-chunks.json` | 预切分 chunks | — | — | Concept Agent 独立测试输入（避免依赖 Chunk 串联） |

来源：手工撰写（对齐 prompt-eval §3.3「用户决策：Few-shot 数据手工撰写」）。

### W2 eval 脚本

文件：`scripts/prompt-eval.ts`

```
bun run scripts/prompt-eval.ts \
  --agent concept \
  --provider glm \
  --model glm-5.2 \
  --fixture rag-chunks \
  --runs 5 \
  --thinking off
```

行为：
- 从 `.env.local` 读 API key；缺 key 则 skip 该 provider
- 对指定 (agent, provider, model, fixture) 跑 N 次 `runAgent`
- 收集指标：Schema 通过率 / JSON 合规率 / Retry 触发率 / 业务约束达成 / 延迟 P50/P95
- 输出 Markdown 报告到 `reports/prompt-eval/{date}-{provider}-{model}.md`（gitignored）

### W3 extraBody 透传

改动面：
- `providers/types.ts`：`ChatRequest` 加 `extraBody?: Record<string, unknown>`
- `providers/openai-compat.ts`：`buildRequestInit` 把 `extraBody` merge 进请求 body
- `agents/config.ts`：`disableThinking` 已就位（M2 已建字段，全 true）
- `agents/_runner.ts`：读 `config.disableThinking`，设 `req.extraBody = { enable_thinking: false }`
- DeepSeek V4 不认 `enable_thinking`（其 thinking 是模型固有），透传后被忽略，无副作用

### W5 映射层

文件：`src/lib/compiler/agents/mappers.ts`

| Agent 输出（snake_case） | domain.ts（camelCase） |
|---|---|
| `feedback.next_action` | `AttemptRecord.nextAction` |
| `feedback.feedback_text` | （运行时临时字段，不入 domain） |
| `quiz-eval.rubricResults[].hit` | （保持，已是 snake 风格枚举值） |

实际上 9 个 Schema 中只有 `feedback` 用了 snake_case 字段（`next_action`/`feedback_text`），其余 schema 字段已是 camelCase（如 `normalizedText`/`conceptOrder`/`rubricResults`）。所以映射层很薄，主要是 feedback + 个别处。M2.5 建统一 mapper + 单测，供 M3 `assemble()` 调用。

### W6 质量门禁

- `prettier`（已在 devDeps）：加 `.prettierrc` + `format` script
- `eslint`：加 flat config + `lint` script（next/core-web-vitals + strict ts）
- `husky` + `lint-staged`：pre-commit 跑 prettier + tsc on staged files

## 3. 验收标准

M2.5 完成的判定（全部 Must 项达标）：

| 验收项 | 目标 | 验证 |
|--------|------|------|
| fixtures 就位 | 6 个文件（3 主题 + 3 边界 + 1 chunks） | 目录存在 + eval 脚本能读 |
| eval 脚本可跑 | 至少一个 (provider×agent×fixture) 组合跑通出报告 | `bun run eval` 退出码 0 + 报告文件生成 |
| extraBody 透传 | GLM 请求 body 含 `enable_thinking:false` | openai-compat 单测断言 |
| A/B 决策记录 | V1/V2/V3 各一份报告 + §4 决策表填写 | `reports/` 下报告 + 本文档 §4 更新 |
| 映射层 | 9 Agent mapper 函数 + 单测 | `mappers.test.ts` 通过 |
| 质量门禁 | `bun run lint`/`format` 零错误，pre-commit 生效 | husky 装好 + 提交触发 |

### 真实 LLM 指标基线（参考 prompt-eval §1，非硬门）

| 指标 | 目标 | 说明 |
|------|------|------|
| Schema 通过率 | ≥ 80%（真实 LLM，prompt-eval §5 放宽自 90%） | mock 是 100%，真实 LLM 更易触发 retry |
| JSON 合规率 | ≥ 90% | |
| Retry 触发率 | ≤ 30% | 放宽自 20% |
| Concept 数 ∈ [2,5] | 同一 fixture 5 次方差 ≤ 1.5 | 放宽自 1 |

未达标不阻塞 M2.5 结项，但触发 Prompt 调优迭代（属于 M2.5 的 W4 决策内容）。

## 4. A/B 决策表（W4 执行后回填）

> 本节由 eval 报告数据驱动填写。M2.5 W4 阶段已完成有限规模 smoke 验证（每个组合 1-3 次），
> 完整统计意义上的 A/B（每组 5+ 次的 P50/P95 收敛）推迟到 M3 编译流水线串联后随实际负载跑出。

| 变量 | 选项 | 决策 | 依据（报告链接） |
|------|------|------|----------------|
| V1 模型（编译主体：Chunk/Concept/Module/Mission/Quiz/Feynman） | `glm-5.2` vs `glm-5-turbo` | **`glm-5.2`** | `glm-5-turbo` 在 chunk agent（2k 字符 md 输入）100% 超时（30s）；concept agent 跑通但延迟 P50=16.6s 偏高。turbo 不适合结构化 JSON 中等负载场景，编译主体用旗舰。报告：`reports/prompt-eval/2026-07-07-glm-glm-5-turbo-chunk-rag-medium.md` / `...-concept-rag-chunks.md` |
| V1 模型（轻量 Agent：Import / Feedback / Feynman-Eval） | `glm-5.2` vs `glm-5-turbo` | **`glm-5-turbo`** | import agent 在 turbo 上 100% schema 通过、P50 ≈ 3s，比旗舰更快更便宜。轻量 Agent 用 turbo。报告：`reports/prompt-eval/2026-07-07-glm-glm-5-turbo-import-edge-very-short.md` |
| V2 Prompt 语言 | 中文 vs 英文 | **维持中文（M2.5 不变）** | W8（英文模板）属 Should 项，M2.5 资源未投入；中文 prompt 在 import/concept 上 100% schema 通过率已达标，V2 A/B 推迟到 V1.1 |
| V3 enable_thinking | on vs off | **off（与 AGENT_CONFIG 默认值一致）** | M2.5 全部 smoke run 均在 `--thinking off` 下完成且通过；thinking on 的对照留待 W4 完整 A/B（需 ≥5 次每组、要观察延迟与 token 成本），M2.5 不阻塞决策。 |

### 决策落地点（M3 接手）

M3 `pipeline.ts` 默认模型表（来自本节决策）：

```ts
const DEFAULT_COMPILE_MODEL = 'glm-5.2'       // 6 个编译主体 Agent
const DEFAULT_LIGHTWEIGHT_MODEL = 'glm-5-turbo' // Import / Feedback / Feynman-Eval
```

## 6. 与 M3 的衔接

M2.5 交付给 M3 的输入：

1. **验证过的默认模型组合**（W4 决策）→ M3 `pipeline.ts` 的 `defaultCompileModel` / `defaultFeedbackModel`
2. **映射层**（W5）→ M3 `assemble()` 直接调用，把 9 Agent 的 snake_case 输出组装成 camelCase `Module`
3. **extraBody 机制**（W3）→ M3 调 runAgent 时 thinking 开关已就位
4. **fixtures**（W1）→ M3 集成测试复用 `rag-medium.md` 作为"输入 Markdown → 输出合法 Module"的测试输入
5. **质量门禁**（W6）→ M3 起所有提交受 lint/format 约束

M3 不必等 M2.5 全部完成：M3 可先做 `/api/compile` 端点骨架与 SSE 协议，集成测试待 M2.5 的 W4 决策出来后再锁定模型跑。

## 7. 风险

| 风险 | 等级 | 缓解 |
|------|------|------|
| 真实 LLM Schema 通过率远低于 80% | 中 | W4 触发 Prompt 调优；必要时回退到 Few-shot（提前 W8/V1.1） |
| A/B 跑出"两个模型差不多"无法决策 | 低 | 引入延迟/成本作为 tie-breaker；默认选 GLM（本仓库主供应商） |
| DeepSeek V4 2026-07-24 旧 ID 退役影响测试 | 低 | W4 前确认用 v4 新 ID；ping 脚本已校验 |
| husky 在 Windows 下 pre-commit hook 不稳 | 低 | 用 husky v9 + `lint-staged`；失败时降级为 CI 门禁 |
