# V2.0.1 实现计划 — 主题学习作答进度丢失修复

> **For Claude/Sisyphus:** REQUIRED SUB-SKILL: 用 `superpowers:executing-plans` 按 task 顺序执行；每 batch 派发 subagent 前先 load 对应 skill。
> 对应设计文档：[`v2.0.1-fix-report.md`](./v2.0.1-fix-plan.md) （**V1.1**，Proposal 已锁，6 项修订全部落地）
> ⚠ 本 plan 的 task 细节以 fix-report V1.1 §5 为准。如 task 描述与 report 冲突，以 report 为准。

**Goal:** 修复 V2.0.0 暴露的主题学习作答进度丢失缺陷（S2/S3 P0 数据丢失 + S4 P1 体验缺陷），并补齐 storage invariant 注释、细粒度进度展示、E2E re-entry 门禁。

**Architecture:** 单点根因——5 处 Topic 入口无差别调用 `startModule` 重置 stage。修复方案 3 件套：
1. `progressStore.resumeModule(moduleId)` 三级 fallback action（snapshot → 全局 blob → startModule）
2. `enterModule({ moduleId, allowResume })` 共享 helper（5 处入口替换）
3. `computeModuleProgress(module, progress)` 纯函数（细粒度 UI 展示）

**Tech Stack:** Next.js 15 + React 19 + Zustand + TypeScript（strict）+ vitest + Playwright

**总周期：** ~2–2.5 天（8 个 task，5 个批次）

**⚠ 执行注意：**
- Plan 中文件路径基于 fix-report V1.1 §2–§5 已核实的引用，但派发 subagent 前必须用 codegraph/grep 再次确认行号（特别是组件 handler 位置）
- 所有新增单测**必须在 Node 环境跑通**（vitest `environment: node`），涉及 browser storage 的需 mock
- **不读 `.env.local`**（ANTI-PATTERNS）
- 本 plan 不引入数据迁移、不改持久化 schema、不动 attempts-store / FSRS schedule

2. 实现逻辑：
   ```ts
   export function enterModule({ moduleId, allowResume = true }: EnterModuleOptions): boolean {
     const storedModule = storage.get<Module>(StorageKeys.module(moduleId))
     if (!storedModule) return false
     useModuleStore.getState().setModule(storedModule)
     if (allowResume) {
       useProgressStore.getState().resumeModule(moduleId)
     } else {
       useProgressStore.getState().startModule(moduleId)
     }
     return true
   }
   ```
3. 文件顶部 JSDoc 明确："依赖 Zustand store，必须在浏览器环境调用；测试需 mock store 与 storage"
4. 单测覆盖：
   - `allowResume=true` + module 存在 → 调 `resumeModule`，返回 true
   - `allowResume=false` + module 存在 → 调 `startModule`，返回 true
   - module 不存在 → 返回 false，不调任何 store action
5. **验收 gate：** 单测全绿；helper 可被 F1-3 的 5 处入口复用；`bun run typecheck` 通过
