## 0. TL;DR

V2.0.0 暴露一个影响主题学习的**作答进度丢失缺陷**：5 处 Topic 入口（TopicCard、TopicTransitionView ×3、ShowcaseHome）进入模块时无条件调用 `progressStore.startModule(moduleId)`，该 action 把 `stage` 强制重置为 `{ kind: 'module_intro' }`，丢弃 `conceptIndex` / `quizIndex` / `feynmanAttempt`。Library 入口不调用此 action，所以保留进度——两条路径的不对称是全部症状的单一根因。

V1.1 在 V1.0 基础上修订了 6 项关键问题：(P0) 明确 storage backend 二分现实并把 invariant 写入前置条件；(P1) `resumeModule` 加入全局 blob fallback；(P1) `computeModuleProgress` 指标重新定义；(P1) 入口范围扩展到 5 处并明确 Showcase 纳入；(P1) E2E 升级为门禁；(P2) 4 处语义定稿。

修复方案：**新增 `progressStore.resumeModule(moduleId)` action**，按"per-module snapshot（LS） → 全局 blob fallback → startModule"三级恢复顺序；把 5 处 Topic 入口对 `startModule` 的调用替换为条件化 `resumeModule`；同时新增 `computeModuleProgress` 纯函数（仅基于 concept 维度，不计算"已答题数"），在题库和主题卡片展示细粒度进度。

实施周期预估：**2–2.5 天**（8 个 task，含 storage invariant 注释、E2E 门禁、双模式 build、回归）。

## 1. 症状与复现

### 1.1 用户报告的 4 个症状

| 编号 | 症状 | 严重度 |
|------|------|--------|
| S1 | 从主题学习进入的模块作答进度，退出后在题库点"继续"能接着做 | 现象（非 bug 本身，见 §2.3） |
| S2 | 在主题卡片点"开始主题学习"会重置当前模块进度 | **P0 数据丢失** |
| S3 | S2 之后题库的"继续"也丢失进度 | **P0 数据丢失（连带）** |
| S4 | 题库未显示模块做到第几个概念；主题学习进度只显示模块级 4 态，颗粒度太粗 | **P1 体验缺陷** |

### 1.2 复现路径

**S2/S3（100% reproducible）：**

1. 主题卡片"开始主题学习" → 进入第一个模块的 `/learn/module/{id}`
2. 做几道题（stage 推进到 `concept(2, 3)` 等）
3. 退出（返回题库、关闭浏览器）→ 题库"继续"**能恢复**（per-module `alc:progress:{id}` 已写入 LS）
4. **再次**点主题"开始主题学习" → `startModule(id)` 触发 → `stage` 被覆盖为 `module_intro`
5. 监听器把 `module_intro` 同步覆盖到 per-module key（见 §2.3）
6. 题库"继续"也丢失进度（S3）

**S4：** 题库卡片仅显示 `"3 概念 · 12 题 · 日期"`，无"做到第几概念"等细粒度；主题卡片/TopicTransitionView 仅显示 `"1/5 模块已完成"`（且当前实现把 skipped 算入完成统计，§5.6 会修正）。

## 2. 根因分析

### 2.1 两条调用链对比

**Library 入口（正确，保留进度）：**

```
ModuleLibraryList.handleOpen(summary)      [src/components/library/ModuleLibraryList.tsx:206-223]
  → moduleStore.setModule(storedModule)     // 只设 currentModule，清 currentQuiz
  // ⚠ 不调 startModule
  → router.push('/learn/module/{id}')       // stage 从 Zustand persist 自动 rehydrate
```

`TopicSection.handleOpen`（题库内 Topic 区块）走相同路径，同样正确。

**Topic 入口（错误，重置进度）——共 5 处调用点（V1.1 修正）：**

| 调用点 | 文件:行 | 触发场景 | 模式 |
|--------|---------|---------|------|
| `TopicCard.handleStartTopic` | `src/components/library/TopicCard.tsx:70` | 主题卡片"开始主题学习"/"继续主题" | production/library |
| `TopicTransitionView.handleContinue` | `src/components/learn/TopicTransitionView.tsx:87` | 模块完成后过渡页"继续" | production/library |
| `TopicTransitionView.handleSkip` | `src/components/learn/TopicTransitionView.tsx:101` | 过渡页"跳过当前" | production/library |
| `TopicTransitionView.handleReenter` | `src/components/learn/TopicTransitionView.tsx:110` | 过渡页"重新进入已 skipped 模块" | production/library |
| `ShowcaseHome.handleStartTopic` | `src/components/home/ShowcaseHome.tsx:77` | 首页 Showcase 主题"开始学习" | **showcase** |

五处的代码模板相同：

```ts
useModuleStore.getState().setModule(moduleData)
useProgressStore.getState().startModule(moduleId)   // ← 罪魁祸首
router.push(`/learn/module/${moduleId}`)
```

### 2.2 罪魁祸首：`startModule` 的无条件重置

**文件**：`src/lib/state/progress-store.ts:104-110`

```ts
startModule: (moduleId) =>
  set({
    moduleId,
    stage: { kind: 'module_intro' },   // ← 强制重置，丢弃 conceptIndex/quizIndex
    updatedAt: Date.now(),
    feynmanAttempt: null,               // ← 清空费曼作答
  }),
```

该 action 的语义是"**首次进入一个全新模块**"，重置到 `module_intro` 是合理的。问题在于调用方无差别使用它处理"恢复已开始的模块"，导致已有进度被覆盖。

### 2.3 数据流追踪：为什么 S1 能"继续"但 S3 失败

`progress-store` 有**双写持久化**（`progress-store.ts:418-437`）：

```
set({ stage }) ─┬─→ Zustand persist 自动写：getStorage() 在 production=SQLite / showcase=LS
                └─→ subscribe 监听器同步写：storage.set (永远 LS) 到 alc:progress:{moduleId}
```

**S1（首次退出后题库能继续）：**
- 用户做到 `concept(2, 3)` 时，subscribe 监听器把 stage 同步到 LS 的 `alc:progress:{id}`
- 题库"继续"用 `storage.get`（LS）读取 → 直接 push `/learn/module/{id}` → progress-store rehydrate ✅

**S2（主题再入重置）：**
- TopicCard 调 `startModule(id)` → `set({ stage: { kind: 'module_intro' } })`
- 监听器立刻把 `module_intro` 同步覆盖到 LS 的 `alc:progress:{id}`
- 原进度被永久覆盖

**S3（题库继续也丢）：**
- LS 的 per-module key 已被 S2 覆盖为 `module_intro`
- 题库"继续"读到的就是被覆盖后的值，进度彻底丢失

## 3. 前置条件：Storage Backend 现状与约束（V1.1 新增，P0）

### 3.1 现状：storage 实例二分（架构债）

`progress-store.ts:26-28` 同时引入两个 storage 实例：

```ts
import { getStorage } from '@/lib/persistence/client/storage'         // production=SQLite ClientFetchStorageRepository / showcase=LS
import { storage } from '@/lib/persistence/client/local-storage'      // 永远是 LS（LocalStorageRepository 单例）
```

| 写入端 | storage 实例 | 物理位置（production） | 物理位置（showcase） |
|--------|-------------|----------------------|---------------------|
| Zustand persist blob `alc:state:progress` | `getStorage()` | SQLite | LS |
| subscribe 监听器 → `alc:progress:{moduleId}` | `storage` | **LS**（非 SQLite） | LS |
| `onRehydrateStorage` → `alc:progress:{moduleId}` | `storage` | **LS** | LS |
| Library/TopicSection 读 per-module | `storage` | **LS** | LS |
| `listStoredModules` 读 per-module | `storage` | **LS** | LS |

**结论：per-module snapshot 永远在 LS，无论模式。** 全局 progress blob 在 production 走 SQLite，在 showcase 走 LS。

### 3.2 V2.0.1 范围内的修复策略：与现有 Library 路径一致（用 `storage` 读 LS）

**关键决策**：`resumeModule` 必须使用 `storage`（LS）读取 per-module snapshot，与 Library 入口、`listStoredModules` 保持一致。**不能用 `getStorage()`**，否则 production 模式下读不到 per-module（数据在 LS 不在 SQLite），修复失效。

这是 V2.0.1 的**最小且必要**修复，不引入架构变更。

### 3.3 Storage Invariant（写入前置条件，实施时必须验证）

实施 F1-0 时必须在 `progress-store.ts` 顶部以注释形式明确以下不变式（invariant），并在 AGENTS.md ANTI-PATTERNS 加入对应条目：

> **Storage Invariant (V2.0.1)**：`progress-store` 的 per-module key (`alc:progress:{moduleId}`) 在所有运行模式下均通过 `storage`（LocalStorageRepository 单例）读写。`getStorage()` 仅用于 Zustand persist 的全局 blob。任何"读 per-module"的新代码必须用 `storage`，否则在 production 模式下会读到空值。

### 3.4 长期改进（V2.0.2 候选，不在本次范围）

把 per-module snapshot 也迁到 `getStorage()`，使全局 blob 和 per-module snapshot 物理同库。这需要：
- 把现有 LS 中的 per-module key 迁移到 SQLite
- `subscribe` / `onRehydrateStorage` 改用 `getStorage()`
- 所有读 per-module 的位置（`module-library.ts:listStoredModules`、`ModuleLibraryList.handleOpen`、`TopicSection.handleOpen`、`IgnoredQuizSection` 等）改用 `getStorage()`

该项作为架构债在 v2.0.2 单独处理，**不在 V2.0.1 范围内**。

## 5. 修复设计

### 5.1 设计原则（V1.1 调整）

1. **不破坏 `startModule` 现有语义**：它仍是"首次进入"的正确实现
2. **显式区分"首次进入"与"恢复"**：新增 `resumeModule` action
3. **恢复优先级**：per-module snapshot → 全局 blob → startModule（fallback）
4. **进度展示只展示可稳定计算的指标**：基于 concept 维度，不展示"已答题数"（因 quizIndex 可能落在 reviewSlot 上，§5.5）
5. **修复点最小化**：不重构 ModuleStage 状态机、不引入新 store、不改持久化 schema
6. **Storage invariant 写入代码**：通过注释 + lint 风格约束防止再次混用

### 5.2 数据层（无 schema 变更）

复用既有 LS key：

| 数据 | Key | Storage 实例 |
|------|-----|-------------|
| per-module 进度快照 | `alc:progress:{moduleId}` | `storage`（LS，所有模式） |
| 当前活跃全局进度 | `alc:state:progress` | `getStorage()`（production=SQLite / showcase=LS） |
| 模块内 concept 总数 | `module.concepts.length` | `storage`（从 `alc:module:{moduleId}`） |

无需新增 StorageKey。

### 5.3 `progress-store.resumeModule` —— 三级恢复顺序（V1.1 重设计）

**文件**：`src/lib/state/progress-store.ts`

**新增 action：**

```ts
/**
 * 恢复模块进度（不重置）。
 *
 * 恢复优先级（V1.1）：
 *   1. 读取 LS 中的 per-module snapshot（storage.get）
 *   2. 若当前全局 blob 的 moduleId === 目标 moduleId，比较 updatedAt，取更新的快照
 *   3. 若快照 stage.kind === 'done' → 走 startModule 语义（让用户从头看）
 *   4. 若无任何有效进度 → 走 startModule 语义
 *
 * Storage Invariant: 必须用 `storage`（LS）读 per-module，不能用 getStorage()
 * 见 §3.3。
 *
 * 已知限制（§9）: feynmanAttempt 在恢复时清空（per-module snapshot 不保存该字段）
 */
resumeModule: (moduleId: string) => {
  // 候选 1: LS 中的 per-module snapshot
  const snapshot = storage.get<ProgressState>(StorageKeys.progress(moduleId))

  // 候选 2: 当前全局 blob（若 moduleId 匹配）
  const current = get()
  const globalMatched =
    current.moduleId === moduleId && current.stage !== null
      ? { stage: current.stage, updatedAt: current.updatedAt }
      : null

  // 选 updatedAt 更新的（per-module 可能落后于全局 blob，见 progress-store.ts:411 注释）
  let winner: { stage: ModuleStage; updatedAt: number } | null = null
  if (snapshot && snapshot.stage && snapshot.stage.kind !== 'done') {
    winner = { stage: snapshot.stage, updatedAt: snapshot.updatedAt }
  }
  if (
    globalMatched &&
    globalMatched.stage.kind !== 'done' &&
    (!winner || globalMatched.updatedAt > winner.updatedAt)
  ) {
    winner = globalMatched
  }

  if (winner) {
    set({
      moduleId,
      stage: winner.stage,
      updatedAt: winner.updatedAt,
      feynmanAttempt: null, // 见 §9 已知不满足项
    })
    return
  }

  // 无有效进度或已完成 → 走 startModule 语义
  set({
    moduleId,
    stage: { kind: 'module_intro' },
    updatedAt: Date.now(),
    feynmanAttempt: null,
  })
},
```

**设计要点（V1.1）：**

- **三级 fallback**：snapshot → 全局 blob → startModule，确保关闭页面未切换的场景也能恢复
- **done 状态走重置**：与 Library handleOpen 的 `summary.completed` 分支一致
- **feynmanAttempt 清空**：明确为已知不满足项（§9），不再标为"低风险"
- **幂等**：多次调用同一 moduleId 不破坏数据
- **storage invariant**：使用 `storage`（LS）读 per-module，与 Library 入口、listStoredModules 一致

### 5.4 入口修复策略（5 处）

**抽取共享 helper**：

**新文件**：`src/lib/runtime/enter-module.ts`

```ts
import { useModuleStore } from '@/lib/state/module-store'
import { useProgressStore } from '@/lib/state/progress-store'
import { storage } from '@/lib/persistence/client/local-storage'
import { StorageKeys } from '@/lib/persistence/shared/keys'
import type { Module } from '@/types/domain'

export interface EnterModuleOptions {
  moduleId: string
  /**
   * 是否允许恢复已有进度（默认 true）。
   * 用户显式选"重做/重新学习"时传 false，走 startModule 语义。
   */
  allowResume?: boolean
}

/**
 * 进入模块的统一入口。
 *
 * 依赖 Zustand store（useModuleStore / useProgressStore），
 * 因此必须在浏览器环境调用，不能在 SSR / 纯 Node 测试中直接调
 * （测试需 mock store 与 storage）。
 *
 * 返回 true 表示成功加载，false 表示模块不存在。调用方负责 router.push。
 */
export function enterModule({ moduleId, allowResume = true }: EnterModuleOptions): boolean {
  const storedModule = storage.get<Module>(StorageKeys.module(moduleId))
  if (!storedModule) return false

  useModuleStore.getState().setModule(storedModule)

  if (allowResume) {
    useProgressStore.getState().resumeModule(moduleId)
  } else {
    useProgressStore.getState().startModule(moduleId)
  }

  return true
}
```

- ~~返回 `route`~~ → 改为返回 `boolean`（成功/失败），跳转由调用方决定（因为不同入口跳转目标可能不同：Library 跳 `/learn/module/{id}` 或 `/learn/overview`，Topic 只跳 `/learn/module/{id}`）

**调用点改写：**

| `TopicTransitionView.handleContinue` | 同上 |
| `TopicTransitionView.handleSkip` | 同上 |
| `TopicTransitionView.handleReenter` | 同上（本版默认恢复；§9 列为后续改进） |
| `ShowcaseHome.handleStartTopic` | 同上 |

### 5.5 `computeModuleProgress` —— 指标定稿（V1.1 重设计）

#### 5.5.1 为什么不能计算"已答题数"

`progress-store.ts:168-170` 显示：

```ts
const quizCount = concept.quizSeries.quizzes.length
const currentReviewSlots = stage.reviewSlots ?? []
const totalSlots = quizCount + currentReviewSlots.length
```

`quizIndex` 是**当前 concept 内的 slot 序号**，可能落在原始题上，也可能落在复习 slot 上。因此：
- `quizIndex` ≠ "已答原始题数"
- 计算"已答题数"需要传入 `attemptsBySlot`（来自 attempts-store），但这会让纯函数依赖外部 state

**V1.1 决策**：本版**不计算"已答题数"**，仅基于 concept 维度计算进度。

#### 5.5.2 指标定义（锁定）

**新文件**：`src/lib/runtime/module-progress.ts`

```ts
import type { Module, ProgressState, ModuleStage } from '@/types/domain'

export interface ModuleProgressInfo {
  /** 已完全完成的 concept 数（不含当前正在做的） */
  completedConcepts: number
  /** 当前正在做的 concept 索引（0-based，-1 表示未开始或已完成） */
  currentConceptIndex: number
  /** 整体 concept 完成百分比（0–100，整数） */
  conceptPercent: number
  /** 简短文案，用于 UI 展示 */
  label: string
  /** 是否已完成（stage.kind === 'done'） */
  done: boolean
  /** 是否已开始（stage !== null 且 !== 'module_intro'） */
  started: boolean
}

/**
 * 计算 module 的展示用进度。
 *
 * 进度语义（V1.1 锁定）：
 *   - stage = null / 'module_intro' → started=false, completedConcepts=0, percent=0, label='未开始'
 *   - stage = 'concept_intro'(i) → 当前 concept 算"已开始"，completedConcepts=i, percent=round((i+0.1)/total*100)
 *     （+0.1 表示"已开始但未完成"，避免显示 0%）
 *   - stage = 'concept'(i, q) → completedConcepts=i, percent=round((i+0.5)/total*100)
 *     （+0.5 表示当前 concept 进行中，按 50% 折算）
 *   - stage = 'challenge' / 'feynman_intro' / 'feynman_step' / 'feynman_final'
 *     → 所有 concept 已完成，completedConcepts=total, percent=95（留 5% 给最终 done）
 *   - stage = 'done' → completedConcepts=total, percent=100, label='已完成'
 *
 * 注意：本函数不计算"已答题数"，因为 quizIndex 可能落在 reviewSlot 上（§5.5.1）。
 * 若未来需要展示"已答题数"，需传入 attemptsBySlot。
 */
export function computeModuleProgress(
  module: Module,
  progress: ProgressState | null,
): ModuleProgressInfo {
  const total = module.concepts.length

  if (!progress || !progress.stage) {
    return {
      completedConcepts: 0,
      currentConceptIndex: -1,
      conceptPercent: 0,
      label: '未开始',
      done: false,
      started: false,
    }
  }

  const stage: ModuleStage = progress.stage

  if (stage.kind === 'done') {
    return {
      completedConcepts: total,
      currentConceptIndex: -1,
      conceptPercent: 100,
      label: '已完成',
      done: true,
      started: true,
    }
  }

  if (stage.kind === 'module_intro') {
    return {
      completedConcepts: 0,
      currentConceptIndex: -1,
      conceptPercent: 0,
      label: '未开始',
      done: false,
      started: false,
    }
  }

  // concept_intro / concept：当前 concept 算"进行中"
  if (stage.kind === 'concept_intro' || stage.kind === 'concept') {
    const i = stage.conceptIndex
    const pct = total > 0 ? Math.round(((i + 0.5) / total) * 100) : 0
    return {
      completedConcepts: i, // 不含当前
      currentConceptIndex: i,
      conceptPercent: Math.min(pct, 99), // 上限 99，留空间给 done=100
      label: `概念 ${i + 1}/${total}`,
      done: false,
      started: true,
    }
  }

  // challenge / feynman_*：所有 concept 已完成，等待最终提交
  return {
    completedConcepts: total,
    currentConceptIndex: -1,
    conceptPercent: 95,
    label: '最后阶段',
    done: false,
    started: true,
  }
}
```

**关键决策**：

- "已完成"与"已跳过"在 UI 上必须**视觉区分**（§5.6）：本函数返回 `done` 表示 stage.kind==='done'，调用方还需查 `topicSession.moduleStatus[moduleId]` 判断是否 `'skipped'`
- conceptPercent 上限 99，避免显示 100% 但 stage 还不是 done 的歧义
- challenge / feynman_* 统一显示 95% + "最后阶段"文案

#### 5.5.3 不在 V2.0.1 范围内

- "已答题数" / "已答 slot 数"：需要 attempts-store 数据，留作 V2.0.2（§5.5.1 说明）
- 时间维度的进度（已学 N 分钟）
- 单 concept 内 quiz 进度（"概念 2/5 · 第 3 题"）

## 9. 已知不满足项（V1.1 新增，从风险表升级）

明确写为**本版不满足**，不再标为"低风险细节"：

1. **feynmanAttempt 在 resumeModule 恢复时清空**：per-module snapshot 不保存该字段（见 `progress-store.ts:418-437` 的 partialize）。本版保证 concept/quiz 进度恢复，不保证 feynmanAttempt。若用户在 feynman 阶段退出后从 Topic 入口再入，费曼作答会丢失。V2.0.2 候选：扩展 per-module key schema。

2. **`handleReenter` 默认恢复，不区分"继续进度"与"重新开始"**：本版统一走 resumeModule。V2.0.2 候选：在 TopicTransitionView 增加"重新开始"按钮，传 `allowResume: false`。

3. **"已答题数"不展示**：因 quizIndex 可能落在 reviewSlot 上（§5.5.1）。V2.0.2 候选：传入 attemptsBySlot 计算实际已答数。

4. **per-module key storage 二分**（§3.1）：本版通过 invariant 注释 + 统一用 `storage` 缓解，但架构债仍在。V2.0.2 候选：把 per-module snapshot 迁到 getStorage()（SQLite）。

5. **Today review / search 入口的进度保留**：本版未核查 `/learn/today/review` 和 SearchDialog 跳转路径是否调用 startModule。F1-3 实施时会一并检查；若发现需要修改，补 E2E。
