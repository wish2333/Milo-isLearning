## 0. 文档说明

V1.0(已废弃)的问题:把单人自用做成了偏平台化的大版本(7 功能 ~9 周),且有几处影响数据正确性的架构缺口。V2.0 收窄范围、修复一致性、删除过度设计。

**核心保留**:FSRS → 今日复习 → 学习反馈这条主线(用户明确认可)
**核心修正**:
- 先锁部署边界 + 数据可靠(P0),这是 FSRS 数据可信的前提
- FSRS SchedulingData 定义为"可重放的派生缓存",不是独立真值
- X1 做成独立 pipeline 入口(不插现有 pipeline)
- 搜索走客户端(数据已在缓存)
- X2(多 Module 拆解)推迟 v2.1.0

| 维度 | 决策 | 理由 |
|------|------|------|
| **部署** | localhost-only(本机 bun start) | 用户明确选择;不暴露公网,不需登录 |

| **FSRS 参数变更** | configRevision 追踪;不一致则全量 replay | review V2.1 P0:参数修改后缓存不过期 |
| **"今天"定义** | **浏览器本地时区**自然日 | review 修正:不用服务器时区 |
| **X1 pipeline** | **独立入口**(不插现有 pipeline) | review P1 修正:绕过 validateInput + stable anchors |
| **知识页** | 可编辑 + 标明 AI 生成 | review P1 修正:自用纠错刚需 |

## 4. Phase 0 详细设计 — localhost 部署可靠性

### 4.1 部署文档

**新增** `docs/v2.0.0/Deploying-Localhost.md`:

- `.env.local` 配置(只放 LLM key,不放敏感数据)

- SQLite 文件位置:`data/alc.db`(本机持久)
- **明确约束**:不暴露公网;如需远程访问必须加门禁(本版不做)

### 4.2 自动备份(复用现有 createSnapshot)

**现有基础设施**(`src/lib/persistence/server/backup.ts`):
- `createSnapshot(db)` — SQLite `VACUUM INTO` 一致性快照(含 WAL 未 checkpoint 数据)
- `BACKUP_DIR = 'data/backup'`,`MAX_AUTO_BACKUPS = 10`,自动轮转
- `listAutoBackups()` — 已有 UI 列表支持

**V2.0 错误**:另建 `data/backups/*.json` 系统,与现有 `data/backup/*.db` 形成两套冗余。

**V2.1 修正**:**复用 createSnapshot(),不建第二套系统**。

**策略**:
- **触发**:"首次成功写入后检查,距上次快照 >24h 则备份"——覆盖日常 FSRS 复习(不只是 module 完成)
- **module 完成 / 迁移前 / 恢复前**:额外强制备份(不受 24h 限制)
- **客户端先 `await flushNow()`**:确认无 failed tasks 后再请求快照(避免备份到旧数据)
- **位置**:`data/backup/alc-snapshot-YYYYMMDD-HHmmss.db`(现有格式)
- **保留**:现有 `MAX_AUTO_BACKUPS = 10` 轮转
- **JSON `BackupPackageV1`**:保留为"手动导出"(全量 JSON 导出,非"增量备份")

**实现**:
- 新增 `src/lib/persistence/server/auto-backup.ts`:`shouldBackup(lastSnapshotTime)` + `triggerSnapshotIfNeeded(db)`
- 客户端 `progress-store` done 态 / attempt 写入后:先 `await flushNow()` → POST `/api/backup/auto`(传 force 标志)
- `/api/backup/auto` 路由:force=true 立即快照;否则检查 24h 阈值

### 4.3 备份验证

**Settings 页加"验证备份"按钮**:
- 读取 `data/backup/` 最新 `.db` 快照
- 执行 `PRAGMA integrity_check`(SQLite 完整性校验)
- 可选:向临时数据库做一次恢复演练(attach 快照 → 抽样查询 → detach)
- 显示结果:"✓ 最近备份 {时间},integrity_check 通过" 或 "✗ {原因}"

### 4.4 StorageStatus 增强(取消 clearFailed)

**现状**(`src/components/StorageStatus.tsx`):已有轮询(500ms)+ failedCount + retryFailed。

**V2.1 修正**:
- 加"单条 retry"—— 扩展 WriteTask 加 `lastError` + `failedAt` 字段
- **取消"清空失败任务"** —— failed tasks 是未落盘的真实写入,清空 = 永久 cache/SQLite 分叉
- 如果确实需要放弃:改为"放弃未保存更改"流程 —— 二次确认 + 从服务端重载对应 key + 恢复客户端 cache
- 页面关闭时存在 failed tasks 必须告警(`beforeunload`)
- 保留现有浮层位置(fixed bottom-right)

**WriteTask 扩展**(`src/lib/persistence/client/write-queue.ts`):
```typescript
export interface WriteTask {
  operationId: number
  key: string
  value: string | null
  attempts: number
  status: 'pending' | 'sending' | 'failed'
  lastError?: string    // 新增:最后一次失败的错误消息
  failedAt?: number     // 新增:进入 failed 状态的时间戳
}
```
**不加 clearFailed** —— failed tasks 是未落盘的真实写入,清空会导致客户端 cache 与 SQLite 永久分叉。

## 5. Phase 1 详细设计 — FSRS 派生缓存 + Today

### 5.1 SchedulingData 类型(派生缓存)

**新增**(`src/types/domain.ts`):

```typescript
/**
 * FSRS 调度状态 — 派生缓存(可从 attempts 重放重建)
 *
 * 不是独立真值。所有字段均可通过 rebuildScheduleForSlot() 重建。
 * 始终维护(enabled 只控制是否消费 due 队列,不影响 schedule 写入)。
 */
interface SchedulingData {
  // === 标识(级联清理用)===
  slotId: string
  moduleId: string
  conceptId: string

  // === ts-fsrs (FSRS-6) 核心字段 ===
  stability: number
  difficulty: number
  elapsed_days: number
  scheduled_days: number
  reps: number
  lapses: number
  state: 'new' | 'learning' | 'review' | 'relearning'
  due: string                       // ISO date
  last_review: string | null        // ISO date

  // === 派生缓存元数据 ===
  schemaVersion: number              // SchedulingData schema 版本(当前 1)
  contentRevision: string            // quiz 核心字段的 hash;编辑后失效,触发重建
  configRevision: string             // fsrs 参数 hash;参数变化后触发全量 replay
  lastAppliedAttemptId: string       // 最后应用的 attempt id(幂等)
}
```

**AttemptRecord 扩展**(向后兼容):
```typescript
export interface AttemptRecord {
  // 现有字段...
  moduleId?: string                  // 新增(可选,旧记录无)
  conceptId?: string                 // 新增(可选,旧记录无)
}
```
旧记录迁移时通过扫 Module 建立 `quiz.id → moduleId/conceptId` 映射并补全上下文；不能假设 `originalQuizId` 具有可解析的复合格式。

**存储**:独立 KV `alc:schedule:{slotId}`
- **新增 repo** `src/lib/persistence/schedule-library.ts`
  - `get(slotId)` / `set(slotId, data)` / `remove(slotId)`
  - `listByModule(moduleId)` — module 删除时级联清理(入口在 `quota.ts:removeModule`)
  - `listDueBefore(date, timezone)` — Today 页面用
  - `listAll()` — 全量扫描(统计用)

### 5.2 ts-fsrs 封装 + Rating 推导

**新增** `src/lib/runtime/fsrs.ts`:

```typescript
import { fsrs, generatorParameters, Rating, createEmptyCard } from 'ts-fsrs'

// ts-fsrs@5.x 实现 FSRS-6 算法(非 FSRS-5)
// 锁定确切版本(如 ts-fsrs@5.4.1)避免算法行为漂移

// 从空状态创建

// 单步推进(内部用,不直接暴露)

// Rating 自动推导
export function inferRating(attempt: AttemptRecord): Rating {
  if (attempt.score < 80) return Rating.Again
  if (attempt.guessed) return Rating.Hard
  if (attempt.timeSpentMs && attempt.timeSpentMs < 5000) return Rating.Easy
  return Rating.Good
}
```

**⚠ 状态测试注意**:新卡 Again 通常是 Learning(不是 Relearning),取决于 `learning_steps`/`relearning_steps` 参数。单测必须先锁定完整 FSRS 参数,再按官方 Card 状态测试,不能假设 Good→review / Again→relearning。

### 5.3 rebuildScheduleForSlot — 调度协调层(核心)

**新增** `src/lib/runtime/fsrs-replay.ts`:

```typescript
/**
 * 调度协调层:为单个 slot 重建 SchedulingData。
 *
 * 这是唯一的 schedule 构建入口。所有触发点都调这个函数。
 * 需要 moduleId/conceptId/quiz 作为显式参数(AttemptRecord 可能缺这些字段)。
 */
export function rebuildScheduleForSlot(params: {
  slotId: string
  moduleId: string
  conceptId: string
  quiz: Quiz                      // 当前 quiz(用于 contentRevision)
  attempts: AttemptRecord[]       // 按 timestamp 升序
  contentRevision: string         // computeContentRevision(quiz)
  configRevision: string          // fsrs 参数 hash
  fsrsConfig: { requestRetention: number; maximumInterval: number }
}): SchedulingData | null {
  // 1. 空 attempts → null
  // 2. 从 createSchedule 开始,逐个 applyRating(inferRating(a))
  // 3. 写 contentRevision + configRevision + lastAppliedAttemptId
}
```

**为什么需要协调层(不能在 attempts-store 内直接调)**:
- `addAttempt(attempt)` 只收单条 attempt,无 moduleId/quiz 上下文
- 需要调用方(ConceptView/ChallengeView/review 页面)显式传入 module/quiz
- `AttemptRecord.moduleId?` 是可选字段(旧记录无),协调层做兜底

### 5.4 全链路触发点接入

**每个触发点都通过协调层调 `rebuildScheduleForSlot`**:

| 触发点 | 位置 | 行为 |
|--------|------|------|
| `addAttempt` | 调用方(ConceptView/ChallengeView)传入 moduleId/conceptId/quiz → rebuildScheduleForSlot | attempts-store 的 addAttempt 只管 attempt;schedule 更新由调用方协调 |
| `markGuessed` | 调用方传入上下文 → rebuildScheduleForSlot | 同上 |
| `unmarkGuessed` | `attempts-store.ts:110-123` + 调用方传入上下文 → rebuildScheduleForSlot | 同上 |
| `reevaluateLastAttempt` | `attempts-store.ts:125-145` + 调用方传入上下文 → rebuildScheduleForSlot | 纠题后 score 变,schedule 必须重算 |
| 忽略 quiz | `module-store.correctQuizAnswer` | quiz.ignored=true → scheduleLibrary.remove(slotId) |
| 恢复 quiz | `module-store.correctQuizAnswer` | quiz.ignored=false → 调用方传入上下文 → rebuildScheduleForSlot |
| 删除 module | **`quota.ts:removeModule`**(非 module-library) | scheduleLibrary.listByModule(moduleId).forEach(remove) |
| 清除进度 | `attempts-store.clearAll` | scheduleLibrary.clearAll |
| **FSRS 参数变化** | settings-store fsrs config 变更 | configRevision 不一致 → 全量遍历 replay |
| **FSRS 开关变化** | settings-store fsrs.enabled 变更 | **不影响 schedule 维护**(始终维护);只影响 Today 是否消费 due |

**始终维护原则**:无论 fsrs.enabled 是否为 true,schedule 都在更新。enabled 只控制:
- Today 页面是否查 due 队列
- adaptive-sequencer 是否注入 due slots
- 首页智能路由是否显示"今日复习"

这避免了"关闭→重开→migrationFlag 已设→不回填→丢数据"的问题。

### 5.5 contentRevision 计算

**新增** `src/lib/runtime/content-revision.ts`:

```typescript
/**
 * 计算 quiz 内容的 hash,用于检测 quiz 是否被编辑。
 * 编辑后(F42),旧 schedule 的 contentRevision 与当前不符,触发重建。
 *
 * ⚠ 浏览器端运行,不能用 Node crypto.createHash。
 * 用 cyrb53(非加密,稳定,浏览器兼容)或 Web Crypto crypto.subtle.digest(异步)。
 * 这里不需要加密强度,只需稳定的变更检测。
 */
export function computeContentRevision(quiz: Quiz): string {
  const relevantFields = {
    stem: quiz.stem,
    answer: quiz.answer,
    options: quiz.options,
    acceptableAnswers: quiz.acceptableAnswers,
  }
  return cyrb53(JSON.stringify(relevantFields)).toString(16)
}

export function computeConfigRevision(config: { requestRetention: number; maximumInterval: number }): string {
  return cyrb53(JSON.stringify(config)).toString(16)
}
```

**检查时机**:每次从 schedule-library 读取时,比对 `schedule.contentRevision !== computeContentRevision(currentQuiz)` 或 `schedule.configRevision !== computeConfigRevision(currentConfig)`。不符则触发 rebuildScheduleForSlot 重建。

```typescript
interface SettingsState {
```

### 5.8 Today 页面 + TodaySession

**新页面** `/learn/today`(概览)+ `/learn/today/review`(复习执行):

## 6. Phase 2 详细设计 — X1 独立 expand pipeline

### 6.1 核心问题与修正

V1.0 的错误:把 expand 插在现有 pipeline 的 import 之后。但:
- `pipeline.ts:109` 的 `validateInput()` 要求 200-20000 字符,5-50 字主题词会被拦
- KnowledgeExpander 先生成 concepts,后续 chunk/concept agent 又重新生成,knowledgePage 无法映射

**修正**:X1 走**独立 pipeline 路径**,不经过 `compileMarkdown()`。

### 6.2 独立 pipeline 架构

```
topic(5-50 字) + 可选约束
    │
    ▼
[KnowledgeExpander Agent] (1 次 LLM)
    │ 输出:ExpandedKnowledge {
    │   title, intro, goal,
    │   normalizedSource,              ← 扩充后的完整 Markdown(交给后续 pipeline)
    │   conceptAnchors: [{             ← 稳定锚点
    │     anchorId, name, knowledgePage
    │   }]
    │ }
    ▼
[compileMarkdown(normalizedSource)]    ← 走现有完整 8 阶段 pipeline
    │ (validateInput 不会拦,因为 normalizedSource 是完整文档)
    ▼
[Module 产出]
    │ concepts 已由现有 pipeline 生成
    ▼
[anchor 回填]
    │ 按 concept.sourceAnchorId 匹配 conceptAnchors
    │ 匹配上 → 写入 concept.knowledgePage
    │ 匹配不上 → 记录 warning(不阻断)
    ▼
[最终 Module(含 knowledgePage)]
```

**关键**:KnowledgeExpander 产出的是 `normalizedSource`(完整文档)+ `conceptAnchors`(锚点,每锚有稳定 `anchorId`)。后续 pipeline 正常跑,Concept Agent 在 schema/prompt 中被强制保留 `sourceAnchorId`,最终回填**按 anchorId 精确匹配**(非 name fuzzy)。未匹配的知识页保留给用户手动关联,不静默丢失。

### 6.3 KnowledgeExpander Agent

**新增**:
- `src/lib/compiler/agents/knowledge-expander.ts`
- `src/lib/compiler/prompts/knowledge-expander.md`

**IO**:
```typescript
interface ExpandedKnowledge {
  title: string
  intro: string
  goal: string
  normalizedSource: string          // 扩充后的完整 Markdown(≥ 1000 字,过 validateInput)
  conceptAnchors: ConceptAnchor[]
```

### 6.4 独立编译入口

**新增** `src/lib/compiler/pipeline/compile-with-expand.ts`:

```typescript
// ⚠ 使用正确的 CompileEvent 事件名(stage_enter/progress/complete/error)
// ⚠ CompileStage 需扩展加 'expand'(见 §6.6)
// ⚠ 必须透传 CompileOptions(checkpoint/resume 能力)

export async function* compileWithExpand(
  topic: string,
  constraints: string | undefined,
  config: CompileConfig,
  options?: CompileOptions,         // 透传 checkpoint/resume
): AsyncIterable<CompileEvent> {
  // 1. expand 阶段(可被 checkpoint 跳过)
  const expandedCheckpoint = options?.checkpointData?.get('expand')
  let expanded: ExpandedKnowledge
  if (expandedCheckpoint && options?.resumeFrom === 'expand') {
    expanded = expandedCheckpoint.artifact as ExpandedKnowledge
  } else {
    yield { kind: 'stage_enter', stage: 'expand' }
    yield { kind: 'progress', stage: 'expand', percent: 10, message: '正在扩充知识网络...' }
    expanded = await runKnowledgeExpander(topic, constraints, config)
    options?.writeCheckpoint?.('expand', expanded)
  }

  // 2. 走现有 pipeline(用 normalizedSource + 透传 options)
  for await (const event of compileMarkdown(expanded.normalizedSource, config, options)) {
    if (event.kind === 'complete') {
      // 3. anchor 按 ID 回填
      const module = backfillKnowledgePagesById(event.module, expanded.conceptAnchors)
      yield { kind: 'complete', module, qualityReport: event.qualityReport }
    } else {
      yield event
    }
  }
}
```

**回填逻辑**(`backfillKnowledgePagesById`):
- 遍历 module.concepts
- 按 `concept.sourceAnchorId` 精确匹配 conceptAnchors(非 name fuzzy)
- 匹配上 → 写 concept.knowledgePage
- 匹配不上 → 保留未匹配 anchor 到 module 元数据,供用户手动关联(不静默丢失)

## 7. Phase 3 详细设计 — 客户端搜索

### 7.1 客户端统一搜索

**核心观察**:production 启动时已通过 `/api/data/bulk` 加载全部 KV 到客户端缓存;showcase 数据在 LocalStorage。**两端数据都在客户端**。

**新增** `src/lib/runtime/search-client.ts`:

```typescript
/**
 * 客户端统一搜索。production 和 showcase 共用同一实现。
 * 数据源:从 storage 读取所有 alc:module:* + alc:topic-index,在内存构建索引。
 */
export class ClientSearchIndex {
  private entries: SearchEntry[] = []

  /** 从 storage 加载所有 module,构建索引 */
  rebuild(storage: StorageRepository): void

  /** 搜索 */
  search(query: string, filters?: SearchFilters): SearchHit[]

}

interface SearchEntry {
  type: 'module' | 'concept' | 'quiz'
  moduleId: string
  conceptId?: string
  quizId?: string
  title: string       // 可搜索标题
  content: string     // 可搜索内容(拼接 definition/keyPoints/stem 等)
}
```

**搜索逻辑**:
- 中文子串匹配(`String.includes()`)+ 简单分词(按空格 AND)
- snippet 截取(match 前后各 30 字)
- **不返回 HTML**(避免 XSS),返回纯文本 snippet,前端用 React 渲染高亮

**索引构建时机**:**每次打开 SearchDialog 时快速 rebuild**(不维护多写入点失效协议)。
- 理由:编译完成、包导入、删除都可能直接写 repository;监听 module-store 不够
- 单人 <100 module,rebuild 成本可忽略(~10ms)
- 避免多写入点的失效协议复杂度

**新增** `src/components/search/SearchDialog.tsx`:

## 11. 风险与缓解

| 风险 | 等级 | 缓解 |
|------|------|------|
| FSRS rebuildScheduleForSlot 性能(slot attempts 多) | 低 | 单人场景 <20 attempts/slot;若极端情况可加缓存 |
| KnowledgeExpander 产出的 anchors 与 pipeline concepts 匹配不上 | 中 | Concept Agent 强制保留 sourceAnchorId;未匹配项保留 warning/手动关联 |
| 客户端搜索索引构建慢(大量 module) | 低 | 每次打开 SearchDialog rebuild;单人 <100 module 成本可忽略 |
| 自动备份占磁盘 | 低 | 轮转保留最近 10 个;JSON 压缩 |
| ts-fsrs 默认参数在中文 quiz 场景不适配 | 中 | Settings 暴露 requestRetention/maximumInterval |
| 知识页可编辑导致内容与 quiz 不一致 | 低 | 编辑只改 knowledgePage,不影响 quiz;"AI 生成"徽标提示来源 |
