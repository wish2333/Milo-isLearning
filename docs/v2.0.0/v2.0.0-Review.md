## 0. 结论

V2.0.0 的四个 Phase 全部完成：localhost production 可靠性、FSRS 派生缓存与 Today、X1 AI 扩充知识页、客户端搜索。当前代码门禁通过：749 个单元测试、typecheck、lint，以及 showcase/production 两种 build 均通过；两种运行模式的搜索浏览器集成验证也通过。

人工发布验证仍是关闭前置条件：需在本机 production 环境验证真实 LLM 扩充、SQLite 自动备份/integrity_check、FSRS 参数变更后的全量回放，以及正式 E2E 套件。

## 2. 架构决策记录

### 2.1 D-1：attempts 是真值，schedule 是可重放缓存

选择：`SchedulingData` 始终维护，但不作为独立学习数据源；每个 slot 的状态可以从 `AttemptRecord[]` 全量重建。

理由：FSRS 开关只决定是否消费 due 队列，不会让关闭期间的历史丢失；纠题、蒙对标记、参数变化都能通过同一 replay 逻辑恢复正确状态。`contentRevision`、`configRevision` 和 `lastAppliedAttemptId` 用于识别缓存是否过期。

### 2.2 D-2：所有 attempt 变更经协调层同步 FSRS

选择：store 保留 attempts 为写入真值，调用方传入 module/concept/quiz 上下文，由 `rebuildScheduleForSlot` 负责 set/remove schedule。

理由：避免把 FSRS 业务散落在多个 store mutation 中，也避免旧 `originalQuizId` 无法解析时错误关联。删除 module、忽略题和恢复题均有明确清理/重建路径。

### 2.3 D-3：AI 扩充使用独立 pipeline 与稳定 anchor ID

选择：扩充先生成带 `sourceAnchorId` 的知识页，再调用既有 Markdown compiler，最后按 ID 回填，不按概念名称模糊匹配。

理由：名称会被模型改写或重复，ID 才能保证知识页回填到正确 concept；未匹配 anchor 保留为 warning，不静默丢弃。知识页编辑与 quiz 内容解耦，降低人工修改风险。

### 2.4 D-4：搜索打开时重建客户端索引

选择：不增加 `/api/search`、SQLite FTS5 或多写入点订阅；SearchDialog 打开时从当前 mode 的 storage 读取并重建索引。

理由：单人 localhost 场景数据量有限，打开时读取能覆盖直接 repository 写入和跨页面更新；production/showcase 只替换 storage 实现，搜索逻辑保持一致。React 文本高亮避免 `dangerouslySetInnerHTML`。

### 2.5 D-5：showcase 与 production 数据边界保持不变

选择：搜索只索引当前模式允许访问的 module；showcase 内容继续只读，production 只显示用户数据。

理由：搜索不能成为绕过 `ContentOrigin` 隔离的旁路，也不能让展示题库被用户编辑或持久化覆盖。

## 4. 已知限制

### 4.1 真实 LLM 扩充仍需人工验证

自动化验证使用确定性的 SSE/mock 链路覆盖状态、回填和编辑保存；没有在提交阶段调用真实 provider，避免依赖 API key 和产生不可控外部费用。发布前需在本机配置 provider 后验证 AI 扩充编译、断点恢复和未匹配 anchor warning。

### 4.2 正式 E2E 套件未纳入本次门禁

本次使用独立 Playwright 浏览器脚本完成 P3 双模式验证，但未把完整 `bun run e2e` 作为本次收尾门禁。手动验证时应优先使用 production build/start，避免 dev server 生命周期问题掩盖应用结果。

### 4.3 搜索暂不支持 FTS5

索引为内存结构，打开时 O(N) 重建；适合当前单人、小规模 localhost 场景。若 module 数量和知识页体积显著增长，再评估 SQLite FTS5、中文分词和增量索引。

### 4.4 发布动作暂缓

本次只完成本地历史消息整理、Review/Plan 文档和代码验证；未创建 PR、未 push、未创建 tag，等待人工验证结果后再决定是否发布。

## 5. 迁移与兼容性说明

### 5.1 旧 attempts 与 FSRS

现有 `AttemptRecord` 的 `moduleId`/`conceptId` 为可选字段；启动回填会扫描 Module 补齐上下文，再生成 schedule。旧 attempts 仍是唯一真值，不需要删除或重写用户学习记录。

### 5.2 旧 Module 与知识页

`knowledgePage`、`sourceAnchorId` 和新的 `concept_intro` 均为可选/新增字段。没有知识页的旧 Module 继续走原 concept 流程；没有 anchor 的 concept 不会被强行匹配。

### 5.3 搜索与存储

客户端搜索不新增持久化 schema 或迁移步骤。每次打开从当前 storage 读取，production SQLite 与 showcase LocalStorage 均沿用既有 `StorageKeys` 和 `ContentOrigin` 规则。

### 5.4 本地 SQLite 数据

production 验证只使用本地 `data/` 数据目录；数据库/WAL 文件未纳入 Git。人工验证和发布前应通过应用自动快照备份，不要直接复制正在写入的 SQLite 主文件。
