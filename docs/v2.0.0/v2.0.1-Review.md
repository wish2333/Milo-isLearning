## 2. 决策记录

### 2.1 D-1：三级 fallback 而非单读 per-module

**选择**：`resumeModule` 按"per-module snapshot（LS） → 全局 Zustand blob（若 moduleId 匹配且 updatedAt 更新） → startModule 语义"顺序恢复。

**理由**：V2.0.0 的架构债是 per-module key（LS）与全局 blob（Zustand persist）二分——production 模式下两者 backend 不同（LS vs SQLite）。仅读 per-module 在某些边界场景（如 SSR rehydrate 时序、跨页未切换）会读到陈旧值；仅读全局 blob 又会丢失 per-module 持久化。三级 fallback 兼顾两者，且 `updatedAt` 比较保证取更新的快照。

**`stage.kind==='done'` 一律走 startModule**：与 Library handleOpen 的 `summary.completed` 分支一致——用户已完成后再进入应从头看，而不是停在 done stage 看空白页。

### 2.2 D-2：抽取 `enterModule` 而非每个入口分散调

**选择**：5 处 Topic 入口统一调 `enterModule({ moduleId, allowResume })`，helper 内部完成 `setModule + resumeModule/startModule`。

**理由**：避免 5 处入口各自维护"`storage.get(Module)` + `setModule` + `resumeModule`"的重复模板；helper 返回 `boolean`（成功/失败），跳转仍由调用方决定（不同入口跳转目标可能不同）。Library 入口（`handleOpen`）保持不变——它已等价于 `allowResume=true`（不调 startModule），是"正确"的参考实现。

### 2.3 D-3：concept 维度进度而非 quiz 维度

**选择**：`computeModuleProgress` 按 `conceptIndex` 计算百分比（`(i+0.5)/total*100`），不计算"已答题数"。

**理由**：`progress-store.ts:168-170` 显示 `quizIndex` 是当前 concept 内的 slot 序号，**可能落在复习 slot 上**（`quizCount + currentReviewSlots.length = totalSlots`），所以 `quizIndex !== 已答原始题数`。计算"已答题数"需要传入 `attemptsBySlot`（来自 attempts-store），但这会让纯函数依赖外部 state。V2.0.1 锁定 concept 维度，"已答题数"留作 V2.0.2 候选。

`concept_intro` 与 `concept` 统一用 `(i+0.5)/total*100` 公式（避免分叉），上限 99（留 1% 给 done=100 区分）。

### 2.4 D-4：`doneCount` 拆分但 `allDone` 逻辑不变

**选择**：UI 分别统计 done/skipped/in_progress 三变量并分开显示，但 `allDone = doneCount + skippedCount === totalCount` 保持不变。

**理由**：F24 skip 流程已假设"skipped 算入完成"（用户主动放弃也是完成的一种），改 `allDone` 会破坏既有 topic session 语义。UI 拆开显示只是让用户看清"3 个完成 vs 1 个跳过"，不影响逻辑判定。主题整体进度百分比基于 `doneCount / totalCount`（**不含** skipped），避免"跳过也算 33% 完成"的误导。

## 4. 已知不满足项

明确写为**本版不满足**，不在验收范围内（沿用 fix-report §9）：

1. **`feynmanAttempt` 在 `resumeModule` 恢复时清空** — per-module snapshot 不保存该字段。单测 `progress-store.test.ts` 分支 g 显式锁定此行为。V2.0.2 候选：扩展 ProgressState schema 或 per-module snapshot 持久化 feynmanAttempt。

2. **`handleReenter` 默认恢复，不区分"继续"与"重做"** — V2.0.1 默认 `allowResume=true`，即使模块之前是 skipped 状态。V2.0.2 候选：加"重新开始"按钮（`allowResume=false`）。

3. **"已答题数"不展示** — `quizIndex` 可能落在 reviewSlot 上（D-3）。V2.0.2 候选：传 `attemptsBySlot`（attempts-store）。

4. **per-module key storage 二分** — 架构债，V2.0.2 候选：统一到 `getStorage()`（production 模式下 SQLite 而非 LS）。

5. **Today review / SearchDialog 入口的进度保留** — F1-3 已核查，两者均不调 `startModule`（仅 `router.push`）。无需修改，但未加 E2E 用例。

## 5. 后续建议（V2.0.2 候选）

按优先级排序：

### 5.1 P0：统一 per-module storage backend
当前 `alc:progress:{moduleId}` 在 production 模式下走 LS、全局 blob 走 SQLite，是 V2.0.0 遗留架构债，也是 V2.0.1 修复的根因。V2.0.2 应迁到统一的 `getStorage()`，让两种 key 走同一 backend。

### 5.2 P1：扩展 ProgressState schema 持久化 feynmanAttempt
让 `resumeModule` 能完整恢复费曼阶段进度（当前清空是妥协）。需要 per-module snapshot schema 升级 + 数据迁移。

### 5.3 P1：handleReenter 区分"继续"与"重做"
为 skipped 模块的"重新进入"加双按钮选项（`allowResume=true` vs `false`），尊重用户意图。

### 5.4 P2：补"已答题数"展示
`computeModuleProgress` 加可选参数 `attemptsBySlot`，UI 传入后展示"概念 2/5 · 第 3 题"。纯函数性不受影响（参数可选）。

### 5.5 P2：补 V2.0.0 baseline E2E 失败修复
"Topic learning flow" 和 "Showcase Home clicking module card" 两个 pre-existing 失败应在 V2.1.0 P0 发布验证时修复（不属 V2.0.1 范围）。
