# 提炼项目简历经历

先还原证据与决策，再写简历。代码能证明实现，不自动证明职位、主导权、用户研究、业务采纳或效果。

所有面向用户的输出必须为中文，包括证据卡、追问和简历内容。

## 工作流

1. **建立仓库地图。** 用 `rg --files` 找到需求、PRD、README、设计说明、issue、测试、配置、演示和入口文件；检查近期 Git 提交。先读需求与设计材料，再沿用户可见的关键路径读代码。不要通读无关的构建产物、依赖目录或生成文件。
2. **还原决策。** 从材料中提取用户/场景、痛点、目标、功能优先级、约束和取舍；用实现、测试或提交验证。AI 项目特别检查模型边界、成本、隐私、错误处理和人工接管。
3. **先输出证据卡。** 读取 [references/output-template.md](references/output-template.md)，按其中的“项目证据卡”格式列出结论及来源。每个重要主张标为：

| 状态     | 使用规则                                     |
| -------- | -------------------------------------------- |
| 已验证   | 有直接仓库证据，可陈述为事实。               |
| 用户确认 | 用户在本轮明确说明，可陈述为事实并保留范围。 |
| 推断     | 仅用谨慎措辞，优先请求确认。                 |
| 未验证   | 不写入正式简历。                             |

4. **做最小访谈。** 若无法确认实际职责、协作对象、目标用户、试用/上线、反馈、指标或测量口径，按模板最多问 5 个中文问题。不要重复询问仓库已经证明的事实。缺少关键事实时，停在证据卡与问题阶段。
5. **生成简历内容。** 收到回答后，严格按模板输出完整版、精简版和表述核对。每条遵循“场景与问题 → 本人决策/职责 → 方案与取舍 → 验证结果及口径”。技术只解释决策，不能成为主语。

## 证据边界

- 不把提交作者、代码行数、目录所有权或“独立开发”字样视为岗位、主导权或业务成果的证明。
- 不杜撰访谈人数、节省时间、准确率、用户数、持续使用、上线范围或“显著提升”。已有数字也要标明样本、时间或任务口径；无法确认时删去或改为待确认。
- 不把 README 的愿景、计划功能或营销文案写成已交付/已验证结果；交叉检查实现、测试、发布记录或用户确认。
- 没有真实效果数据时，可以写已完成的交付、可用能力或确认过的试用设置；不能用空泛的“提升体验/效率”替代结果。
- 若材料涉及密钥、个人信息或敏感业务，只概述与简历相关的最小必要信息，绝不复述敏感内容。

## 常见偏差

| 偏差                       | 改法                               |
| -------------------------- | ---------------------------------- |
| 直接输出四条“成果感”要点   | 先证据卡，缺证据先追问。           |
| 罗列框架、模块数、代码行数 | 提取用户问题、决策和取舍。         |
| 把推断写成事实             | 标为“推断”，用谨慎表述或请求确认。 |
| 用“负责”掩盖具体贡献       | 写清做出的判断、操作和证据。       |

在开始输出前始终读取 [references/output-template.md](references/output-template.md)。
