# 检索增强生成（RAG）入门

## 什么是 RAG

检索增强生成（Retrieval-Augmented Generation，RAG）是一种把外部知识检索与大语言模型生成结合的架构模式。当用户提问时，系统先从知识库中检索相关文档片段，再把这些片段作为上下文喂给语言模型，让模型基于检索到的内容生成答案。

RAG 解决的核心问题是：大语言模型的训练数据有截止时间，且对私有领域知识一无所知。通过外挂知识库，模型可以回答它从未见过的问题，同时显著降低"凭空编造"（hallucination）的发生率。

一个典型的 RAG 流程包含三个阶段：**索引**（离线把文档切片并向量化入库）、**检索**（在线用 query 检索 top-k 片段）、**生成**（把片段拼进 prompt 让 LLM 作答）。

## 索引阶段

索引阶段的目标是把原始文档变成可被快速检索的向量。具体步骤是：先把长文档按语义切成 200-800 字符的小块（chunk），然后对每个 chunk 调用 embedding 模型得到一个高维向量，最后把这些向量存入向量数据库（如 FAISS、Milvus、Chroma）。

切片粒度是关键权衡：切得太碎会丢失上下文，切得太粗会降低检索精度。常见做法是用滑动窗口配合重叠（overlap 50-100 字符），并在边界处保留段落完整性。

embedding 模型选择直接影响检索质量。常用选项：OpenAI 的 text-embedding-3-small（便宜、多语言）、bge-large-zh（开源、中文效果好）、e5-mistral-7b（更强但更慢）。MVP 阶段建议先用 text-embedding-3-small 跑通流程。

## 检索阶段

用户提问时，系统把问题用同一个 embedding 模型编码成向量，然后在向量数据库里做近邻搜索，返回余弦相似度最高的 top-k（通常 k=3 到 k=10）个 chunk。

纯向量检索有一个已知短板：它擅长语义匹配但不擅长精确匹配。比如搜"GPT-4"时，向量检索可能返回"GPT-3"的文档。生产系统常采用**混合检索**（hybrid search）：把 BM25 等基于关键词的稀疏检索分数与向量密集检索分数加权融合。

检索之后还有一步 **rerank**：用 cross-encoder 模型（如 bge-reranker-large）对 top-k 候选重新打分，把真正相关的片段排到前面。rerank 会增加 100-300ms 延迟，但能显著提升最终答案质量。

## 生成阶段

生成阶段把检索到的 chunks 拼成上下文字符串，连同用户问题一起喂给 LLM。Prompt 通常长这样：

```
你是一个严谨的问答助手。请只基于以下参考资料回答问题，不要使用外部知识。
如果资料中没有答案，请直接回答"我不知道"。

参考资料：
{retrieved_chunks}

问题：{user_query}
```

"请只基于以下资料"这一句很关键。它强制 LLM 把注意力放在检索内容上，而不是从自己的参数化记忆里编答案。这能大幅降低幻觉率。

但即使有这句话，LLM 仍可能在资料不全时编造细节。生产实践里常用 **citation**（让模型标注每个论断来自哪条 chunk）来审计答案的可追溯性。

## 评估 RAG 系统

RAG 系统的评估分为三个独立维度：

**检索质量**：召回率（recall@k）和精度（precision@k）。recall@k 衡量"该被检索到的相关文档是否在前 k 个结果里"，这是 RAG 系统最关键的指标——如果检索阶段就漏掉了正确答案，再强的 LLM 也救不回来。

**生成质量**：答案是否忠实于检索内容（faithfulness）、是否直接回答了问题（answer relevance）。常用 RAGAS 等自动评估框架。

**端到端效果**：用户主观满意度、答案完整度、是否产生幻觉。这一维最接近真实价值，但也最难自动测量。

一个常见反模式是只看最终答案好不好，发现答案不好就换更大的 LLM。但 80% 的情况下，问题出在检索阶段——embedding 模型太弱、切片粒度不合理、没有 rerank，都会让 LLM 拿到无关上下文从而胡编。

## RAG 的边界

RAG 不是万能的。它适合"知识查询"类任务（事实型问答、文档摘要），但不适合"推理"类任务（多步数学证明、复杂规划）。对于后者，模型自身的推理能力比外挂知识更重要。

RAG 也不擅长处理需要跨多文档综合、时间序列推理的场景。比如"过去三年这家公司营收增长趋势如何"需要先检索三年财报，再自己做时序对比——这超出了一次性检索的能力范围，需要 Multi-hop RAG 或 Agentic RAG 等更复杂架构。
